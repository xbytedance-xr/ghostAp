# Workflow 结果简报卡设计

## 背景

当前 `/wf` 终态流程同时承担两种冲突职责：在飞书卡片中展示结果正文，以及通过 HTML/Markdown 附件保存完整报告。`src/workflow_engine/renderer.py` 会把 `project.result` 按 4,000/2,000 字符限制格式化，因此同一张完成卡中可能出现多处“内容已截断”。随后 `_enforce_card_size()` 还可能再次按字符裁剪或从尾部删除元素。

这使用户无法判断被省略的是论据、结论还是关键风险。已有完整报告附件虽然保证了原始信息可追溯，但卡片没有稳定地展示本次任务最重要、可直接行动的信息。

## 目标

- 终态卡首屏展示本次任务的关键结论、发现、验证、交付物和下一步。
- 卡片中的每个语义条目必须完整，不显示半句、半个对象或“内容已截断”。
- 任意长度的原始结果、证据、Agent 输出和 Workflow 状态继续完整写入 HTML/Markdown 报告。
- 新生成的 Workflow 和既有模板都能得到稳定的结果简报。
- 结果语义、卡片布局和完整报告保持单向、清晰的依赖边界。

## 非目标

- 不把任意长度的完整结果正文塞进一张飞书卡片。
- 不发送多张结果续接卡。
- 不增加一次终态 LLM 总结调用。
- 不在本轮重做 HTML 报告的信息架构。
- 不改变运行中进度卡、Workflow 调度和 Agent 执行语义。

## 方案选择

评估过三种产品方向：

1. 结果简报卡：卡片展示结构化关键结果，附件展示完整证据。
2. 多卡连续结果：把完整关键结果拆成 2–4 张卡。
3. 单卡折叠全文：把长正文放入折叠面板。

选择方案 1。它能保持“一张终态卡 + 一份完整附件”的消息结构。方案 2 会增加话题噪音；方案 3 的折叠只改变显示状态，不减少 payload，无法规避飞书卡片大小限制。

## 信息结构

终态卡采用固定骨架、动态内容。没有内容的栏目自动隐藏。

1. 标题：Workflow 名称与终态。
2. 结果统计：耗时、阶段、验证结论，以及与任务最相关的风险数或交付物数。
3. 结论：一句完整、可行动的最终判断。
4. 关键发现：按严重度和相关性排序的完整条目。
5. 验证：通过/未通过状态及完整验证条目。
6. 交付物：文件、测试、文档或其他可识别产物。
7. 下一步：完整的后续行动条目。
8. 执行过程：折叠在结果之后，保留阶段与 Agent 统计。
9. 报告状态：附件已发送、附件发送失败或报告生成失败。

预览文件：`ux/workflow-result-brief.html`。

## 结果契约

新生成的 Workflow 应优先返回结构化 `card_summary`：

```json
{
  "card_summary": {
    "verdict": "passed | needs_attention | failed | unknown",
    "conclusion": "完整结论",
    "findings": [
      {"severity": "high | medium | low | info", "text": "完整发现"}
    ],
    "verification": [
      {"status": "passed | failed | warning | info", "text": "完整验证结果"}
    ],
    "deliverables": [
      {"type": "code | test | document | artifact | other", "text": "完整交付物"}
    ],
    "next_steps": ["完整后续行动"]
  },
  "result": "或其他完整业务结果",
  "verification": "完整验证对象"
}
```

`card_summary` 是展示契约，不替代完整业务结果。报告仍保存原始 `project.result`，不从简报反向生成完整报告。

## 组件边界

新增 `src/workflow_engine/result_brief.py`，独立承担：

- Pydantic 简报模型与枚举。
- `card_summary` 验证和标准化。
- 既有 JSON/纯文本结果的确定性字段映射。
- 完整条目排序、选择和遗漏计数。
- 卡片 payload 预算内的整体条目装配。

`src/workflow_engine/renderer.py` 只把标准化简报转换为飞书元素，不再解析任意深度的业务结果，也不再对终态结果正文做字符截断。

`src/workflow_engine/reporting.py` 继续从完整 `WorkflowProject` 生成报告。它可以复用简报标题和顺序，但不能只保存简报内容。

`src/workflow_engine/script_gen.py` 在生成提示中声明 `card_summary` 输出契约。这个契约用于提升新 Workflow 的结果质量，但运行时不能假设所有模板都已迁移。

## 标准化与兼容

标准化按以下优先级运行：

1. 验证并使用顶层 `card_summary`。
2. 从已知字段映射：`conclusion`、`summary`、`verification`、`findings`、`issues`、`risks`、`deliverables`、`artifacts`、`next_steps`、`recommendations`。
3. 对纯文本或无法可靠识别的长结果，使用明确的中性结论“任务已完成，完整结果见报告”，不猜测正文含义。

旧字段映射只接受字符串、标量或具有已知语义的对象字段。未知的大对象不展开到卡片中，仍完整保存在报告里。

## 不截断与预算规则

“不截断”定义为：卡片不显示被字符切断的句子、列表项或对象字段；它不等于卡片包含全部原始结果。

- 每个候选条目先验证为完整语义单元，再按 UTF-8 payload 预算决定是否纳入。
- 单条内容本身超过简报条目上限时，整条移出卡片并计入遗漏数，绝不裁掉尾部。
- 关键发现按 high、medium、low、info 排序；同级保持原始顺序。
- 超出预算的条目统一显示“另有 N 条完整内容，详见报告”。
- 始终为结论、验证状态、报告状态和失败错误预留空间。
- 执行过程和低优先级可选栏目最先整体降级。
- 如果完整简报仍无法满足 28KB 安全预算，渲染最小终态卡：状态、可靠结论、遗漏计数和报告状态。

通用 `_enforce_card_size()` 继续保护进度卡等既有调用，但完成卡在进入通用保护层前必须已经满足预算。完成卡的回归测试禁止出现按字符裁剪产生的 `...` 或“内容已截断”。

## 终态数据流

1. JS runtime 返回完整结果，Engine 写入 `project.result`。
2. `on_done()` 生成完整 HTML/Markdown 报告并尝试上传附件。
3. `build_result_brief(project.result)` 构造标准化结果简报。
4. `fit_result_brief()` 按完整条目和 payload 预算选择卡片内容。
5. `render_completion_card()` 渲染一张终态卡并替换运行中卡片。
6. 迟到的 progress 回调继续被终态门禁忽略。

消息数量保持为一张被更新的 Workflow 卡片，以及成功上传时的一份报告附件。

## 异常处理

- `card_summary` 非法：记录诊断并回退到旧字段映射，不影响 Workflow 完成状态。
- 无可靠结论：使用中性结论并引导查看报告。
- 单条或栏目超量：整体省略并累计遗漏数。
- 报告上传失败：结果简报正常展示；卡片准确显示本地报告状态和完整文件名/路径，不声称附件已发送。
- 报告生成失败：结果简报仍正常展示，并显示简短、脱敏的失败原因。
- 完成卡预算计算异常：使用最小终态卡，不回退到包含原始结果前 500 字符的文本消息。

## 测试策略

遵循 TDD，先写以下失败回归：

- 标准 `card_summary` 能生成固定骨架，空栏目隐藏。
- 旧 JSON 字段能确定性映射到对应栏目。
- 纯文本或未知对象使用中性降级，不把长正文塞入卡片。
- 超长单条被整体省略，遗漏计数正确，未出现半句。
- 超多条目按严重度选择，完整报告仍包含所有哨兵内容。
- 完成卡序列化后不超过 28KB，且保留结论、验证和报告状态。
- 完成卡不包含“内容已截断”或字符切割产生的省略尾巴。
- 附件成功、附件失败和报告失败状态文案准确。
- `on_done()` 仍只更新一张终态卡，迟到进度不能覆盖它。
- HTML/Markdown 报告继续保存完整原始结果与 Agent 输出。

定向验证覆盖 `tests/test_workflow_result_brief.py`、`tests/test_workflow_renderer.py` 和 `tests/test_workflow_confirm.py`，随后扩大到 Workflow 相关测试、ruff、配置校验和 diff 检查。

## 长期维护原则

- 业务结果字段扩展只进入标准化层，不在渲染器中新增分支。
- 新模板优先生产 `card_summary`；旧模板通过兼容层逐步迁移。
- 卡片展示契约使用类型和测试约束，不依赖 prompt 文案约定。
- 完整报告是事实归档，结果简报是行动界面，两者不可互相替代。
- 所有超限处理以完整语义单元为粒度，禁止重新引入字符硬截断。
