# ACP 模型选择异步激活设计

日期：2026-07-10

## 背景与根因

当前普通编程模式在用户确认 ACP 模型后，同步调用
`_enter_mode_with_acp_model()`，后者继续调用 `enter_mode()` 或
`switch_model()` 并等待 ACP 会话启动完成。模型选择卡只有在该调用返回后才会
PATCH 成终态。

最新失败日志中，Codex ACP 启动先后耗尽 20 秒和 30 秒超时，导致模型确认回调
阻塞约 51 秒。与此同时，`_enter_mode_with_acp_model()` 只表示“找到了工具处理器”，
不会判断被 `silent=True` 吞掉的启动失败，因此调用方仍会把卡片更新成“已就绪”。

模型列表探测的 single-flight、正负缓存属于上一层问题，当前代码已经处理；本设计
只解决最终确认模型后的会话激活延迟和假成功，不再次修改模型探测策略。

## 目标

- 用户确认模型后快速结束卡片回调，不再同步等待 ACP 冷启动或重试。
- 原模型选择卡准确展示“初始化中、已就绪、初始化失败”三种状态。
- 只有会话真实可用时才显示“已就绪”。
- 保留当前 ACP 启动超时、重试和会话管理语义。
- 项目聊天暂存的首条需求只能在激活成功后转发一次。
- 同一聊天、项目和工具的重复确认不得并发启动多个会话。

## 非目标

- 不缩短全局 ACP 启动超时或重试次数。
- 不修改 ACP 模型探测、模型级联或 Workflow/Worktree/Spec 的选模流程。
- 不引入新的进程级执行器；复用已有 `TaskScheduler`。
- 不改变正常编程消息的 session recovery 语义。

## 方案比较

### 方案 A：快速回执并后台激活（采用）

确认模型后立即 PATCH“正在初始化”，再把会话激活提交给 `TaskScheduler`。后台任务
结束后 PATCH 同一张卡片为成功或失败终态。

优点是选择交互立即返回、仍可提前发现后端不可用、不会把初始化延迟完全转移到第一条
消息。代价是需要显式管理卡片中间态和后台任务幂等。

### 方案 B：纯懒启动

确认模型只保存配置并切换模式，等第一条编程消息再启动会话。实现较简单，但第一条消息
承担完整冷启动延迟，且选择阶段无法发现工具不可用。

### 方案 C：缩短选择阶段超时

继续同步启动，只降低超时和重试。改动最小，但仍阻塞回调，并会把正常冷启动误判为
失败，因此不采用。

## 详细设计

### 1. 卡片状态

在 `src/card/builders/system.py` 与 `CardBuilder` 外观增加 ACP 编程初始化卡片，保持
原模型选择卡原地 PATCH：

- `initializing`：展示已选工具和模型，并明确“正在启动会话”。
- `ready`：复用现有 `build_acp_programming_ready_card()`。
- `failed`：展示工具、模型和安全化错误摘要，并提供重试模型初始化和返回模型选择入口。

实现前先在 `ux/` 更新 HTML 预览，确认三态文案、按钮和移动端布局，再使生产卡片与预览
一致。

### 2. 调度边界

`SystemHandler.handle_select_acp_model()` 只做同步轻量工作：

1. 校验并持久化工具/模型选择，清除对应的旧 session snapshot。
2. 原地 PATCH `initializing` 卡片。
3. 捕获 `thread_root_id`、项目和暂存需求等不可依赖后台线程上下文的信息。
4. 向现有 `TaskScheduler` 提交一个 ACP 激活任务并立即返回。

激活任务使用稳定的 `queue_key`，粒度为聊天、项目和工具。相同 key 按已有 scheduler
串行规则执行，避免重复确认造成并发启动。任务元数据保留原消息 ID 和项目 ID，便于
追踪、取消与错误定位。

### 3. 激活结果契约

把 `_enter_mode_with_acp_model()` 从“处理器是否存在”的布尔值改为“激活是否成功”的
明确结果。普通 `enter_mode()` 和活动会话的 `switch_model()` 必须向调用方返回成功或
失败，不能依赖 `silent=True` 后的副作用推断。

最低契约为布尔结果：

- `True`：目标会话真实存在，且对应模式/线程上下文已建立。
- `False`：工具不支持、路径无效、启动/切换失败或会话未建立。

失败细节继续写入现有日志；卡片只展示经过 `get_error_detail()` 和现有错误安全边界处理
的摘要，不暴露命令行参数中的敏感信息。

### 4. 后台完成处理

- 成功：PATCH 原卡为 `ready`，随后转发暂存需求。
- 失败：PATCH 原卡为 `failed`，不得显示“已就绪”，也不得转发暂存需求。
- 卡片更新失败：记录 warning，不新建重复终态卡；用户后续消息仍走现有恢复路径。
- 暂存需求在提交激活任务前从槽位取出，并由该任务独占；任务结束后不重新入槽，防止
  重试时重复执行用户需求。

### 5. 竞态与顺序

- 相同聊天、项目和工具的激活任务串行。
- 后一次模型确认可以排在前一次之后；后台任务开始时重新核对项目当前工具/模型。若
  已被更新，则将旧任务视为 stale 并退出，不覆盖新卡片状态或启动旧模型。
- `thread_root_id` 显式传入 `enter_mode()`，不依赖后台线程的 thread-local。
- 第一条新消息若早于激活完成，继续由项目/聊天 scheduler 串行和现有
  `handle_message()` recovery 兜底，不绕开 `ACPSessionManager` 的 session-key 锁。

## 测试设计

遵循测试先行：先添加能稳定复现当前阻塞/假成功语义的失败测试，再实现最小改动。

### 处理器测试

- 最终模型确认不会在当前调用栈执行 `enter_mode()`，而是先 PATCH 初始化卡并提交任务。
- 后台激活成功后 PATCH `ready`。
- `enter_mode()` 返回失败时 PATCH `failed`，且不出现“已就绪”。
- 暂存需求只在成功后转发一次，失败时不转发。
- 后台任务获得显式 `thread_root_id`。
- 旧模型任务发现项目选择已变化时不启动、不覆盖卡片。
- 已在目标模式时，模型切换成功/失败都返回准确结果。

### 卡片测试

- 初始化、成功、失败三态 JSON 合法。
- 失败卡具有重试和返回选择动作，payload 只包含允许字段。
- 三态工具/模型文案一致，默认模型不显示空字符串。

## 验收标准

- 模型确认处理器不等待 ACP 启动超时；在单元测试中同步路径不调用阻塞启动函数。
- 日志中的 20 秒/30 秒 ACP 启动超时只发生在后台任务，不再占用卡片回调。
- 启动失败后模型卡显示失败态，不显示“编程模式已就绪”。
- Traex 正常启动仍最终显示 ready，并能处理暂存需求。
- 未引入重复会话、重复需求转发或旧任务覆盖新选择。
