# 卡片管线迁移 FAQ

> **另见**: [迁移设计文档](./2026-04-30-card-refactor-design.md) | [CHANGELOG](../CHANGELOG.md) | [ADR: Pipeline 边界](./adr-card-pipeline-boundary.md)

本文档回答从 DirectCardSession + CardBuilder 迁移到 CardSession dispatch 管线过程中的常见问题。

---

## Q1: 升级后历史卡片会怎样？

已发送到飞书的历史卡片**不受影响**。它们是静态的 JSON 消息，已持久化在飞书服务端。迁移仅影响后续新创建的卡片会话——新会话将使用 CardSession 管线渲染和投递。正在进行中的旧会话（如果有）会在自然结束后释放，不会产生冲突。

## Q2: 如何回滚到旧版本？

如果新版本出现严重问题：

1. 切换到迁移前的分支：`git checkout <旧分支名>`
2. 重新安装依赖：`uv sync --group dev`
3. 重启服务：`uv run python -m src.main`

旧版本的 DirectCardSession 代码仍保留在 git 历史中，可随时恢复。回滚不会影响已发送的卡片消息。

## Q3: 升级期间进行中的会话是否受影响？

建议在升级前等待所有活跃引擎会话（Deep/Loop/Spec/Worktree）自然结束或手动停止。如果在会话运行中重启服务：

- 飞书端的卡片会停留在最后一次更新的状态
- 用户可重新发送引擎命令（如 `/deep`、`/loop`）启动新会话
- 不会产生数据损坏或状态不一致

## Q4: 旧版本客户端（飞书 App）是否兼容？

卡片渲染完全在服务端完成，飞书客户端只负责展示。任何版本的飞书 App（桌面端、移动端）都能正常显示新管线生成的卡片，无需用户升级客户端。

## Q5: 按钮文案变化后会影响正在使用的用户吗？

文案变更仅影响**新创建**的卡片。已存在的卡片按钮文案不会自动更新（它们是静态 JSON）。用户下次启动新引擎会话时会看到更新后的按钮文案。但旧卡片的交互按钮将不再响应操作（点击后提示已失效）。

**用户点击旧卡片按钮时会看到的提示（Toast 消息）：**
- 已完成的任务："[引擎名]任务已完成，按钮不再可用。发送 [命令] 可开始新任务"
- 已超时关闭的任务："[引擎名]已超时关闭，按钮不再可用。重新发送 [命令] 可开始新任务"

**用户接下来该怎么做：** 直接在对话中发送对应引擎命令（如 `/deep`、`/loop`、`/spec`）即可开始新的会话，无需其他操作。

> 💡 提示：始终与对话中最底部（最新）的卡片交互，旧卡片按钮已失效。

## Q6: 如何验证迁移是否成功？

升级部署后，可通过以下步骤确认新管线正常工作：

1. **基本功能验证**：在群聊中发送 `/deep test`，观察卡片是否正常渲染（应显示进度状态、按钮）
2. **检查启动日志**：运行 `uv run python -m src.main --validate`，确认输出 "配置校验通过" 及参数摘要
3. **排除旧模块残留**：在项目目录运行 `grep -r 'DirectCardSession' src/`，应无结果（墓碑文件已删除）
4. **检查运行时日志**：启动服务后观察日志，确认无 `ImportError` 或 `DeprecationWarning` 涉及 `direct_session` / `adapter`
5. **多引擎覆盖测试**：依次触发 `/deep`、`/loop`、`/spec`、`/wt` 各一次，确认卡片能正常创建、更新和关闭
