## Slock 普通任务自动多角色协作

- 背景：用户反馈 Slock 模式里普通任务经常只有一个角色回复，没有自动拆解问题、分配多角色、触发讨论并共同完成任务。
- 根因：
  - `SlockHandler.handle_message()` 对普通非命令文本最终落到 `_execute_routed_message()`，即智能路由到单个 Agent；已有 `/task assign <content>` 无角色路径虽然能创建任务看板和协作计划，但普通消息没有复用这条链路。
  - `DiscussionManager._find_best_discussion_partner()` 只读 `_task_router`，而当前 `SlockEngine` 实际路由器字段是 `_router`，导致基于技能关键词的讨论对象选择退化。
  - 协作计划步骤 `_dispatch_collaboration_task()` 调用 `execute_task(..., callbacks=None)`，执行结果和后续讨论缺少可见卡片回调，计划链路难以在群里持续展示多角色产出。
- 修复：
  - `Settings` 新增 `slock_autonomous_task_planning_enabled=True`，用于控制普通任务消息是否自动进入任务看板和多角色规划。
  - `SlockHandler` 在命令/NLI/失败命令之后、Smart fallback 之前新增普通任务自动协作入口：非 @ 指定角色、非 shell-like 文本、`TaskClassifier` 判断为 `task` 时，直接复用 `assign_task(..., role_name="")`，进入已有任务看板、链式模板匹配、`CollaborationOrchestrator.create_plan()` 和计划卡片路径。
  - shell-like 文本复用 `IntentRecognizer` 判定，避免 `pytest tests/ -q`、`cd` 等命令被 Slock 任务规划抢走。
  - `DiscussionManager` 讨论对象选择兼容 `_router` 与 `_task_router`，恢复按技能关键词匹配 Security/Reviewer/Tester 等角色。
  - `SlockEngine._dispatch_collaboration_task()` 为计划步骤构造 `SlockEngineCallbacks`，将卡片 send/update 和 Agent done 结果卡接回群聊，便于后续自动讨论和步骤产出可见。
- 验证：
  - 红灯覆盖：普通任务应走 `engine.add_task()` + `create_plan()` 而不是 `_execute_routed_message()`；显式 `@Coder` 仍直达指定 Agent；shell-like 文本不被抢；讨论对象按 `_router.extract_skill_keywords()` 选择；计划步骤执行必须携带可见 callbacks。
  - `.venv/bin/python -m pytest tests/test_slock_autonomous_collaboration.py -q` -> `5 passed`。
  - `.venv/bin/python -m pytest tests/test_slock_autonomous_collaboration.py tests/test_slock_handler_nli_integration.py tests/test_slock_async_message.py tests/test_slock_e2e_integration.py tests/test_slock_task_execution.py tests/test_slock_discussion.py -q` -> `150 passed, 12 warnings`。warnings 为既有 async 测试里 `IntentRouter.classify_intent` coroutine 未 await。
  - `.venv/bin/python -m src.main --validate` -> 配置校验通过。
  - `git diff --check` -> 通过。
- 风险/后续：
  - 真实多角色效果仍依赖群里已创建/配置对应角色；如果只有一个可用角色，计划步骤中缺角色的部分仍会跳过。
  - 另行抽测 `tests/test_slock_dispatch_loop.py::TestTimeoutDetection::test_fresh_task_not_timed_out` 仍失败，原因是旧测试用 `SlockEngine.__new__` 绕过 `__init__` 后没有设置 `_task_mgr`，不是本轮普通任务自动协作路径引入。

## Workflow 预算重复计数修复

- 背景：Workflow 新增预算显示与预算耗尽保护后，`WorkflowEngine` 在 agent 执行期间通过 `_on_token_usage()` 累加 `project.budget.used`，而 `WorkflowProgressRenderer.on_agent_done()` 又在渲染层对同一个 `budget.used` 再加一次相同 token 数。
- 根因：预算状态被两个层级同时维护——执行层负责真实 token 记账，渲染层本应只同步 agent/metrics 展示，却额外修改了共享 `BudgetState`，导致每个真实 agent 调用都被双倍计入。
- 修复：删除 `src/workflow_engine/renderer.py` 中 `on_agent_done()` 对 `self._project.budget.used` 的重复累加，保留 `WorkflowEngine._on_token_usage()` 作为预算已用量的唯一写入点。
- 测试：在 `tests/test_workflow_budget_lock.py` 新增回归，用 fake executor 模拟一次真实 agent 调用链路（先触发 `_on_token_usage(120)`，再走 `_handle_agent_call()` + renderer 更新），断言 `budget.used == 120` 而不是 240；同时确认 `metrics.total_tokens` 仍正确为 120。
- 验证：`uv run python -m pytest tests/test_workflow_budget_lock.py tests/test_workflow_renderer.py -q` -> `21 passed`。
- 风险/后续：本次只修复 Python 侧双重累加；Workflow JS runtime 侧预算同步仍值得后续单独复核，但不属于本轮已确认缺陷的直接修复范围。

## Workflow 按钮 Schema 2.0 迁移与错误反馈增强

- 背景：飞书卡片 Schema 2.0 废弃了 `{tag: 'action', actions: [...]}` 容器，要求使用 `column_set` 布局；同时按钮回调值需要通过 `behaviors[0].value` 读取，而不是旧的 `button.value`。
- 修复：
  - `src/card/render/buttons.py` 新增 `build_responsive_button_row(buttons)` 辅助函数，根据按钮数量自动选择合适的 `column_set` 布局（1个全宽、2个平分、3个三等分、>3个双行排列）。
  - `src/feishu/handlers/workflow.py` 所有 `{tag: 'action', actions: [...]}` 容器全部替换为 `build_responsive_button_row()` 调用，包括推荐工具按钮、更多工具面板内按钮、预算选择按钮、确认/取消按钮。
  - 所有按钮 dict 新增 `'behaviors': [{'type': 'callback', 'value': <value>}]` 字段，与 `value` 字段保持一致。
  - `src/feishu/ws_client.py` `_process_card_action_async()` 优先读取 `action.behaviors[0].value`，降级到 `action.value`。
  - `src/feishu/ws_card_action_handler.py` `value_dict()` 和 `dedup_fingerprint()` 同样优先读取 `behaviors[0].value`。
  - `src/feishu/action_registry.py` `_resolve_project()` 当 `pid` 为空时降级到 `get_active_project(cid)`，避免卡片按钮无 `project_id` 时无法解析项目。
  - `src/feishu/handlers/workflow.py` `handle_workflow_select_tool()` 和 `handle_workflow_select_budget()` 校验失败时从 `reply_text()` 改为 `reply_error()`，提供更醒目的错误卡片反馈。
- 测试更新：
  - `tests/test_workflow_confirm.py` 按钮收集逻辑兼容 `column_set` 内直接放置的 `button` 标签。
  - `tests/test_workflow_renderer.py` 更新三个测试：`test_budget_section_uses_markdown` 检查 `💰` 而非 `Budget`；`test_confirm_card_has_tool_buttons` 和 `test_confirm_card_has_confirm_cancel_buttons` 兼容 Schema 2.0 布局。
  - `tests/test_workflow_renderer.py` `_progress_bar` 导入改为 `_pct`，测试方法重命名为 `test_pct_*`。
- 验证：
  - `uv run python -m pytest tests/test_workflow*.py -q` -> `238 passed`。
  - `uv run python -m pytest tests/test_workflow*.py tests/test_ws_card_action_handler.py tests/test_action_dispatcher.py tests/test_card_schema_contract.py -q` -> `258 passed`。
- 风险/后续：
  - `tests/test_feishu_dispatcher.py` 中有 7 个测试失败，涉及编程模式转发，经检查与本次 Schema 迁移无关，为既有问题。
  - 其他引擎（Deep/Spec/Worktree/Slock）的按钮仍使用旧的 `action` 容器，后续需要统一迁移到 `build_responsive_button_row()`。

## Workflow AC19 错误分类单元测试

- 背景：AC19 验收项要求验证所有回调验证失败返回标准化错误卡片，覆盖 `session_expired`、`invalid_state`、`invalid_argument`、`forbidden` 四种错误类别。
- 实现：
  - 新增 `tests/test_workflow_error_categories.py`，共 13 个测试：
    - 工具选择回调（5 个）：missing engine → session_expired、wrong state → invalid_state、session key mismatch → session_expired、wrong initiator → forbidden、missing tool_name → invalid_argument
    - 预算选择回调（5 个）：missing engine → session_expired、wrong state → invalid_state、invalid tokens → invalid_argument、session key mismatch → session_expired、wrong initiator → forbidden
    - 错误卡片 UI（3 个）：通过 reply_card 可见、所有类别有 UI 文本、正确使用对应 UI 文本
  - 测试使用 `WorkflowHandler.__new__` 绕过 `__init__`，mock `ctx.workflow_engine_manager.get()`、`_reply_workflow_error()` 等依赖，通过 `@patch("src.thread.get_current_sender_id")` 模拟当前用户。
  - 修复测试编写过程中发现的两个问题：
    - 卡片结构从 `card["elements"]` 改为 Schema 2.0 的 `card["body"]["elements"]`
    - 移除不存在的 `internal_error` 类别，只保留已实现的 4 种错误类别
- 验证：
  - `uv run python -m pytest tests/test_workflow_error_categories.py -v` -> `13 passed in 2.26s`
  - `uv run python -m pytest tests/test_workflow*.py -q` -> 所有 workflow 测试通过
- 风险/后续：
  - `_reply_workflow_error` 目前只在 `handle_workflow_select_tool` 和 `handle_workflow_select_budget` 中使用，后续其他回调（如 confirm_tools、confirm_start、cancel）也应统一使用该错误响应机制。
