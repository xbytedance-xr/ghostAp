## 修复卡片流式输出速度缓慢/卡顿问题
### 任务描述
排查卡片流式输出（Card Streaming）字速度缓慢、卡顿的问题，并进行修复。

### 执行内容
1. **原因排查**：深入分析了卡片流式输出机制，发现在 `StreamingCardManager.update_content` 中触发时，如果文本长度满足阈值或者时间到达流控间隔，则会立刻调用飞书的同步 API `self._client.im.v1.message.patch(req)` 进行更新。而 `update_content` 的调用方在 ACP 会话事件处理回调 `_dispatch_event` 中，这意味着飞书的同步网络调用阻塞了处理底层的 `asyncio` 事件循环（stdout 流读取）。这导致了代理进程输出堆积且无法被实时读取，呈现肉眼可见的卡顿。
2. **异步化优化**：在 `StreamingCardManager` 中引入了 `ThreadPoolExecutor`。`update_content` 目前仅处理内存状态（如截断、判断流控）并立即返回。实际的请求构建及 HTTP 发送被推入单独的后台任务 `_do_patch_task` 中执行。
3. **并发安全控制**：为 `StreamingCard` 新增 `lock`、`pending_update`、`is_inflight` 字段，确保同一张卡片的请求保持顺序执行且不发起重叠的 PATCH 调用，避免网络竞态导致渲染顺序错乱。
4. **测试修复**：在相关涉及 mock 调用次数及更新延迟的测试用例（`test_streaming.py` 和 `test_streaming_logic.py`）中加入了非阻塞等待，确保异步任务能够正确被 pytest 验证。

### 技术要点
- 将同步阻塞的网络 IO 与实时推流（asyncio pipe reader）隔离开，解决流式输出速率的瓶颈。
- 利用基于状态和锁的单背景任务模式（drain queue）来避免多线程提交时的 API 频控及乱序问题。
