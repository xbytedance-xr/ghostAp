## 审查超时 in-cycle auto-retry 机制实现

### 任务描述
落实「审查异常：当前系统较繁忙，操作已超时」场景的改进建议。此前文案承诺"稍后自动重试"，但实际无对应实现——review 全超时后仅靠 cycle 循环隐式重执行，若 max_cycles 耗尽则需用户手动 `/spec resume`。本次新增 in-cycle delayed retry 机制，使文案与实际行为对齐。

### 执行内容

**工具层——新增 `compute_retry_delay()` 函数：**
- 位于 `src/utils/review_helpers.py`，公式 `min(base_delay * 1.5^n, max_delay)`
- 基于 `consecutive_timeouts` 计算渐进等待时间（5s → 7.5s → 11.25s → ... → 30s 封顶）

**配置层——新增 2 个配置项：**
- `spec_review_auto_retry_enabled: bool = True` — 主开关
- `spec_review_retry_max_delay: int = 30` — 最大延迟上限（秒）

**核心逻辑——`_conduct_review_pipeline()` in-cycle retry：**
- 全超时分支检测 `all_timeout=True && auto_retry_enabled`
- `sleep(compute_retry_delay())` 后用缩短的 budget 重跑全部视角（最多 1 次）
- retry budget 使用 `compute_adaptive_timeout(consecutive+1)` 计算，低于首次 budget
- retry 成功：替换 outcomes、重置所有 circuit 计数器（consecutive_timeouts/failures/backoff/skips 全归零）
- retry 失败/异常：走原有 circuit 递增路径，行为与修改前一致

**文案对齐——消除虚假承诺：**
- `timeout_busy_spec` 文案从"建议：稍后自动重试，或通过 /spec resume 手动恢复"改为"将在当前轮次内自动重试一次，若仍失败请通过 /spec resume 手动恢复"
- `review.py` L433 处硬编码文案改为引用 `UI_TEXT["timeout_busy_spec"]`

**可观测性——retry 追踪字段：**
- `circuit.last_review_failure_diag` 新增 `retry_attempted`/`retry_succeeded` 字段
- `handle_review_exception` metrics 默认携带 `retry_attempted=False`/`retry_succeeded=False`

### 技术要点
- retry 在 circuit 计数器更新**之前**执行，成功时直接重置，避免 retry 与熔断器竞争
- retry 使用全新 `CycleBudget` + `run_review_pipeline`（内部创建 `EphemeralReviewSession`），不复用首次超时的 session
- `normalize_review_diagnostics` 使用白名单 `REVIEW_DIAG_STABLE_KEYS`，retry 字段在 normalize 后手动附加到 dict
- `time.sleep` 在函数内通过 `import time as _time` 导入，测试中通过 `patch("time.sleep")` 拦截

### 修改文件

- `src/config.py` — 新增 `spec_review_auto_retry_enabled` / `spec_review_retry_max_delay`

## 重构：提取 `_handle_pipeline_errors_with_retry` 降低 `_conduct_review_pipeline` 圈复杂度

**简化 `_conduct_review_pipeline` else 分支：**
