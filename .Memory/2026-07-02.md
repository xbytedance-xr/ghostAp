## WF 选模后自动执行

- 背景：用户反馈 `/wf` 选完主编排 Agent 与评审 Agent/Auto 后，再展示人工确认卡会中断节奏；期望模型选完后直接生成编排脚本并开始执行，运行过程中由 workflow 动态调整即可。
- 实现：
  - `WorkflowHandler._generate_and_show_confirm_card()` 生成脚本后不再构建确认卡；它仍先写入 `PendingConfirmation`、记录脚本 hash、保留 selected tools / orchestrator / review bindings，然后调用 `_start_pending_workflow_execution()` 自动启动。
  - 新增 `_start_pending_workflow_execution()`，服务端自动启动时临时绑定 pending 中保存的发起人身份，并复用既有 `handle_workflow_confirm_start()` 路径，保留脚本重读、hash 校验、脚本安全验证、工具白名单修正、immutable temp copy 和进度卡切换逻辑。
  - 生成阶段优先保留旧 pending 的 `initiator_user_id`，避免异步生成线程缺少 `get_current_sender_id()` 时把发起人清空。
  - Workflow stepper、入口卡、帮助文本、旧工具选择按钮文案从“脚本确认/确认执行”同步改为“自动生成并执行”；底层确认卡与 `workflow_confirm_start` 兼容入口保留，继续由既有测试覆盖。
  - 顺手修正 `test_agent_call_times_out` 的测试 deadline：mock future 立刻抛 `TimeoutError` 时必须将 `SESSION_CREATE_TIMEOUT_S` 缩短，否则会等到 pytest-timeout，而非测试真正的超时返回语义。
- 测试：

  - 绿灯：新增用例通过，并更新 regenerate / entry-help / tool-select 断言匹配自动执行流程。

## WF 嵌套 race 取消状态修复

- 背景：race() 原语在嵌套 workflow（模板内调用 race）中，loser agent 的取消状态无法向上传播到父引擎 state_manager 和进度卡片，导致 loser 一直显示为"执行中"。
- 根因：`src/workflow_engine/bridge.py` 中 `_execute_sub_workflow` 构造子 RuntimeBridge 时，只传了 `on_agent_call`/`on_phase`/`on_log`，未传 `on_agent_aborted`。子 bridge 收到 JS runtime 的 `agent_aborted` 通知后因回调为 None 被静默丢弃。
- 修复：子 RuntimeBridge 构造补充 `on_agent_aborted=self._on_agent_aborted`，使取消通知穿透嵌套层级。
- 测试：
  - 新增 `test_nested_workflow_race_abort_propagates_to_parent`（真实 Node 子进程）：验证父 bridge `on_agent_aborted` 收到子 workflow 内 race loser 的通知、slow 的 cancel_event 被 set、workflow 在 10s 内完成。
  - 红灯验证：移除修复后测试失败（aborted_labels=[]），确认测试准确捕获 bug。
  - 已有 `test_race_cancel_real_node_process`（test_workflow_bridge_transport.py）覆盖单层级 race 取消，保持绿灯。
- 修改文件：
  - `src/workflow_engine/bridge.py`（第 844 行新增 on_agent_aborted 参数）
  - `tests/test_workflow_race_cancel.py`（新增第 8 节 + `import pytest`）

## Deep/Spec grill-me 自动质询接入

- 背景：需要借鉴 mattpocock/skills 的 grill-me 思路，让 Deep 与 Spec 在无需用户二次确认的情况下自动追问、给推荐答案，并采纳能提升方案质量的建议；Spec 还需要结合既有多角色/多模型 review 机制做综合判断。
- 外部参考：`https://github.com/mattpocock/skills/tree/main/skills/productivity/grill-me` 当前核心描述是进入 `/grilling` 式 relentless interview；本地落地为 prompt 内部协议，避免新增运行时外部依赖。
- 实现：
  - 新增 `src/grill_me.py` 作为 Deep/Spec 共用 prompt 片段，定义自动质询、推荐答案、自动采纳、多角色 review 和 completion_control 综合采纳判断规则。

  - 传统五视角 review 与 adaptive role review prompt 注入多角色 grill-me 规则；`completion_control` 专门增加“综合采纳判断”，只阻断有证据、需要自动采纳的 blocker/major 建议。
- 测试：

## Spec adaptive review 既有失败测试修复

- 背景：用户要求修复上一轮扩大验证发现的失败 case，即使不是 grill-me 接入引入的问题。
- 根因：
  - `ReviewRoleSpec` 是 frozen dataclass，`tests/test_adaptive_review_timeout.py` 的旧测试在构造后写 `role.blocking = ...`，导致 `FrozenInstanceError`，未实际测到 timeout 降级语义。
  - `AdaptiveRoleReviewStrategy` 创建 `EphemeralReviewSession` 时会传 `startup_timeout`；`tests/test_spec_review_agent_selection.py` 的 fake session 构造签名未跟生产接口同步，导致 fake 在进入 fallback/agent 分配逻辑前抛 `unexpected keyword argument 'startup_timeout'`。
- 修复：
  - `_role()` 测试 helper 增加 `blocking` 参数，在构造 `ReviewRoleSpec` 时写入 frozen 字段，删除所有后置赋值。
  - 两个 `FakeReviewSession` 补 `startup_timeout: float | None = None` 参数，保持测试继续只关注 agent/model 捕获和 fallback 顺序。
- 验证：
  - 红灯复现：`uv run python -m pytest tests/test_adaptive_review_timeout.py tests/test_spec_review_agent_selection.py -q` → 21 failed, 25 passed。
