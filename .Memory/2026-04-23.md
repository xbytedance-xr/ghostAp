## 消除 DeepEngineCallbacks 重复定义，统一 HasOnError Protocol 兼容性
### 任务描述
`src/deep_engine/callbacks.py` 中存在与 `src/deep_engine/engine.py` 重复的 `DeepEngineCallbacks` 定义。`callbacks.py` 版本的 `on_error` 类型为 `Callable[..., None]`（默认 `_NOOP`），与 `engine_base.py` 中 `HasOnError` Protocol 声明的 `Optional[Callable[[str], None]]` 不兼容，导致静态类型检查（mypy/pyright）下无法满足 Protocol 约束。

### 执行内容
1. **删除 `src/deep_engine/callbacks.py`** — 该文件仅包含重复的 `DeepEngineCallbacks` 定义和 `_NOOP` 辅助，且 `src/` 中无其他模块导入该文件。`__init__.py` 公开导出来源为 `engine.py`，`engine.py` 版本的 `on_error: Optional[Callable[[str], None]]` 与 `HasOnError` Protocol 类型完全兼容。
2. **更新 `tests/test_deep_callbacks_metrics.py`** — 将两处 `from src.deep_engine.callbacks import DeepEngineCallbacks` 改为 `from src.deep_engine.engine import DeepEngineCallbacks`；同步更新模块 docstring。
3. **验证** — 目标测试 11 passed；全量 2761 tests passed，零回归。

### 技术要点
- `engine.py` 中的 `DeepEngineCallbacks` 使用 `Optional[Callable[[str], None]]` 类型，默认值为 `None`，与 `HasOnError` Protocol 声明一致。
- `callbacks.py` 中的版本使用 `Callable[..., None]` 加 `_NOOP` 默认值，虽然运行时 `@runtime_checkable isinstance` 通过（仅检查属性存在），但静态分析下签名不兼容。
- 删除重复定义是最干净的方案，因为 `__init__.py` 已从 `engine.py` 导出 `DeepEngineCallbacks`，不影响任何公开 API。
