## Thread 感知模式路由修复（R3-Fix1 + R3-Fix2）

### 任务描述
修复 `ws_client.py` 中线程（Thread）场景下模式路由的两个问题：当 `auto_enter_mode`（来自 ThreadContext）被设置时，`/exit` 和编程入口命令无法被正确拦截；`_process_with_intent` 和 `_dispatch_empty_text` 无法感知 Thread 级别的模式状态。

### 执行内容

#### R3-Fix1: `_dispatch_message_logic` 拦截控制命令
- **问题**: `auto_enter_mode` 非空时所有消息直接进入编程 handler，`/exit` 和 `/coco`、`/claude` 等命令无法退出话题编程会话
- **修复**: 在 `auto_enter_mode` 具体模式分支前，新增统一拦截：
  - `/exit` → `_exit_current_mode()` 退出当前模式
  - 编程入口命令 → 友好提示"当前话题已在编程模式中"

#### R3-Fix2: `_get_effective_mode` 辅助方法 + 两处调用替换
- **问题**: `_process_with_intent` 和 `_dispatch_empty_text` 通过 `_mode_manager.get_mode()` 获取模式，无法感知 ThreadContext 中的模式覆盖
- **修复**:
  - 新增 `_get_effective_mode(chat_id, project_id)` 方法：优先从 `ThreadContextManager` 获取当前 thread 的模式，回退到 `_mode_manager`
  - `_process_with_intent`: 替换 `get_mode()` + `is_programming_mode()` 为 `_get_effective_mode()`
  - `_dispatch_empty_text`: 替换 `get_mode()` 为 `_get_effective_mode()`

### 技术要点
- `_get_effective_mode` 利用已有的 `get_current_thread_id()` thread-local 机制获取当前线程上下文
- 返回 `(InteractionMode, bool)` 二元组，与原有 `get_mode()` + `is_programming_mode()` 的使用模式一致
- 当 ThreadContext 模式为 "smart" 时不覆盖，回退到全局模式管理器

### 文件变更
- `src/feishu/ws_client.py`: +3 处修改（新增 `_get_effective_mode` 方法、`_dispatch_message_logic` 拦截逻辑、两处调用替换）

## R4-Fix4 + R4-Fix5: Thread 编程会话恢复与淘汰清理

### 任务描述
修复线程（Thread）多项目编程模式下的两个问题：(1) `handle_card_resume` 恢复会话时未重新注册 ThreadContext；(2) ThreadContext 被淘汰/移除时未清理对应的 ACP Session，导致资源泄漏。

### 执行内容

#### R4-Fix4: handle_card_resume re-register ThreadContext
- **问题**: 在话题中通过卡片恢复编程会话（resume）时，session 成功启动但 ThreadContext 未重新注册，导致后续消息路由丢失线程上下文
- **修复**: 在 `handle_card_resume` 的 Claude 路径和 ACP 路径成功后，新增 `if thread_id and project: self._register_thread_context(thread_id, chat_id, project, session)`
- **文件**: `src/feishu/handlers/programming.py` (+4 行)

#### R4-Fix5: ThreadContext on_evict 回调触发 ACP Session 清理
- **问题**: ThreadContext 过期淘汰或手动移除后，对应的 ACP Session 无人清理，造成后端进程泄漏
- **修复**:
  - `ThreadContextManager.__init__` 新增 `on_evict: Optional[Callable[[ThreadContext], None]]` 参数
  - `_evict_expired`/`remove`/`remove_by_chat`/`close` 4 个方法中，在 lock 外调用 `self._on_evict(ctx)`，均有 try/except 保护
  - `FeishuWSClient.__init__` 中设置 `self._thread_manager._on_evict = self._on_thread_evicted`
  - 新增 `_on_thread_evicted` 方法：遍历 6 个 session manager（coco/claude/aiden/codex/gemini/ttadk），调用 `end_session(chat_id, project_id, thread_id=thread_root_id)` 清理关联会话
- **文件**: `src/thread/manager.py` (+42 行), `src/feishu/ws_client.py` (+15 行)

#### 附带修复: test_handlers.py settings mock
- **问题**: `_make_handler_context()` 使用 `MagicMock()` 作为 settings，`thread_programming_enabled` 属性返回 truthy MagicMock，导致 `_thread_enabled` 为 True 时 `_enter_mode_on_manager` 不被调用，测试断言失败
- **修复**: 显式设置 `settings.thread_programming_enabled = False`
- **文件**: `tests/test_handlers.py` (+2 行)

### 技术要点
- on_evict 回调必须在 lock 外调用，避免死锁（回调可能反向访问 manager）
- on_evict 回调的 try/except 防止单个回调异常阻断清理线程或其他回调
- 遍历所有 6 个 session manager 是 safe 的，因为 `end_session` 对不存在的 session key 返回 None

## Thread 编程 ACP Session rebind_thread 修复

### 任务描述
修复 Thread 编程模式下 ACP session 查找失败的 bug：`enter_mode` 创建 ACP session 时 `thread_id=None`（因为 thread 尚不存在），但后续 thread 消息以 `thread_id=response_id` 查找 session，导致 key 不匹配找不到 session。

### 执行内容
1. **ACPSessionManager.rebind_thread()** — 新增方法，将 session 从 `chat_id:project_id`（thread_id=None）迁移到 `chat_id:project_id:t:thread_id`
2. **ProgrammingModeHandler.enter_mode()** — 在两个 `_register_thread_context` 调用点前（恢复路径+新会话路径）调用 `rebind_thread`，确保 session key 与后续 thread 消息的查找 key 一致
3. **TestRebindThread** — 3 个测试用例：session 迁移成功、缺失时返回 False、不影响其他 key

### 技术要点
- session key 格式：无 thread 时为 `{chat_id}:{project_id}`，有 thread 时为 `{chat_id}:{project_id}:t:{thread_id}`
- rebind 发生在 `reply_message_with_id` 返回 `response_id` 之后、`_register_thread_context` 之前
- 操作在 `_lock` 下原子执行，保证线程安全
