## Deep 子任务卡片冻结与最终汇总卡优化

- 问题：Deep 进入多任务/子任务卡片后，agent/subagent 子任务的运行中摘要和 `TOOL_CALL_START` 进度仍可能 PATCH 到父任务或最初任务卡；普通计划任务完成后也会继续接收后续任务列表广播，导致已完成卡片还在刷新。
- 修复：`TaskOrchestrator` 新增终态任务冻结集合；普通 plan task 完成/失败时先给对应任务卡发送 `completed/failed`，随后从后续 `TASK_LIST_UPDATED` 广播中排除；已终态的子任务后续 ACP update 会被消费但不再落到旧卡。
- 修复：agent/subagent 子任务注册和完成不再广播到父任务卡，也不再通过 `TOOL_MODEL_CHANGED(subagents=...)` 实时 PATCH 父卡摘要；子任务内容只更新自己的子任务卡。`DeepRenderer` 先路由 agent/subagent `TOOL_CALL_START` 以创建子卡，再把进度事件发给该子卡，避免父卡 footer 被子任务进度刷新。
- 修复：Deep 多卡任务完成后不再只关闭现有任务卡；`TaskOrchestrator.finish_with_summary()` 会先停止/终态化各任务卡，再创建 `任务总结` 新卡承载最终完成/失败摘要。

## Deep 子任务独立卡片路由修复

- 问题：Deep 多任务已经接入 `TaskOrchestrator`，但并行子任务如果以 ACP `agent/subagent` 工具调用形式出现，后续 `TOOL_CALL_START/UPDATE/DONE` 仍按最近活跃主任务解析，容易表现为子任务卡片只有任务列表，详情和最终结果集中在最初或某一张主任务卡片。
- 修复：`TaskOrchestrator.route_acp_event()` 在普通 plan task resolver 前先识别 agent/subagent 工具调用（`title=agent/subagent` 或内容包含 `子代理：`），按 `tool_call.id` 注册独立子任务 CardSession，并把 start/update/done 事件路由到该子任务卡。
- 修复：子任务 done 时向独立卡片同步 `completed/failed` 终态和最终内容；后续优化已取消父任务卡的实时子任务摘要 PATCH，避免子任务进展继续刷新父卡。
- 边界：未改 `DeepEngine`、`render`、`delivery`；仍通过 `CardEvent` 和 `CardSession.dispatch()` 走现有 `handler -> session -> render/delivery` 管线。
