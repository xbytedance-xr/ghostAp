# 2026-06-03

## Traex 模型列表 lazy-init 修复

- 背景：飞书里触发 `/traex` 后模型选择卡显示“获取 Traex 模型列表失败”；`logs.log` 只记录到 `[ACP] selecting tool=traex ...` 和 `refresh_acp_models` 点击，没有模型 probe 成功日志。
- 根因：`src/acp/helper.py::probe_acp_models()` 直接从 `tool_registry.get_provider("traex")` 取 provider，但 ACP providers 是 lazy 注册的；如果当前进程里还没有其他路径调用 `get_providers()`，registry 为空，Traex probe 直接返回空列表。真实 `traex acp serve` 的 `new_session` 响应其实包含 27 个模型，位于 `config_options` 的 `category="model"` 中，既有 `_extract_models_from_config_options()` 可以正确解析。
- 修复：
  - `src/acp/helper.py` 引入并调用 `get_providers()`，在 `probe_acp_models()` 读取 provider 前确保 lazy provider registry 已初始化。
  - 同步让 `list_acp_tools()` 也先触发 provider 初始化，避免 `/acp` 工具列表路径遇到同类空 registry 问题。
  - 新增 `tests/test_acp_model_probe_timeout.py::test_probe_acp_models_initializes_lazy_providers_for_traex`，重置 provider registry 后用 fake ACP stdio 响应验证 Traex 模型 probe 不依赖外部预初始化。
- 验证：
  - 红灯：新增测试在修复前返回空列表失败。
  - `uv run python -m pytest tests/test_acp_model_probe_timeout.py -q` -> `12 passed`。
  - `uv run python - <<'PY' ... fetch_acp_models('traex') ... PY` -> 返回 `count 27`，默认模型 `c_o_new_thinking`。
  - `uv run python -m pytest tests/test_model_command.py tests/test_acp_provider_extensions.py tests/test_worktree_tool_discovery.py -q` -> `67 passed`；后续合并 probe 测试后相关集 `79 passed`。
  - `uv run python -m src.main --validate` -> 配置校验通过。
  - `uv run ruff check src/acp/helper.py tests/test_acp_model_probe_timeout.py` -> 通过。
  - `git diff --check` -> 通过。
