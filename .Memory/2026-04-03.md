## 修复话题编程持续对话失败 — thread_root_id 使用原始 message_id

### 任务描述
话题内后续消息无法继续编程模式（回退到 SMART 模式提示"无法理解你的意图"），根因是 `_dispatch_to_thread` 使用 `reply_message_with_id()` 的返回值作为 `thread_root_id`，但飞书话题 `root_id` 等于被回复的原始 `message_id`，导致 ThreadContext 查找失败。

### 执行内容
1. **root cause**: `reply_message_with_id()` 返回的是回复消息的 ID（reply_msg_id），而飞书后续话题消息的 `root_id` 是原始 `message_id`。ThreadContext 以 reply_msg_id 注册，后续消息以 message_id 查找 → 查找失败
2. **修复**: 在 `_dispatch_to_thread` 中将 `thread_root_id = message_id`（原始消息ID），`reply_message_with_id` 返回值仅用于成功/失败检查
3. **测试**: 新增 `test_dispatch_to_thread_uses_message_id_as_root` 验证 enter_mode/get_session/register_thread_context 均使用 message_id

### Skills 调查结论
用户反馈"没有加载coco的skills"。经调查：`~/.coco/skills/` 包含 20 个 lark-xxx skills；ACP session 继承 `os.environ.copy()`（HOME 保留）；截图显示 GetDocument 成功调用（lark-doc skill）。实际问题是飞书 API 权限不足（"权限不足"错误），非 skills 加载问题。

### mloop 迭代记录
- Round 1: CLEAN — 修复最小化且正确
- Round 2: CLEAN → 收敛 (2/2 连续干净轮次)

### 技术要点
- 飞书话题语义：`reply_in_thread=True` 回复 `message_id` 时，新话题的 `root_id = message_id`（被回复消息），不是 reply 消息自身的 ID
- 3 行核心修复：`reply_id = handler.reply_message_with_id(...)` + `thread_root_id = message_id`

## 话题内持续编程模式 — 跳过 enter_mode 防止 session 被旧 snapshot 覆盖

### 任务描述
话题创建后第一条消息能正常编程，但第二条消息回退 SMART 模式（"无法理解你的意图"）。用户需求：话题内一直保持编程状态直到显式 /exit。

### 根因
`_dispatch_message_logic` 每条话题消息都调用 `enter_mode`，而 `enter_mode` 使用 project snapshot 的旧 session_id 调用 `ensure_session`，导致 thread 已有的 ACP session 被杀掉重建。具体链路：
1. 话题消息到达 → auto_enter_mode = "coco"
2. `_dispatch_message_logic` → `_enter_coco_mode` → `enter_mode`
3. `enter_mode` 读取 project.coco_session_snapshot（旧 session_id）
4. `ensure_session` 发现已有 thread session 的 session_id 不匹配 → 杀掉 thread session
5. 新 session 可能创建失败或上下文丢失

### 执行内容
1. **FIX 1** (ws_client.py _dispatch_message_logic): auto_enter_mode 有值时（话题消息），通过 `_get_mode_handler(InteractionMode(auto_enter_mode))` 直接调用 `handler.handle_message`，跳过 `enter_mode`。取代原 6 个 elif 分支（_enter_xxx_mode + _handle_xxx_message）。
2. **FIX 2** (programming.py enter_mode L252): 条件改为 `snapshot.is_resumable and not thread_id`，话题内不使用旧 snapshot session_id 作为安全网。
3. **FIX 3** (ws_client.py _dispatch_message_logic): 话题内 /exit 退出路径补充 `_should_defer_exit` 判断，与主对话退出逻辑对齐。

### mloop 迭代记录
- Round 1: CLEAN(产品/架构) + SUGGESTION(工程/测试) → 补充 defer_exit + 测试覆盖
- Round 2: CLEAN(产品+架构) + 轻微建议(工程+测试) → 不影响目标达成
- Round 3: 四合一审查，产品/架构 No material → 收敛 (2/2 连续无实质建议)

### 技术要点
- 话题消息路由：auto_enter_mode → 三层拦截（退出/入口/引擎限制）→ handler.handle_message
- enter_mode 的 snapshot 干扰：thread 内不应使用主对话的旧 session snapshot
- handle_message 自身有 session fallback（session 不存在时自动 enter_mode 恢复）
