## Traex 复合模型 ID 后端规范化

### 背景 / 动机
用户反馈 Deep 执行失败，卡片显示 `Model metadata for c_o_new_thinking/max/max not found ... Internal error`。日志复盘确认：普通 Traex 会话在 `session/setModel` 中直接传入 UI 级联选择值 `c_o_new_thinking/max/max`，随后 Deep 以同一项目保存的模型值启动 Traex，并在 prompt 阶段重试两次后 `Internal error`。本质不是模型服务不可用，而是 Traex ACP 后端只接受 slug/base 模型名（如 `Test-O-New-Thinking`），不能接受 UI 的 `config_name/profile/effort` 复合值。

### 改了什么
1. **Provider 统一规范化入口**
   - `src/acp/provider.py` 协议新增 `normalize_model_name()`。
   - `src/acp/providers/__init__.py` 新增 `normalize_acp_model_name(tool, model)`，默认 provider 只 trim，Traex provider 复用既有“剥离 variant suffix + config_name→slug”逻辑。
   - Traex `get_serve_command()` 改为走 `normalize_model_name()`，保持启动参数和协议参数同源。

2. **所有后端执行入口使用规范化模型**
   - `src/feishu/handlers/programming.py`：普通编程模式在线 `session/setModel` 和失败后重启会话都传后端模型名，卡片仍显示用户选择值。
   - `src/agent_session/factory.py`：`create_sync_session()`、`create_engine_session()`、`create_review_session()` 在 ACP 启动前规范化模型，覆盖 Deep/Spec/Review/Slock 等直接会话路径。
   - `src/acp/manager.py`：普通模式首次进入、会话复用和模型变化判断先规范化，避免首次选择或复用检查继续使用复合 ID。

3. **回归测试**
   - 新增 `tests/test_acp_model_normalization.py`，覆盖 Traex helper、普通 `switch_model()`、Deep/engine factory 启动模型规范化。

## WF 总超时可设不限时 + 运行态停止按钮

### 背景 / 动机
用户反馈：动态 Workflow 属复杂长任务，可能跑很久，总超时应可设更长甚至不设；同时希望实时看到状态、由用户自行选择停止时机。原实现 `workflow_total_timeout_s` 默认 3600s 且 `Field(ge=60)` 禁止 0；`RuntimeBridge` 无条件用 `started+total` 作硬 deadline；运行态进度卡只能靠 `/stop_wf` 命令停止，卡片上无按钮。

### 改了什么
1. **0=不限制总时长**
   - `src/config/settings.py`：`workflow_total_timeout_s` 的 `Field` 由 `ge=60` 改为 `ge=0`，说明补充「0 表示不限制总时长，跑到结束或用户停止；per-agent 超时与 MAX_TOTAL_AGENTS 熔断仍生效」。
   - `src/workflow_engine/constants.py`：`WORKFLOW_TOTAL_TIMEOUT_S` 注释补充 0=unlimited 语义。
   - `.env.example`：`WORKFLOW_TOTAL_TIMEOUT_S` 范围说明改为 `>= 0（0=不限制）`。
   - `src/workflow_engine/bridge.py`：
     - `_ensure_workflow_deadline` 识别 `total<=0` → monotonic deadline 置 `None`、`deadline_unix_ms=0`（JS 侧 falsy→Infinity），仍填充 `started_*`；加 `_workflow_started_monotonic` 幂等守卫避免二次解析。
     - `run()` 循环：`deadline is None` 时跳过总超时检查，`_pop_message` block 仍上限 1s 以便及时响应取消/进程死亡。
     - 既有 `_remaining_workflow_budget_s`/`_reject_if_workflow_budget_exhausted`/`_cap_agent_timeout_to_remaining_budget` 在 deadline=None 时已返回 None/不裁剪，无限模式下 per-agent timeout 自然回落到 Settings 默认。JS `runtime.js` 无需改（已处理 deadline=0→Infinity）。

2. **运行态进度卡「停止」按钮**（由并行 subagent 实现）
   - `src/card/actions/dispatch.py`：新增 `WORKFLOW_STOP_RUNNING = "workflow_stop_running"`。
   - `src/feishu/handlers/workflow.py`：新增 `handle_workflow_stop_running`，委托既有 `stop_workflow`（复用发起者/管理员鉴权与状态校验）。
   - `src/feishu/handlers/workflow_script.py`：`_build_workflow_callbacks.on_progress` 在渲染前经 `_inject_workflow_stop_button` 给运行卡追加 danger「⏹️ 停止 Workflow」按钮（带确认弹窗），try/except 保护不影响进度更新。
   - `src/feishu/action_registry.py` + `src/feishu/router.py`：接线新 action → `handle_workflow_stop_running`。
   - `src/feishu/ws_card_action_handler.py`：`SYSTEM_CARD_ACTIONS` 补 `workflow_stop_running`（API contract 测试要求所有 WORKFLOW_* 均注册）。

3. **测试**
   - `tests/test_config_validation.py`：删除旧 `ge=60` floor 断言，改为 `test_total_timeout_zero_allowed_means_unlimited`（0 通过）+ `test_total_timeout_negative_rejected`（-1 拒绝）。
   - `tests/test_workflow_bridge_transport.py`：新增 `test_ensure_deadline_unlimited_when_total_timeout_zero`（deadline=None、unix_ms=0、started_* 已填、剩余预算=None、不拒绝）。
   - `tests/test_workflow_stop_button.py`（新建，subagent）：常量/处理器委托/路由映射断言，6 passed。
