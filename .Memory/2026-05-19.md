### AC-15: get_escalation() 暴露可变引用

- **问题**: `engine.get_escalation()` 直接返回内部 `_escalation` 引用，handler 层持有引用后可意外突变 engine 内部状态。
- **修复**:

### AC-16: _resolve_escalation() 成功路径冗余 fallback 文本

- **问题**: `update_card()` 成功后仍无条件 `send_text_to_chat()`，用户在群内收到两次反馈。
- **修复**: `src/feishu/handlers/slock.py` — fallback 文本发送改为条件 `if not card_updated`。
- **测试**: `tests/test_slock_handler_escalation.py` — `test_admin_click_resolves_successfully` 改为 `send_text_to_chat.assert_not_called()`；新增 `test_resolve_success_no_redundant_text` 显式验证。

### AC-17: slock_team_name_prefix → suffix 字段重命名兼容

- **问题**: 字段重命名后，已部署 `.env` 中 `SLOCK_TEAM_NAME_PREFIX=...` 失效。初始方案在 `model_validator(mode="before")` 中检查 `data` dict，但 Pydantic BaseSettings **不会将未定义字段对应的环境变量传入 `data`**。

- **测试**: `tests/test_slock_settings_compat.py` — 5 tests:

  2. `test_new_suffix_env_takes_priority` — 新 env 优先于旧 env
  3. `test_prefix_emits_deprecation_warning` — 触发 DeprecationWarning
  4. `test_no_prefix_no_warning` — 无旧 env 无警告

## Slock 生产就绪加固：有界并发 + 队列超时 + 任务上限 + 安全错误 + 双语升级

针对 Slock 模式的 6 项生产就绪改进，覆盖并发控制、安全性和国际化。

### 1. 有界并发控制（BoundedExecutor）

- **新增文件**: `src/slock_engine/bounded_executor.py`
  - `QueueFullError` 自定义异常（携带 `max_queue_size` 上下文）
  - `BoundedExecutor` 包装 `ThreadPoolExecutor`，原子检查 `pending_count >= max_queue_size` 后拒绝提交
  - 每个 future 附带 `enqueue_time = time.time()` 用于队列等待超时检测
  - `threading.Lock` 带 leaf lock 注解满足 `test_lock_ordering` 扫描门禁
- **修改**: `src/slock_engine/engine.py` — `_get_executor()` 返回 `BoundedExecutor`，参数从 settings 读取

### 2. 配置字段（settings.py）

新增 4 个 `Field(ge=1)` 校验字段：
- `slock_max_parallel_agents` (default=4)
- `slock_max_queue_size` (default=8)
- `slock_queue_wait_timeout` (default=60s)
- `slock_max_open_tasks` (default=50)

### 3. 开放任务上限

- `engine.add_task()` 返回 `Optional[SlockTask]`，open_count >= limit 时返回 `None` 并 WARNING
- Handler 层 `handle_message()` 捕获 `None` 返回值更新卡片提示"团队繁忙"

### 4. 队列等待超时检测

- Handler `_async_execute()` 在 agent 真正开始执行前检查 `time.time() - future.enqueue_time > slock_queue_wait_timeout`，超时则跳过执行并回滚任务状态

### 5. 安全错误消息

- **新增**: `src/utils/errors.py::_safe_error_message(exc)` — 已知异常类型映射到中文友好文案，未知异常返回"内部错误，请联系管理员"，杜绝内部路径泄露
- Handler 层 `_async_execute()` 和 `assign_task()` 的错误卡片改用 `_safe_error_message(e)` 替代 `get_error_detail(e)`

### 6. 双语升级解决选项

- `src/slock_engine/models.py` 新增 `RETRY_OPTIONS/SKIP_OPTIONS/ABORT_OPTIONS` frozenset 常量，包含中英文选项
- `engine.resume_after_escalation()` 使用 `resolution in RETRY_OPTIONS` 替代 `== "Retry"`
- `card_templates.py` 按钮类型判断使用 `option in ABORT_OPTIONS`

### 7. 权限校验去重

- Handler `_resolve_escalation()` 提取 `_check_slock_permission(engine, message_id, chat_id)` 方法，复用权限校验逻辑

## Slock 架构加固：异步执行统一 + 事件驱动取消 + Rate-Limit 安全层

针对最终 review 反馈的 3 项架构缺陷和 1 项安全问题，完成全部修复和验证。

### 1. execute_parallel 事件驱动取消替代 time.sleep（架构缺陷 D-01）

- **问题**: `execute_parallel()` 超时后 `time.sleep(5.0)` 固定阻塞主线程，无论 future 是否已完成。
- **修复**: `src/slock_engine/engine.py` — 改用 `concurrent.futures.wait(incomplete_futures, timeout=5.0)` 事件驱动等待，提前感知完成。
- **测试**: `tests/test_slock_parallel_cancel.py` — 验证 futures_wait 被调用。

### 2. EscalationManager Retry 路径异步化（架构缺陷 D-02）

- **问题**: 升级重试路径同步调用 `execute_task`，在 handler 的 BoundedExecutor 模型中可能阻塞主线程。
- **修复**: `src/slock_engine/escalation_manager.py` — Retry 分支先尝试 `get_executor_fn()` 异步提交，失败时 fallback 同步执行。
- **测试**: `tests/test_slock_escalation_resolve.py::TestResumeRetryAsync` — 4 tests（异步提交、返回 None、executor 错误 fallback、无 executor 同步）。

### 3. Handler 重复异步执行代码提取（架构缺陷 D-03）

- **问题**: `handle_message()` 和 `_submit_task_execution()` 各含 ~90 行几乎相同的异步执行闭包，DRY 违规。
- **修复**: `src/feishu/handlers/slock.py` — 新增 `_execute_async()` 公共方法（~50 行），两个调用点重写为 lambda 回调 + 调用 `_execute_async()`。
- **测试**: `tests/test_slock_handler_async.py` — 7 tests（helper 正常/异常/空结果/队列满/无 engine、两个调用点委托验证）。

### 4. 非管理员 Rate-Limit 安全层（安全 S-01）

- **问题**: 任何群成员可不受限制地 `/task assign`，造成 DoS 风险。
- **修复**:
  - `src/config/settings.py` — 新增 `slock_assign_rate_limit: int = Field(default=5, ge=1)` 配置字段。
  - `src/feishu/handlers/slock.py` — 新增 `_check_assign_rate_limit()` 方法：滑动窗口 60s 内最多 N 次提交，admin/owner 豁免。
