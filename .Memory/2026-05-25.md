## Slock passive 路由不再抢项目群编程

### 背景

项目群自由文本在普通编程模式下被 Slock passive routing 抢走，表现为项目群无法继续正常 `/coco`、`/codex` 等编程会话，消息进入 Slock 团队群处理。

### 根因

`MessageDispatcher.process_with_intent()` 的 Slock 分支排在普通编程态转发之前：

- managed chat 的自由文本会先调用 `_handle_slock_message()`。
- unmanaged group 的任务型文本会先触发 `_auto_activate_slock()`。
- 这两个判断没有排除 `ModeManager` 已确认的普通编程态，也没有排除 Deep/Spec/WT 话题态或已有项目上下文。

### 修复

- 在 `src/feishu/dispatcher.py` 增加 `slock_context_allowed` 门禁：Slock passive/legacy 自由文本消费只允许发生在非普通编程态、非 Deep/Spec/WT 话题态。
- 增加 `slock_auto_activate_allowed`：passive auto-activate 只允许在无 project 上下文的普通群里触发，避免已有项目上下文的 SMART/编程意图被 Slock 抢占。
- 在 `src/feishu/ws_client.py` 增加 `_is_topic_engine_context()`，用当前 Feishu thread context 判断是否属于 `worktree/deep/spec` 话题策略。
- 保留显式 `/slock` 命令入口，不影响用户主动进入 Slock。

### 回归测试

- `tests/test_slock_dispatcher_routing.py`
  - managed Slock chat + 普通编程态：自由文本转发到当前编程 handler，不进入 Slock。
  - unmanaged task-like 文本 + 普通编程态：不触发 Slock auto-activate。
  - task-like 文本 + project 上下文：走 SMART intent recognition，不触发 Slock auto-activate。
- `tests/test_ws_client_routing.py`
  - WT 话题 + managed Slock chat：仍进入 WT，不被 Slock 抢走。
  - 项目群 + Coco 编程态 + managed Slock chat：仍进入 Coco handler。

### 验证

- `uv run python -m pytest tests/test_slock_dispatcher_routing.py tests/test_ws_client_routing.py tests/test_feishu_ws_client_routing.py -q` → `80 passed`
- `uv run ruff check src/feishu/dispatcher.py src/feishu/ws_client.py tests/test_slock_dispatcher_routing.py tests/test_ws_client_routing.py` → passed
- `uv run python -m src.main --validate` → passed
- `git diff --check` → passed

### 注意

尝试运行宽口径 `uv run python -m pytest tests/test_slock*.py -q`，但当前工作树中的 Slock 专项存在大量与本次路由改动无关的既有失败，代表性错误包括 `MemoryManager` 缺少 `shutdown()`，并伴随记忆、安全、信任、任务链等专项失败；本轮不把该宽口径结果作为路由修复成败依据。

## Coco 模型选择不再恢复旧会话快照

### 背景

线上反馈最近 `/coco` 无法使用。日志显示 `/coco` 入口和模型选择都正常：

- 23:25:59 进入 `tool=coco` 模型选择。
- 23:26:09 ACP session started。
- 随后又 `Session loaded: 521fae82`，即加载了项目旧 `coco_session_snapshot`。
- 23:29:12 `session.send_prompt()` 收到 ACP server `Internal error`，用户侧表现为 Coco 执行失败。

本机持久化 `/data00/home/jiataorui/.ghostap/projects.json` 中 `iloveu.coco_session_snapshot.session_id=521fae82...`，最后请求是旧的 `pwd`，且 `coco_mode=false`。因此问题不是 Slock 抢路由，也不是 `/coco` 未进编程 handler，而是显式模型选择后仍自动 resume 陈旧 Coco 会话。

### 根因

`ProgrammingModeHandler.enter_mode()` 在启动前只要项目存在 resumable snapshot 且不在 thread，就无条件把 `snapshot.session_id` 传给 `ensure_session()`。近期 `/coco` 统一走模型选择卡后，用户以为选择模型是在开启新会话，但 handler 仍加载旧 ACP session；`session/load` 可以成功，后续 prompt 再由 Coco server 返回 `Internal error`。

另一个二阶问题是：即使阻止了 `ensure_session(session_id=旧值)`，后续恢复分支仍会根据本地 `snapshot` 把项目状态写回旧 session_id。

### 修复

- `SystemHandler.handle_select_acp_model()` 在模型卡确认后清理当前工具的旧 `*_session_snapshot`，使“选模型进入”语义固定为新 ACP session。
- `ProgrammingModeHandler.enter_mode()` 在存在显式 `model_name` 时不使用旧 snapshot，也不会进入恢复分支写回旧 session_id。
- 显式恢复按钮不受影响：`handle_card_resume()` 仍直接使用用户选择的 `session_id` 走 resume 流程。

### 回归测试

- `tests/test_handlers.py::TestProgrammingModeEnterExit::test_enter_mode_with_selected_model_does_not_resume_stale_snapshot`
  - 红灯复现：显式模型进入仍传 `session_id=stale-session`，并把项目状态写回旧 session。
  - 修复后：`ensure_session(session_id=None, model_name="fresh-model")`，项目状态写入新 `fresh-session`。
- `tests/test_model_command.py::TestHandleSelectAcpModelPendingPrompt::test_default_model_selection_clears_stale_tool_snapshot`
  - 覆盖默认模型 `model_name=None` 分支，确认模型卡选择同样清理旧 Coco snapshot。
