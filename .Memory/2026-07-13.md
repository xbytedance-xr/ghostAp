## Durable Ingress Task 4：员工附件安全暂存

### 更改与原因

- ACK 路径仍只把 exact typed resource descriptor 写入员工加密 Ingress Blob，新增
  URL、绝对/本地/遍历路径拒绝，并把 `resource_type` 与 `img_`/`file_` key 空间
  绑定；不在三秒回调内下载。官方资源 adapter 只把已验证的 message ID、resource
  key、type 交给 `GetMessageResourceRequest`。
- 新增 `AttachmentStagingService`。只有 Router 后续构造的
  `AuthorizedAttachmentStagingRequest` 才能进入；它精确解析目标员工
  credential/agent/app，构建员工 `lark-oapi` client，绝无 Manager Bot fallback。
  credential、credential ref、resource ID 和用户文件名均不进入 Journal。
- 下载前先锚定服务端随机 temp/final 路径和 tenant/employee/envelope owner；下载后
  校验 count、单文件/总大小、deadline、实际大小、sha256、MIME magic 与 executable
  magic/后缀，再以 0700 目录、0600 文件、dir-fd + `O_NOFOLLOW`、file+parent fsync
  原子发布。用户文件名仅作验证元数据，从不参与路径。
- staging/cleanup 的 started/completed/failed 全部是可重放 Journal 状态。崩溃后
  started staging 转 terminal failed 并清理；cleanup started 继续删除。父/叶被
  symlink 或非可信对象替换时保持 cleanup started，绝不伪写 completed。只有
  completed、未清理且重开后仍是 0600 regular/hash 匹配的服务端路径可交给 Gateway。
- 最终 blocker review 把 lifecycle reducer 收紧为显式状态机：parent 只能在 live
  staging/cleanup-none 绑定，completed 要求 parent + 全部 leaf identity，failed 必须早于
  cleanup，aggregate cleanup completed 要求全部 leaf 已 durable disposed。每次 commit 在
  `transaction_guard` 内同步后先对复制投影 preflight，双实例 stale parent/completion 事件
  会在 Journal/anchor 增长前拒绝，不能先污染持久日志再由 replay 报错。
- 每个 leaf 先创建并 fsync 0600 空 temp、锚定 exact dev/inode，之后才向同一打开 fd
  写入用户字节并原子 rename，所以 temp/final crash window 共享同一 durable identity。
  cleanup 对 exact fd 先锚定逐叶 intent，再 `ftruncate(0)+fsync`，锚定逐叶 completed 后
  保留零字节墓碑直到 aggregate cleanup completed；外移、replacement、unexpected name、
  temp+final 双名均 fail-close。
  未锚定崩溃窗口最多残留 same-owner/0600/nlink1/size0 planned temp，其他形态不自动删除。
- 第三轮复审补齐 hardlink capability 边界：identity anchor 后写入前、rename 后完成前、
  trusted export 读前/后均同时要求 `st_nlink == 1` 和 bound parent 扫描只存在 expected
  temp/final 单名。发现 planned-temp/unexpected alias 时，失败清理仍通过 exact fd 把共享
  inode `truncate+fsync` 为零，但不锚定 leaf/aggregate completed。unbound leaf cleanup 也
  改为读取 durable 当前态续跑：none 才写 started、started 才写 completed、completed 只移除
  重新验证的零字节 planned temp，避免 started 崩溃后重复事件永久卡死；这个当时的删除步骤
  已由第六轮改为永久保留零字节墓碑。
- 第四轮复审将 cleanup target 单独持久化为 exact dev/inode 或显式 absent sentinel；
  unbound completed 前后都通过 no-follow fd/fstat 匹配 target，same-UID 非空或同属性空
  replacement 均不能替换原 temp，absent target 后出现任何 planned leaf 也拒绝。当时的
  completed leaf pathname unlink 在第六轮因无法消除 verify→unlink TOCTOU 而被移除。
- `after_publish` 后、`staging_completed` 前新增所有 leaf 的 current-root-path 全量验证：parent/
  leaf identity、0600 regular、UID、nlink1、单名和 hash 任一异常都会先写 failed，绝不暴露
  completed terminal。trusted read 读后重复完整 predicate，并从 root fd 重开当前 pathname；
  读前/后/current-reopen 的 dev/inode/mode/uid/nlink/size/mtime/ctime generation 必须一致，
  因此读中 chmod、普通目录替换或同 inode 同长度改写都 fail-close。
- 第五轮复审把 cleanup target 扩为 exact dev/inode + `temporary|final` 路径种类，并禁止在
  aggregate cleanup completed 前 unlink。所有 completed leaf 会在聚合提交前从当前 root
  重开、逐一验证 exact path、0600 regular、UID、size0、nlink1 与 parent 单名；期望名缺失、
  外部 alias 或目录替换都保持 cleanup started。
- 第六轮证明聚合完成后的 `verify; unlink(pathname)` 仍有不可关闭的 TOCTOU：Linux/Python
  没有按 dev/inode 条件删除目录项的原语，再加一次同类 lstat 仍可在检查与 syscall 间被换名。
  因此 completed cleanup 现在只 best-effort 观察 tombstone，绝不再修改 pathname。零字节、
  0600、单链接目录项可以永久保留；cleanup completed 的合约只表示服务通过 exact open fd
  `truncate(0)+fsync` 证明用户字节已擦除，不承诺目录项删除。completed recovery 的观察失败
  不写新 lifecycle failure，也不追踪或删除被移走的原 tombstone。

## 员工 Ingress Journal + 独立加密 Blob 持久化

### 更改与原因

- 新增 `src/autonomous/ingress/projection.py` 与 `service.py`，员工入口不复用
  legacy `manager.DurableInbox`。`employee.ingress.accepted` 只写严格安全元数据、
  authenticated Blob reference/hash；规范化 payload 仅进入 Ingress 独占的
  AES-GCM BlobStore，并复用员工 Data keyring 的 key resolver，而不共享 Data
  BlobStore/live-set。
- acceptance 在单一 service mutex + `JournalWriter.transaction_guard()` 内先同步
  投影和查重，再 publish/readback Blob、fsync Journal、完成 anchor、应用 frame，
  最后才返回 ACK。随机 nonce Blob 只在确认非 duplicate 后创建；并发 32 次投递
  只产生一个 Blob、一个 Journal acceptance，其余 ACK 复用稳定 acceptance 并绑定
  当前 generation/connection/request。
- 同 dedup key 的 semantic digest 或 sender/chat/action provenance 漂移会 durable
  fail-close；fallback action 必须额外提供与 metadata 一致的 server correlation，
  不从用户 action JSON 推导。Blob AAD labels 绑定 schema、tenant、employee、
  envelope、dedup key 与 semantic digest。
- 启动重放在恢复 admission 前验证所有 nonterminal authenticated Blob；缺失、损坏
  或 duplicate readback 失败会关闭对应员工 ingress。Blob publish 后的 readback、
  Journal fsync 或 anchor 失败都会隔离新 Blob 且不应用 acceptance。terminal
  disposition 先写 durable tombstone 再进入 GC/quarantine，历史 acceptance 保留。
- `EI-IPC-01` selector 已由 pending 改为 collectable。真实 `multiprocessing spawn +
  Pipe + FileAnchor + Journal fsync` harness 证明子进程只在 anchor sequence/hash 与
  acceptance 一致后发 ACK；本次精确结果为 `0.014952s <= 1.5s`。

### TDD、审查与验证

- 初始 RED：两个指定测试文件对缺失 projection/service 的存在性断言均失败
  (`2 failed`)；行为测试随后因缺少 `IngressProjectionState` 在收集阶段失败。
- 首轮 GREEN 后自审新增 Blob readback mismatch 回归，先复现“已发布 Blob 未隔离”
  的失败，再扩展统一 quarantine 路径。两轮最终自审无剩余实质建议，并锁定 service
  close 幂等且只关闭自己的 BlobStore。
- 新增/Ingress contract `84 passed`；连同 JournalWriter、BlobStore、Data projection/
  commit/composition 相邻回归共 `185 passed in 4.43s`。scoped Ruff 与
  `git diff --check` 通过。
- Task 2 review 追加真实 restart RED，复现 publish→commit 前崩溃 Blob 遗留、
  tombstone anchor→物理隔离前崩溃无法重试，以及非法 disposition 已进入 Journal
  才被 reducer 拒绝。修复后唯一 Ingress owner 在每次 rebuild/startup 严格扫描
  live-set 并隔离所有无引用 Blob，已 tombstone 但物理 Blob 仍存在时 GC 可重试；
  disposition 在 commit 前先构造完整模型验证。review 回归 `4 passed`，Task 2 与
  相邻 Journal/Data 最终 `189 passed in 4.39s`。
