## `/hire` 生产链路实现与安全门禁（保持关闭）

### 更改内容与原因

- 新增 `ProductionEmployeeHireService` 和冻结的 Hire 投影：以权威 tenant + message 生成稳定 intent，所有注册、Vault、Bot 绑定、Slash、Channel 和激活状态均写入 Journal；外部 Effect 在派发前完成 PREPARED/EXECUTING fsync+anchor，重启从 Journal 重放，未知远端结果进入 `ACTION_REQUIRED`，不盲目重建应用。
- 新增 Journal/Vault durable Saga 与 callback bridge：Vault receipt 可幂等收养，员工与 Bot principal 原子绑定；首次调度失败允许同一 intent 安全重投，关闭时停止 admission 并等待后台任务收敛。
- 新增每员工 fresh-interpreter Channel 子进程：secret 只经一次性 pipe，禁止出现在 argv/env/普通 IPC，协议为严格限长版本化 NDJSON；默认使用 bwrap 最小只读文件系统与 user/mount/PID namespace，attestation 在 secret 下发前验证项目凭证路径不可见，PID/generation fencing、发送身份回执、崩溃恢复和重新验证均由父进程监管。
- 新增官方 `lark-oapi Client.arequest(BaseRequest)` Slash adapter 与 GET→diff→POST/PATCH/DELETE→GET 精确 reconciliation；最终 observed set/hash 不一致则拒绝激活。
- 新增激活验证路由：一次性 nonce/TTL 与 tenant、agent、app、generation 绑定；只有真实员工 Bot DM `/status`、Slash/Channel 真实时间证据、员工回执中的 app/message/generation/connection 全匹配、且独立主 Bot 审计返回零发送，才可消费 nonce 并提交 `ACTIVE`。缺审计 provider 时 composition fail-close；员工 Channel 崩溃后 durable generation +1 并强制重新验证。
- 新增单一生产 composition 并接入 `FeishuWSClient`：权威 `tenant_key` 通过 TaskSpec/ContextVar 传递，runtime 启动恢复和关闭纳入 WS 生命周期；limit=0 的全新环境保持纯 dormant，已有 Journal 可离线重放但禁止新 admission 或无签名外部恢复。
- 新增 Employee 专用 17 项租户验收清单、追加式 fsync 哈希链、外部 checkpoint 及 Ed25519 独立 QA 签名；签名绑定 environment/release/commit/service/staging/prod tenant，最长 24 小时。该 bundle 仅作验收候选证据；因尚无不可变构建/工作负载证明、固定 QA trust root 与外部单调消费账本，runtime 对 config-only release 恒 fail-close，默认 CLI 只返回 `PENDING`。

## `/hire` 管理员 DM 卡片权限与 readiness 反馈修复

### 根因与实现

- 日志证明 `/hire` 首次消息已经通过管理员与 DM 检查，`global_hire` 也完整保留到最终模型确认；真正失败点是卡片回调从官方模型不存在的 `P2CardActionTrigger.context.chat_type` 推导 `is_p2p=False`。官方事件文档的权威结构字段是 `im.message.receive_v1.event.message.chat_type`；`GET /im/v1/chats/{chat_id}` 的结构字段是 `chat_mode`，而该 API 的 `chat_type` 仅表示 `private/public` 可见性。
- 消息入口现在把 origin message、chat、operator、tenant 与事件 `chat_type` 保存到 `MessageLinker`。卡片回调优先验证完整 origin/chat/operator/type 四元组；查询异常、残缺记录、跨 chat/operator/origin、记账覆盖与并发冲突均保持 fail-close。
- 只有服务端明确没有来源记录（典型为重启后的旧卡片）时才查询 Chat API `chat_mode`。回退结果必须通过 `register_trusted_origin_if_absent()` 原子 compare-and-set 写回完整来源；空 operator、已有残缺/冲突记录、写入 no-op 或失败均不得授予 DM。成功回填可支撑 `/hire` 后续多步卡片且不会重复查询 API。
- 卡片路径的 request-id 记账不再携带 callback chat，避免拒绝跨 chat 回调后反向覆盖可信来源。
- `HandlerContext` 新增只读 readiness provider。`/hire` 只有在 provider 明确返回 `ready=True` 且 blockers 为空时才派发；缺失、异常、畸形或未就绪均默认拒绝，并向管理员展示 visible limit、QA evidence、anchor、sandbox、recovery 等具体门禁。
