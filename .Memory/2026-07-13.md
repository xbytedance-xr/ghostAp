## Agent Department completion audit 与 `/hire` 安全收口

### 审计结论

- 三个并行只读审计分别检查最终目标、`/hire` 生产调用链和 release gate，结论一致：整体目标仍 active。现有数据面、Context、Provisioning、Channel、Slash、Router、Response 和 Fire 多数是独立模块或内存脚手架，生产 composition 与真实租户证据缺失。
- 现场 `/hire` 没有调用一键创建 SDK；它仍路由到 `SlockHandler.create_role()`。首张工具卡丢失 `global_hire`，最终模型回调也未传回该标记；非 Slock 群被错误 gate 拦截，Slock 群则创建旧 `AgentIdentity` 并绿色假成功。带完整参数的 global 路径还会访问 `engine=None`。
- `AgentDepartmentBootstrap` 在 limit=0 或非 0 时都常量设置全部 readiness=True；Channel/Slash/Hire 测试全部使用 fake。acceptance manifest 77 项仍 pending，不能用本地测试代替真实 Bot、双租户、50 Bot soak 和故障注入。

### 更改与原因

- 将 `lark-oapi` 从 `1.6.5` 升级并锁定到官方 `1.7.1`；新增签名 contract，强制 `register_app/aregister_app` 支持 `app_preset`、`addons`、`create_only`、`app_id`。
- 新增 `LarkAppRegistrar`：通过官方 `aregister_app` 使用 minimal preset、版本化最小 scopes/events/callback 与 `create_only=True`；同步转发十分钟链接和状态，严格校验 credential result，错误与 repr 不暴露 secret。
- 新增 `EmployeeHireRequest/EmployeeHireService` 类型边界并注入 `HandlerContext`。`/hire` 的工具/模型/effort 标记全程保留，只允许配置管理员在主 Bot 私聊发起；没有 production service 时明确拒绝，绝不回退旧 global registry。`/new-role` 的 Slock 虚拟角色行为保持不变。
- `AgentDepartmentBootstrap` 新增 dormant 状态与六个具名 readiness probe；缺 probe、probe 异常或任一 unavailable 均 fail-close，`healthy` 同时检查 data/context/provisioning/channel/router/response。
- 更新 `docs/goals.md`，移除“数据 Task 1 未实现”等过时状态，明确局部模块证据与 production completion 的区别。
