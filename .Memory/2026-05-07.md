## 重构：统一消费 `CommandMatch`（Slash 命令 SSOT）
### 背景
- 项目已引入统一解析器（`SlashCommandParser`/`CommandMatch`），但消费链路仍存在多处“手工解析/切片/兜底字段”并行：
  - `ChatLockManager.should_block()` 仍接受 `raw_text/has_args` 并在无结构化输入时推断参数。
  - `SystemHandler` 对 `/switch`、`/new`、`/close` 仍存在 `startswith()+切片` 的参数提取逻辑。
- 这会导致 SSOT 失效：同一条指令在不同模块的“是否有参数/参数内容”判定可能漂移，进而影响锁逻辑与路由一致性。

### 执行内容
- 解析变为 request-scoped：在 `FeishuWSClient` 清洗文本后只解析一次，得到 `command_match` 并沿链路透传。
- `ChatLockGate.check()` 仅接收 `command_match`（或 `None`），不再传 `command/raw_text`；阻断与 fail-close 路径统一基于同一份结构化信息。
- `ChatLockManager.should_block()` 改为 `command_match` 入参（`Protocol` 解耦 core 与 feishu 层），删除 `raw_text/has_args` 兜底推断逻辑：
  - 锁定场景下 “裸 `/worktree`（无参数）放行 / 带 goal 阻断” 由 `CommandMatch.has_args` 驱动，避免 `strip()` 推断漂移。
- `SystemHandler` 的系统命令参数消费统一改用 `CommandMatch.args`：
  - `/switch`、`/new`、`/close` 走专用 `_handle_*_args()`，不再通过文本切片提取参数。
- `WorktreeHandler` 增加 `handle_worktree_command_match()`：直接消费 `CommandMatch.args` 作为 goal，旧入口保留为薄兼容层。
- 测试同步迁移：`chat_lock`/`chat_lock_gate`/`ws_client_routing`/`worktree_command_routing` 全部以 `SlashCommandParser.parse()` 产出的 `CommandMatch` 作为输入构造，避免再写重复解析规则。

### 回归风险清单（重点关注）
- `command_match=None` 的非 slash 消息：链路需保持“正常不拦截、不误判 has_args”。
- 文本清洗顺序：必须在清洗后解析，避免 `\t/\n` 等空白字符导致 args 提取不一致。
- 锁逻辑一致性：拦截卡提示与实际阻断判定需基于同一份 `CommandMatch`，避免“提示放行但实际阻断”的错配。
- handler 兼容：卡片 action 或旧调用点仍能走到兼容层（若存在），且不会再次触发二次解析。

### 验证
- 全量测试：`5645 passed, 1 skipped, 122 warnings`（约 195s）。
