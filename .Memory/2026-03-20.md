## Spec 停机并发导致模型失败与崩溃修复

### 任务描述
2026-03-20 06:27:39
排查 `logs.log` 中 Spec 模式连续模型切换与 `'NoneType' object has no attribute 'cycles'` 异常，定位服务停机阶段的并发清理问题并完成修复。

### 执行内容
1. 日志排查与时间线确认：
   - `logs.log:344-345` 出现 `Spec执行异常: 'NoneType' object has no attribute 'cycles'`。
   - `logs.log:323-344` 显示在停机后仍持续触发模型切换（`gpt-5 -> gpt-5.1 -> gpt-5.2 -> gemini-3-pro`）。
   - `logs.log:324` 附近出现 `服务已停止`，说明这不是普通在线请求失败，而是停机流程与运行中 Spec 线程并发冲突。
2. 根因定位：
   - 停机时 `ws_client.close()` 先 `engine.stop()` 后立即 `cleanup_all()`，可能与运行中的 Spec 循环并发。
   - `SpecEngine.cleanup()` 原先会直接清空 `self._project`，运行线程后续访问 `self._project.cycles` 时触发 `NoneType` 异常。
   - `_run_phase()` 在停止态异常时仍可能进入 `_try_switch_model()`，导致停机阶段继续“模型失败切换”噪音。
3. 代码修复：
   - `src/spec_engine/engine.py`
     - `_run_phase()`：在 `EngineRunState.STOPPING` 下，phase 异常直接按中断返回，不再切模型/保存失败任务。
     - `_try_switch_model()`：增加运行态门禁，仅 `RUNNING` 时允许切换模型。
     - `cleanup()`：只要引擎未回到 `IDLE`，仅请求 `STOPPING + cancel`，不立即清空 `self._project`。
   - `src/feishu/ws_client.py`
     - `close()` 增加 `_wait_engines_stopped()` 短等待（grace period），在 `stop()` 后先等待 Deep/Loop/Spec 引擎退出运行循环，再执行 `cleanup_all()`。
4. 测试补充：
   - `tests/test_spec_engine.py`
     - 新增活跃态 cleanup 回归：`test_cleanup_while_active_only_requests_stop`
     - 新增停止态 phase 回归：`test_run_phase_when_stopping_skips_model_switch_and_failed_task`
     - 新增停止态模型切换门禁回归：`test_try_switch_model_returns_false_when_not_running`
   - `tests/test_ws_client_patch.py`
     - 新增停机等待流程回归：`test_close_waits_engine_shutdown_before_cleanup`
5. 验证结果：
   - 定向与相关回归通过：
     - `uv run python -m pytest tests/test_spec_engine.py -q` → `163 passed`

### 技术要点
- 必须区分“真实模型失败”与“停机取消异常”：停机路径应抑制模型切换与失败任务落盘，避免误判为模型问题。
- 引擎 `cleanup` 在活跃态不能破坏运行中状态对象；应先协作式停止，再由运行线程自然收敛。
- 停机入口层应提供短暂停稳窗口，减少 `stop` 与 `cleanup_all` 的并发竞争面。
