## WF 生成卡片卡住修复

### 背景 / 症状
- 用户在 `/wf` 三步流程中选完主编排/评审模型后，飞书卡片停在 `Workflow — 生成脚本中...`。
- 日志显示脚本生成实际已经返回并触发校验失败处理：
  - `Generated script failed validation`
  - 随后 `update_card` 被飞书拒绝：
    `unknown property, property: template, path: ROOT -> body -> elements -> [12](tag: collapsible_panel) -> header`

### 根因
1. `src/workflow_engine/renderer.py::_collapsible_panel()` 和
   `src/feishu/handlers/workflow.py::_build_confirm_card()` 在 `collapsible_panel.header`
   下写入 `template`。飞书 Schema 2.0 不支持该字段，导致确认/错误卡片 patch 失败，旧 loading 卡保留。
2. `_generate_and_show_confirm_card()` 对 `update_card(gen_msg_id, confirm_card)` 的返回值不做处理；
   即使后续再出现卡片结构问题或飞书临时 patch 失败，用户仍会看到旧的“生成脚本中”卡。
3. 模板校验失败分支在发送 loading 卡后调用旧 `reply_error()` 并直接返回，没有替换或关闭 loading 卡；
   同时 pending 状态未清理，用户可能继续被“已有 Workflow 等待操作”挡住。

### 改动
- `src/workflow_engine/renderer.py`：
  - `_collapsible_panel()` 统一移除/忽略 `header.template`。
  - 状态色改放到合法的 `collapsible_panel.border.color`。
- `src/feishu/handlers/workflow.py`：
  - 确认卡、脚本预览、子 Workflow 预览、更多工具、详细信息面板移除 `header.template`，改用 `border.color`。
  - 新增 `_replace_or_send_workflow_card()`：更新旧卡失败时发送新卡，避免 loading 卡永久卡住。
  - 模板校验失败改为更新 loading 卡为结构化错误卡，并把 Workflow 状态重置为 `IDLE`、清空 pending。
- 测试：
  - 新增进度卡/确认卡递归断言：禁止 `collapsible_panel.header.template`。
  - 更新旧 Workflow progress 契约：颜色断言从 `header.template` 改为 `border.color`。
  - 新增 loading 闭环回归：`update_card(False)` 新发确认卡；模板校验失败替换 loading 卡并清 pending。

## 全仓 lint 债清理与测试稳定性

### 背景 / 症状
- `/wf` 修复后提交前执行 `uv run ruff check .`，发现全仓有大量既有 lint 债。
- `ruff --fix`/`--unsafe-fixes` 自动清理后，仍剩少量导入位置、类型注解、重复定义和模糊变量名问题。
- 全量 pytest 首轮暴露 `tests/test_slock_engine.py::TestSlockEngineParallelExecution::test_execute_parallel_runs_multiple_tasks`
  偶发失败：`activate_channel()` 启动的后台 dispatch/idle/patrol worker 会抢先消费测试里的 `_run_acp_session`
  mock `side_effect`，导致手动 `execute_parallel()` 第二个任务拿到 `None`。

### 改动
- 对全仓执行 ruff 自动修复，清理未使用导入/变量、import 排序和空白行等 lint 债。
- 手动修复剩余 lint：
  - `src/card/render/atoms.py`、`src/workflow_engine/renderer.py` import 位置。
  - `src/feishu/handlers/workflow.py`、`src/feishu/renderers/base.py` 类型检查导入。
  - `src/slock_engine/card_templates/role.py` 模糊变量名。
  - `src/slock_engine/card_templates_legacy.py` 历史重复定义加局部 `noqa`。
  - `tests/test_workflow_parallel_perf.py` 补回 `ThreadPoolExecutor` 导入。
- `tests/test_slock_engine.py` 的手动并发/dispatch 测试在 `activate_channel()` 后停止后台 worker，
  保留 channel 初始化副作用，但避免后台线程与测试主体争用同一个 mock。

- `uv run python -m pytest tests/test_slock_engine.py -q` → 128 passed。
- `uv run python -m src.main --validate` → 配置校验通过。
- `uv run python -m pytest tests/ -q` → 9411 passed, 8 skipped, 45 subtests passed。
- `git diff --check` → 通过。

### 后续风险
- 全量测试仍报告既有 warning：部分测试 mock 导致 coroutine 未 await、后台线程 timeout 参数为
  `MagicMock`。这些 warning 本轮未扩大处理，因为不影响 `/wf` 卡片修复与 lint 清理目标。
