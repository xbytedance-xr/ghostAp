# 2026-07-03

## WF 兜底脚本串行超时修复

- 背景：现场 `/wf` 自动执行后卡片显示 `route-classify traex 1m33s — TimeoutError: ACP prompt 执行超时 (90s)`，随后 `execute-primary traex 3m2s — TimeoutError: ACP prompt 执行超时 (180s)`，总耗时约 270s 且 token 为 0。
- 根因：
  - `.ghostap/workflow_scripts/generated_workflow.js` 是 `generated-dynamic-workflow`，说明本次走的是 `generate_simple_script()` 静态兜底模板，而不是模型生成出的专用编排脚本。
  - 旧兜底模板先用 `classify()` 调 `primaryTool` 做路由，超时 90s；失败后 runtime 默认落到 simple 分支，再用同一个 `primaryTool` 做 `execute-primary`，超时 180s。
  - 用户选择了多个工具时，旧 simple 分支仍只串行等待第一个工具，例如 `traex`；`fallbackTool` 只在 synthesis/verification 使用，不能救执行阶段超时。
- 修复：
  - `generate_simple_script()` 的 Routing 阶段改为本地启发式 `chooseRoute()`，不再产生 `route-classify` ACP 调用。
  - simple 执行分支改为对去重后的前 3 个 selected tools 使用 `race()` 并发执行，先返回有效结果即取消其他请求，避免单个慢后端串行拖满超时。
  - parallel split 阶段也改为 `race()` 多工具尝试，失败则降级单任务执行。
  - 执行提示显式保留用户约束：如果用户要求“只分析/不要改代码”，必须只输出分析并停止，避免兜底模板把分析请求误解释为实现任务。
- 测试：
  - 新增回归断言：多工具兜底脚本必须包含 `race()`/`candidateTools`，不得包含 `await classify(`、`label: "route"` 或 `route-classify`。
  - 新增回归断言：兜底执行 prompt 必须包含 analysis-only / do-not-change-code 防护语义。
- 验证：
  - `uv run python -m pytest tests/test_workflow_subagent.py -q` → 37 passed, 6 skipped。
  - `uv run python -m pytest tests/test_workflow_subagent_ac5.py -q` → 7 passed。
  - `uv run python -m pytest tests/test_workflow_token_efficiency.py -q` → 18 passed。
  - `uv run python -m pytest tests/test_workflow_confirm.py::TestValidateGeneratedScriptRegression -q` → 7 passed。
  - `uv run python -m pytest tests/test_workflow_race_cancel.py -q` → 10 passed。
  - `uv run python -m pytest tests/test_workflow_runtime_primitives.py -q` → 20 passed。
- 后续风险：
  - 如果用户只选择一个本身不可用或极慢的工具，执行阶段仍会等待该工具自己的 bounded timeout；本次修复消除的是“多工具已选择但兜底脚本仍串行卡在第一个工具”的问题。
