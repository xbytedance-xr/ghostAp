## WF 兜底脚本串行超时修复

- 背景：现场 `/wf` 自动执行后卡片显示 `route-classify traex 1m33s — TimeoutError: ACP prompt 执行超时 (90s)`，随后 `execute-primary traex 3m2s — TimeoutError: ACP prompt 执行超时 (180s)`，总耗时约 270s 且 token 为 0。
- 根因：
  - `.ghostap/workflow_scripts/generated_workflow.js` 是 `generated-dynamic-workflow`，说明本次走的是 `generate_simple_script()` 静态兜底模板，而不是模型生成出的专用编排脚本。
  - 旧兜底模板先用 `classify()` 调 `primaryTool` 做路由，超时 90s；失败后 runtime 默认落到 simple 分支，再用同一个 `primaryTool` 做 `execute-primary`，超时 180s。
  - 用户选择了多个工具时，旧 simple 分支仍只串行等待第一个工具，例如 `traex`；`fallbackTool` 只在 synthesis/verification 使用，不能救执行阶段超时。
- 修复：
  - `generate_simple_script()` 的 Routing 阶段改为本地启发式 `chooseRoute()`，不再产生 `route-classify` ACP 调用。
  - simple 执行分支改为对去重后的前 3 个 selected tools 使用 `race()` 并发执行，先返回有效结果即取消其他请求，避免单个慢后端串行拖满超时。
  - parallel split 阶段也改为 `race()` 多工具尝试，失败则降级单任务执行。
  - 执行提示显式保留用户约束：如果用户要求“只分析/不要改代码”，必须只输出分析并停止，避免兜底模板把分析请求误解释为实现任务。
- 测试：
  - 新增回归断言：多工具兜底脚本必须包含 `race()`/`candidateTools`，不得包含 `await classify(`、`label: "route"` 或 `route-classify`。
  - 新增回归断言：兜底执行 prompt 必须包含 analysis-only / do-not-change-code 防护语义。
- 验证：
  - `uv run python -m pytest tests/test_workflow_subagent.py -q` → 37 passed, 6 skipped。
  - `uv run python -m pytest tests/test_workflow_subagent_ac5.py -q` → 7 passed。
  - `uv run python -m pytest tests/test_workflow_token_efficiency.py -q` → 18 passed。
  - `uv run python -m pytest tests/test_workflow_confirm.py::TestValidateGeneratedScriptRegression -q` → 7 passed。
  - `uv run python -m pytest tests/test_workflow_race_cancel.py -q` → 10 passed。
  - `uv run python -m pytest tests/test_workflow_runtime_primitives.py -q` → 20 passed。
- 后续风险：
  - 如果用户只选择一个本身不可用或极慢的工具，执行阶段仍会等待该工具自己的 bounded timeout；本次修复消除的是“多工具已选择但兜底脚本仍串行卡在第一个工具”的问题。

## 消息/模型选择巨慢修复（async bridge 单环路瓶颈）

- 背景：`ce57198`~`de22d4f` 一轮架构重构后，每条消息处理都变慢，模型选择尤其慢（logs.log 反复出现 `ACP claude/aiden 模型探测超时 (15.0s)`）。
- 根因（3 个并行 subagent 调查 + git diff 定位）：
  - `6ba03b7` 把 `run_async()` 从"每次 `asyncio.run()` 在调用线程自建事件循环"改成"全进程共享单个 daemon 线程事件循环 `_BRIDGE_LOOP`"。
  - 结果：ACP 模型探测（`src/acp/helper.py:205`）、Coco 模型探测（`src/coco_model/manager.py:179`）、dispatcher 热路径的 slock 自主消解（`src/slock_engine/gateway.py:96`，`b5683a3` 加了 15s timeout）、slock NLI/engine 全部挤到同一个 loop。每个调用方 `future.result()` 阻塞自己的 worker 线程，且互相 head-of-line 阻塞——本来并行的独立操作被串行化。
  - 复现证据：5 个并发 1s 协程在旧实现下串行 ~5s，属于经典单环路瓶颈。
- 修复（`src/utils/async_helpers.py`）：
  - `run_async` 改为 thread-local 事件循环（每个 worker 线程复用自己的 loop，`run_until_complete` 驱动），恢复重构前的线程级并行，同时保持"每线程一个 loop、集中复用"，不回退到"散乱竞争 loop"。
  - `timeout` 分支用 `asyncio.wait_for` 包一层并保证非空 TimeoutError 文案。
  - 保留 `_get_bridge_loop`：仅供需要"另一线程里已运行的持久 loop + `run_coroutine_threadsafe`"的调用方（slock NLI 分类 `handlers/slock.py`、slock engine 自主消解 `engine.py`）。
- 验证：
  - 并发验证脚本：5 个并发 1s 协程从串行 ~5s 降到 ~1.0s，PASS（不再串行）。
  - `tests/test_async_helpers.py`+`test_safe_wait_for_integration.py` 14 passed。
  - `tests/test_acp_client.py`+`test_empty_error_guard.py` 126 passed。
  - `test_coco_model.py`+`test_acp_model_probe_timeout.py`+`test_acp_models.py` 53 passed。
  - slock NLI/intent 相关 286 passed。
  - dispatcher/gateway/autonomous/ws_client 相关 254 passed。
  - imports OK，`ruff check src/utils/async_helpers.py` 通过。
- 后续风险：
  - 模型探测 15s 超时（`acp_model_probe_timeout`）本身仍在；首次冷探测非 coco 工具仍会付一次真实 probe 成本。本次修复消除的是"所有 async 操作被单环路串行化互相拖慢"，不改探测超时值。
  - thread-local loop 随 worker 线程存活期常驻（pytest 下有 benign ResourceWarning），线程池 worker 复用，不会泄漏无界 loop。
