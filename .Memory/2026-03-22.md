## 飞书 WS 长连接静默失活导致“服务不再处理消息”根因修复
### 任务描述
排查线上服务运行一段时间后进程仍存活、但不再处理飞书消息的问题，要求做根因修复而不是简单绕过。

### 执行内容
- 结合 `logs.log`、进程树和 `sample` 结果确认：业务 Python 进程仍存活、线程大多处于事件循环/队列等待，未见异常栈、ERROR 或调度器死锁证据。
- 下钻 `lark_oapi.ws.client.Client` 实现，确认其仅在 `recv()/send()` 显式报错时才会进入 `_disconnect() -> _reconnect()`；对静默失活的 half-open socket 没有 pong 新鲜度/idle watchdog。
- 在 `src/feishu/ws_client.py` 引入 `_ObservedLarkWSClient`，观测 `connected/data/pong/disconnected` 活动并记录最近连接、最近消息帧、最近 pong 时间。
- 新增 WS watchdog：若连接空闲时间超过 `max(300s, 2 * ping_interval + 30s)`，则主动调用底层 `_disconnect()`，借用 `lark_oapi` 现有重连链路恢复连接，而不是永久卡在 `recv()` 等待。
- `start()` 改为使用 observed client 并启动 watchdog；`close()` 增加 watchdog 清理，避免后台线程残留。
- 在 `tests/test_ws_client_patch.py` 增加回归：覆盖 stale connection 触发断开重连、recent pong 不误判重连。

### 技术要点
- 根因不是业务 handler、卡片发送链路或 `TaskScheduler` 卡死，而是飞书 SDK 长连接客户端只处理“显式断连”，没有处理“连接仍存活但已不再收到事件”的静默失活场景。
- 修复方式不是外层定时重启进程，而是在现有长连接层补上活性观测和主动断连重连，让服务能对半开连接自愈。

### 验证记录
- `uv run pytest -x -q tests/test_ws_client_patch.py` → `15 passed`
- `uv run pytest -x -q` → `1655 passed, 10 skipped, 1 warning`

## 飞书卡片 schema 2.0 根级 `elements` 发送失败修复
### 任务描述
修复飞书回复消息创建 interactive card 时出现 `ErrCode: 200621`、`unknown property, property: elements, path:` 的问题，确保 schema 2.0 卡片发送给飞书时不会携带非法根级 `elements` 字段。

### 执行内容
- 在 `src/feishu/handlers/base.py` 增加 `_normalize_interactive_card_content()`，对 outgoing interactive payload 做统一规范化。
- 对 schema 2.0 卡片：若同时存在根级 `elements` 与 `body.elements`，发送前自动移除非法根级 `elements`；若只有根级 `elements`，则先下沉到 `body.elements` 再移除。
- 将该规范化逻辑接入 `_inject_ref_note()` 前置阶段，使所有通过 `BaseHandler.reply_message()/send_message()` 发出的 interactive card 共用同一修复路径，而不是在单个卡片 builder 上打补丁。
- 补充 `tests/test_handlers.py` 回归测试，验证 schema 2.0 payload 会在发送前被规整为飞书可接受结构。

### 技术要点
- 根因不是重试逻辑或飞书 SDK，而是内部同时维护了“测试/兼容读取方便”的根级 `elements` 与 schema 2.0 正式 `body.elements`；飞书服务端只接受后者。
- 选择在发送层统一规范化，而不是逐个 builder 清理，一次性覆盖所有 `BaseHandler` interactive 回复路径，避免同类错误在其他卡片构建入口重复出现。
