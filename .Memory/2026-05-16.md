## Feishu 官方卡片元素/字节上限守卫

- 问题：线上又出现兜底错误卡片 `code=230099; element exceeds the limit`，说明渲染出的飞书卡片仍会超过官方结构限制，delivery 层只能发出错误提示。
- 官方限制：飞书开放平台 Card JSON 2.0 文档明确单张卡片最多支持 200 个元素或组件；消息卡片发送/更新接口的卡片内容最大 30 KB，`230099` 是创建卡片内容失败的总错误码，需要结合返回的具体原因定位。
- 日志证据：`logs.log` 中同一窗口出现 `Rendered card page ... has 240/402/293 nodes (exceeds Feishu 200-element limit)`，renderer 只提示 `payload_truncator will attempt truncation at delivery layer`，但投递前守卫并没有保证节点数最终低于预算。
- 根因：分页预算依赖 `RenderAtom.node_count`，但 active tool panel、activity digest、subagent dispatch 等 atom 都把真实 3-4 个 tagged node 低估为 1，导致分页前低估元素数；`check_and_truncate_payload()` 遇到节点超限时只记录并尝试文本截断，若字节未超限或文本截断后节点仍超限，仍可能返回超过 200 元素的卡片。
- 修复：把上述 atom 的 `node_count` 调整为真实结构节点数；payload truncator 在字节未超限分支也会执行 table guard 后继续检查节点数；智能截断后如果字节或节点仍超预算，会降级为单 markdown 的 compact fallback card，并二次确认低于内部 180 节点/27 KB 预算和官方 30 KB 上限。

## Feishu 流式正文/思考块合并与引用左对齐

- 问题：线上截图中 `/spec`/编程卡片正文和灰色“思考完成”引用块被切成大量短段，部分引用块内容只有 1-5 个字符且视觉上靠右；长卡片随后还出现内容错乱/截断风险。

- 修复：流式桥接层改为同一逻辑流内分别稳定追加正文和思考块，只有 `TOOL_CALL_START`、bind/close 等硬边界才关闭当前块并开启下一组唯一 ID；reasoning 面板改为单列灰底 markdown 并显式 `text_align=left`，避免引用内容靠右；reasoning atom `node_count` 调整为 3，使分页预算与真实 Schema 节点一致。
- 回归：新增 soft alternation 不碎片化、工具调用硬边界、reasoning 左对齐、reasoning 多块分页不超过 200 节点的测试。

## 多页飞书卡片历史页冻结

- 问题：同一个 `CardSession` 内容超过飞书容量后会渲染成多条消息卡片；后续 live ticker/runtime 变化会让 `CardDelivery` 遍历 PATCH 所有已绑定 page，导致历史卡片标题耗时也持续刷新，一个话题里同一时刻可能有多条飞书消息在更新。
- 根因：分页是 render-time 结果，delivery 层按 `RenderedCard.page_index` 逐页比较签名；ticker 会改变结构签名，所有 page 都被判定需要更新。历史 page 已经承载旧内容和续接关系，不需要继续保持标题运行时间最新。
- 修复：`CardDelivery` 在 session 已有 binding 且当前 render 结果为多页时，把非最后一页视为 frozen history page，直接返回 `history_page_frozen` skip；只有最后一页继续 create/update/element update。这样卡片达到容量并续页后，旧消息不再实时刷新，最新消息继续展示最新运行时间。
- 兼容：如果最新页遇到 stale binding，仍会丢弃整个 session binding 并在下一次投递重建；page shrink 时原有 stale page 清理逻辑保持不变。
