## Spec 多角色 Review 面板与 task 简介展示

- 问题：`/spec` 多角色 review 完成后仍主要以长 markdown 文字展示审查结果，用户不容易看清本轮有哪些角色、每个角色用了哪个工具/模型，以及各自的具体建议；Build 阶段 active `task` 蓝条在泛名输入场景下也可能只显示 `task`。
- 根因：卡片状态层没有结构化的 review role block，`SpecStreamProcessor.on_review_done()` 只写入一个 `_review_done_*` text block；`task` 工具的 late update 只追加到 `ToolBlock.content`，没有回填 `tool_summary`，渲染层也刻意隐藏了 literal `task` 标签。
- 修复：新增 `REVIEW_RESULT_UPDATED` 事件、`ReviewRoleBlock`、`src/card/render/review.py` 和 `src/spec_engine/review_display.py`，把每个 `PerspectiveReview` 转成独立 `collapsible_panel`，按角色分别展示状态、工具/模型、结论和建议，并使用不同浅色背景/边框；Spec 运行态和 review 状态查看页都走同一结构化 payload，Review 阶段可见文案同步从“多视角审查”改为“多角色审查”。`task` 工具行改为 `task：<任务简介>`，并在 `TOOL_DELTA` / `TOOL_DONE` 时用真实描述回填摘要。
- 截断边界：review 结果不再把所有角色合并进一个长 markdown atom；每个角色是独立 atom，分页时可以按角色分布到不同飞书卡页，降低整段 review 被 payload 限制一起截断的风险。

## 多页飞书卡片历史页冻结

- 问题：同一个 `CardSession` 内容超过飞书容量后会渲染成多条消息卡片；后续 live ticker/runtime 变化会让 `CardDelivery` 遍历 PATCH 所有已绑定 page，导致历史卡片标题耗时也持续刷新，一个话题里同一时刻可能有多条飞书消息在更新。
- 根因：分页是 render-time 结果，delivery 层按 `RenderedCard.page_index` 逐页比较签名；ticker 会改变结构签名，所有 page 都被判定需要更新。历史 page 已经承载旧内容和续接关系，不需要继续保持标题运行时间最新。
- 修复：`CardDelivery` 在 session 已有 binding 且当前 render 结果为多页时，把非最后一页视为 frozen history page，直接返回 `history_page_frozen` skip；只有最后一页继续 create/update/element update。这样卡片达到容量并续页后，旧消息不再实时刷新，最新消息继续展示最新运行时间。
- 兼容：如果最新页遇到 stale binding，仍会丢弃整个 session binding 并在下一次投递重建；page shrink 时原有 stale page 清理逻辑保持不变。
