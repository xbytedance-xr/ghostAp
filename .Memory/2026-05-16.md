## Deep/Spec 首卡启动延迟修复

- 问题：用户反馈“启动一个任务时间很长”。最近一次日志显示 `/deep 检查下项目的各种能力，每个模块有哪些问题` 在 `17:41:34.994` 收到，`17:41:35.915` 已进入 Deep 并开始 `create_engine_session`，`17:41:37.428` 左右 ACP session ready；但 `src.card.delivery.pool: delivery pool created` 直到 `17:42:49.036` 才出现，首张卡片/首个可见更新晚了约 71 秒。

## 多页飞书卡片历史页冻结

- 问题：同一个 `CardSession` 内容超过飞书容量后会渲染成多条消息卡片；后续 live ticker/runtime 变化会让 `CardDelivery` 遍历 PATCH 所有已绑定 page，导致历史卡片标题耗时也持续刷新，一个话题里同一时刻可能有多条飞书消息在更新。
- 根因：分页是 render-time 结果，delivery 层按 `RenderedCard.page_index` 逐页比较签名；ticker 会改变结构签名，所有 page 都被判定需要更新。历史 page 已经承载旧内容和续接关系，不需要继续保持标题运行时间最新。
- 修复：`CardDelivery` 在 session 已有 binding 且当前 render 结果为多页时，把非最后一页视为 frozen history page，直接返回 `history_page_frozen` skip；只有最后一页继续 create/update/element update。这样卡片达到容量并续页后，旧消息不再实时刷新，最新消息继续展示最新运行时间。
- 兼容：如果最新页遇到 stale binding，仍会丢弃整个 session binding 并在下一次投递重建；page shrink 时原有 stale page 清理逻辑保持不变。
