## 2026-05-08 19:20:00 修复 /coco 在群里选模型后会话启动失败 + 双重错误消息

### 任务描述
用户反馈所有项目群 `/coco` 命令选完模型后没进入编程模式，反而连续展示两条提示：
1. ⚠️ "已开启 Coco 编程模式 直接发送你的编程需求,将自动创建编程话题"
2. ⚠️ "Coco 会话启动失败,请重新发送 /coco 开始"

且飞书侧弹"操作已受理，请勿重复点击" toast。

### 排查
- 第一条 = `mode_already_in_thread_msg` (UI_TEXT)，仅在 `enter_mode()` silent=False 路径输出
- 第二条 = `mode_session_fail_msg` (UI_TEXT)，仅在 `programming.py::handle_message` 中 session 仍为 None 时输出
- 用户在所有项目群都复现，意味着是普遍的代码逻辑问题而非状态污染

追踪发现：
1. `_handle_enter_coco(pending_prompt=text)` 流程：自然语言 → stash pending → 弹模型选择卡
2. 用户选模型 → `handle_select_acp_model` 取出 pending
3. **本应** 走 `_dispatch_pending_prompt_to_thread`（因为 thread mode + 未在 thread）
4. **但** 若 `_enter_mode_with_acp_model` 在 thread mode 下走 `enter_mode(silent=True, thread_id=None)`：
   - 该函数发现 `thread_programming_enabled and not thread_id` → `_thread_enabled=True` → 进 line 226+ 分支
   - 该分支只用 `_enter_mode_on_manager` 标记 mode、`_set_mode_on_project`，**不调用 ensure_session 启动会话**
   - silent=True，line 232 跳过提示输出，line 255 直接 return
5. 然后 fall-through `handle_message(pending)` 被调：
   - `session = get_session(...)` → None（因为 step 4 没启 session）
   - `enter_mode(silent default False)` recovery → 出"已开启..."（程序判断用户已在 mode）
   - 第二次 `get_session` → 仍为 None（recovery 走相同 thread-mode 分支不启 session）
   - 输出 `mode_session_fail_msg`

### 修复
两处防御性补丁：

1. **`src/feishu/handlers/programming.py::handle_message`** recovery 路径
   - 原: `self.enter_mode(message_id, chat_id, project=project, thread_id=thread_id)` (silent 默认 False)
   - 改: `self.enter_mode(message_id, chat_id, silent=True, project=project, thread_id=thread_id)`
   - 效果：即便 enter_mode 没起 session，不再输出迷惑性"已开启..."；最终统一由 `mode_session_fail_msg` 给出错误

2. **`src/feishu/handlers/system.py::handle_select_acp_model`** fall-through 路径
   - 在 `_enter_mode_with_acp_model` + handle_message(pending) 之前加 session 健康检查
   - 若 session 缺失 + thread_programming_enabled + 不在 thread → 回退走 `_dispatch_pending_prompt_to_thread` 创建新话题并启动 thread session
   - 这覆盖了 silent enter_mode 在 thread mode 下未真正启 session 的边缘情况

### 测试覆盖
- `tests/test_model_command.py::TestHandleSelectAcpModelPendingPrompt`
  - 新增 `test_fallthrough_in_thread_with_missing_session_calls_handle_message`：in-thread 状态下 fall-through 仍交给 handle_message，由其内部 silent recovery
  - 新增 `test_fallthrough_falls_back_to_thread_when_session_missing`：fall-through 检测到 session 缺失 + thread 模式 + 不在 thread → 重建话题（防御性兜底）

### 验证
- `tests/test_model_command.py` 25 passed
- 全量 `pytest tests/` 6240 passed, 1 skipped, 125 warnings

### 技术要点
- `enter_mode` 在 `thread_programming_enabled and not thread_id` 分支只标记 mode 不起 session，是为了把会话启动延后到用户在 thread 中真正发消息——但这个语义被 `_enter_mode_with_acp_model + handle_message(pending)` 直接级联破坏：pending 还没到 thread 就被强制 dispatch
- 治本应在 `_enter_mode_with_acp_model` 增加"thread mode 下启动 thread session"分支；本次保守做法是 fall-through 路径兜底以最小改动回滚回归
- silent=True 在 recovery 路径是必须的：当 session 缺失我们已经准备输出 `mode_session_fail_msg`，前面再输出"已开启..."只会让用户困惑

## 2026-05-08 18:50:00 Worktree 工具发现修复 ACP provider lazy-init 缺失

### 任务描述
18:25 那一轮已经把 worktree tool_discovery 改成强制 `skip=False` 并在 handler 放宽 short-circuit 条件，但用户实测仍出现 "已添加 Coco / 工具内置模型" + 第二次添加被 dedup 的现象。说明 ACP 分支根本没走到，Coco 落到了 CLI 分支。

### 根因
- `src/acp/providers/__init__.py` 中的 ACP provider 是 **lazy 注册** 的：`_ensure_providers()` / `get_providers()` 必须被显式调用一次，才会把 Coco/Aiden/Codex/Claude/Gemini provider 注册到模块级 `tool_registry`
- `src/worktree_engine/tool_discovery.py` 直接 `from ..acp.providers import tool_registry` 然后调用 `tool_registry.get_provider("coco")`，但**没有触发 lazy 注册**，所以 registry 默认为空
- 结果：所有 known tool 都落到 CLI 分支：`provider_type="cli"` + `supports_model=False`，进而 `selection_key="acp:coco:default"`（其实是 `cli:coco:default`，但 display_label 走 `supports_model=False` 分支输出"工具内置模型"），第二次添加被 `add_item` dedup 阻止
- 之前 18:25 的修复（强制 `skip=False`、放宽 `len(models)<=1`）针对的是 ACP 分支已生效后的二级问题；ACP 分支没生效时这些改动都不起作用

### 验证根因
```python

tool_registry._providers  # {} 空
get_providers()
tool_registry._providers  # {'coco': ..., 'claude': ..., 'aiden': ..., 'codex': ..., 'gemini': ...}
```

### 执行内容
1. **`src/worktree_engine/tool_discovery.py`**
   - import 改为 `from ..acp.providers import get_providers, tool_registry`
   - `get_available_tools()` 入口处显式调用 `get_providers()` 触发 lazy 注册（idempotent，重复调用无副作用）

2. **测试**
   - `tests/test_worktree_tool_discovery.py` 新增 `test_get_available_tools_triggers_acp_provider_lazy_init`，使用 `mock.patch("...get_providers")` 断言 `get_available_tools()` 调用一次

### 技术要点
- ACP provider lazy 注册是为了避免冷启动时执行 binary 探测 + help blob 加载（昂贵），仅在确实有 ACP 调用时才初始化
- 但 worktree 入口只是想读 registry 里"已知"的 provider 列表，因此触发 lazy 注册是必要前置——否则用户必须先用过 `/acp` 等命令让 ACP 路径预热，才能在 `/wt` 时看到 ACP 模式
