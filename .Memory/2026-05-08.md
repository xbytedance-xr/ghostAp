## 2026-05-08 00:58:00 修复模型选择后 pending prompt 在线程模式下启动失败

### 任务描述
用户反馈上一轮修复后仍不对：项目群自由文本触发 Coco 模型选择后，点击模型按钮仍出现"已开启Coco编程模式"提醒，并随后提示"Coco 会话启动失败"。

### 执行内容
1. **根因定位**
   - 日志确认模型按钮回调为 `action=select_acp_model, project_id=ghostap, value_keys=['action', 'model_name', 'project_id', 'tool_name']`，缺少 `thread_root_id`
   - 日志确认随后出现飞书 API 错误：`content is not a string in json format`，对应代码把 `build_switching_status_card()` 返回的文本用 `reply_card()` 发送
   - 代码链路确认 pending prompt 被直接转发给 `ProgrammingModeHandler.handle_message()`，在线程编程模式下没有已创建的 thread session，因此触发"已开启" already-in 提醒与"会话启动失败"

2. **修复内容**
   - `src/feishu/handlers/system.py` 移除模型切换前的无意义"切换中"回复，避免把文本当卡片发送
   - `SystemHandler.handle_select_acp_model()` 在 pending prompt + thread programming + 无当前 thread 时，直接创建编程话题、启动线程 session，并把原始需求投递到该 session
   - `src/card/builders/system.py` 与 `src/card/builder.py` 为 ACP 模型选择卡 action payload 增加 `thread_root_id`，保证话题内选择模型时能恢复线程上下文
   - `_resolve_current_acp_tool()` 补传 `project_id`，避免项目级模式判断落到 chat 级状态

3. **测试覆盖**
   - 新增测试覆盖 `/model list` 卡片携带 `thread_root_id`
   - 新增测试覆盖 `/model <name>` 不再发送进度卡/文本卡
   - 新增测试覆盖 pending prompt 选择模型后创建线程并启动 session，而不是进入顶层 thread-pending 模式

### 技术要点
- `build_switching_status_card()` 返回 `("text", msg)`，调用方不能用 `reply_card()` 发送
- 线程编程模式下，pending prompt 不能在主会话直接 `handler.handle_message()`；必须先有 thread session
- 卡片 action 的 `thread_root_id` 由 `ws_client._process_card_action_async()` 恢复到 thread-local，是按钮回调中保持话题上下文的关键字段
