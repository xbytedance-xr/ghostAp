### 技术要点

1. **RLock vs Lock**: `ProjectManager` 使用 `threading.RLock` 而非 `threading.Lock`，允许 `validate_project_path` → `_get_project_unchecked` 嵌套锁获取。
2. **frozen=True dataclass**: `RepoLockInfo` 设为不可变，消除跨线程读取时的竞态窗口。
3. **异常分层**: `LockConflictError` 在 `_safe_execute_engine` 中优先于 `except Exception` 捕获并穿透，防止锁冲突被静默吞掉。
4. **锁粒度细化**: ProgrammingModeHandler 从"锁整个 streaming 会话"改为"仅锁提交阶段"，streaming 期间通过 30s heartbeat `touch()` 保持 repo lock 活跃。
5. **TOCTOU 防御**: 所有锁状态读取在 `with self._mu:` 保护下完成，避免 Time-of-check-time-of-use 竞态。
6. **`/wt` `/worktree` 白名单**: 运行时解析 `raw_text`，无子参数时放行（只看锁状态），有子参数时走正常锁检查。
7. **open_id 防泄露**: 卡片展示 `locked_by` 时截断为前 8 字符 + `…`，避免完整 open_id 暴露。

## 重构：提取 HMAC 签名逻辑到独立模块 src/utils/signing.py（SRP）

### 任务描述
`src/card/builders/lock.py` 混合了 UI 卡片构建与密码学签名两种职责，违反单一职责原则（SRP）。将 HMAC 签名相关函数提取到独立模块 `src/utils/signing.py`，保持向后兼容。

### 执行内容

1. **新建 `src/utils/signing.py`**: 包含 `_get_signing_key`、`_compute_command_sig`、`verify_command_sig`、`_verify_legacy_sha256_fallback`、`_PROCESS_START_DATE` 五个导出
2. **修改 `src/card/builders/lock.py`**: 移除函数实现体和 `hashlib`/`hmac` 未用导入，添加 re-export 块（`from src.utils.signing import ... # noqa: F401`）确保现有调用方无感迁移
3. **新建 `tests/test_signing.py`**: 17 个单元测试覆盖全部签名函数（`TestGetSigningKey` 3 用例、`TestComputeCommandSig` 5 用例、`TestVerifyCommandSig` 4 用例、`TestLegacySha256Fallback` 4 用例 + 1 集成用例）
4. **更新 `tests/test_lock_cards.py`**: ~20 处 `@patch("src.card.builders.lock._get_signing_key")` → `@patch("src.utils.signing._get_signing_key")`，1 处 `@patch("src.card.builders.lock.logger")` → `@patch("src.utils.signing.logger")`
5. **更新 `tests/test_action_retry.py`**: 所有 `@patch("src.card.builders.lock._get_signing_key")` → `@patch("src.utils.signing._get_signing_key")`
6. **更新 `tests/test_retry_handler.py`**: 所有 `@patch("src.card.builders.lock._get_signing_key")` → `@patch("src.utils.signing._get_signing_key")`

### 技术要点

1. **Re-export 向后兼容**: `lock.py` 通过 `from src.utils.signing import (...)` 重导出所有签名函数，现有 `from src.card.builders.lock import verify_command_sig` 调用方零改动
2. **Mock patch 路径语义**: 函数移到 `signing.py` 后，`verify_command_sig` 内部调用 `_get_signing_key` 时走 `signing` 模块的名字绑定，必须 patch `src.utils.signing._get_signing_key` 而非 `src.card.builders.lock._get_signing_key`
3. **常量时间比较**: `hmac.compare_digest()` 防时序侧信道攻击
4. **Legacy SHA-256 回退窗口**: 通过 `sig_compat_deploy_date` + `sig_compat_window_days` 配置控制，过期后自动关闭

## 重构：移除 RepoLockGuard 中间层，统一 repo lock 使用模式

### 任务描述
解决 `RepoLockGuard` 与 `BaseHandler._with_repo_lock` 职责重叠问题。原有三种 repo lock 使用模式（guard/`_with_repo_lock`/直接 `acquire/release`）统一为两种：`_with_repo_lock` 用于 context-manager 场景，`_acquire_repo_lock`/`_release_repo_lock` 配对方法用于需要跨 streaming 持有的长生命周期场景。

### 执行内容

1. **`BaseHandler._with_repo_lock` 内联**: 移除对 `RepoLockGuard` 的依赖，直接内联 `repo_lock_mgr.hold()` + `LockConflictError` 转换逻辑
2. **新增配对方法**: `BaseHandler._acquire_repo_lock` / `_release_repo_lock` 封装 `is_p2p` 读取、`repo_lock_manager` 获取和空值短路判断
3. **重构 ProgrammingModeHandler**: `handle_message` 中裸 `acquire/release` 替换为 `_acquire_repo_lock`/`_release_repo_lock`，消除重复逻辑
4. **删除 `src/repo_lock_guard.py`**: 移除 80 行纯委托中间层
5. **重写测试**: `TestRepoLockGuard`（5 用例）→ `TestRepoLockHold`（5 用例），`test_concurrent_guard_execute` → `test_concurrent_hold_execute`，直接测试 `hold()` context manager

### 技术要点

1. **两种统一模式**: `_with_repo_lock`（短生命周期 context-manager 包裹）和 `_acquire/_release`（长生命周期显式配对 + touch 心跳）
2. **`_acquire_repo_lock` 返回三元组**: `(AcquireResult | None, repo_lock_mgr, needs_release)`，调用方通过 `needs_release` 决定 `finally` 块行为
3. **P2P bypass 统一收口**: 两种方法内部都通过 `get_current_is_p2p()` thread-local 读取，调用方无需关心
