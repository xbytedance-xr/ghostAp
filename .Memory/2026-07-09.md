# 2026-07-09

## WF 完整报告附件与 cache 存储

### 背景
- `/wf` 已能跑完，但终态卡片受飞书卡片大小限制，结构化结果、验证摘要、执行过程等内容会被 `_enforce_card_size()` 和完成报告分段截断。
- 用户明确要求最终结论数据不能被截断，并接受用静态 HTML 临时报告作为飞书附件发送；本地报告文件应统一放到 `~/.cache/ghostAp`，跟 Spec/Workflow 状态等本地数据位置保持一致。

### 改动
- 新增 `src/workflow_engine/reporting.py`：
  - 生成完整 Markdown 与自包含 HTML 报告，包含运行摘要、最终报告/结果/验证/评审/风险/后续、原始结果 JSON、agent 原始输出和完整 Workflow 状态 JSON。
  - HTML 仅使用内联 CSS/JS，不依赖外部资源；动态内容通过 `html.escape()` 转义，支持搜索、展开/折叠和复制。
  - 报告落盘到 `~/.cache/ghostAp/<项目绝对路径>/workflow_reports/`，测试可通过 `cache_root` 注入临时 cache。
- `FeishuIMClient` 增加 `upload_file()` 和 `reply_file()`：
  - 使用飞书 IM file create API 上传本地 HTML；
  - 以 `msg_type=file` 回复当前消息，并复用现有 `_execute_with_retry()`。
- Workflow `on_done()` 回调：
  - 先生成/发送完整 HTML 附件，再渲染完成卡；
  - 成功时卡片显示“完整 HTML 报告已发送”；
  - 附件上传/回复失败时，Workflow 仍按成功完成展示，并在卡片显示降级原因和本地 HTML/Markdown 路径。
  - 有完整报告文件时，终态卡片不再嵌入长结果正文，只保留报告入口和执行过程简表，避免再次出现“内容已截断”的误导。
- `render_completion_card()` 新增可选 `report_status`，保持无参数调用兼容。
- 同步修正 `WorkflowHandler` 和 `WorkflowScriptMixin` 两处重复 callback 实现，避免后续调用路径回退到无附件版本。
