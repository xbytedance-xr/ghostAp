# 2026-07-24 ACP 图片产物进入飞书任务卡片

## 背景与结论

- 既有 ACP 客户端只消费文本、思考、工具和计划事件；ACP 0.11 已提供的 `ImageContentBlock`、工具内容图片及资源图片会落入 `Unhandled content type`，因此生成截图/图片不会进入飞书消息流。
- 飞书卡片渲染层已经支持 `img` 元素，但此前缺少“ACP 图片 → 官方 SDK 上传 → `image_key` → CardState”的桥接。
- 本次建立类型化图片事件和共享媒体发布器，普通编程模式及 Deep/Spec 的主任务、子任务卡片均可在执行中展示图片；完成态将文本答案与图片放在执行历史之前。

## 实现

- ACP：
  - 新增冻结的 `ACPImageInfo` 与 `ACPEventType.IMAGE_CHUNK`。
  - 支持消息/思考中的直接图片、工具内容中的图片与 blob，以及 ACP 根目录内的本地资源图片。
  - 只接受 PNG/JPEG/GIF/WebP/BMP，解码后上限 10 MiB；按内容 SHA-256 去重。
  - 拒绝远程 URL、越出 ACP 根目录的路径、无效 base64 和不支持的 MIME。
- 飞书上传：
  - `FeishuIMClient.upload_image_bytes()` 使用官方 `lark-oapi` `CreateImageRequest` 上传消息图片并返回 `image_key`。
  - Handler 上传边界再次校验 MIME、base64 和大小，不记录二进制或 base64 内容。
- 卡片：
  - 新增 `IMAGE_ADDED` / `IMAGE_FAILED`、冻结 `ImageBlock`、纯 reducer 和 `img` render atom。
  - CardState 与卡片 JSON 只保存飞书 `image_key`，不保存图片载荷。
  - 上传失败显示非敏感降级提示，不把父任务判为失败；同一流中的相同图片只上传/展示一次。
  - `ACPImagePublisher` 接入 `ProgrammingCardSession`、`ACPStreamBridge`、Deep 与 Spec 的主/子任务流；图片是文本/思考的硬边界，避免内容顺序错乱。
- 设计资料：
  - 实施计划：`docs/2026-07-24-acp-image-card-stream-plan.md`
  - UI 预览：`ux/programming-image-artifacts.html`

## 后续风险与边界

- 当前自动展示 ACP 原生图片块、工具内容图片，以及工具/CLI 明确引用且相对本轮 prompt 基线新增或变更的本地图片；不再遍历工作目录广泛发布。扫描不完整、忽略目录、非图片后缀、符号链接或并发父子根快照冲突均默认拒绝。
- 图片上传在事件顺序点同步完成，保证图片与前后文本顺序稳定；超大或大量图片会增加该事件的延迟，但受 10 MiB、SDK 重试和内容去重约束。
- 尚未使用真实飞书租户凭据做端到端视觉验收；上线后应验证机器人具备图片上传权限，并观察 CardKit 中运行态、完成态和失败降级三种外观。

## Deep 普通 execute 输出误生成子任务修复

### 现场证据与根因

- 截图中的失败卡对应 ACP 调用 `call_zXAT0JlJc0dqRewiUJK8nHYL`。原始 rollout 表明它实际是普通 `exec_command`：运行
  `tests/test_programming_completion_guards.py::test_execute_output_containing_subagent_marker_stays_parent_tool`，约 2.1 秒即按预期红灯失败；卡片显示的 `35m06s` 是父 Deep 会话累计耗时，不是该命令耗时。
- ACP 工具终态会把 `ToolCallInfo.content` 替换为 `raw_output`。该 pytest 输出恰好包含被测源码字符串 `子代理：`，而 `TaskOrchestrator._is_agent_task()` 旧逻辑不检查 `kind`，仅凭该文本标记就把 `kind="execute"` 的普通工具误注册为子任务。
- 误注册后，失败输出中的 JSON、`formatted_output` 与 jq 命令又被当成任务标签更新，因此出现截图中的伪子任务卡、JSON/jq 任务名和泛化的“子任务”错误摘要。问题源头是路由误判，不是 pytest 实际运行了 35 分钟，也不是卡片布局本身。

### 修复

- `TaskOrchestrator` 现在优先信任 ACP 语义身份：`kind == "agent"` 或显式 `agent`/`subagent`/`task` 标题仍按子任务路由。
- 兼容旧 provider 的 `子代理：` 文本兜底只允许 `kind == "other"`；`execute`、`read`、`edit` 等具体工具的 stdout/stderr 不再触发子任务识别。
- 没有调整卡片布局或文案；删除错误路由后，伪子卡、JSON/jq 标题和泛化摘要在源头一并消失。
- 实施计划：`docs/2026-07-24-deep-false-subtask-routing-fix-plan.md`。

## 编程终态、媒体归属与并发安全巡检

### 发现与修复

- **稳定子任务路由**：子任务首次按 ACP agent 形状绑定后，后续事件只按稳定
  `tool_call.id` 路由；即使终态事件退化成 `kind="execute"`、标题改成
  `shell`，仍会把失败终态送到原子卡。修复了
  `test_known_agent_call_keeps_routing_when_terminal_shape_changes` 中子卡永久
  `running` 的现场错误。
- **终态和关闭生命周期**：Programming/Deep 取消先更新子任务摘要再终态化父卡；
  Deep 的 `PAUSED` 正确收敛为 `CANCELLED`，已完成子任务保持完成；CLI
  统一完成/失败/取消/超时结果；CardOrchestrator 超时调用改成 daemon worker，
  不再因阻塞 SDK 线程拖住进程退出。
- **ACP prompt 隔离**：异步与同步 ACP session 都增加非阻塞 prompt 互斥，拒绝同一
  session 并发覆盖 event handler 或 active future；关闭时取消 future、清 handler
  并释放图片 snapshot lease。尾部事件 drain/终态屏障已覆盖，原 Backlog B050
  因此移除。
- **媒体严格归属**：取消 prompt 结束时的全工作目录图片扫描。普通文本路径只有在
  工具明确引用、位于允许根内且相对 prompt baseline 确认新增/变更时才可发布；
  同根或父子根重叠、扫描预算耗尽、忽略目录和伪装后缀均 fail closed。图片 MIME
  以实际字节判定，本地读取使用 no-follow 与大小上限；文件名、alt 和渲染哈希会
  清理未配对 surrogate/控制字符，上传重试每次重建字节流。
- **Deep pending 上下文**：异常以及返回型 timeout/cancel/failed/incomplete plan
  都保留尚未消费的用户指导并返回失败结果，避免一次失败静默吞掉下一轮调整。
- **Shell 锁与按钮防重放**：嵌套 cwd 的 Git probe 失败时只接受可验证的祖先
  `.git` marker；锁冲突重试全链路传递真实 `chat_id`。命令按钮统一使用绑定聊天、
  过期时间和 nonce 的 v2 签名；真实回调拒绝 v1，`/status` 只读入口保持豁免。
  已消费 nonce 通过 flock、no-follow、原子替换和 fsync 持久化到
  `~/.ghostap/used-command-nonces.json`，服务重启后仍不可重放；存储损坏或容量耗尽
  均默认拒绝。
- **Autonomous Context 投影竞态**：员工激活 ACK 的后台 tombstone 可能在 Context
  组装中推进共享 Journal head，旧实现把“员工/principal/Channel 语义未变”误判为
  `SCOPE` 或 `MEMORY`。现在完整组装最多做 3 次乐观重试并整次重读 L1/L2/source；
  成员、凭据、ACL 或 Channel 变化仍立即 fail closed。完整与 canonical partial
  共用 admission/drain lease，partial 在 ledger 返回后再次校验权限和投影头。
- **子任务卡片移动端收敛**：共享 display helper 清理 opaque call ID、转义控制文本
  和 raw JSON；子卡只显示可读任务名、状态、分支号与父卡号，整体进度折叠为紧凑
  摘要，footer 去掉重复目录和 subagent 标记；父卡布局保持不变。
