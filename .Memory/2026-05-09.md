## 2026-05-09 12:35:11 修复 Worktree 选工具时 ACP 模型探测卡住

### 任务描述
用户要求直接在飞书 Web 页面验证 `/wt`。实际复现后，Worktree 卡片按钮能触发服务端，但点击工具后按钮一直 loading，卡片无法进入模型选择或已选组合下一步。

### 执行内容
1. **根因定位**
   - 在允许的测试群中复现 `/wt`，发现主群只展示卡片摘要，真实按钮在右侧话题面板中。
   - 点击工具后日志停在 `卡片按钮点击: action=worktree_select_tool`，没有后续卡片 dispatch。
   - 对照 `WorktreeHandler.handle_worktree_select_tool()`，卡住点位于同步 `_get_models_for_tool()`；ACP 模型探测通过 `spawn_agent_process → initialize → new_session` 获取模型，但没有硬超时。
   - Coco fallback 又会进入 `CocoModelManager._load_models_via_acp()`，同样没有硬超时，可能导致回调二次卡住。

2. **修复内容**
   - `src/acp/helper.py::fetch_acp_models()` 使用 `asyncio.wait_for()` 包裹 ACP 模型探测，超时后走 fallback。
   - Coco fallback 改为读取 `DEFAULT_MODELS` + 当前配置模型，不再调用 `get_models()` 触发第二次 ACP 探测。
   - `src/coco_model/manager.py::_load_models_via_acp()` 同样加 `asyncio.wait_for()`，避免其他路径直接调用时卡死。
   - 新增 `tests/test_acp_model_probe_timeout.py` 覆盖超时后快速返回 current model、Coco 超时后快速返回静态默认模型。

### 技术要点
- 飞书卡片 callback 必须尽快返回；选工具阶段不能同步等待无界 ACP 启动/握手。
- Worktree 选择卡的模型探测属于增强体验，失败或超时应降级到默认模型/静态模型，不能阻塞选择状态机。

## 2026-05-09 11:44:34 修复 Worktree 多工具点击被误判重复

### 任务描述
用户反馈 `/wt` 卡片中已添加 Aiden 后点击 `+ 添加 Coco` 仍无法加入，飞书弹出“操作已受理，请勿重复点击”。

### 执行内容
1. **根因定位**
   - 日志已证明 `worktree_select_tool` 回调能到达服务端，说明上一轮 `behaviors` 修复后按钮触发链路已恢复。
   - 继续追踪 `_handle_card_action()` 入口去重逻辑，发现去重 key 只包含 `chat + message + operator + action`。
   - Worktree 工具列表中所有“添加工具”按钮 action 都是 `worktree_select_tool`，因此同一张卡上先点 Aiden 再点 Coco 会被当成重复点击拦截。

2. **修复内容**
   - `src/feishu/ws_client.py` 为卡片去重 key 增加稳定 payload fingerprint，纳入 `value` 以及可用的 `option/options/form_value/input_value`。
   - 保留原本的快速重复点击保护：完全相同 payload 的二次点击仍返回“操作已受理，请勿重复点击”。
   - 不在 Worktree handler 做特殊绕过，避免破坏入口层统一防抖语义。

3. **测试覆盖**
   - `tests/test_button_gate_and_dedupe.py` 新增回归：同一卡片连续点击 `worktree_select_tool` 的 Aiden 与 Coco payload 应提交两次；重复点击同一个 Coco payload 才应被去重。

### 技术要点
- 卡片 action 级去重不能只按 action 名称做 key；对于一张卡内复用同一 action 的多按钮场景，必须把具体 payload 纳入 key。
- 入口层区分 payload 后，Worktree 多工具选择、模型选择、多选式表单提交这类交互都能避免被误判为重复点击。

### 执行内容

   - 这些内嵌按钮仅携带 `value`，缺少飞书 Schema 2.0 callback 所需的 `behaviors: [{"type": "callback", "value": ...}]`。
   - 后续 Worktree 阶段的 `ButtonSpec` 通用渲染路径同样只输出 `value`，存在下一阶段按钮继续不可交互的风险。

   - `src/card/render/worktree.py` 新增 `_callback_button()`，让工具添加、模型选择、移除、清空、确认按钮都输出 `value` + `behaviors`。
   - `src/card/render/buttons.py` 的通用 callback 按钮输出 `behaviors`，覆盖 Worktree 后续确认/执行/合并/清理按钮。
   - 保留原 `value` 字段，避免破坏现有路由和测试中对 legacy value 的读取。

3. **测试覆盖**
   - `tests/test_worktree_render_direct.py` 增加断言：Worktree 工具选择、模型选择、移除、清空、确认按钮必须带 callback behaviors，且与 `value` 一致。
   - `tests/test_card_render_components.py` 增加通用按钮 callback behaviors 断言。

### 技术要点
- Schema 2.0 卡片按钮需要 `behaviors` 才能作为前端 callback 交互生效；仅保留 `value` 不足以保证客户端触发回调。
- 兼容策略是同时输出 `value` 和 `behaviors`，让旧的服务端解析与新的客户端交互字段同时存在。
