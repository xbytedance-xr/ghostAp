## 2026-05-09 17:59:22 优化 GhostAP 重启脚本降低固定等待

### 任务描述
用户反馈日志中 Spec Review 重试预算 warning 后，实际重启到服务恢复工作太慢。需要区分 warning 与重启慢的关系，并优化 `restart.sh`。

### 执行内容
- 确认 `重试实际预算可能超限` 只是 `src/config.py` 的 Spec Review retry 软约束 warning，不会阻塞服务启动。
- 优化 `restart.sh`：
  - 将远程重启前置等待从固定 3 秒改为 `GHOSTAP_RESTART_GRACE_DELAY`，默认 1 秒。
  - 将 SIGTERM 等待从固定 2 秒改为 `GHOSTAP_TERM_GRACE_DELAY`，默认 0.8 秒。
  - 将残留进程清理等待从固定 1 秒改为 `GHOSTAP_RESIDUAL_GRACE_DELAY`，默认 0.2 秒。
  - 将启动后检查等待从固定 2 秒改为 `GHOSTAP_START_CHECK_DELAY`，默认 0.3 秒。
  - 优先使用 `.venv/bin/python -m src.main` 启动，缺失时回退 `uv run python -m src.main`。
  - 本机无 `setsid` 时使用 `launchctl submit` 脱离 Codex/终端执行器，避免 `nohup` 子进程在执行器退出后被清理；远程 worker 也使用独立 `launchctl` label 兜底。
  - 远程 worker 复用主 `restart.sh restart` 逻辑，不再复制一份 stop/start 代码，避免两套逻辑漂移。
  - 增加 `[RESTART]` 阶段日志，记录 stop/start/spawn 状态和启动命令。

### 技术要点
- 重启慢分成两段：脚本固定等待和 Python 应用冷启动。此次把脚本返回耗时降到约 1.36 秒；本地 restart 实测服务从 stop begin 到飞书 WS connected 约 15 秒，remote worker 路径验证通过但仍受 `src.main` 顶层 import/config 冷启动波动影响。
- macOS 环境没有 `setsid`，仅靠 `nohup` 在 Codex 执行器中会短暂启动后被清理；`launchctl submit` 可以让服务进程独立保持运行。

- 重启后 `logs.log` 显示 `17:58:36` 飞书 WS `connected`

- `GHOSTAP_RESTART_GRACE_DELAY=0 ./restart.sh remote-restart` → worker status=0，重启后飞书 WS `18:01:57` connected，PID 29424

## 2026-05-09 17:22:24 修复 Worktree 同工具不同模型二次添加被卡片去重误拦

### 任务描述
用户反馈在 `/wt` 里已选 `Coco / Test-O-New-Thinking` 后，再点击 `+ 添加 Coco` 想选择 Coco 的另一个模型时，飞书弹出“操作已受理，请勿重复点击”。期望是同一个工具可以重复进入模型选择，从而选择不同模型组成并行组合。

### 根因
Worktree selection 状态机本身已经使用 `selection_key = "{provider}:{tool_name}:{model_name|default}"`，天然支持同工具不同模型多次添加；真正的误拦发生在 `FeishuWSClient` 的卡片 action 去重层。卡片刷新回工具列表后，`+ 添加 Coco` 按钮的 callback value 与第一次点击前完全相同（同 message_id + 同 action + 同 payload），因此短 TTL action dedupe 把它当成同一张卡的快速重复点击，提前返回“请勿重复点击”，handler 没机会再次弹模型选择卡。

### 执行内容
- `src/card/render/worktree.py`：
  - `_selected_tool_keys()` 纳入 `selection_key`，保留 tool/name/display_name 兼容旧 selected 形态。
  - 新增 `_selected_signature()`，把当前已选组合生成 12 位短签名。
  - 工具按钮 callback value 增加 `_selection_sig`；当已选组合变化后，同一个 `+ 添加 Coco` 按钮 payload 也随之变化，卡片去重层会放行。模型按钮 value 不变。
- `tests/test_button_gate_and_dedupe.py`：新增 `test_same_worktree_tool_after_selection_change_is_not_deduped`，锁定“同工具但 selection_sig 变化 → 放行；同一 selection_sig 再点 → 仍去重”。
- `tests/test_worktree_render_direct.py`：新增 `test_tool_button_payload_changes_after_selection_changes`，确保渲染层真的把已选组合变化反映到工具按钮 payload。

### 技术要点
- 这次不能移除 action dedupe：它仍需要挡住同一张卡/同一 payload 的快速重复点击，避免重复提交任务。
- 正确做法是把“卡片状态版本”纳入 payload，使刷新后的合法二次点击与原始点击可区分。
- `_selection_sig` 是 UI/去重层内部字段，Worktree handler 忽略未知 value 字段，不影响后端 tool/model 语义。

## 2026-05-09 14:00:00 修复 Worktree Coco 模型列表与 /coco 不一致

### 任务描述
新版 Worktree 模型选择卡上线后，用户反馈"模型列表是错误的"，附图只显示 6 个静态 DEFAULT_MODELS（gpt-5.2/gpt-4.1/claude-3-opus/claude-3.5-sonnet/claude-3.7-sonnet/doubao-1.5-pro），并指出 `/coco进入模型选择` 才是正确的获取方式。

### 根因
1. `acp_healthcheck_timeout` 默认 2.0s，但 ACP `spawn_agent_process → initialize → new_session` 拿到 `available_models` 实测要 3-4s。两秒就 `asyncio.wait_for` TimeoutError，`fetch_acp_models` 落 fallback 直接返回 `DEFAULT_MODELS`。
2. `/coco` 系列里 `show_coco_status` 走的是 `CocoModelManager.get_models()`：内部有 5 分钟 cache，第一次成功 probe 之后所有调用都拿真模型；而 `/coco` 模型选择 / `/wt` 模型选择都走 `_fetch_acp_models` → `fetch_acp_models`，绕过这个 cache，每次都要重新 probe 并被 2s 超时坑。
3. Coco fallback 也只看 `DEFAULT_MODELS`，没有把 manager 已经缓存好的真模型当作降级首选。

### 执行内容
- `src/config.py`：新增 `acp_model_probe_timeout: float = 6.0`，与 `acp_healthcheck_timeout`（2s healthcheck）拆开。Healthcheck 仍保持短，模型 list 拉取走更宽的窗口。
- `src/acp/helper.py::fetch_acp_models`：
  - Coco 入口先调用新加的 `_coco_models_from_manager()`，命中 manager 真实 ACP 模型缓存就直接返回，跳过本次 probe。
  - 命中失败时再做 ACP probe（用 `acp_model_probe_timeout`）。
  - probe 超时后再次尝试 `_coco_models_from_manager()`（兼顾"我们 probe 失败但 manager 在并发路径已 cache"），全失败才回退到 `DEFAULT_MODELS`。
  - `_resolve_acp_model_probe_timeout()` 优先读 `acp_model_probe_timeout`，回退到 `acp_healthcheck_timeout` 兼容旧配置。
- `src/coco_model/manager.py::_load_models_via_acp`：probe 同样改用 `acp_model_probe_timeout`，让 manager 自己第一次成功率提升。
- `tests/test_acp_model_probe_timeout.py`：
  - 改 `test_fetch_coco_models_timeout_uses_static_defaults`，让 fake manager 返回静态 DEFAULT_MODELS（保证 fetch_acp_models 不死循环），断言仍能落到 DEFAULT_MODELS。
  - 新增 `test_fetch_coco_models_uses_manager_cache_when_real_models_present`：fake manager 提前缓存 25 个真模型时，必须直接返回真列表，绝不 spawn probe。

### 技术要点
- "/coco_status 显示真模型 / /coco 模型选择显示 fallback"是同一进程里两条 model fetch 路径并存的副作用：show_coco_status 走 manager（带 cache），select 走 helper（每次 probe）。统一到 manager 缓存源后，`/coco` 选模型、`/wt` 选模型、`/coco_status` 三路读到的就是同一份。
- 单独的 `acp_model_probe_timeout`（6s）只影响"列出可用模型"这一步；ACP 服务健康检查仍保持 `acp_healthcheck_timeout`（2s），不会拉长无关探测路径。
- 第一次 probe 后 5 分钟内同一进程的 Coco 模型选择都不再 spawn 子进程，`/wt` 的二次添加 Coco 体验从"3-4s 等待"压到 0s。
