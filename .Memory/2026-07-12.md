## 生产级自主 Agent 部门设计收敛

### 现状审计
- 三路只读子代理分别审计身份/生命周期、飞书 transport、运行时/路由；共同确认当前生产仍是“主 Bot + Slock 虚拟 Agent”，`src/autonomous/` 的 577 项测试验证安全内核组件，但没有证明 production composition、独立员工 Bot、Channel、Slash、Router、`/fire`、按日历史或 Thread Context。
- 当前 `lark-oapi==1.6.5` 的真实 `register_app/aregister_app` 签名不含 `app_preset/addons/create_only/app_id`；隔离验证官方 `1.7.1` wheel 已包含这些参数。设计固定升级并以签名 contract + 真租户行为双门禁 fail-close，不凭网页文档猜 API。
- `lark-channel-sdk==1.1.0` 的底层 WS Client 使用模块级全局 event loop，多 Bot 同进程存在 ownership 风险；设计采用每员工 fresh-interpreter Channel 子进程，主进程保留 Journal/Router/Slock runtime。

### 方案与关键决策
- 比较“直接扩展旧 Slock”“v5 Journal SSOT + Slock runtime adapter”“每员工完整 GhostAP 服务”三案，采用第二案：Journal 是员工/Saga/membership/Inbox/Outbox/Effect 唯一事实源，AgentRegistry 和文件布局是可重建 Projection，独立员工消息通过类型化 gateway 复用原 `_run_acp_session`。
- `AgentIdentity` 只保存 `app_id + credential_ref`；secret 使用 AES-GCM 文件 Vault、`0700/0600`、原子 fsync/rename、key ID/轮换/orphan receipt，master key 来自 `src/config/.env`。这是对原“凭证存入 AgentIdentity”的生产安全等价解释。
- Hire/Fire 都是可恢复 Saga；所有外部步骤 PREPARED/EXECUTING 先 fsync+anchor，未知结果进入 ACTION_REQUIRED，禁止假成功或盲目重试。现有 ACP 无跨崩溃幂等/查询能力，因此 RPO=0 只承诺事件不丢、不重复建 task，不宣称 ACP 副作用 exactly-once。
- Channel child 通过一次性 pipe 仅取得自己的 secret，常规 IPC 不含凭证；fresh interpreter、FD allowlist、dump/core/ptrace/proc/no-new-privileges 均是 VISIBLE employee 放行门禁。
- Thread Context 固定 Thread → 群最近 → L1 → L2；分页失败/缺 scope 明确 CONTEXT_UNAVAILABLE，不用群 replay 冒充。History 终态以 Journal+blob 为权威，`history/YYYY-MM-DD.jsonl` 是可重建投影。
- 状态卡首次通过稳定 UUID 幂等创建，再由 child-owned CardStreamController 更新；主 Bot delivery 与员工 delivery 永不 fallback。READY_PENDING_VERIFICATION 使用受限 VerificationRouter 避免激活死锁。
- 设计文档：`docs/2026-07-12-autonomous-agent-department-design.md`，包含 24 节架构、数据、权限、迁移、SLO、故障注入和真实租户发布门禁。

## 员工凭证 AES-GCM Vault 与 Key Rotation

### 更改与原因
- 新增 `CredentialKeyring`，从 `SecretStr` 内的严格版本化 JSON 加载 AES-256 rotation set；拒绝未知顶层字段、非 v1、重复 key ID、非法 Base64、非 32-byte key、空配置和缺失 active key，异常不携带 key material。
- 新增 `CredentialVault` / immutable `CredentialReceipt`。凭证 ref 仅由 `hire_intent_id|attempt_id` 的 SHA-256 生成，secret 使用 AES-GCM 加密，canonical 非秘密身份字段作为 AAD，错误仅暴露异常类和 credential ref。
- envelope 固定 11 个字段，支持 resolve、active-key rewrap、orphan receipt 扫描和幂等 destroy；后续 Hire/Channel 只能消费 `credential_ref`，不读取 envelope JSON。
- 密文目录强制 `0700`，同目录临时文件强制 `0600`，写入执行 file flush/fsync、`os.replace`、directory fsync；删除后 directory fsync。配置新增 fail-closed keyring/active key 和凭证目录，依赖锁定 `cryptography==49.0.0`。
- 保持 `autonomous_visible_employee_limit=0` 的生产限制不变；本 Foundation task 不增加不完整的 readiness attestation，也不提前实现 workforce Journal/authority 接线。

## Journal 员工投影与身份物化

### 更改与原因
- `ProjectionState` 扩展为 Journal-backed workforce 投影，重放员工、Bot Principal、名称保留、legacy alias/source hash 与 authority epoch；冻结域对象只通过 `dataclasses.replace()` 更新。
- 所有 workforce 命令先校验实时投影 cursor 与 Journal head 一致，再在隔离容器副本上验证完整事务；单写者锁内使用 Journal 条件 head commit，认证且 anchored 后才应用实时投影。并发推进或语义拒绝均不会追加毒化帧，也不会污染实时状态。
- Bot 绑定强制同一事务同时包含 employee aggregate 的 `employee.bot_principal_bound` 与 principal aggregate 的 `bot_principal.bound`，expected-version 栅栏覆盖两个 aggregate；名称按 tenant + casefold 唯一，ARCHIVED 名称保留 tombstone，App ID 不可绑定到另一未归档员工。
- 重放在应用 frame 前验证 Bot 配对、canonical 且不相交的 `agt_`/`bot_` ID、跨租户/空绑定与重复事件类型；同 aggregate 的不同事件仍可原子提交，员工/principal 版本最终以 Journal frame 的 aggregate version 为准。ARCHIVED 为员工终态，任何 profile/membership/alias/binding 变更都不能释放名称 tombstone。
- `EmployeeIdentityMaterializer` 使用显式非秘密字段 allowlist，排除 `budget_template` 等任意嵌套数据；只接受 canonical `agt_` 路径组件，并以逐组件 dir-fd + `O_NOFOLLOW` 锚定 root/员工目录和临时文件，执行 `0600` file fsync、原子 rename 和 directory fsync，从投影确定性重建身份文件。Task 4+ 的 Authority/Registry runtime 未提前接线。

## 全局员工 Registry 与 Writer Authority 栅栏

### 更改与原因
- 新增只读 `ProjectedAgentRegistry`，所有 `get/find_by_name/list_agents/as_slock_identity` 查询均强制携带 `tenant_key`；群成员过滤只改变视图，不复制或重定义全局员工身份。Slock adapter 每次生成新的 `AgentIdentity`，只投影工具、模型、persona、权限、成员和可重建路径，不暴露 App ID 或凭证引用。
