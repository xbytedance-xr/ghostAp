## 生产级自主 Agent 部门设计收敛

### 现状审计
- 三路只读子代理分别审计身份/生命周期、飞书 transport、运行时/路由；共同确认当前生产仍是“主 Bot + Slock 虚拟 Agent”，`src/autonomous/` 的 577 项测试验证安全内核组件，但没有证明 production composition、独立员工 Bot、Channel、Slash、Router、`/fire`、按日历史或 Thread Context。
- 当前 `lark-oapi==1.6.5` 的真实 `register_app/aregister_app` 签名不含 `app_preset/addons/create_only/app_id`；隔离验证官方 `1.7.1` wheel 已包含这些参数。设计固定升级并以签名 contract + 真租户行为双门禁 fail-close，不凭网页文档猜 API。
- `lark-channel-sdk==1.1.0` 的底层 WS Client 使用模块级全局 event loop，多 Bot 同进程存在 ownership 风险；设计采用每员工 fresh-interpreter Channel 子进程，主进程保留 Journal/Router/Slock runtime。

### 方案与关键决策
- 比较“直接扩展旧 Slock”“v5 Journal SSOT + Slock runtime adapter”“每员工完整 GhostAP 服务”三案，采用第二案：Journal 是员工/Saga/membership/Inbox/Outbox/Effect 唯一事实源，AgentRegistry 和文件布局是可重建 Projection，独立员工消息通过类型化 gateway 复用原 `_run_acp_session`。
- `AgentIdentity` 只保存 `app_id + credential_ref`；secret 使用 AES-GCM 文件 Vault、`0700/0600`、原子 fsync/rename、key ID/轮换/orphan receipt，master key 来自 `src/config/.env`。这是对原“凭证存入 AgentIdentity”的生产安全等价解释。
- Hire/Fire 都是可恢复 Saga；所有外部步骤 PREPARED/EXECUTING 先 fsync+anchor，未知结果进入 ACTION_REQUIRED，禁止假成功或盲目重试。现有 ACP 无跨崩溃幂等/查询能力，因此 RPO=0 只承诺事件不丢、不重复建 task，不宣称 ACP 副作用 exactly-once。
- Channel child 通过一次性 pipe 仅取得自己的 secret，常规 IPC 不含凭证；fresh interpreter、FD allowlist、dump/core/ptrace/proc/no-new-privileges 均是 VISIBLE employee 放行门禁。
- Thread Context 固定 Thread → 群最近 → L1 → L2；分页失败/缺 scope 明确 CONTEXT_UNAVAILABLE，不用群 replay 冒充。History 终态以 Journal+blob 为权威，`history/YYYY-MM-DD.jsonl` 是可重建投影。
- 状态卡首次通过稳定 UUID 幂等创建，再由 child-owned CardStreamController 更新；主 Bot delivery 与员工 delivery 永不 fallback。READY_PENDING_VERIFICATION 使用受限 VerificationRouter 避免激活死锁。
- 设计文档：`docs/2026-07-12-autonomous-agent-department-design.md`，包含 24 节架构、数据、权限、迁移、SLO、故障注入和真实租户发布门禁。

## `/hire` / Slock 员工级联选模回调恢复

### 根因与修复
- Autonomous 生产缺口审计扩大到 Slock 交互回归时，稳定复现带 Profile/Effort 的 `/hire` / `/new-role` 模型选择在 `handle_new_role_select_tool()` 读取未初始化 `model_page`，直接抛 `UnboundLocalError`。
- 更深一层根因是 7 月 12 日把 Slock 接入共享 ACP 级联卡时，只复用了卡片外观；级联的 group/profile/effort/confirm/refresh action 仍硬编码为普通编程模式，员工名和 `global_hire` payload 也没有贯穿。因此即使绕过异常，确认模型也会进入普通 ACP 模式，而不是继续员工创建。
- 共享 `build_acp_model_cascade_card()` 新增默认保持兼容的 action/value 扩展点；Slock 使用独立的 group/profile/effort action，并在每次无状态重绘中保留 `role_name`、`global_hire`、工具和 pending 选择。只有最终确认 action 才进入现有角色创建回调，刷新仍回到 Slock 工具/模型发现链路。
- `model_page` 在分支前统一解析，普通分页卡与级联卡均可安全到达共同的投递逻辑。

### TDD 与验证
- RED 1：原回归测试先稳定得到 `UnboundLocalError: model_page`。
- RED 2：修复变量作用域后，测试继续证明级联卡没有 `slock_new_role_select_model`、员工 payload，且不存在 Slock 级联重绘 handler。
- GREEN：员工级联确认携带 `gpt-5/high` 或重绘后的 `gpt-5/xhigh`，dropdown change 不创建角色；action registry 覆盖三种新回调。
- `tests/test_slock_role_creation.py tests/test_model_cascade.py tests/test_action_dispatch_mapping.py tests/test_card_builders.py` → `143 passed`。
- 相关生产/测试文件 `ruff` 与 `git diff --check` 通过。
