## Spec 指令机制优化：系统指令优先级 + 卡片交互 + BUG 修复

### 任务描述
优化 Spec 模式下的指令优先级体系、卡片交互流畅度，修复 resume 方法中的严重 BUG。

### 执行内容
1. **系统指令门控精细化**：新增 `_READONLY_CARD_ACTIONS` 集合（18 个只读动作），重组 `_handle_card_action` 中 gate check 逻辑顺序，只读展示类卡片动作（展开/折叠/模式切换）不受 `/help`、`/exit` 处理期间的门控阻塞
2. **引擎控制动作优先级提升**：在 `_is_system_card_action` 中添加 9 个引擎控制动作（pause/stop/resume × deep/loop/spec），获得 HIGH 优先级和独立系统线程池调度
3. **修复 resume max_cycles 零轮执行 BUG**：将 `max_cycles` 从绝对值改为相对追加（`max_cycles = start_cycle + additional_cycles - 1`），确保 resume 后总能获得额外循环
4. **修复 resume 终止处理不一致**：抽取 `_handle_max_cycles_termination()` 方法，统一 execute/resume 的三分支终止逻辑（核心标准满足→PAUSED、infinite_mode→PAUSED、其他→ABORTED）
5. **代码质量清理**：清除 execute() 中双重 `from ..utils.errors import get_error_detail` 和双重 `import gc` + `gc.collect()`

### 技术要点
- 系统指令在调度层已有四重保障（队列隔离、独立线程池、项目串行豁免、HIGH 优先级），无需修改调度器
- 卡片动作的 gate check 需在 action_type_preview 提取之后执行（代码重排序）
- resume 的 max_cycles 应基于 start_cycle 相对计算，而非使用全局绝对值
- `test_card_deduplication::test_cache_cleanup` 为预存在 flaky 测试（git stash 验证原代码也失败），与本次变更无关
