## 修复 Deep 错误降级回复参数错位

### 任务描述
- Deep 模式执行异常后，日志出现 `BaseHandler.reply_text() missing 1 required positional argument: 'text'`
- 错误降级再次失败时，日志继续出现 `BaseHandler.reply_text() got multiple values for argument 'message_id'`

### 执行内容
- **根因 1**: `src/feishu/renderers/base.py` 的 `BaseRenderer.create_session()` 在 CardSession 创建失败兜底时，只传了文案，漏传 `message_id`
- **根因 2**: `src/feishu/handlers/engine_base.py` 的 `BaseEngineHandler._on_engine_error()` 在纯文本降级分支把正文作为位置参数、同时又用关键字传 `message_id`，触发参数冲突
- **修复**: 统一改为 `reply_text(message_id, text)` 调用顺序，并补充两处回归测试锁定降级路径
- **涉及文件**: `src/feishu/renderers/base.py`、`src/feishu/handlers/engine_base.py`、`tests/test_base_renderer_session_failure.py`、`tests/test_renderer_refactor.py`

### 技术要点
- `BaseHandler.reply_text` 的签名是 `reply_text(message_id, text, *, reply_in_thread=None)`，所有降级路径都必须严格按该约定传参
- 降级异常路径也需要单测覆盖，否则主流程测试通过并不能发现错误处理链路的参数错位

## 修复话题回复 pending 误判 + 卡片 text_color 属性不兼容

### 任务描述
1. 用户在话题中回复消息时，bot 错误地返回"请发送编程需求，将自动创建编程话题"提示，而不是实际执行编程
2. 用户输入指令后卡片一直显示"正在思考..."，不产生任何回复

### 执行内容

#### Bug 1: 话题回复 pending 误判
- **根因**: `_dispatch_empty_text` 中，当消息来自话题（有 `root_id`）但话题不在 `_thread_manager` 中时（如 Spec 引擎创建的话题），`_is_one_shot_pending` 仍然判定为 pending 并返回无用提示
- **修复**: 在 `_dispatch_empty_text` 中增加 `root_id` 参数，当 `root_id` 存在（用户已在话题中）时跳过 pending 检查，直接转发给 handler 处理
- **涉及文件**: `src/feishu/ws_client.py`

#### Bug 2: 卡片 text_color 属性不兼容
- **根因**: 飞书卡片 markdown 组件不支持 `text_color` 属性（错误码 200621），导致卡片创建失败，后续所有更新都因 message_id 无效而失败
- **修复**: 移除所有 `text_color` 属性
- **涉及文件**: `src/card/render/footer.py`、`src/card/render/renderer.py`、`src/card/render/worktree.py`、`tests/test_worktree_render_direct.py`

### 技术要点
- 飞书卡片 markdown 组件支持 `text_size`、`text_align`，但**不支持** `text_color`
- `_dispatch_empty_text` 路径只在文本解析为空时触发（post 类型消息中 @bot 后没有 text 元素时）
- Spec 引擎通过 `default_reply_mode="thread"` 在话题中展示结果，但不注册 ThreadContext，导致后续在该话题中的回复无法被 thread_manager 识别
