## 员工群账本重试、幽灵 Slock 与空卡片修复

### 日志证据与根因

- 08:03 的真实 TraeX 员工任务已在私有 workspace 成功启动，模型和 `reasoning_effort` 均应用成功；员工模式当前阻塞点已不是 TraeX 启动。
- 08:04:45 同一条 `/role list` 由主 Bot 和两个员工 Bot 分别观察；员工 Ingress 中两个 app-scoped Open ID 不同，但 `sender_union_id` 相同。员工路径直接将 Open ID 写入已由主 Bot 锚定的 Group Ledger，触发 `group event idempotency conflict`。
- 未终结 Ingress 在 worker 每约 5 秒重试，投影失败又发生在 Router 之前，因此两条记录永久无法进入 disposition，形成无限 `GroupLedgerError` 日志。
- 飞书 200621 明确指向 `ROOT -> body -> elements -> [2]`；本地复现发现 `build_empty_state_card()` 将 `build_responsive_layout()` 返回的 list 用 `append` 塞入 `elements`，导致顶层元素变成二维数组。
- 启动恢复的两个 Slock 来自两个本地 `.slock_channel.json`。Journal 序号 800 已确认员工从旧群 `oc_bb609...` 移除，但主 Bot `bot_deleted` 回调是空实现，旧 marker 未归档，导致重启不断复活“女主天团”引擎并产生空 Registry bootstrap 警告。
- 09:10 重启后的 `main Bot reply recipient scope is unavailable` 来自同一个持久化 `hire.notification:*:ready` Effect：进程重启会清空内存 MessageLinker，而恢复器继续回复旧 `/hire` 消息，出站审计因缺少可信 recipient provenance 正确地默认拒绝并指数重试。
- 同一窗口的 `employee data recovery failed closed: WorkspaceLintError` 来自 TraeX 在私有 `runtime/trae-home` 内生成的后端会话目录。私有边界仍为 `0700`，但后端子目录使用常规 `0755`/`0644`；旧 lint 错误地要求所有后代目录都为 `0700`，将安全的后端布局误判为恢复阻断。

### 实现

- 员工派发在 Router 前先消费员工专用控制，再将主 Bot 所有的群斜杠命令以 `main_bot_group_command` 幂等标记为 ignored；普通群文本以及内部 `direct:` / `team:` Ingress 仍保持既有队列语义。
- 员工 Group Ledger 投影前，使用激活 Hire 状态中的 owner principal 和可信 `sender_union_id` 将 employee-app Open ID 归一为主 Bot owner Open ID；无法证明同一 owner 时默认拒绝投影。
- 空状态卡片将响应式布局由 `append` 改为 `extend`，保证 `body.elements` 顶层只包含卡片 element object。
- 主 Bot 收到群移除事件时，无条件先原子归档 Slock marker，再按存在性停止内存 Engine；即使 Engine 未加载也会清理 marker。归档或拆除失败不再吞异常，保留事件失败/重投语义。
- 群移除的归档、全索引 Engine 拆除和路由注销统一持有 Slock 激活锁；创建、激活和登记改为原子 manager API，并在当前进程中永久拒绝重新激活已收到删群事件的 chat，关闭 marker 被并发重建的窗口。
- 空 Registry 仅在确实有排队任务时执行 bootstrap 重试并记录 `tasks retained` WARNING；没有排队任务的恢复群直接进入待命，避免将正常空闲状态误报为故障。
- Hire 通知恢复前从持久 Hire 状态恢复 message/chat/requester/tenant provenance，并通过官方 Chat API 验证会话类型；已有 provenance 只接受完全一致的作用域，冲突继续默认拒绝。回复期间显式恢复 tenant 上下文，结束后还原线程上下文。
- Workspace lint 继续严格要求员工根、`.trae` 与 `runtime/trae-home` 私有边界为 `0700` 并拒绝 symlink，但允许这些私有边界内部的后端自有后代使用常规文件/可执行权限。

## 员工 TraeX 独立状态与持久 ACP 工作目录修复

### 背景与根因

- 最近一次真实员工消息已完成 Journal、Ingress 和 Router 派发，但 TraeX 在模型配置后连续拒绝 `reasoning_effort=high`，最终以 `employee_session_failed` 终态结束。
- 对管理进程与员工进程分别做无提示词 ACP 探测后确认：模型/Profile/Effort 组合本身合法；员工 `HOME` 下的 TraeX 没有登录态和模型能力，`traex login status` 为未登录，因此只暴露空模型占位并拒绝 Effort。
- 仅共享管理进程的整个 `TRAE_HOME` 虽可恢复模型配置，但会让员工共享状态目录，破坏员工记忆与数据隔离，也可能把认证带入离职归档。
- 进一步核对真实调用链发现，持久 Employee Actor 虽有独立 `HOME` 和 workspace，ACP 进程仍以 Slock 项目根作为 `cwd`；这会让默认会话状态落在团队项目，而不是员工自己的持久数据目录。
- 同一日志窗口的 Lark keepalive/open handshake 错误均自动重连；旧 Slock 空 registry 警告来自兼容群，与本次持久员工模型启动失败无直接因果关系。

### 实现

- 新增 `AUTONOMOUS_EMPLOYEE_TRAEX_AUTH_HOME`，它只描述本机 TraeX 登录元数据来源，不再作为员工进程共享状态目录。
- TraeX 员工派发时创建独立 `<employee>/runtime/trae-home`，将验证后的 `cli/auth.json` 原子投影到员工目录，并设置该路径为进程 `TRAE_HOME`；Coco、Codex、Claude、Gemini 和 TTADK 等其他后端不会收到或创建 TraeX 状态目录。
- 将员工 canonical `workspace/AGENTS.md` 同步写入员工 `TRAE_HOME` 根目录，作为 TraeX 在该数据根读取的约束。目录使用 `0700`，认证与约束文件使用 `0600`。
- 投影入口使用 no-follow 打开、当前用户所有权与权限检查、大小上限、重复 JSON key 拒绝、非空 token 检查、原子替换和 fsync；错误对外统一为不含认证内容的安全错误。
- Fire 原子归档在生成 manifest 和移动员工目录前，以 descriptor-relative/no-follow 方式删除员工 TraeX 认证副本；`AGENTS.md` 与非秘密员工数据继续正常归档。
- Employee Actor 和兼容的一次性员工 ACP 会话都以规范化后的 `agent.workspace_path` 作为启动 `cwd`；workspace 不存在时 fail-close。普通非员工 Slock 会话保持项目根目录语义。
- 全部修改直接在 `dev` 完成，没有新建开发分支。
