## 修复 Spec Engine 与卡片交互的两个 KeyError 及状态同步问题

### 任务描述
发现日志中存在以下两个紧密相关的错误：
1. `Spec执行异常: Phase build 失败，任务已保存(task_id=...): Internal error`
2. `操作失败 (spec_expand): 'completed'` 以及 `❌ 操作失败 (spec_expand_ac): 'completed'`

在卡片交互和异步阶段更新期间发生状态冲突和键错误，导致整个 Phase 崩溃，产生中断与任务保存。

### 执行内容
1. **分析错误链**:
    - 分析发现 `❌ 操作失败 (spec_expand_ac): 'completed'` 是由 `_process_card_action_async` 捕获的异常信息，错误根源为 `KeyError: 'completed'`。
    - 追溯到 `src/feishu/renderers/spec_renderer.py` 的 `_render_status_view` 方法，其中 `progress_info["completed"]` 和 `progress_info["total"]` 试图访问 `progress_info` 中不存在的键（应为 `satisfied_count` 和 `total_criteria`）。
    - 这个异常由于在 `SpecEngine` 处于 `RUNNING` 状态下伴随着内部进度汇报触发（经由 `renderer.process_event` 或卡片动作 `show_spec_status`），导致未捕获异常冒泡到 `SpecEngine._run_phase` 中。
    - `SpecEngine._run_phase` 会把未捕获异常转为 `Phase {phase.value} 失败...` 的 RuntimeError，从而向外抛出。

2. **修复 ActionDispatcher 漏注问题**:
    - 检查发现 `ws_client.py` 中 `_FORWARDING_MAP` 字典里遗漏了 `_toggle_spec_ac` 方法的转发声明。
    - 补充 `"_toggle_spec_ac": ("_spec_handler", "toggle_spec_ac")` 至映射中，确保 `spec_expand_ac` / `spec_collapse_ac` 能够正确被底层 `BaseHandler` 分发并调度。

3. **修复 SpecRenderer 字典键错误**:
    - 修改 `src/feishu/renderers/spec_renderer.py` 中 338 行附近的进度条生成代码：
      ```python
      # 修改前
      progress_bar = self._generate_progress_bar(progress_info["completed"], progress_info["total"])
      # 修改后
      progress_bar = self._generate_progress_bar(progress_info["satisfied_count"], progress_info["total_criteria"])
      ```
    - 清除了由于点击卡片而意外触发引擎阶段崩溃的问题。

4. **测试验证**:
    - 执行了 `uv run python -m pytest tests/test_spec_interaction.py -v` 验证卡片动作分发。
    - 执行了 `uv run python -m pytest tests/test_spec_engine.py -v` 验证全量 Spec 引擎（155个测试）。
    - 所有测试通过，无警告（系统已知警告除外）。

### 技术要点
- **异常冒泡中断 Engine 循环**: UI 渲染层（Renderer）或者回调的微小异常，在异步环境中如果不加处理地冒泡，会导致核心控制流（如 Engine 的 Phase 循环）被终止，从而造成宏观上的“内部错误”。
- **动态方法查找风险**: 在基于统一的 `_dispatch_standard_card_action` 模式里，方法查找依赖字典注册、反射和精确的关键字解包，重构和调整时若未覆盖完整（如 `_FORWARDING_MAP` 的增补），会出现很难排查的 Silent Error 或者 Action NotFound 异常。

4. **体验升级：优雅降级 (Graceful Degradation)**
   - **问题**：原逻辑在 `get_models()` 失败时直接返回错误卡片，导致用户无法看到当前已选工具，且无法进行"切换工具"等补救操作，体验生硬。
   - **方案**：
     - 修改 `SystemHandler.show_ttadk_info`：捕获模型获取错误，不中断流程，而是将错误信息传递给卡片构建器。
     - 更新 `SystemBuilder.build_ttadk_info_card`：支持 `model_error` 参数。当模型获取失败时，在"当前模型"栏显示 `⚠️ <错误信息>`，但仍保留"当前工具"展示和"切换工具"按钮。
     - **效果**：即使后端模型服务暂时不可用，用户仍能看到工具状态，并有机会切换到其他工具，实现了"体面失败"。
