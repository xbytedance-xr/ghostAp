## Spec Build 工具流式与多角色审查降级修复

- 背景：现场 Spec 卡片在执行构建阶段长时间无更新，用户截图显示多角色审查里有大量超时和 `role output was not valid JSON` 作为可见 evidence。复核发现 Build 阶段只在 `TOOL_CALL_DONE` 后统计工具，`TOOL_CALL_UPDATE` 若缺少 start 块会被 reducer 忽略；`total=0` 的 progress label 也被清空，因此"工具调用次数/文件数"不会渲染到飞书卡片。
- 修改内容：
  - `src/feishu/renderers/_spec_stream_processor.py`：Build 阶段按唯一 tool id 统计工具调用；对 `TOOL_CALL_UPDATE`/`TOOL_CALL_DONE` 缺少 start 的情况合成 `TOOL_STARTED`，保证流式 delta 有可更新的工具块；从 `locations` 和工具内容中提取文件引用，持续派发 `🔨 N 次工具调用 · M 文件` 进度摘要。
  - `src/card/state/reducer.py`：`PROGRESS_UPDATED(total=0, label=...)` 保留文本型进度，不再清空 footer progress。
  - `src/spec_engine/adaptive_review.py`：角色输出非 JSON 时先用文本 fallback 提取 PASS/FAIL 与 bullet 建议；完全不可解析时作为非阻断 infra skip，不再把 `role output was not valid JSON` 暴露成审查建议 evidence。
  - `src/spec_engine/review_strategy.py`：adaptive role review 的选中 agent 超时时按候选 review agents 和默认 agent 降级重试；非超时异常仍保持原错误路径。
  - `src/spec_engine/review_display.py` / `src/spec_engine/reporter.py`：将汇总文案改为"多角色审查建议"，明确它来自同一轮多角色审查，并驱动下一轮优化，不是独立审查流程。

## Slock `/role move` 执行顺序修复 + L1 记忆跨群加载验证

- 背景：API 契约 Review 发现 `move_role()` 执行顺序存在缺陷——notification card 在 `registry.move_agent()` 之前发送，若 move 失败则目标群已收到假通知；同时需验证 L1 memory 在跨群 move 后能被独立 MemoryManager 正确加载。
- 修改内容：
  - `src/feishu/handlers/slock.py` — `move_role()` 方法重写执行顺序：
    - 旧流程：send notification card → move_agent → update context → confirm card
    - 新流程：move_agent → refresh target cache → send notification (best-effort) → update context → confirm card
    - `engine.registry.move_agent()` 现在在 `send_card_to_chat()` 之前执行，消除假通知风险
    - move 失败：直接 error reply，不发送任何 notification
    - notification 发送失败：降级警告 "✅ 移动成功，但目标群通知发送失败。目标群可通过 /role list 查看新成员。"，继续执行 context update 和 confirm card（不 rollback）
  - `tests/test_slock_move_agent.py` — 更新和新增测试类：
    - `TestNotificationSendFailureAbortsMove` 重写为新语义：move_agent 已执行，notification 失败不 rollback，发降级警告
    - `TestMoveRoleExecutionOrder`：验证 move_agent 失败阻止 notification、验证成功路径 call ordering
    - `TestNotificationFailureDegradation`：notification 返回 None 时源群收到降级警告、confirm card 仍发送
    - `TestL1MemoryFullyLoadableFromIndependentManager`：独立 MemoryManager 实例完整加载 L1、migration record 可见
    - `TestSystemPromptIdenticalAfterMoveStrengthened`：role/key_knowledge/system_prompt 在 move 前后一致
  - `tests/test_slock_move_agent_concurrency.py` — 新增 `test_concurrent_move_notification_isolation`：两线程竞争移动同一 agent，只有赢者的目标群收到 notification
- 设计决策：
  - "move first, notify second"：registry 持久化是权威操作，notification 是 best-effort UI 副作用
  - 不做 rollback：notification 失败时 rollback registry 会引入更高复杂度和风险（registry 已持久化成功），用降级警告 + `/role list` 查看替代
  - L1 memory 无需文件迁移：按 `agent_id` 寻址于 `~/.ghostap/slock/agents/{agent_id}/MEMORY.md`，天然跨群持久
- 验证：
  - `uv run python -m pytest tests/test_slock_move_agent.py tests/test_slock_move_agent_concurrency.py tests/test_slock_card_move_notification.py tests/test_slock_slash_commands_move.py -q` → 通过
  - Slock 专项 762 passed
  - 非 slock 核心测试采样通过，变更隔离于 handler 层，回归风险低
