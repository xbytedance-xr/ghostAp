## Deep/Loop/Spec 引擎优化：限速自适应 + 统一状态 + 时长显示 + task_id

### 任务描述
优化三种执行模式（Deep/Loop/Spec）的执行过程体验，解决三个 FAIL 验收标准：
1. **CRITERIA_2**: 实时执行时长和详细过程细节显示
2. **CRITERIA_4**: Claude API 限速自动暂停等待恢复
3. **CRITERIA_5**: 统一任务列表查询 + 按 task_id 查询详情

### 执行内容

**Step 1: 基础工具**
- `src/utils/text.py` 添加 `format_duration()` — 支持小时/分/秒格式化
- `src/utils/text.py` 添加 `generate_task_id()` — `{name}_{YYYYMMDD}_{HHMMSS}_{4hex}` 防碰撞

**Step 2: TaskScheduler task_id 支持**
- `TaskSpec` + `TaskEvent` 添加 `task_id` 字段
- `TaskScheduler` 添加 `_by_task_id` 索引 + `get_state_by_task_id()` 支持精确/模糊匹配

**Step 3: 引擎模型 task_id**

- `src/agent_session.py` 添加 `RateLimitAwareSession` 包装类：
  - 显式 SyncSession 协议委托（非 __getattr__）
  - 可中断 sleep（cancel_event 1秒轮询）
  - `rate_limit_until` 属性用于状态查询
  - 回调异常安全（try/except 包裹 on_rate_limit）
- `create_engine_session()` 接受 `on_rate_limit`/`cancel_event`，自动包装
- `tests/test_rate_limit.py` — 32 个测试覆盖检测/重试/取消/回调安全/包装

**Step 5: Reporter 时长显示**

**Step 6: Handler 集成 task_id + rate_limit**
- `deep.py`/`loop.py`/`spec.py` — `start_*()` 生成 task_id 并传入 TaskSpec + 引擎
- 三个 handler 创建 `_on_rate_limit` 回调闭包 → 发送限速通知卡片
- 三个引擎 `execute()`/`plan_and_execute()` 接受 `on_rate_limit` 并传给 `create_engine_session()`
- resume 路径也复用 `_on_rate_limit`（通过 `getattr(self, "_on_rate_limit", None)`）

**Step 7: 统一 /status 命令**
- `diagnostics.py` 添加 `show_unified_status()` — 跨 Deep/Loop/Spec 引擎聚合任务列表
- `diagnostics.py` 添加 `_show_task_detail()` — 按 task_id 搜索引擎实例 + scheduler 兜底
- `system.py` — `/status` 和 `/status <arg>` 路由到 `diagnostics_handler.show_unified_status()`
- 帮助文档更新，`is_interceptable_command` 添加 `/status ` 前缀匹配

### 技术要点
- **RateLimitAwareSession 设计决策**：显式协议委托而非 `__getattr__`（避免动态调度风险，ARCHITECT 审查建议）
- **task_id 碰撞防御**：4 位随机 hex 后缀（ARCHITECT 审查建议）
- **限速检测**：编译正则 + `\b429\b` 词边界避免误匹配（如 14291 不触发）
- **可中断等待**：`cancel_event.wait(timeout=1.0)` 轮询代替 `time.sleep()`（ARCHITECT 审查建议）
- **统一状态查询**：按 running→paused→done 排序，task_id 支持精确/后缀/子串模糊匹配（PRODUCT 审查建议）

## 编程模式卡片空白修复 + 完成摘要优化

### 任务描述
修复普通编程模式（Coco/Claude）卡片变成空白的 bug，并添加完成摘要。

### 根因
`handle_response()` → `renderer.get_final_content()` 返回 `""` → `close_streaming(card, final_content="")` 用空内容覆盖卡片。触发场景：Claude 只发 THOUGHT_CHUNK、所有 tool call title 为空、ACP 后端崩溃/超时。

### 执行内容

**Bug Fix 1: handle_response() 多级兜底** (`src/feishu/handlers/programming.py`)
- `renderer.get_final_content()` 返回空时，fallback 到 `result.text`
- `result.text` 也为空时，兜底 `"✅ 执行完成"`

**Bug Fix 2: close_streaming() 空字符串保护** (`src/card/streaming.py`)
- `final_content is not None` → `final_content`（空字符串也 fallback 到 `card.last_content`）

**产品优化: 完成摘要** (`src/acp/renderer.py` + `programming.py`)
- 新增 `render_summary()` 方法：返回 "🛠️ N 次工具调用  ·  🗂️ M 个文件" 格式
- `handle_response()` 在 close_streaming 前追加摘要行

**测试新增** (6 个)
- `test_acp_renderer.py`: render_summary 三场景 + get_final_content 空场景两个
- `test_streaming.py`: close_streaming 空字符串 fallback

### 技术要点
- 三级 fallback: renderer → result.text → 默认文本
- `close_streaming` 的 `final_content is not None` 改为 truthy 检查，空字符串等价于 None
