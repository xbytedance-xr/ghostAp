# 2026-06-17

## WF 模型选择 & 跨工具模型串台修复

### 背景 / 症状
- `/wf` 模式下选编排/评审 Agent 时，traex、coco 等 ACP 工具的**模型列表为空**，只能用默认模型。
- 日志反复出现：
  - `ProbeStrategy: output is not an invalid model error for tool=traex (rc=1)`
  - `[TTADK] models_fallback: tool=traex source=defaults`
- deep（coco 后端）执行报 `执行异常: Internal error`（`src.engine_base`），retry 2 次后失败。

### 根因
1. **WF 模型 provider 写死**：`src/feishu/handlers/workflow.py` 的 `_get_workflow_models_for_tool()` 调用
   `WorktreeToolDiscovery.get_models_for_tool(tool, provider="workflow", ...)`。但
   `get_models_for_tool` 只对 `provider=="acp"` 走 ACP 模型获取，其它一律走 TTADK 子进程探测。
   ACP 工具（traex/coco/...）被 TTADK probe 抓模型 → probe 失败 → 返回 `[]` → WF 卡片无模型可选。
2. **跨工具模型串台导致 Internal error**：`src/engine_base.py:_resolve_identity()` 缺少 `traex`
   分支，`engine_name="Traex"` 落入 `else` → `agent_type="coco"`。而 deep/spec 又单独从项目读取
   traex 的 `acp_model_name`（如 `c_o_new_thinking`）并透传，结果 **coco 后端收到 traex 的模型** →
   coco 不认识 → `Internal error`（用 coco 合法模型 `openrouter-3o` 则正常）。

### 改动
- `src/feishu/handlers/workflow.py`：`_get_workflow_models_for_tool` 先用
  `WorktreeToolDiscovery.get_available_tools()` 查出该 tool 的真实 provider（acp/cli/ttadk），
  再传给 `get_models_for_tool`，使 ACP 工具走 ACP 模型路径。
- `src/engine_base.py`：`_resolve_identity` 增加 `normalized.startswith("traex") -> InteractionMode.TRAEX`
  分支，使 Traex 引擎解析为 `agent_type="traex"`，agent 与 model 一致，不再把 traex 模型喂给 coco。
- 注：`engine_identity.resolve_engine_identity` 的 COCO 分支早已有 `acp_tool_name=="coco"` 归属
  校验，无需改动（双重保险：即便上游误传，coco 也不会采用他工具模型）。

### 验证
- `uv run python -m pytest tests/test_engine_identity.py tests/test_engine_base_di.py
  tests/test_worktree_tool_discovery.py tests/test_workflow_pending_model.py
  tests/test_workflow_tool_select.py tests/test_acp_provider_extensions.py -q` → 82 passed。
- `tests/test_workflow_orchestrator_select.py / test_workflow_security.py /
  test_workflow_selection_controller.py / test_deep_engine.py` → 122 passed, 1 skipped。
- 行为验证：`_resolve_identity('Traex')` 现为 `('Traex','traex',None)`（修复前是 coco）；
  COCO mode + traex 模型 → 模型被丢弃；`get_available_tools()` 中 traex/coco provider=acp。

### 后续风险
- WF 模型列表现依赖 `get_available_tools()` 的 provider 判定；若新增 ACP 工具需确保其在
  `_KNOWN_TOOLS` 注册，否则会回退 ttadk provider。
