## 修复 Spec Engine 收敛检测误判：review 异常轮次不应参与收敛比较

### 任务描述
Spec Engine 在循环 2 轮就触发收敛退出，但实际两轮都有 5 条审查建议应该继续循环。

### 根因分析
- `spec_convergence_window = 2`，只需连续 2 轮 suggestion 集合相同即判定收敛
- 循环 1 和循环 2 的 review 阶段都因 `TimeoutError` 失败
- 失败时 `conduct_review` 生成固定模板 fallback suggestions：每个视角一条 `"审查执行异常: TimeoutError (empty message)"`
- 两轮 fallback 文本完全相同 → `detect_convergence()` 的 `len(set(suggestion_sets)) == 1` 为 True → 误判为收敛
- 实际上这两轮根本没有执行真正的审查，不应参与收敛判定

### 执行内容
1. **修复 `src/spec_engine/convergence.py:detect_convergence()`**
   - 在遍历 recent cycles 构建 suggestion_sets 时，检查 `c.review_decision` 是否以 `"review_failed"` 开头
   - 如果是（timeout、熔断等异常情况），直接返回 `False`（不收敛）
   - 确保正常 review 的收敛检测逻辑不受影响

2. **新增测试 `tests/test_convergence.py::TestDetectConvergenceReviewFailed`**
   - `test_consecutive_review_failures_not_converged`：连续 2 轮 timeout 不判定为收敛
   - `test_one_failed_one_normal_not_converged`：1 轮异常 + 1 轮正常也不判定为收敛
   - `test_circuit_breaker_decision_not_converged`：熔断决策同样不参与收敛
   - `test_normal_reviews_still_converge`：正常 review 相同建议仍正确判定为收敛

### 技术要点
- review 异常时的 `review_decision` 字段固定以 `"review_failed"` 开头（`review_failed_continue`、`review_failed_open_circuit` 等）
- 收敛检测的本质是"LLM 反复给出相同建议却无法解决"，但异常产生的 fallback 文本不代表 LLM 的真实建议
- 修复方案最小化：仅在收敛检测中跳过异常轮次，不改变 review 异常处理本身的逻辑

## 改进 Spec Engine 审查超时体验：解决 TimeoutError (empty message) 不友好文案

### 任务描述
Spec Engine review 阶段 ACP 调用超时后，用户看到 `"审查执行异常: TimeoutError (empty message)"` 这样的不友好建议文案，且连续超时会浪费大量时间（每轮 120s×3=360s）。

### 根因分析
- `SyncACPAdapter.future.result(timeout=120)` 超时抛出的是 Python 原生 `TimeoutError()`，不带任何消息
- `review.py` 的 `_extract_error_text()` 拿到空字符串后生成 `"TimeoutError (empty message)"` 兜底文案
- review timeout 120s 硬编码，无法配置
- 熔断器默认关闭，连续 TimeoutError 时每轮都浪费 360s

### 执行内容
1. **`src/acp/sync_adapter.py`** — `except TimeoutError` 分支改为 `raise TimeoutError(f"ACP prompt 执行超时 ({timeout}s)") from e`
2. **`src/config.py`** — 新增 `spec_review_timeout: int = 120` 配置项；`spec_review_failure_circuit_enabled` 默认改 `True`
3. **`src/spec_engine/review.py`** — 三处改动：
   - `conduct_review` 中 `timeout=120` 硬编码改为 `timeout=settings.spec_review_timeout`
   - `build_review_exception_diagnostics` 中 `fail_reason=="timeout"` 时 error_text 改为 `"审查超时，将在下一轮重试"`
   - fallback ReviewResult 中 `fail_reason=="timeout"` 时 suggestions 改为 `"审查超时，跳过本轮审查继续执行"`
4. **`tests/test_spec_engine.py`** — 新增 7 个测试：
   - `test_timeout_error_with_message_diagnostics_has_friendly_text` — TimeoutError 带消息时 diagnostics 正确
   - `test_timeout_error_empty_message_diagnostics_uses_friendly_text` — TimeoutError 空消息时使用中文友好文案
   - `test_non_timeout_error_empty_message_still_uses_empty_message_fallback` — 非 timeout 异常不受影响
   - `test_spec_review_timeout_config_exists_and_defaults` — 配置项存在且默认 120
   - `test_spec_review_timeout_passed_to_send_prompt` — 配置项正确传递到 send_prompt
   - `test_timeout_fallback_review_result_has_friendly_suggestion` — timeout fallback 使用友好文案
   - `test_non_timeout_fallback_review_result_keeps_generic_format` — 非 timeout fallback 保持原格式

### 技术要点
- `_infer_fail_reason` 使用 `isinstance(err, TimeoutError)` 判断，不依赖消息内容，sync_adapter 修改不影响
- 收敛检测已通过 `review_decision.startswith("review_failed")` 跳过异常轮次（commit e1b99c4），文案变更不影响
- 熔断器默认开启（`max_consecutive=3, cooldown=3`），连续超时时自动跳过审查节省时间

## Worktree Engine TimeoutError 加固：推广 spec_engine 三层防御到 worktree_engine

### 任务描述
将 spec_engine 中已修复的 TimeoutError(empty message) 防御模式推广到 worktree_engine，避免相同漏洞在并行执行引擎中复现。

### 根因分析
- `worktree_engine/dispatcher.py` 的 `_run_single_unit` 中没有 `except TimeoutError` 分支
- 空消息 `TimeoutError("")` 经过 `str(exc)` 后变为空串，导致 `unit.error` 为空
- `execute_units` 外层和 `manager.py` 的 `state.last_error` 也存在同样的空串问题
- `reporter.py` 虽有 `unit.error or "执行失败"` 兜底，但错误信息丢失影响排查

### 执行内容
1. **`src/worktree_engine/dispatcher.py`** — 两处修改：
   - `_run_single_unit`：在 `except Exception` 前新增 `except TimeoutError as te` 分支，`unit.error = str(te).strip() or f"执行超时 ({timeout}s)"`
   - `execute_units`：外层 `except Exception` 中 `unit.error = str(exc).strip() or "执行异常"`

2. **`src/worktree_engine/manager.py`** — `execute_goal` 的 `except Exception` 中 `state.last_error = str(exc).strip() or "执行异常"`

3. **`tests/test_worktree_dispatcher.py`** — 新增 3 个测试：
   - `test_timeout_error_empty_message_unit_has_friendly_error`：`TimeoutError('')` → unit.error 非空且含"超时"
   - `test_timeout_error_with_message_preserves_original`：`TimeoutError('ACP prompt 执行超时 (120s)')` → 保留原始消息
   - `test_generic_exception_empty_message_has_fallback`：`Exception('')` → unit.error 非空（兜底文案）

4. **`tests/test_worktree_reporter.py`** — 新增 1 个测试：
   - `test_reporter_timeout_unit_shows_timeout_summary`：超时失败 unit 的 summary 展示包含超时信息

### 技术要点
- `TimeoutError` 是 `Exception` 子类，`except TimeoutError` 必须在 `except Exception` 之前
- `str(TimeoutError(""))` 在 Python 3.12 中返回空串，`.strip() or fallback` 模式确保非空
- worktree 并行执行中单 unit 超时不影响其他 unit，错误隔离完好
- 与 spec_engine 的区别：worktree 无收敛检测和熔断器需求（单轮并行执行，非迭代循环）

## Loop Engine 结构化审查诊断：与 Spec Engine 对齐

3. **`src/loop_engine/engine.py`（修改）** — 三处改动：
   - `LoopReviewCircuitState` 新增 `last_review_failure_diag: Optional[dict] = None` 字段
   - `_conduct_review` except 分支调用 `build_review_exception_diagnostics` 生成结构化 diag dict，存储到 `circuit.last_review_failure_diag`
   - 日志改用 `format_review_exception_log_line(diag, diag_json=..., prefix="[Loop]")` 结构化格式
   - 熔断器打开时补充 `decision/review_circuit_open/open_until_iter/consecutive_failures` 到 diag
   - try 前 `circuit.last_review_failure_diag = None` 清除旧值（与 Spec 一致）
4. **`tests/test_loop_review_circuit.py`（修改）** — 新增 6 个测试：
