## Loop 多视角审查输出解析 — 三级容错策略

### 任务描述
Loop 模式多视角审查频繁出现"审查输出解析失败，请检查实现质量"错误，所有四个视角（架构师/产品经理/用户/测试）都显示解析失败。原因是 LLM 输出格式变体超出了现有两级正则解析的覆盖范围。

### 执行内容

**根因分析**：`_parse_review_output()` 有两级正则解析：
1. 严格模式：`[TAG]\nPASS/FAIL` 格式
2. 容错模式：`[TAG] PASS`、`## [TAG]`、中文标题格式

但 LLM 经常产出以下未覆盖格式：
- `**[ARCHITECT]**: PASS` (加粗方括号)
- `**ARCHITECT**: PASS` (加粗无方括号)
- `### ARCHITECT` (Markdown 标题无方括号)
- `ARCHITECT: PASS` (裸标签加冒号)
- `**架构师**: PASS` (加粗中文名)
- `架构师审查: PASS` (中文名带后缀)
- `1. 架构师: PASS` (编号列表)

**解决方案 — 三级容错**：

1. **Tier 1（原有）**：严格正则 `_REVIEW_SECTION_PATTERN` — 标准格式快速路径
2. **Tier 2（增强）**：`_REVIEW_HEADER_EN_PATTERNS`（5个模式列表）+ 增强版 `_REVIEW_HEADER_ZH_PATTERN`
   - 新增支持：加粗方括号、加粗无方括号、Markdown标题、裸标签冒号、加粗中文、编号列表、中文后缀
3. **Tier 3（新增）**：LLM 兜底解析 `_parse_review_with_llm()`
   - 当正则全部失败时，用 ChatOpenAI 提取 JSON 格式审查结果
   - 遵循项目 "AI Native first" 原则
   - 解析 JSON 中代码块包裹的情况

**诊断日志增强**：
- 正则全部失败时记录原文前500字符，方便调试

### 修改文件
- `src/loop_engine/engine.py` — 正则模式增强 + LLM 兜底方法 + 诊断日志
- `tests/test_loop_engine.py` — 新增 12 个测试用例

### 技术要点
- EN 模式从单个 `_REVIEW_HEADER_EN_PATTERN` 改为 `_REVIEW_HEADER_EN_PATTERNS` 列表（5个模式），避免复杂交替正则的分组序号混乱
- `_split_review_sections()` 使用 `seen_positions` 集合避免多模式重复匹配
- ZH 模式增加 `\*{0,2}` 支持加粗、`(?:审查|评审|视角)?` 支持中文后缀、`(?:\d+[.、)]\s*)?(?:[-*]\s*)?` 支持编号/列表
- LLM 兜底解析复用已有 ChatOpenAI 配置（ark_api_key / ark_model），temperature=0.0 确保稳定输出
- `_extract_reviews_from_llm_response()` 处理 JSON 代码块包裹、无效 JSON、PASS 视角清空 suggestions

### 测试覆盖
- 814 测试全通过（原 802 + 新增 12）

## Deep 模式 TodoWrite 卡片内容丢失修复

### 任务描述
Deep 模式执行过程中，消息卡片更新时出现大量 "TodoWrite..." 条目，但实际的 todo 修改内容（任务列表）从未展示。最终结果卡片信息完整，但过程中卡片只显示工具名称和省略号，用户无法看到 agent 的任务管理进展。

### 根因分析
1. **`_parse_tool_call()`** — 从未提取 `raw_input`：ACP SDK 的 `ToolCallStart`/`ToolCallProgress` 有 `raw_input` 字段包含工具参数，但解析时只提取 title/kind/status/locations，`ToolCallInfo.content` 始终为空
2. **`ACPEventRenderer`** — 无 TodoWrite 意识：活跃工具统一渲染为 `{icon} {title}...`，完成后追加 `{icon} {title} ✅` 到 `_text_chunks`
3. **文本缓冲区污染** — TodoWrite 完成行（`🔧 TodoWrite ✅`）大量写入 `_text_chunks`，`_tail(text_content, 1400)` 截断后有意义的文本被挤掉

### 执行内容
**方案：TodoWrite 内容提取 + 独立渲染区块**

1. **`src/acp/client.py`** — 新增 `_format_todo_content()` 和 `_is_todo_tool()` 辅助函数
   - `_is_todo_tool`: 通过 title 含 "todo" 或 raw_input 含 "todos" key 检测
   - `_format_todo_content`: 将 TodoWrite 的 raw_input 格式化为可读 checklist（✅/🔄/⏳ 图标）
   - `_parse_tool_call`: 提取 `raw_input`，对 TodoWrite 工具填充 `ToolCallInfo.content`

2. **`src/acp/renderer.py`** — 新增 `_todo_content` 跟踪 + 独立渲染
   - `_todo_content: str` 字段：持久保存最新 TodoWrite 内容
   - TOOL_CALL_START/UPDATE/DONE：若 `tool_call.content` 非空则更新 `_todo_content`，完成时不写入 `_text_chunks`
   - `_render()`: 新增 "📝 任务进度" 区块（plan→todo→active tools→text）
   - `render_plan_view()`: 同步包含 todo 区块
   - `_render_active_tools()`: 跳过有 content 的工具（已在 todo 区块展示）
   - 新增 `todo_content` property

3. **`tests/test_acp_renderer.py`** — 新增 12 个测试
   - TodoWrite 内容跟踪（start/done/update）
   - TodoWrite 不污染 text_chunks
   - TodoWrite 不出现在 active tools 渲染
   - plan_view 包含 todo 内容
   - todo 内容跨工具调用持久化
   - 多次 TodoWrite 只保留最新
   - raw_input 解析（含 activeForm 优先展示）

### 修改文件
- `src/acp/client.py` — `_parse_tool_call` 增强 + 2 个新辅助函数
- `src/acp/renderer.py` — `_todo_content` 跟踪 + 渲染逻辑
- `tests/test_acp_renderer.py` — 12 个新测试

### 技术要点
- TodoWrite 检测双保险：title 匹配（`"todo" in title.lower()`）+ raw_input 结构匹配（`"todos" in raw_input`）
- in_progress 状态优先使用 `activeForm` 展示（更具可读性）
- `_todo_content` 在 TodoWrite 之间持久保留，不会被其他工具调用清除
- 完成的 TodoWrite 不写入 `_text_chunks`，避免文本缓冲区被工具完成行稀释

### 测试覆盖
- 827 测试全通过（815 + 新增 12）
