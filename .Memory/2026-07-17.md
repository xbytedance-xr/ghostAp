## SMART Shell 路由与飞书热路径修复

### 现象与根因

- SMART、无项目、未管理聊天中，`pwd`、`ls -la`、`echo hi` 会先进入被动 Slock 分类并弹出“是否是任务”的澄清卡片；`git status`、`./restart.sh rr` 可能直接激活 Slock。提交 `a09bd51a` 引入的被动分类位于普通 intent/shell 识别之前，而 shell 快速队列只改变调度，不改变路由优先级。
- 澄清卡片发送后缺少 `return`，同一条消息还会继续进入普通 intent，造成重复处理。
- `logs.log` 中真实 `pwd` 请求先发送澄清卡，再执行 Contact API；Contact GET 返回 400 且一次耗时约 7 秒，随后普通 shell 路由仍继续执行。卡片回调还会在 `ActivationGuard` 中因生产配置的 `frozenset` 被当作字符串调用 `.split()` 而崩溃。
- 运行进程环境核查只发现 `NO_PROXY=127.0.0.1`，未发现 `http_proxy`/`https_proxy`；因此“生产进程继承代理并放大 open.feishu.cn 延迟”的假设不受当前进程证据支持。

### 修改

- 将 `IntentRecognizer.looks_like_shell()` 提升为可直接复用的类方法，`SystemHandler`、Slock handler 和 dispatcher 共用同一 shell 检测器，避免规则漂移。
- dispatcher 在被动/遗留 Slock 路由前识别 SMART shell 并直接保留给普通 shell intent；显式 Slock 命令、已激活上下文和话题引擎语义不变。澄清卡片发送后立即返回。
- 普通消息、卡片入口以及 Slock 角色/任务/升级回调全部改为非阻塞用户名解析；后台刷新使用进程级 single-flight 和最多 32 个并发刷新。飞书稳定权限错误族负缓存 1 小时，空响应、服务端错误、导入或网络等瞬态异常只负缓存 60 秒。
- `ActivationGuard` 对 `admin_user_ids` 同时接受逗号字符串和 `list`/`tuple`/`set`/`frozenset`，与生产 Settings 契约一致。
- 新增跨模块回归：SMART、无项目、未管理聊天中的 `pwd`、`ls -la`、`echo hi`、`git status`、`./restart.sh rr` 只能进入 shell；澄清卡片必须终止路由；卡片回调不得同步查询 Contact；Feishu handler 禁止带 API factory 调用同步用户名解析；生产 `frozenset` 管理员配置可正常鉴权。

## 常驻员工、团队协调与知识工作目录改造方案

### 现状结论

- 当前真正常驻并可在重启后恢复的是每名员工的飞书 Channel 子进程；员工 ACP/CLI 会话仍由 Slock 在每个任务中创建、单次发送 prompt 后立即关闭。
- 可见 `/hire` 只把名字、工具、模型、Profile 和 Effort 传入正式 Hire request；领域模型已有的 role/persona/traits/capabilities/permissions 没有被 UI 完整固化。
- Team 模式不是主协调 Agent，而是按 `agent_id` 排序后固定执行“第一名 analysis、第二名 review、第一名 synthesis”。TeamRun 只持久化任务 digest，重启无法恢复原始 instruction。
- 本次错误 `context_unavailable` 的具体链路是：内部 Team assignment 仍要求员工 Bot 重新抓取并严格验证飞书 Thread/Group 排序；`ORDERING` 有界重试耗尽后被统一终止，模型尚未收到任务。

### 自动 Grill 决策

- “员工常驻”定义为可恢复的逻辑 Actor、durable mailbox、任务/会话 checkpoint 和知识目录；ACP 支持时保留热会话，不支持时允许从同一身份和 checkpoint 冷启动。
- Journal 与加密 Blob 保持唯一事实源；`AGENTS.md`、`IDENTITY.md`、`NOW.md` 和 Wiki 是可重建投影，员工不可通过文件修改身份、权限或任务状态。
- 借鉴 Karpathy LLM Wiki 的 immutable source / compiled wiki / schema 结构，并采用 ingest、query、lint；GhostAP 增加独立 runtime state 层，不首期引入向量库。
- 主 Bot 维护全群 durable event ledger，普通消息对协作系统可查但不自动唤醒所有模型；直接 @、Coordinator assignment、CAS claim 或显式职责规则才唤醒员工。
- 内部 Team assignment 使用已锚定的 canonical group context；远端 ordering/pagination/revision 失败降级为可披露的 partial context，scope/permission/current-message 等 authority 失败仍 fail-close。
