## Codex ACP fallback + Feishu 卡片内容非法兜底

### 任务描述
用户要求在 Refactoring Analysis 之外继续审查项目潜在问题，并点名 Codex 模式支持异常；同时 Backlog 中仍有 B018：Feishu `page_mutator.update_page()` 把 `230099` 内容非法错误当作可通过重建恢复的 permanent transport error，存在反复删除 binding、重建同一份非法 JSON 的刷屏风险。

### 根因分析
- 当前本机 `codex acp serve --help` 返回 `error: unrecognized subcommand 'serve'`，实际可用子命令包含 `mcp-server` / `app-server`，因此 GhostAP 的 Codex native ACP 命令会在启动阶段失败。
- `@zed-industries/codex-acp@0.14.0` 提供 `codex-acp` bin，`npx --yes @zed-industries/codex-acp@0.14.0 --help` 可用，并支持 `-c model="..."` 配置覆盖。
- `TransportError.is_permanent` 同时覆盖 `99992354` 与 `230099`，但两者恢复策略不同：`99992354` 表示消息/card binding 失效，删除 binding 后重建有意义；`230099` 多数是卡片内容非法，重建同一 payload 只会再次失败。

### 执行内容
- `src/acp/providers/__init__.py` 新增 `CodexACPProvider`：native `codex acp serve` 可用时保留原路径；不可用但存在 `npx` 时自动返回 `npx --yes @zed-industries/codex-acp@0.14.0`；传入模型时追加 `-c model="..."`。
- `restart.sh` 新增 `prepare_codex_acp_dependency()`：服务启动前若 native `codex acp serve` 不可用，会执行 `npx --yes "$CODEX_ACP_NPM_PACKAGE" --help` 预热固定版本 ACP bridge，让新环境启动脚本自动拉取 Codex fallback 依赖；可用 `GHOSTAP_PREPARE_CODEX_ACP=0` 跳过。
- `src/card/delivery/types.py` 将 transport error 分类拆成 `needs_recreate` 与 `is_content_invalid`：仅 `99992354` 走重建；`230099` 或消息中含 `ErrCode: 200621/200861` 视为内容非法。
- `src/card/delivery/page_mutator.py` 在内容非法时不再移除 page binding，而是 patch 一张已知合法的 fallback 卡，提示“卡片内容不符合飞书卡片格式，已停止自动重建以避免重复刷屏”，并把当前 rendered signature 标记为已处理，避免下一轮继续推同一份非法结构。
- `.Memory/Backlog.md` 移除 B018；随后 B017 也通过字号改造移除，当前 Backlog 无开放条目。

### 技术要点
- 群聊默认卡片正文遵循最小披露：诊断详情只能通过用户显式点击“查看详情”进入脱敏/截断反馈链路。

### 技术要点
- ACP startup 不再以“集中方法 + 后续待拆”作为最终状态；本轮通过协作者入口和测试断言证明后端选择、cwd、TTADK 预检、retry、diagnostics、telemetry 可分别验证。
- 异常治理不追求全仓一刀切删除 `except Exception`，而是对启动/分发/WebSocket 关键路径建立分类语义和源码级 guard。
- 兼容入口保留不是“暂不处理”：每个保留项必须说明调用方迁移到哪里、何时删除或是否长期保留。
- Memory 的 Refactoring Analysis 验收入口保留一个结构化口径；状态字段只表达“存在 / 不存在 / 已被其他改动解决”，处理结果与兼容策略放在非状态字段。
- `STARTUP_OPERATIONAL_EXCEPTIONS` 作为启动边界命名捕获契约，仅覆盖进程/stdio/握手等明确运行期失败；`TypeError`、`AssertionError` 与未知业务异常默认穿透，不进入 retry/degrade 语义。
- TTADK transport 仍遵循 CLI bridge-only：本次只把遗留降级 helper 从 manager 搬到 startup 边界，没有在 `ttadk_*` 启动失败时新增 Coco ACP 自动 fallback；`tests/test_acp_client.py::test_acp_manager_ttadk_start_failure_no_coco_acp_fallback` 已锁定该约束。
