## Team step timeout 根因与竞态收敛加固

### 生产证据与根因

- 最近一次真实失败是 `teamrun_dd9a…`，并非 07-16 01:55 的再次上报消息。Journal
  sequence 334 于 2026-07-15 15:05:31 创建 run；首个 analysis assignment 在 sequence
  342 进入 Router `queued(position=1)`，但从未出现 `router_dispatching` 或
  `employee.execution_attempt.*`。
- `logs.log` 从 15:05:33 起持续出现 `ContextUnavailableError`。事故时的旧 coordinator
  让普通群 Context 失败逃到 dispatch worker；worker 只记录错误并退避，没有推进 Router
  终态，因此 `_RuntimeTeamBackend.result()` 始终返回 `None`。
- 600 秒后 sequence 343 把首个 dispatch effect 标为 `action_required`；retry 随后在
  sequence 351 进入 `queued(position=2)`，但旧 assignment 仍占队首。第二个 600 秒窗口
  结束后，sequence 352 把 retry effect 标为 `action_required`，sequence 356 才写入
  `team.run.action_required(error_code=team_step_timeout)`，总计约 20 分钟。
- 事故发生在 `f69c325c` 部署前。该提交已支持普通群 singleton Context、三次 durable
  `context_retry`、失效 Team effect 检查和 recovery 顺序；重启后两个旧 acceptance 最终以
  `authority_stale` 终结。此次没有调整 timeout 数值。

### 本轮加固

- `EmployeeDispatchCoordinator` 在 Router 选出 Team assignment 后、任何 Context/API 工作前，
  先校验 canonical team payload 与 owning `employee_dispatch` effect；已失效任务立即持久化为
  `team_step_inactive`。post-render late check 和两道 Journal-head CAS 保留，用于关闭检查后
  stop-vs-dispatch 竞态。
- 原 CAS 回归因新增 early check 一度变成假覆盖；测试改为第 2 次（late）检查后注入
  `action_required`，并断言共发生 3 次检查，证明 Journal head 变化触发 retry，再由 retry
  early check 终结旧任务。
- `EmployeeTeamService` 在自身 deadline 到达且 stop 未触发时，增加恰好一次 backend final
  poll；若 Gateway 终态刚刚可见，复用唯一 `_terminalize_step_result()` 路径保存真实 status、
  history id 和 `slock_session_timeout` 等错误码，只有 final poll 仍为 `None` 时才生成
  `team_step_timeout`。

## Deep 卡片问题摘要抬头

### 变更与原因

- 根因是 Deep 为复用 Spec-style 阶段结构固定派发 `cycle_started(1, 1)`，共享 v2 header
  因而把首行渲染成“第 1 轮”；原用户问题其实已在启动入口可用，无需额外模型调用。
- 新增确定性 `summarize_question_title()`：折叠连续空白，短问题原样保留，超长取前
  14 个字符加 `…`，包含省略号严格不超过 15 字；空问题回退为“Deep 任务”。
- Deep handler 在创建 CardSession 前透传原 requirement，renderer 将摘要写入独立、不可变的
  `CardMetadata.question_title`；共享 header 只对 Deep 主卡优先使用该字段，并在渲染边界
  再次限长，Spec/Worktree 的轮次标题不变。
- 首次 STARTED、运行、完成、失败、冻结分页均保持摘要标题；失败状态由红色模板和正文表达，
  不在摘要前追加通用错误前缀；冻结分页副标题只保留卡号/页码，不重复摘要。
- `ux/card_preview.html` 同步更新运行、完成、失败和分页示例及 Header 布局规范。

### 验证与收敛

- TDD RED：helper 缺失、header 仍为“第 1 轮”、renderer 不接受 requirement、handler 未透传；
  GREEN 后首轮目标集 `269 passed`。
- 相邻及状态模型扩大回归先后 `334 passed`、`387 passed`；最终 Deep/card 相关目标集
  `489 passed in 13.57s`。
- 修改范围 `ruff check`、`uv run python -m src.main --validate`、`git diff --check` 均通过；
  配置校验仅保留既有 `slock_default_roles` 为空提示。
- mloop 以产品/UI、架构/工程、测试/质量三角色并行审查 5 轮，修复首次投递、失败态、
  冻结分页重复和 metadata 语义问题后，Rounds 4–5 连续无实质建议并收敛。
