## 话题编程模式 chat_id 降级防御（第六轮终结）

### 任务描述
前五轮修复（thread_root_id 切换、跳过 enter_mode、四层防御、reply_id 根因、双键注册）均未彻底解决飞书话题编程模式持续对话降级问题。用户在线程内的后续消息仍出现"无法理解你的意图"。

### 执行内容

#### 真正根因（第六轮确认）
1. `_dispatch_to_thread` 处理完首条消息后调用 `ModeManager.exit_to_smart(chat)`
2. 模式状态从 **ModeManager** 转移到 **ThreadContext**（仅通过 root_id 可查）
3. 后续消息若 root_id 与注册的 reply_id/message_id **均不匹配**，ThreadContext 查找失败
4. `auto_enter_mode = None` → 走意图识别 → **"无法理解你的意图"**
5. 核心问题：模式状态存在**单点依赖**（root_id），无任何降级路径

#### 修复方案（第三层防御 — chat_id 降级）
在 root_id 查找全部失败后，通过 `get_by_chat(chat_id)` 找到同 chat 的 ThreadContext：

| 防御位置 | 新增逻辑 |
|---|---|
| `_resolve_message_context` | root_id miss → `get_by_chat(chat_id)` → 取 `chat_ctxs[0]` |
| `_process_message_async` safety-net | root_id safety-net 也 miss → 同上 chat_id 降级 |
| `_handle_message` | root_id miss + 无 project_id → chat_id 降级设置 queue key |

#### 三层防御完整总结
| 层 | 机制 | 防御场景 |
|---|---|---|
| L1 | 双键注册 (reply_id + message_id) | root_id 是两个已知 ID 之一 |
| L2 | root_id 直接查找 (get) | 正常路径 |
| **L3** | **chat_id 降级 (get_by_chat)** | **root_id 为 None 或未知值（最终兜底）** |

### 技术要点
- `get_by_chat()` 已做 `id(ctx)` 去重（第五轮修复），多别名不会重复返回
- 系统设计为单话题 per chat（`_dispatch_to_thread` 会清理旧话题），chat_id 降级误匹配风险极低
- 即使误匹配到"错误"话题，也比回退 SMART 模式好（至少保持编程状态）

## 话题编程模式双键注册修复（第五轮）

### 任务描述
前四轮修复（thread_root_id 切换、跳过 enter_mode、四层防御、reply_id 根因修复）均未彻底解决飞书话题编程模式持续对话降级问题。用户在线程内的后续消息仍走意图识别而非编程 handler。

### 执行内容

#### 根因分析
1. ThreadContext 用单一 key 注册（无论 message_id 或 reply_id），但飞书后续消息的 root_id 具有不确定性（可能是用户原始 message_id 或 bot reply_id）
2. 任意单键总有一侧不匹配，导致 `_thread_manager.get(root_id)` 返回 None
3. 查找失败后 `auto_enter_mode = None`，消息被错误路由到意图识别

#### 修复方案（三层）
1. **ThreadContextManager 双键注册**: `register()` 新增 `alias_keys` 参数，同一 ThreadContext 以 reply_id(主键) + message_id(别名) 双存到 `_contexts` dict
2. **消息处理链路使用 canonical thread_root_id**: `_handle_message`、`_process_message_async`、`_resolve_message_context`、safety-net 查找到 thread_ctx 后均用 `thread_ctx.thread_root_id`（canonical）做 `set_current_thread_id`
3. **诊断日志**: 关键查找点记录 msg_root_id 和 canonical thread_id

#### 附带修复
- `get_by_chat` / `active_count` 对双键做 `id(ctx)` 去重，避免重复计数
- `remove()` 先通过 `_aliases` 规范化到 canonical 再全清理
- `remove_by_chat` / `close` / `_evict_expired` 同步清理别名映射

### 技术要点
- 双键注册 = 同一 ThreadContext 对象存储在 `_contexts` 的两个键下（引用共享），`_aliases` dict 维护 alias→canonical 映射
- `get()` 方法无需改动（直接 dict lookup），两个键都能命中同一 ThreadContext
- `remove()` 规范化：先查 `_aliases` 解析到 canonical，再清理 canonical 条目 + 所有指向它的 alias 条目
- mloop 3 轮审查收敛（Round 1: 修复 get_by_chat 去重、remove 规范化、补测试；Round 2-3: No material suggestion）

## 话题编程模式持续对话交付（reply_id 根因修复）

### 任务描述
用户反馈飞书话题内编程模式仍然只能单次对话，后续回复会掉回 SMART；期望行为是当前话题在当前项目中持续保持编程模式，直到用户输入 `/exit` 才退出，并要求最终 commit & push 交付。

### 执行内容

#### 根因分析
在已落地的 thread context 多层防御之外，继续追踪 `_dispatch_to_thread → enter_mode → get_session → handle_message` 链路，发现仍有一个关键断裂点：

1. **_dispatch_to_thread 使用了错误的 thread root**：创建话题后，后续回复实际锚定的是 `reply_message_with_id()` 返回的 reply message id；但旧逻辑仍把原始 `message_id` 当作 `thread_root_id` 传给 `enter_mode` / `get_session` / `_register_thread_context`，导致后续 root_id 查不到正确 ThreadContext，消息再次回退到 SMART
2. **project=None 路径 session 查找不稳**：创建话题时若调用点未携带 project，旧逻辑会直接用空 `project_id` 查 session，错过 active project 下刚建立的 thread session
3. **handler 恢复路径仍需 project 回退**：`handle_message` 在初始 `project=None` 且首次 `get_session()` miss 时，需要再次回退 active project 才能找回 thread session

#### 修复实现

**Fix 1 — `_dispatch_to_thread` 以 reply_id 作为真实 thread root (`src/feishu/ws_client.py`)**
- 创建话题成功后，把 `reply_message_with_id()` 返回值赋给 `thread_root_id`
- 用该 `thread_root_id` 统一传给 `enter_mode(..., thread_id=...)`、`get_session(..., thread_id=...)`、`_register_thread_context(...)`
- 若存在 project，同时把 `reply_id` 注册到 `MessageProjectMapper`，保证后续从回复消息也能反查项目

**Fix 2 — `_dispatch_to_thread` 在 `project=None` 时回退 active project (`src/feishu/ws_client.py`)**
- `enter_mode` 之后若当前 `project` 仍为空，调用 `get_active_project(chat_id)` 恢复真实项目
- 后续 `get_session()` 与 `exit_to_smart()` 都使用恢复后的 `project_id`

**Fix 3 — `handle_message` 补强 active project 恢复 (`src/feishu/handlers/programming.py`)**
- 首次 `get_session()` miss 后，若初始 project 缺失，先尝试 `get_or_create_project_for_path()`
- 若仍未拿到有效 project，再显式回退 `get_active_project(chat_id)` 重查 thread session
- 仅在两轮恢复都失败时才提示“会话启动失败”

#### mloop 审查
- Round 1：产品/架构/实现/测试视角复核当前代码与测试，未发现需要新增修改的实质问题
- Round 2：再次按相同 Goal Snapshot 独立复核；虽然有 1 个审查 agent 因 worktree 锁失败，唯一成功输出的建议与当前代码状态不符（代码已覆盖 thread 优先恢复、图片路径安全网、active project fallback），复读当前实现后判定无新增实质问题
- 收敛结论：连续两轮无有效实质建议，停止 mloop

### 技术要点
- 飞书话题持续编程的真正锚点不是首次用户消息 id，而是机器人创建出来的 reply message id
- `ThreadContextManager`、`MessageProjectMapper`、session manager 三者必须共享同一个 thread root id，否则恢复链会断裂
- 文本与图片路径均经 `_resolve_message_context` + `_process_message_async` 安全网统一恢复 thread mode
- `handle_message` 的 active project fallback 仍然必要，用于修复 project 透传缺失时的 session 回收

## 话题编程模式多层防御修复（第三轮）

### 任务描述
用户反馈话题内编程模式仍被降级到 SMART 模式（第三次报告），要求彻底修复。前两轮修复（thread_root_id 矫正 + 跳过 enter_mode）未能解决根因。

### 执行内容

#### 根因分析
通过完整追踪消息处理链路 `_handle_message → _process_message_async → _resolve_message_context → _dispatch_message_logic`，发现多个断裂点：

1. **_resolve_message_context**：当 `thread_ctx` 存在但 `get_project(thread_ctx.project_id)` 返回 None 时，`auto_enter_mode` 丢失（因为仅在 `if project:` 内赋值），导致 fall through 到 `_resolve_project_from_message`，后者因 project mode flags 被 `exit_to_smart` 重置而返回 `auto_enter_mode=None`
2. **_dispatch_to_thread**：`_register_thread_context` 只在 `if project:` 块内调用，若 project 为 None 则 ThreadContext 从未注册
3. **handle_message**：当 session 未找到且 project 为 None 时直接返回 "会话已过期" 错误，不尝试恢复
4. **_register_thread_context**：project=None 时使用 session.session_id 作为 project_id，导致后续 get_project 永远返回 None

#### 四层防御修复

**Layer 1 — _resolve_message_context (ws_client.py)**
- 将 `auto_enter_mode` 提取与 project 查找解耦
- thread_ctx 存在时始终 return（不 fall through 到 `_resolve_project_from_message`）
- project 回退链：`get_project(thread_ctx.project_id)` → `get_active_project(chat_id)`

**Layer 2 — _process_message_async 安全网 (ws_client.py)**
- 在 `_dispatch_message_logic` 调用前添加安全网
- 若 `auto_enter_mode` 仍为 None 但存在注册的 thread context，强制设置 mode
- 覆盖文本和图片两条消息路径

**Layer 3 — _dispatch_to_thread (ws_client.py)**
- `_register_thread_context` 调用移出 `if project:` 块
- 无论 project 是否为 None 都注册 ThreadContext

**Layer 4 — handle_message + _register_thread_context (programming.py)**
- `handle_message`：移除 `if project:` / `else:` 分支，统一走 `enter_mode` 恢复路径
- `_register_thread_context`：project=None 时通过 `get_active_project(chat_id)` 获取真实 project_id

#### mloop 审查
- Round 1：4 角色审查，发现 _resolve_message_context fall-through、session.session_id 作 project_id、handle_message 分支逻辑等问题 → 全部修复
- Round 2：补充图片路径安全网测试、ensure_session 断言
- Round 3：所有角色 No material suggestion
- Round 4：确认收敛 (2/2 CLEAN)

### 技术要点
- `_get_interaction_mode()` 是各子类硬编码方法（返回固定 InteractionMode 枚举），不受 ModeManager 状态影响
- ThreadContext 的 mode 值来源于 handler 子类的 `_get_interaction_mode().value`，始终正确
- 安全网是必要的冗余防御，覆盖图片消息路径和任何未预见的 `_resolve_message_context` 失败场景
- per-chat 顺序执行保证避免了 `_find_active_thread + remove` 的竞态条件
