## 话题编程模式多层防御修复（第三轮）

### 任务描述
用户反馈话题内编程模式仍被降级到 SMART 模式（第三次报告），要求彻底修复。前两轮修复（thread_root_id 矫正 + 跳过 enter_mode）未能解决根因。

### 执行内容

#### 根因分析
通过完整追踪消息处理链路 `_handle_message → _process_message_async → _resolve_message_context → _dispatch_message_logic`，发现多个断裂点：

1. **_resolve_message_context**：当 `thread_ctx` 存在但 `get_project(thread_ctx.project_id)` 返回 None 时，`auto_enter_mode` 丢失（因为仅在 `if project:` 内赋值），导致 fall through 到 `_resolve_project_from_message`，后者因 project mode flags 被 `exit_to_smart` 重置而返回 `auto_enter_mode=None`
2. **_dispatch_to_thread**：`_register_thread_context` 只在 `if project:` 块内调用，若 project 为 None 则 ThreadContext 从未注册
3. **handle_message**：当 session 未找到且 project 为 None 时直接返回 "会话已过期" 错误，不尝试恢复
4. **_register_thread_context**：project=None 时使用 session.session_id 作为 project_id，导致后续 get_project 永远返回 None

#### 四层防御修复

**Layer 1 — _resolve_message_context (ws_client.py)**
- 将 `auto_enter_mode` 提取与 project 查找解耦
- thread_ctx 存在时始终 return（不 fall through 到 `_resolve_project_from_message`）
- project 回退链：`get_project(thread_ctx.project_id)` → `get_active_project(chat_id)`

**Layer 2 — _process_message_async 安全网 (ws_client.py)**
- 在 `_dispatch_message_logic` 调用前添加安全网
- 若 `auto_enter_mode` 仍为 None 但存在注册的 thread context，强制设置 mode
- 覆盖文本和图片两条消息路径

**Layer 3 — _dispatch_to_thread (ws_client.py)**
- `_register_thread_context` 调用移出 `if project:` 块
- 无论 project 是否为 None 都注册 ThreadContext

**Layer 4 — handle_message + _register_thread_context (programming.py)**
- `handle_message`：移除 `if project:` / `else:` 分支，统一走 `enter_mode` 恢复路径
- `_register_thread_context`：project=None 时通过 `get_active_project(chat_id)` 获取真实 project_id

### 技术要点
- `_get_interaction_mode()` 是各子类硬编码方法（返回固定 InteractionMode 枚举），不受 ModeManager 状态影响
- ThreadContext 的 mode 值来源于 handler 子类的 `_get_interaction_mode().value`，始终正确
- 安全网是必要的冗余防御，覆盖图片消息路径和任何未预见的 `_resolve_message_context` 失败场景
- per-chat 顺序执行保证避免了 `_find_active_thread + remove` 的竞态条件
