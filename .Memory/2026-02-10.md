## Loop Engine 架构补全 — 从 MVP 骨架到完整实现

### 任务描述
根据 `docs/loop_mode_architecture.md` 设计文档，对比当前实现的差距，补全所有未实现的 Loop Engine 功能。当前为 MVP 骨架版（约 30%），需实现角色系统、终止判定器、需求解析器、标准回写、上下文集成等。

### 执行内容

**新建 3 个源文件**:
- `src/loop_engine/roles.py` — `RoleRouter` 动态角色选择器，6 个角色的 system prompt，6 级优先规则引擎
- `src/loop_engine/termination.py` — `TerminationChecker` 多维终止判定器，6 种信号按优先级评估，增强收敛检测（条件A AND (B OR C)）
- `src/loop_engine/analyzer.py` — `RequirementAnalyzer` LLM 驱动需求解析，降级为文本解析

**重写 1 个核心文件**:
- `src/loop_engine/engine.py` — 完全重构 `execute()` 流程：
  - 构造器新增 `_role_router`、`_termination_checker`、`_context_manager`
  - 删除 `_user_guidance`、`_parse_requirement`、`_build_initial_prompt`、`_build_iteration_prompt`、`_detect_convergence`
  - 新增 `_build_iteration_state()`、`_build_role_prompt()`、`_evaluate_criteria()`（正则解析+batch_update回写）、`_apply_termination()`
  - `inject_guidance()` 委托给 `_context_manager`
  - `LoopEngineCallbacks`: `on_iteration_start` 增加 `RoleSelection` 参数，新增 `on_criteria_update`

**修改 3 个文件**:
- `src/loop_engine/__init__.py` — 新增 RoleRouter, ROLE_PROMPTS, TerminationChecker, RequirementAnalyzer 导出
- `src/feishu/handlers/loop.py` — 适配新回调签名，显示角色信息，新增 criteria 进度卡片
- `src/config.py` — 新增 `loop_max_context_tokens`、`loop_default_max_retries`

**重写测试文件**:
- `tests/test_loop_engine.py` — 新增 TestRoleRouter(8)、TestTerminationChecker(10)、TestRequirementAnalyzer(7) 三个测试类，更新 TestLoopEngine 适配新 API

### 技术要点
- **角色选择优先级**: 首轮→ARCHITECT, 连续失败≥2→DEBUGGER, 满足率≥0.6→TESTER, 满足率≥0.8且有TESTER→INTEGRATOR, 迭代≥3无REVIEWER→REVIEWER, 默认→DEVELOPER
- **终止信号优先级**: USER_STOP > FATAL(连续3次失败) > MAX_ITER > COMPLETE > CONVERGED > CONTINUE
- **收敛检测**: 条件A(最近N轮无新标准满足) AND (条件B(输出长度差异<20%) OR 条件C(连续同角色))
- **标准回写**: `_evaluate_criteria()` 用正则 `CRITERIA_{i+1}\s*[:：]\s*(PASS|FAIL)` 解析，通过 `batch_update()` 写回 CriteriaTracker
- **角色 prompt**: 每轮包含角色 system prompt + 标准状态([x]/[ ]) + 上下文历史 + 用户指导

## Bug 修复：流式错误恢复元组解包崩溃 + 配置字段名修正

### 任务描述
全面代码审查发现两个问题：
1. `ClaudeSession._handle_streaming_error_recovery()` 返回 4 元组但 `BaseSession.send_prompt_streaming()` 用 3 变量解包，导致 `too many values to unpack (expected 3)`
2. `.env` 中的 `REPLY_MODE` 字段名与代码中的 `smart_reply_mode`/`default_reply_mode` 不匹配，配置实际无效

### 执行内容
**Bug 1 — 元组解包不匹配**:
- `src/session/base.py` — `_handle_streaming_error_recovery` 返回类型改为 4 元组 `(output, stderr, timed_out, stopped)`，解包处增加 `recovered_stopped` 处理
- `src/session/claude.py` — 同步更新类型注解

**Bug 2 — 配置字段名不匹配**:
- `.env` — `REPLY_MODE=thread` 改为 `SMART_REPLY_MODE=direct` + `DEFAULT_REPLY_MODE=thread`
- `.env.example` — 同步更新

### 技术要点
- `_run_streaming_process` 返回 4 元组 `(output, stderr, timed_out, stopped)`，错误恢复路径也应保持一致
- pydantic-settings 的 `extra="ignore"` 会静默忽略不匹配的环境变量，不会报错但配置不生效
- 触发条件：Claude 会话过期（闲置 > 30min） + Deep 模式使用 Claude 引擎
