## ACP 协议重构实施 — subprocess CLI→ACP 结构化通信
### 任务描述
将 GhostAP 中 Coco 和 Claude 的调用方式从 shell 命令行子进程模式（subprocess.Popen）重构为 ACP（Agent Client Protocol）协议模式（JSON-RPC 2.0 over stdio），实现结构化的 agent 通信。

### 执行内容

**新增 `src/acp/` 模块（7 文件）:**
- `models.py` — ACPEvent, ACPEventType (6种事件), ToolCallInfo, PlanInfo, ACPSessionState, PromptResult
- `client.py` — GhostAPClient 实现 ACP Client 接口，session_update 分发为 ACPEvent，自动审批权限
- `session.py` — ACPSession 异步生命周期（spawn_agent_process → initialize → session/new → prompt → cancel → close）
- `sync_adapter.py` — SyncACPSession: daemon thread + asyncio event loop 桥接同步线程世界
- `manager.py` — ACPSessionManager: 统一 per-chat 会话管理，替代 CocoSessionManager/ClaudeSessionManager
- `renderer.py` — ACPEventRenderer: ACP 事件→飞书 Markdown（计划清单 + 工具状态 + 文本流）

**重写 `src/deep_engine/`（6→4 文件）:**
- 删除 `parser.py`、`planner.py`、`executor.py`（agent 通过 ACP 自主规划执行）
- 新增 `progress.py` — DeepProgress 从 ACP 事件提取计划/工具/文件进度
- 重写 `engine.py` — 单 ACP prompt 驱动完整执行，DeepEngineCallbacks 用 on_event 替代 on_task_start/done
- `models.py` 新增 EngineRunState 枚举

**重写 `src/loop_engine/`（7→4 文件）:**
- 删除 `roles.py`、`termination.py`、`analyzer.py`（简化为内联逻辑）
- 新增 `tracker.py` — IterationTracker 从 ACP 事件提取单次迭代信息
- 重写 `engine.py` — 单 ACP 会话多轮 prompt，收敛检测，标准评估

**更新集成层:**
- `handler_context.py` — 类型从 CocoSessionManager/ClaudeSessionManager→ACPSessionManager
- `ws_client.py` — ACPSessionManager("coco"/"claude") 替代旧 session manager 实例
- `handlers/programming.py` — 使用 ACPEventRenderer + on_event 回调替代 on_chunk 文本流
- `handlers/deep.py` — 使用 DeepEngineCallbacks.on_event 转发 ACP 事件
- `handlers/loop.py` — 使用 LoopEngineCallbacks 事件转发

**新增 `src/utils/errors.py`:**
- GhostAPError 基类 + fmt_error/fmt_exception/fmt_warning/fmt_timeout 统一错误格式化

**删除整个 `src/session/` 目录:**
- base.py、coco.py、claude.py、manager.py、__init__.py

**测试:**
- 新增: test_acp_models.py, test_acp_client.py, test_acp_renderer.py, test_deep_engine.py, test_loop_engine.py
- 删除: test_claude.py (旧 session 测试), test_executor_robustness.py (旧 executor 测试)
- 更新: test_ws_client_patch.py (patch 路径更新)
- **704 测试全部通过**

### 技术要点
- ACP SDK `agent-client-protocol==0.8.0` 提供 `spawn_agent_process()`、`text_block()`、Client/Agent 接口
- Coco 通过 `coco acp serve` 原生支持 ACP；Claude 通过 `--no-interactive` 模式兼容
- SyncACPSession 使用 daemon thread + `asyncio.run_coroutine_threadsafe()` 桥接同步-异步
- 权限控制: GhostAPClient 自动审批（选择 allow_once 选项），等价于旧 --dangerously-skip-permissions
- File/Terminal 操作使用 stub（agent 直接访问宿主文件系统，不需要 Client 端中转）

### 技术要点
- **角色选择优先级**: 首轮→ARCHITECT, 连续失败≥2→DEBUGGER, 满足率≥0.6→TESTER, 满足率≥0.8且有TESTER→INTEGRATOR, 迭代≥3无REVIEWER→REVIEWER, 默认→DEVELOPER
- **终止信号优先级**: USER_STOP > FATAL(连续3次失败) > MAX_ITER > COMPLETE > CONVERGED > CONTINUE
- **收敛检测**: 条件A(最近N轮无新标准满足) AND (条件B(输出长度差异<20%) OR 条件C(连续同角色))
- **标准回写**: `_evaluate_criteria()` 用正则 `CRITERIA_{i+1}\s*[:：]\s*(PASS|FAIL)` 解析，通过 `batch_update()` 写回 CriteriaTracker
- **角色 prompt**: 每轮包含角色 system prompt + 标准状态([x]/[ ]) + 上下文历史 + 用户指导

## Bug 修复：流式错误恢复元组解包崩溃 + 配置字段名修正

### 任务描述
全面代码审查发现两个问题：
1. `ClaudeSession._handle_streaming_error_recovery()` 返回 4 元组但 `BaseSession.send_prompt_streaming()` 用 3 变量解包，导致 `too many values to unpack (expected 3)`
2. `.env` 中的 `REPLY_MODE` 字段名与代码中的 `smart_reply_mode`/`default_reply_mode` 不匹配，配置实际无效

### 执行内容
**Bug 1 — 元组解包不匹配**:
- `src/session/base.py` — `_handle_streaming_error_recovery` 返回类型改为 4 元组 `(output, stderr, timed_out, stopped)`，解包处增加 `recovered_stopped` 处理
- `src/session/claude.py` — 同步更新类型注解

**Bug 2 — 配置字段名不匹配**:
- `.env` — `REPLY_MODE=thread` 改为 `SMART_REPLY_MODE=direct` + `DEFAULT_REPLY_MODE=thread`
- `.env.example` — 同步更新

### 技术要点
- `_run_streaming_process` 返回 4 元组 `(output, stderr, timed_out, stopped)`，错误恢复路径也应保持一致
- pydantic-settings 的 `extra="ignore"` 会静默忽略不匹配的环境变量，不会报错但配置不生效
- 触发条件：Claude 会话过期（闲置 > 30min） + Deep 模式使用 Claude 引擎
