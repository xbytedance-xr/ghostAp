# 2026-06-29

## WF 卡片乱码 + 进度反馈 + Traex 默认模型修复

### 背景 / 症状
- 用户反馈 `/wf` 执行过程中飞书卡片出现乱码。
- 脚本生成阶段卡片半天不动，无实时进度。
- Traex 模型在 Workflow executor 中报 `Internal error` 和 300s 超时。

### 根因定位

1. **卡片乱码根因 A**：`src/workflow_engine/bridge.py:456` 使用 `json.dumps(result)` 默认 `ensure_ascii=True`，中文字符被编码为 `\uXXXX` 后直接渲染到卡片 markdown 中。
2. **卡片乱码根因 B**：`src/card/delivery/feishu_client.py` 的 CardKit 流式路径（create_card/update_card/create_streaming_card）缺少 surrogate 代理对清理，unpaired surrogates 导致 lark_oapi 内部 UTF-8 编码崩溃。
3. **卡片 schema 不匹配**：`workflow.py` 中 `handle_workflow_view_workflow_ref`（line 3482）和 `handle_workflow_add_workflow_ref`（line 3700）两处内联构建的卡片缺少 `"schema": "2.0"` 且使用 v1 格式（根级 `elements`），导致飞书返回 `ErrCode: 200830; ErrMsg: schemaV2 card can not change schemaV1`。
4. **脚本生成无进度**：`_generate_script_via_ai` 中 `session.send_prompt()` 阻塞最长 300s，期间无任何卡片更新。
5. **Traex Internal error**：`create_engine_session(agent_type="traex", model_name=None)` 启动 `traex acp serve` 时不带 `-c model=`，TraeX 使用不可用的默认模型导致 `RequestError: Internal error`。

### 改动

- `src/workflow_engine/bridge.py`：`json.dumps(result, ensure_ascii=False)`
- `src/card/delivery/feishu_client.py`：新增 `_sanitize_card_content()` 并应用于 3 个 API 入口
- `src/feishu/handlers/workflow.py`：
  - 两处内联卡片加 `"schema": "2.0"` + `"body": {"elements": ...}`
  - `_generate_and_show_confirm_card` 增加 heartbeat 守护线程，每 8s 更新卡片显示已等待时间
  - `_generate_script_via_ai` 增加 `progress_callback` 参数，在会话创建/响应收到时主动推进状态
- `src/acp/providers/__init__.py`：`TraexACPProvider.get_default_model()` 从 `~/.trae/cli/models_cache.json` 读取首个可用 slug
- `src/agent_session/factory.py`：`agent_type == "traex"` 且 `model_name=None` 时调用 `get_default_model()` 填充
- `src/workflow_engine/renderer.py`：新增 `_escape_md()` 对用户需求文本做 markdown 转义

### 后续风险
- heartbeat 线程在模板路径（非 AI 生成）也会启动但几乎瞬间 stop，代价极小。
- `get_default_model()` 依赖 `~/.trae/cli/models_cache.json` 存在；如缓存不存在仍回退 None（原行为）。
- renderer `_escape_md` 仅对 completion card 的 requirement 做转义；progress card 中的 agent label/tool/error 未转义（它们来自内部，风险低）。
