## Spec 过程文件迁移到用户 cache

### 背景

用户指出 Spec 的 `.spec_engine/<project_id>/cycle_*`、`generated_specs`、`history.jsonl` 等过程文件直接写在项目目录下，会被项目 Git 管理扫描到；期望迁移到 `~/.cache` 下，并在项目群 `/spec_status` 时按目录实际情况统计有多少 Spec 任务，同时提供一键还原能力。

### 实现

- 新增 `src/spec_engine/storage.py` 作为 Spec 存储路径 SSOT，默认根目录为 `~/.cache/ghostAp`；项目绝对路径会被镜像到 cache 根下，例如 `/Users/a/repo` → `~/.cache/ghostAp/Users/a/repo`。
- `persistence.get_state_path()`、`artifact_root_dir()`、cycle artifact、generated spec、history log 均改为写入 cache project root；默认 `save_state()` 还会在每个 run 目录写 `state.json`，用于按 run 恢复。
- `SpecEngineManager.load_or_create_from_disk()` 优先读 cache 状态文件，并保留旧项目目录 `.spec_engine_state.json` 只读 fallback；新增 `load_or_create_from_state_file()` 支持按具体 run 的 `state.json` 恢复。
- `SpecEngine.resume()` 恢复 review circuit 时同样使用 cache 优先 + legacy fallback，避免旧暂停任务无法继续。
- 失败任务快照目录从 `~/.ghostap/spec_tasks` 迁到 `~/.cache/ghostAp/spec_tasks`。
- `/spec_status` 在无内存任务时会扫描 cache 中该项目的 `.spec_engine/<run_id>` 目录，显示发现任务数、最近 run、状态、cycle 和目标摘要；卡片提供“还原最新 Spec”按钮，回调 `spec_restore_run` 按 run id 加载对应 `state.json`，若状态为暂停/待澄清则直接走原 `/spec_resume` 继续执行。

## Spec 运行态卡片与 Review 可见性修复

### 背景

用户线上观察到 Spec engine 日志已经进入第 2/3 轮，但飞书卡片没有同步更新；卡片 sticky 文案长期停在 `📋 Spec · cycle ?/— · 0s`。后续还指出 review 角色的建议卡片不可见，并希望把每个审查角色实际使用的工具/模型也显示出来。日志中同时出现 `BaseSubprocessTransport.__del__` 在 event loop 关闭后清理 subprocess pipe 的异常。

### 实现

- `CardState` 增加 `RuntimeStats`，reducer 根据 `cycle_started`、`phase_started`、`phase_done` 和事件 `_now` 维护 Spec cycle、phase/perspective 与从 `session_started_at` 开始计算的总耗时；`CardSession` 在事件入队前统一补 `_now`。
- `SpecRenderer.create_spec_callbacks()` 在首张 Spec metadata 中记录整次 Spec 的 monotonic `session_started_at`，后续 cycle/task 轮转卡继承同一个开始时间。
- Spec review 阶段开始前显式触发 `on_phase_start(cycle, review)`，因此主卡会在角色审查 session 创建前更新到 review 阶段；review 完成时把格式化结果作为 `phase_done` 输出，避免后续阶段收尾事件把“用户建议/角色建议”内容覆盖掉。
- `AdaptiveRoleReviewStrategy` / `MultiPerspectiveStrategy` 给每个 `PerspectiveReview` 标注实际 `review_agent_type`、`review_model_name` 和展示 label；`SpecReporter` 在“测试/设计/产品/架构”等角色标题后追加 `（工具 / 模型）`。
- `ACPSession.close()` 与 `SyncACPSession.close()` 在关闭 event loop 前 drain pending callbacks / async generators，避免 review 子进程 session 回收时出现 loop closed 噪声异常。
- v2 header 对最新非封存 Spec 循环卡追加 `总耗时 XmYs`，只在卡片其他模块刷新时随渲染更新；旧封存卡不显示该实时总耗时，避免历史卡片被反复改动。

## Spec Review 选择卡去 WT 旅程化

### 背景

用户反馈 `/spec <目标>` 后的 review-agent 选择卡虽然只应该借用 WT 的“工具/模型选择”组件，但实际卡片内容继承了 Worktree 四步骤旅程：显示 `步骤 1/4`、`Worktree`/等待目标语义，且按钮 action 后续用按钮消息作为 reply 目标，可能让 Spec 流程从原 `/spec` 话题中跳出。

后续用户再次用线上截图确认仍有漏点：Spec 选择卡点击模型后会在群里另发一张 Worktree 等待目标卡。根因是模型按钮网格 `_render_worktree_model_option_grid()` 仍把模型按钮 action 硬编码为 `worktree_select_model`，导致 Spec review 的模型选择回调进入 `WorktreeHandler`，进而走到 WT confirm/awaiting-goal 链路。

### 实现

- `worktree_tool_select` 事件新增 `show_stepper`，默认 `True` 保持 WT 原体验；Spec review 选择 payload 设置为 `False`，渲染层只隐藏旅程步骤器，不复制 WT confirm/等待目标流程。
- Spec review 工具选择卡仍复用 WT 工具/模型列表和按钮渲染，但卡片 body 不再出现 `步骤 1/4`、`(1/4)`、`Worktree`、`等待目标` 等 WT 旅程文案。
- Spec review 的工具、模型、返回、移除、清空、Auto、确认 action 都优先读取 payload `thread_root_id`，刷新原 `/spec` 话题卡片；确认选择后直接以原始 `/spec <目标>` 启动 Spec 主流程，并把所选工具/模型作为后续多角色 review agent pool。

- `SpecRenderer.get_or_create_session()` 对 `(chat_id, project_id, thread_root_id)` 复用选择 session，使工具/模型选择阶段持续 PATCH 同一张 Spec Review 卡。
- WT 原流程默认 `show_stepper=True`，不影响 `/wt` 的四步骤卡片、等待目标和执行旅程。
- UX 预览补充 `ux/spec-review-card-flow.html`，并把同卡选择流程嵌入 `ux/spec-flow.html`；候选工具展示补齐 Aiden，保持 Coco/Aiden/Codex/Claude/TTADK 的顶层口径。

## 日志卡片 Schema V2 BUG 修复

### 背景

用户要求顺便修复日志中的 BUG。`logs.log` 中最新反复出现的卡片错误为：

- `ErrPath: ROOT -> body -> elements -> [6](tag: note)`
- `ErrMsg: cards of schema V2 no longer support this capability`
- fallback 再次失败：`schemaV2 card can not change schemaV1`

根因是超大卡片进入 `payload_truncator` 后追加了旧 Schema V1 `note` 元素；同时 invalid-card fallback 缺少 `schema: "2.0"` 和 `config.update_multi`，导致原卡为 Schema V2 时 fallback patch 被飞书判定为切换 schema。

### 实现

- `payload_truncator` 的截断提示从 `tag=note` 改为 `tag=markdown` + `text_size=notation`，保留 `/deep`、`/spec`、`/wt` 等命令提示语义。
- aggressive truncation fallback 卡补齐 `schema: "2.0"`，并强制 `config.update_multi=True`，避免降级卡片被识别为 Schema V1。
- `PageMutator` 的 invalid-card fallback 同样补齐 Schema V2 身份，防止 rendered card invalid 后的 fallback patch 二次失败。
- `UnifiedCardLayout` 的 sticky message 也从 `note` 改为 notation markdown，消除同类 Schema V1 元素来源。
