## 状态流转产品契约固化

### 背景

继续核查智能状态、普通编程状态、Deep/Spec/WT 工作状态的边界。目标不是新增模式，而是把当前实现和后续开发原则对齐：默认 SMART 能识别简单意图和 shell；普通工具编程按群保持；Deep/Spec/WT 按话题保持。

### 当前实现

- SMART 是默认 chat/project 状态，`MessageDispatcher` 在非编程模式下先处理 slash、工作流命令、锁、项目意图与 shell intent。
- `/coco`、`/codex`、`/aiden`、`/claude`、`/gemini`、`/ttadk` 这类普通工具入口进入 chat+project 级编程状态，后续普通消息持续进入当前工具，直到 `/exit` 回到 SMART。
- Deep、Spec、Worktree 通过 Feishu `thread_root_id/root_id` 绑定 topic-scoped engine context；它们属于执行策略，不覆盖 chat+project 级普通编程工具状态。
- SMART/project chat 的 shell heuristic 保留 `./restart.sh rr`、`bash ./restart.sh rr`、`uv run ...` 等 shell-like 文本为 shell 执行，避免被项目群自由文本编程入口抢走。

### 长期原则

`AGENTS.md` 的 Strategy And Transport 部分新增状态作用域原则：SMART 是默认 chat/project 状态；普通工具入口是持久 chat+project 编程状态；Deep/Spec/WT 是 topic/root-thread scoped engine strategy；shell-like 文本在 SMART 中必须继续走 shell。
