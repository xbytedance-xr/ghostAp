## Spec 多工具 Review 选择与随机分配

### 背景

用户希望 Spec 启动后不要只沿用主任务 agent 的工具/模型做全部多角色 review，而是在 `/spec <需求>` 后先提供一个可选的 review 工具/模型选择步骤：可直接 Auto 保持原流程，也可像 WT 一样选择多个工具模型组合，后续多角色 review 随机分配给这些工具并尽量在单轮 review 中覆盖已选工具。

### 实现

- Spec 启动路径新增 review-agent 选择卡：`/spec <需求>` 先绑定 topic context、检查同项目运行中任务，然后复用 `worktree_tool_select` 卡片渲染逻辑展示工具/模型选择；新增 `spec_review_use_auto` 可直接保持原有主 agent review 流程。
- 选择状态复用 WT 的 `WorktreeSelectionController` / `WorktreeSelectionItem` / 工具模型 discovery 语义，但独立存放在 `ProjectContext.spec_review_selection_state`，只共享模型选择这一步的卡片和选择模型逻辑，不共享 WT 执行/分支/worktree 流程。
- Worktree 选择卡渲染扩展为可配置 action ids、标题与 Auto 按钮；WT 原 action 和 UI 保持默认，Spec 通过 `SPEC_REVIEW_SELECT_TOOL` / `SPEC_REVIEW_SELECT_MODEL` / `SPEC_REVIEW_FINISH_SELECTION` 等 action 接入同一渲染逻辑。
- `SpecEngine` 新增 review agent pool；`AdaptiveRoleReviewStrategy` 在每轮 role review 前随机给角色分配 `ReviewAgentBinding`，当角色数量不少于已选工具数量时先保证每个已选工具至少用一次，剩余角色再随机复用。
- `create_review_session()` 修正默认模型语义：只有 Coco 在未显式传模型时读取 Coco 当前模型；Codex/Aiden/Gemini 等 ACP 工具的默认模型保持 `None`，交给对应工具自身默认配置。

## 状态流转产品契约固化

### 背景

继续核查智能状态、普通编程状态、Deep/Spec/WT 工作状态的边界。目标不是新增模式，而是把当前实现和后续开发原则对齐：默认 SMART 能识别简单意图和 shell；普通工具编程按群保持；Deep/Spec/WT 按话题保持。

### 当前实现

- SMART 是默认 chat/project 状态，`MessageDispatcher` 在非编程模式下先处理 slash、工作流命令、锁、项目意图与 shell intent。
- `/coco`、`/codex`、`/aiden`、`/claude`、`/gemini`、`/ttadk` 这类普通工具入口进入 chat+project 级编程状态，后续普通消息持续进入当前工具，直到 `/exit` 回到 SMART。
- Deep、Spec、Worktree 通过 Feishu `thread_root_id/root_id` 绑定 topic-scoped engine context；它们属于执行策略，不覆盖 chat+project 级普通编程工具状态。
- SMART/project chat 的 shell heuristic 保留 `./restart.sh rr`、`bash ./restart.sh rr`、`uv run ...` 等 shell-like 文本为 shell 执行，避免被项目群自由文本编程入口抢走。

### 长期原则

`AGENTS.md` 的 Strategy And Transport 部分新增状态作用域原则：SMART 是默认 chat/project 状态；普通工具入口是持久 chat+project 编程状态；Deep/Spec/WT 是 topic/root-thread scoped engine strategy；shell-like 文本在 SMART 中必须继续走 shell。

## 状态流转二次复核

### 发现

- 普通编程状态实现已基本符合契约：`/coco`、`/codex` 等经模型选择后进入 chat+project 级 `ModeManager` 状态，普通消息直接转发到当前工具，`/exit` 清回 SMART。
- Worktree 已真正接入 topic-scoped context：`/wt` 入口绑定 `ThreadContext(mode="worktree")`，卡片 action payload 携带 `thread_root_id`，运行态存在 `WorktreeSessionStore(project_id, chat_id, thread_root_id)` 中。
- Deep/Spec 的 topic-scoped 目前只有路由层占位，没有看到启动层调用 `ThreadContextManager.bind_engine()` 注册 `mode="deep"` 或 `mode="spec"`；相关测试也主要手工构造 `auto_enter_mode="spec"`，未覆盖真实启动后的话题续聊。
- topic engine 切换拦截只识别 `/deep`、`/spec`、`/worktree` 三个根命令；`/spec_resume`、`/spec_guide`、`/deep_update`、`/stop_deep` 等前缀命令在其它 engine 话题内可能绕过同话题不同 engine 拦截。
- ACP pending prompt 暂存 key 只有 `chat_id:tool`，不含 `project_id`、`thread_root_id` 或 origin message；同一 chat 内多个项目/多张模型选择卡并发时可能覆盖或错投。并且 `handle_select_acp_model()` 在确认进入模式前先 pop pending，启动失败会丢失原始需求。
- ACP 工具/模型状态在 session 启动或模型切换成功前已写入 project/handler；如果启动失败或 switch fallback 失败，项目上保存的工具/模型可能与实际会话不一致。
- Worktree 自动清理成功会 reset runtime state 并关闭卡片 session，但没有移除 `ThreadContext(mode="worktree")`；按“engine topic 是持续策略状态”的产品预期，同话题后续消息不应走 SMART，而应继续按 WT 策略启动/追加下一轮 WT 任务。当前 `is_awaiting_goal()` 只在 PENDING/AWAITING_GOAL/AUTO_EXECUTING 阶段拦截，COMPLETED/FAILED/IDLE 都会掉回 SMART。

### 契约校正

用户澄清：WT/Deep/Spec 话题状态不是一次性运行期上下文，而是该 Feishu 话题的持续编程策略。一个任务完成后，在同一话题继续发送普通文本，应继续进入原 engine 策略（WT 继续 WT，Deep 继续 Deep，Spec 继续 Spec），除非用户显式退出或在主对话/新话题切换策略。

按这个契约，当前实现的核心缺口是：

- WT topic context 已注册，但续聊判定错把 `WorktreeManager.is_awaiting_goal()` 当作唯一入口；该 helper 只适合“空 `/wt` 等待第一个目标”，不适合“已完成后同话题追加下一任务”。
- Deep/Spec 既缺启动时 `bind_engine()`，也缺 `auto_enter_mode in {"deep", "spec"}` 下普通文本直接追加/重启对应 engine 的分发路径。
- `/exit` 在 engine-only topic 内当前会尝试按 `InteractionMode` 退出；对 `worktree/deep/spec` 这类非 `InteractionMode` 值没有清晰的 topic-context remove/stop 语义。

## WT/Deep/Spec 话题持久策略落地

### 变更

- `FeishuWSClient._dispatch_message_logic()` 在命中 `auto_enter_mode in {"worktree", "deep", "spec"}` 且不是 slash/system command 时，不再回到通用 intent 路由：WT 普通文本直接进入 `handle_worktree_execute()`，Deep/Spec 普通文本分别进入 `start_deep_engine()` / `start_spec_engine()`，从而保证同一 Feishu topic 完成上一任务后继续保持原 engine 策略。
- topic engine 切换拦截从根命令扩展到命令家族：WT 识别 `/wt`/`/worktree`，Deep 识别 `/deep`、`/deep_update`、`/deep_status`、`/stop_deep`，Spec 识别 `/spec_*` 常用命令与 `/stop_spec`；同 topic 内同 engine 命令仍可用，不同 engine 命令会提示另开话题/主对话。
- `BaseEngineHandler` 新增 `_ensure_topic_engine_context()`，Deep/Spec 启动时绑定 `ThreadContext(mode="deep"|"spec")`；如果命令不在已有话题内，命令消息自身成为该 engine 话题 root。
- engine-only topic 的 `/exit` 现在移除对应 `ThreadContext` 并清空当前 thread-local，后续该话题消息回到 SMART；不隐式停止已运行 engine，停止仍由 `/stop_deep`、`/stop_spec` 或 WT 卡片/流程命令负责。
- `WorktreeManager.cleanup_worktrees()` 成功清理后保留 topic 内已选择的 WT 工具/模型组合，只清空 worktree units/git 运行态，确保同话题下一条普通目标能复用此前 WT 编程配置。
