# 2026-07-22

## 普通编程模式迁移到官方 Lark Channel SDK

### 更改内容

- 主 Bot 入站 WebSocket 从 `lark-oapi` WebSocket 客户端迁移到 `lark-channel-sdk`，保留既有事件分发、活动观测、重连健康检查和优雅关闭语义；普通 REST 消息能力仍使用官方 `lark-oapi`。
- 新增 `LarkChannelCardAPIClient`，普通 Coco/Claude/Aiden/Codex/Gemini/TraeX/TTADK 编程卡片通过共享 `FeishuChannel` 使用 CardKit：创建流式实体、发送卡片引用、文本元素增量更新、结构化实体更新和终态 `finish`。
- 保留 `CardSession`/`CardDelivery` 为卡片状态唯一事实源，因此项目、工具/模型、任务清单、当前进展、工具调用、子代理独立卡、总执行时长和分页信息继续由原渲染管道提供。
- 普通编程卡从首帧开始预分配流式实体；纯文本增量使用 element update，结构变化使用 entity update。所有 CardKit 操作使用单调 sequence，历史分页冻结前和终态均显式 finish。
- 出站审计在创建远端实体之前执行；Channel 路径默认拒绝退回普通内联卡片，引用发送失败时回收孤立实体，避免未审计写入或把 `om_*` 消息 ID 当作 CardKit 实体 ID。
- 首卡不可见、终态投递失败时中止剩余重试并以文本兜底；主卡和全部子代理卡完成终态投递后任务才退出，确保共享 Channel 不会先于投递线程池关闭。
- 修正 stale `expected_conn` 断连误报，并让健康监控读取 Channel SDK 的事件循环。

### 设计原因

- 用户要求普通模式使用官方 Channel SDK 的流式能力，同时不能丢失现有富卡片信息。复用现有纯渲染和交付状态机，仅替换普通编程模式的传输适配层，可以保持执行策略与传输维度分离。
- CardKit 实体和消息引用是两阶段操作；审计、孤儿回收、sequence 冲突处理及终态屏障必须在适配器和交付层显式建模，不能依赖 SDK 隐式重试。
- Deep/Spec/Worktree/Workflow 和静态文本/媒体未纳入本次传输切换，避免扩大共享路由改动范围。

## 编程卡片过程摘要与工具活动收敛

### 更改内容

- 修复普通编程卡在每次 `TOOL_STARTED` 时强制结束 reasoning block 的行为；同一来源在一次任务内复用稳定的过程摘要块，工具调用不再把短摘要切成多张“思考完成”面板。
- 共享 `ACPStreamBridge` 仍在工具边界切分正文流，但不再切断 reasoning；Deep、Spec 等共享桥接调用因此保持正文顺序，同时获得一致的摘要聚合语义。
- reducer 支持对既有 reasoning block 重新激活，避免复用稳定 block ID 时追加重复块。
- 将“深度思考中 / 思考完成”改为“正在分析 / 过程摘要”，准确表达模型提供的是可展示摘要而非原始思维链；不对模型原文进行启发式翻译或改写。
- 连续工具调用仍完整保留在活动审计中，并由现有 digest 聚合为单个探索/工具面板。
- 新增 `ux/programming-reasoning-summary.html`，对照展示修复前后的卡片层级和信息密度。

### 根因

- 旧实现为规避共享 block ID 导致内容覆盖，在工具开始事件上同时关闭正文和 reasoning，并递增 reasoning 边界序号；渲染器又会在任意非工具 atom 前冲刷待聚合工具，因此形成“单个工具面板 + 单句思考面板”的交替瀑布。
- 事件本身没有重复；问题是边界建模过度、标题语义不准确和视觉层级过强。
