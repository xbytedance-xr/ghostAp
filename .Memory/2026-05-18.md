## Slock 文档方案缺口补齐

- 背景：按飞书方案文档 `HdCxd5VHuoep8vxzYU3cVXI3nIg` revision 42 重新复核 slock 当前实现。方案明确要求 `/slock` 激活后团队内命令可用、状态/记忆落项目本地 `.ghostap/slock`、每个 Agent 具备 Directive 并初始化 L1 `MEMORY.md`、消息自动归档为 JSONL。
- 发现缺口：
  - `/slock` 在当前群激活后只创建 engine/channel，但未 `register_managed_chat(chat_id)`；dispatcher 的 `/new-role`、`/role`、`/task`、`/team` 捕获依赖 managed chat，因此快速开始路径实际会掉到下游。
  - `SlockEngine` 未显式指定 `memory_base_path` 时走 `~/.ghostap/slock`，不符合方案中的项目本地 `.ghostap/slock` 存储原则。
  - 激活 channel 只写 legacy `root/slock/{chat_id}/.slock_channel.json` marker，未写 canonical `.ghostap/slock/groups/{chat_id}/.slock_channel.json`。
  - `/new-role` 默认 `system_prompt` 为空，也未初始化 L1 `MEMORY.md`；方案第十二章要求 Agent 必须有 Directive 并零人类交互。
  - `SlockEngine._execute_agent()` 无消息归档，方案第 6.7 要求按群 JSONL 归档。
- 修复：
  - `SlockHandler.activate_slock()` 激活当前群后注册 managed chat，使团队内命令可被 dispatcher 捕获；`stop_slock_engine()` 既有 unregister 保持收口。

  - `MemoryManager` 新增 `message_archive_path()` / `append_message_archive()`；Agent 执行时归档 user 输入与 agent 输出到 `{base}/archives/{channel_id}/messages.jsonl`。

## 架构决策：接受证据替代继续删除

### 决策背景

`simple` 分支代码精简实测 707 行（0.34%），远未达到 20% 目标。代码库 205K 行中约 23K+ 行属于六大高风险只读区，继续精简必然触及并发正确性边界、外部协议契约、飞书 schema 规范等不可安全删除的区域。

### 架构决策

**接受 blocking evidence 作为精简终止条件**，不再继续追求 20% LOC 削减比例。转而为高风险区建立契约测试门禁（contract test gate），使后续精简具备自动化安全网。
