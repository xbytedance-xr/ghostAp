## TTADK ACP 输出噪声过滤与 JSON 提取修复
### 任务描述
修复 TTADK 编程模式下下游代理（如 claude 等）输出混入前缀噪声（路径/状态/error 文本）导致 ACP 解析异常的问题，要求从返回字符串中提取所有合法 JSON，并剔除非 JSON 内容。

### 执行内容
- 下钻 `src/utils/ttadk_wrapper.py`，确认旧实现只在“首个以 `{` 开头的行”后改为原样透传，导致两类问题：
  - JSON 前缀噪声（同一行前缀）无法进入透传；
  - JSON 开始后的后续噪声会被原样透传，污染 ACP 流。
- 新增 `src/utils/ttadk_wrapper.py:_extract_json_objects_from_line()`：
  - 对每一行先做 ANSI 清理；
  - 用 `json.JSONDecoder().raw_decode` 扫描整行，提取所有 JSON object/array；
  - 忽略同一行中的前后噪声文本。
- 重写 `src/utils/ttadk_wrapper.py:pump_filtered_stream()`：
  - 改为始终逐行读取；
  - JSON 起始前将无 JSON 行写入 `banner_tail`；
  - JSON 起始后继续过滤所有非 JSON 内容，只透传提取出的 JSON 行。
- 补充回归测试 `tests/test_acp_stdio_integration.py`：
  - `test_ttadk_wrapper_extracts_all_json_from_noisy_lines`：覆盖“前缀/中缀/后缀噪声 + 多个 JSON 同行”场景；
  - `test_ttadk_wrapper_drops_noise_after_json_started`：覆盖“JSON 开始后仍混入噪声”场景。

### 技术要点
- 根因是“基于首字符 `{` 的一次性切换 + 后续原样透传”策略过于脆弱，不适配 TTADK 代理不同下游工具时的混杂输出。
- 修复策略是把过滤粒度从“阶段切换”改为“逐行 JSON 提取”，确保 ACP 输入通道只包含结构化 JSON。
- 保留 `banner_tail` 失败诊断能力，不影响非 0 退出时的可观测性。

### 执行内容
- 审计关键模块并交叉核对：
  - `src/feishu/ws_client.py`（模式路由、handler 组合、系统命令串行化）
  - `src/feishu/handlers/programming.py`（普通编程模式模板与各工具子类）
  - `src/deep_engine/engine.py`、`src/loop_engine/engine.py`、`src/spec_engine/engine.py`（策略引擎）
  - `src/acp/manager.py`、`src/acp/provider.py`、`src/agent_session.py`（ACP/CLI 分流实现）
- 关键结论确认：
  - `ttadk_*` agent type 在 `ACPSessionManager` 中被强制分配 `SyncTTADKCLISession`，不走 ACP Server。
  - `coco/aiden/codex/gemini` 主链路走 ACP provider + `SyncACPSession`。
  - `claude` 当前常规编程链路走 `SyncClaudeCLISession`（CLI），并非“全量 ACP 直连”。
  - Deep/Loop/Spec 是编排策略层；底层会话能力与普通模式解耦。

  - 新增“Programming Strategy Layers”章节，明确四层策略与传输层分离。

  - 设计模式补充 Hybrid transport（ACP + CLI）和 `ttadk_*` 隔离规则。
  - 修正原有“Claude via ACP”等不准确描述。

### 技术要点
- 架构核心是“策略层 × 传输层”二维模型：策略负责任务编排复杂度，传输层负责工具协议适配。
