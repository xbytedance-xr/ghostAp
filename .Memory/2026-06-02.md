# 2026-06-02

## Traex ACP 编程工具接入

- 背景：本机 `traex` 命令支持 ACP server，`traex acp serve --help` 显示通过 `-c model="..."` 指定模型；需求是把 Traex 按 Coco/Codex 类 ACP 编程工具接入 GhostAP，并让 `/traex` 在普通编程、Deep、Spec、Worktree 等路径可用。
- 修复：
  - `src/acp/providers/__init__.py` 新增 `TraexProvider`，命令为 `traex acp serve`，模型参数风格为 `-c model="gpt-5"`；`src/acp/helper.py`/`src/acp/provider.py` 将 traex 纳入 ACP 工具列表和预热目标。
  - `src/mode/manager.py`、`src/project/context.py`、`src/project/unified_context.py` 新增 Traex 模式、项目级状态和 `traex_session_snapshot`，复用通用 programming mode/snapshot 序列化。
  - `src/feishu/handlers/programming.py` 新增 `TraexModeHandler`；`ws_client`、`session_hub`、`handler_context`、`router`、`dispatcher`、`intent_recognizer`、`system` 命令路由接入 `/traex`、`/enter_traex`、`/exit_traex`、`/traex_info`，其中 `/traex` 与 `/codex` 一样先进入 ACP 模型选择卡。
  - Card/UI 侧补齐 Traex 标题、按钮、帮助文案、header/tool display、mode footer 和 renderer shared mapping。
  - Deep/Spec/Worktree 侧通过 `resolve_engine_identity()`、Spec manager/engine 推断和 `worktree_engine/tool_discovery.py` 静态工具注册，让 Traex 作为 ACP backend 可被策略路径识别；Slock intent router 也把 `traex` 纳入已知工具名，避免自然语言工具名处理遗漏。
