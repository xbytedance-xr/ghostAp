## Durable Ingress Phase 3 Task 5：最终权限栅栏与终态附件处置

### 复审发现与修复

- 第二轮独立审查发现 authority provider 采样与最终 queue/rebalance/dequeue 提交之间仍可锚定
  workforce 撤权事件。最终 writer guard 现在用采样的 projection sequence/hash 对最新 Journal 做
  纯本地 fence；发现任何漏过的 workforce event 时 newcomer durable `authority_stale`，不得排队、
  驱逐合法 victim 或取得 dispatch grant。
- requester ACL 只接受精确布尔 `True`；整数、字符串、任意 truthy object 与异常统一 durable
  `requester_denied`。parent-trusted Channel status 要求完整 tenant/bot/app/agent/connection 字符串、
  enum-backed READY state，以及非 bool 的正整数 generation；`3.0`、`True`、raw string state 等畸形
  shape 默认拒绝。
- attachment staging 不再二次读取未经校验的 registry。Router 使用私有冻结
  `_AuthorityResolution` 临时携带已验证 credential ref；它不进入 repr、Journal、snapshot 或 dispatch
  grant。外部 staging 前重复完整 registry/binding/employee/Channel/membership/ACL 校验，staging 后再做
  第三次校验；credential 轮换或撤销时终止并清理，不进入队列。

### 终态附件清理所有权

- 自审证明 Task 4 completed staging 在 Router 因 `queue_full`、`queue_rebalanced`、dequeue 撤权或
  dispatch finish 进入 terminal 后没有其他清理 owner，用户附件字节会残留。Router 现在仅通过 Task 4
  的 `completed_for_acceptance`/`cleanup`/`recover` 公共端口清理 terminal records；不直接读取
  attachment service `.state`，不清理 queued、dispatching 或当前 grant。
- `route`、`dequeue`、`finish` 在 Router mutex 与 Journal transaction guard 全部释放后的 `finally`
  执行 terminal-only sweep。清理异常不覆盖已经提交的 Router 结果，只写入无密
  `RouterAttachmentCleanupReport`；显式 `recover_terminal_attachments()` 先收敛 Task 4 中断状态再重扫，
  重复调用幂等。
- Task 4 `completed_for_acceptance()` 对 cleanup-completed tombstone 返回 `None`，因为它已不再是可复用
  stage。真实 `AttachmentStagingService` 跨服务测试证明 stage → Router queue-full terminal → cleanup，
  随后两次 recover 均无 failed/cleaned 重复项，Journal 只有一次 cleanup-completed。
