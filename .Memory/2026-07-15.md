# 2026-07-15

## 团队模式直接可用与真实员工协作闭环

### 问题与根因

- `logs.log` 中 `/hire` 的 Slash 对账持续失败。真实 Lark GET 响应会省略可选的
  `description.i18n`，旧解码器却要求字段集合完全相等，导致员工卡在
  `CONFIGURING`，未决 Slash effect 保持 `EXECUTING`。
- 启动恢复对失败 intent 做无限指数退避，且在全部 intent 收敛前不调用
  `mark_runtime_recovered()`；单个坏员工因此会永久关闭整个 `/hire` admission。
- `/hire`、`/role add` 和扩展命令面板仍发出 Schema 2.0 已移除的
  `tag: action`，对应日志连续出现卡片 Schema 拒绝。
- `/fire` 只解析 `ACTIVE` 员工，无法清理已经拥有 BotPrincipal/credential、但卡在
  `CONFIGURING`/`VALIDATING`/`ACTION_REQUIRED` 的真实员工。
- 普通团队任务仍进入 legacy Slock `AgentRegistry`；真实可见员工不在该 registry，
  因而团队即使已雇佣员工也会报告没有角色。
- 主 Bot 审计文件打开失败时退化为常量零，激活验证存在 fail-open；注册回调也会在
  employee event loop 中同步执行阻塞式飞书回复。

### 实现

- Slash GET 解码保持未知字段严格拒绝，但允许缺省可选 `i18n`。
- 恢复改为 5 次有界退避；耗尽后把该 hire 的全部未决 effect 锚定为
  `ACTION_REQUIRED(error_code=recovery_exhausted)`，隔离单个员工并重新开放全局
  admission。新 hire 同样对可重试配置故障做有界收敛。
- 主 Bot send audit 打开失败现在直接阻断 runtime composition；不再使用常量零。
- `/hire` 工具/模型、`/role add` 和扩展面板改用 Schema 2.0
  `select_static`、`column_set`、`form`；`ux/slock-team-employee-flow.html` 是生产卡片
  对齐预览。
- Fire authority 允许管理员清理所有已绑定真实凭据的非终态可见员工；退役 admission
  在同一 Journal frame 中先处置未决 hire effects，再提交 `RETIRING` 和 ingress
  closure。
- `AsyncCallbackBridge` 把同步回调放到 worker thread；hire ready/action-required
  通知通过独立的 `hire.notification.*` PREPARED/EXECUTING/COMMITTED effect 锚定后发送。
- 新增 `src/autonomous/team/`：普通任务使用真实 ACTIVE 可见员工完成
  analyst -> reviewer -> synthesizer 三段协作。每次 dispatch 和通知均先写 Journal
  effect；协调器使用专用 `ghostap.team.assignment.v1` 加密 ingress，继续经过 tenant、
  group membership、requester ACL、Channel generation、ProjectedAgentRegistry 和唯一
  ACP gateway 校验，不写入 legacy registry，也不放宽 Bot-loop 拒绝。
- 团队预算固定为 handoff 8、depth 4、fanout 4；失败步骤最多换一个员工重试，仍失败
  则终止为 `team.run.action_required` 并通知用户。任务和交接正文只进入加密 ingress；
  Journal 只保存摘要、坐标和 history record id。

## Slash 图标响应导致退役与同名重聘失败

- 线上 `/fire 柳七月` 在 `slash_cleanup` 进入 `ACTION_REQUIRED(outcome_unknown)`；随后同名
  `/hire` 被 Workforce 唯一名称约束拒绝，但处理器只记录 `HireAdmissionError` 类型，
  用户只能看到无操作价值的通用失败提示。
- 字段级只读探针确认飞书 Slash GET 返回 HTTP 200，命令项额外包含
  `icon: {icon_key: ...}`；当前 `lark-oapi==1.7.1` 的 `AppSlashCommand` 模型未声明该线上
  响应字段，严格解码因此错误拒绝整个合法响应。
- 解码器仅新增 `icon`/`icon_key` 精确白名单和类型验证，继续拒绝其他未知字段；线上只读
  `observe_empty` 已从 `Slash GET response schema is invalid` 恢复为 `False`，准确表示仍有
  命令可供退役流程清理。
- `/hire` 对同名冲突和容量不足给出 `/roster`、`/fire` 操作指引；`/fire` 若停在 Slash
  清理未知结果，会明确提示再次安全执行 `/fire <名字>`，且归档前不要同名重聘。
- 该员工在配置阶段失败，从未创建本地 Agent 工作区；退役归档现在把这种状态收敛为包含
  空文件清单的原子归档，而不是在凭证销毁后因源目录不存在再次卡入人工处理。
- 回归覆盖官方响应 icon 变体、未知 icon 字段拒绝、同名重聘提示和 Slash 清理重试提示。
