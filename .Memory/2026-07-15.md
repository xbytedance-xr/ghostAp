## 已有应用员工 `/status` 无响应与跨应用身份修复

### 问题与根因

- 线上 Journal 证明用户通过官方注册页选择已有应用后，应用注册、凭证入 Vault、
  Slash 对账和员工 Channel 启动均已提交，员工已进入
  `READY_PENDING_VERIFICATION`；已有应用注册流程本身没有中断。
- 两条员工私聊 `/status` 均被 Channel 接收，但主应用与员工应用会为同一个用户分配
  不同的 app-scoped `open_id`。旧验证把员工应用 sender `open_id` 与主应用 hire
  requester `open_id` 直接比较，因不相等而在回复前静默拒绝。
- 启动恢复还会遍历历史 `ACTION_REQUIRED` fire intent，并尝试把已由后续 fire
  成功归档的员工重新写为 `ACTION_REQUIRED`，触发 archived terminal 投影错误并设置
  `fire_recovery` blocker。

### 实现

- 主 Bot 消息和卡片回调从官方事件提取 `union_id`，经 `TaskSpec` 与 request-scoped
  ContextVar 传到 `/hire`；缺失跨应用身份时 fail-closed，不再创建无法安全激活的员工。
- `employee.created`、`DurableHireState` 和 verification challenge 持久化 requester
  `union_id`。旧 Journal 事件继续兼容重放，并提供锚定的
  `hire.requester_identity_bound` 事件给存量 pending hire 做一次性迁移。
- 员工 Channel 的 RAW 事件只跨 IPC 保留 event/tenant/message/union 四项权威元数据；
  验证用 `union_id` 授权，用员工应用 `open_id` 作为实际回复目标。激活证据同时记录
  sender `union_id`，不放宽 tenant、P2P、`/status`、员工发送回执或主 Bot 零代发门禁。
- Fire authority 对已归档员工的历史 `mark_action_required` 恢复请求做幂等 no-op，
  不再试图突破 ARCHIVED 终态。

### 验证

- 跨应用回归明确使用不同 `open_id`、相同 `union_id`，验证员工回复后进入 ACTIVE；
  不匹配 union、伪造员工发送回执及缺失 RAW 权威元数据仍全部拒绝。
- 定向回归：`220 passed`、`128 passed`、`170 passed` 等批次均通过。
- Autonomous 全量：`1858 passed, 1 skipped, 1 warning`（6m57s；warning 为
  `lark_channel` 内 protobuf 的 `utcfromtimestamp` 弃用告警）。
- `ruff check`、`python -m src.main --validate`、`git diff --check` 均通过。

### 线上迁移与风险

- 现网存量 pending hire 早于本修复创建，没有持久化 requester `union_id`。部署时在
  服务停止且 Journal 单写者释放后，从同一主应用已验证事件中解析唯一 open/union
  映射，并已把 `hire.requester_identity_bound` 锚定为 Journal sequence 68。
- 新进程 PID `2123554` 已连接主 Bot 与员工 Channel；恢复流程提交 sequence 69–78，
  重新完成 Channel generation、Slash 证据和 activation challenge。sequence 79–84
  同时处置并擦除修复前两条无法授权的 `/status` ingress。
- 新启动日志没有 `employee retirement recovery failed`、runtime composition error 或
  traceback；历史 fire intent 对已归档员工的恢复已验证为 no-op。
- 新 challenge 晚于修复前两条 `/status`，因此旧消息不会被不安全重放；管理员需在员工
  私聊中重新发送一次 `/status` 完成最终 ACTIVE 转换。
- `union_id` 仍按飞书官方跨应用身份语义 fail-closed；若员工应用事件不提供该字段，
  系统不会退回不安全的 app-scoped `open_id` 比较。

## 团队模式直接可用与真实员工协作闭环

### 问题与根因

- `logs.log` 中 `/hire` 的 Slash 对账持续失败。真实 Lark GET 响应会省略可选的
  `description.i18n`，旧解码器却要求字段集合完全相等，导致员工卡在
  `CONFIGURING`，未决 Slash effect 保持 `EXECUTING`。
- 启动恢复对失败 intent 做无限指数退避，且在全部 intent 收敛前不调用
  `mark_runtime_recovered()`；单个坏员工因此会永久关闭整个 `/hire` admission。
- `/hire`、`/role add` 和扩展命令面板仍发出 Schema 2.0 已移除的
  `tag: action`，对应日志连续出现卡片 Schema 拒绝。
- `/fire` 只解析 `ACTIVE` 员工，无法清理已经拥有 BotPrincipal/credential、但卡在
  `CONFIGURING`/`VALIDATING`/`ACTION_REQUIRED` 的真实员工。
- 普通团队任务仍进入 legacy Slock `AgentRegistry`；真实可见员工不在该 registry，
  因而团队即使已雇佣员工也会报告没有角色。
- 主 Bot 审计文件打开失败时退化为常量零，激活验证存在 fail-open；注册回调也会在
  employee event loop 中同步执行阻塞式飞书回复。

### 实现

- Slash GET 解码保持未知字段严格拒绝，但允许缺省可选 `i18n`。
- 恢复改为 5 次有界退避；耗尽后把该 hire 的全部未决 effect 锚定为
  `ACTION_REQUIRED(error_code=recovery_exhausted)`，隔离单个员工并重新开放全局
  admission。新 hire 同样对可重试配置故障做有界收敛。
- 主 Bot send audit 打开失败现在直接阻断 runtime composition；不再使用常量零。
- `/hire` 工具/模型、`/role add` 和扩展面板改用 Schema 2.0
  `select_static`、`column_set`、`form`；`ux/slock-team-employee-flow.html` 是生产卡片
  对齐预览。
- Fire authority 允许管理员清理所有已绑定真实凭据的非终态可见员工；退役 admission
  在同一 Journal frame 中先处置未决 hire effects，再提交 `RETIRING` 和 ingress
  closure。
- `AsyncCallbackBridge` 把同步回调放到 worker thread；hire ready/action-required
  通知通过独立的 `hire.notification.*` PREPARED/EXECUTING/COMMITTED effect 锚定后发送。
- 新增 `src/autonomous/team/`：普通任务使用真实 ACTIVE 可见员工完成
  analyst -> reviewer -> synthesizer 三段协作。每次 dispatch 和通知均先写 Journal
  effect；协调器使用专用 `ghostap.team.assignment.v1` 加密 ingress，继续经过 tenant、
  group membership、requester ACL、Channel generation、ProjectedAgentRegistry 和唯一
  ACP gateway 校验，不写入 legacy registry，也不放宽 Bot-loop 拒绝。
- 团队预算固定为 handoff 8、depth 4、fanout 4；失败步骤最多换一个员工重试，仍失败
  则终止为 `team.run.action_required` 并通知用户。任务和交接正文只进入加密 ingress；
  Journal 只保存摘要、坐标和 history record id。

## 员工 Channel bwrap 验真与 macOS 隔离兼容

### 根因与安全审查

- Linux warning 的根因不是 Bubblewrap 或 user namespace 不可用，而是 supervisor 用
  `Popen.pid` 验真。该 PID 是宿主 namespace 内的外层 bwrap monitor；真实 worker PID
  由 bwrap `--info-fd` 的 `child-pid` 返回。
- `--json-status-fd` 是生命周期多帧通道，提前关闭会让 bwrap 退出写最终帧时收到
  SIGPIPE；改用一次输出后 EOF 的 `--info-fd`，并以 10 秒、4 KiB 上限严格解析。
- 初版 macOS 方案还暴露三个凭证后失败风险：worker 硬拒绝非 Linux、SDK preflight
  需要读取 `pyproject.toml`/`uv.lock`、controlled import cache 需要临时目录读写。经
  grill-me 与三轮子代理审查后一起修复。

### 实现

- Linux bwrap 增加 `--as-pid-1` 与每次启动独立 info pipe；outer PID 只用于
  wait/terminate，inner PID 用于 user/mount/PID namespace、rootfs 与稳定
  `/proc/<pid>/stat` starttime 验真。沙箱精确只读绑定 `src`、`pyproject.toml`、
  `uv.lock`，继续隐藏 `.env`、`.git`、`.Memory` 与 `AGENTS.md` canary。
- macOS 使用固定 `/usr/bin/sandbox-exec`、deny-default Seatbelt profile、随机 nonce
  proof FD 与 PID 精确绑定。worker 先完成 RLIMIT_CORE、`PT_DENY_ATTACH`、SDK identity
  和 cache preflight，再证明 source/runtime 可读且真实 `AGENTS.md` 以 EACCES/EPERM
  被拒绝；父进程验真后才解析 Vault secret 和写 bootstrap pipe。
- 每个 macOS Channel 创建独立 `0700` 临时目录，只把该目录以 Seatbelt 参数、
  `GHOSTAP_CHANNEL_TMP` 和 `TMPDIR` 开放；正常停止、crash、proof/launch/setup 失败均
  回收目录和全部 protocol/metadata FD。生产平台不能通过测试 override 改变 fallback
  策略；macOS 缺少/拒绝 Seatbelt 时 fail-closed，Linux 保留既有明确
  `verified=false` process fallback。
- `restart.sh` 启动前先用 `uv sync --check --group dev` 判断环境，缺失/漂移时自动同步；
  Linux 缺 `/usr/bin/bwrap` 时通过 apt/dnf/yum/zypper/pacman/apk 自动安装
  `bubblewrap`，无交互 sudo 使用 `-n` 避免挂起；macOS 只探测系统
  `/usr/bin/sandbox-exec`，不安装 Linux bwrap。

### 验证与剩余风险

- 定向安全、合约与 restart 回归：`60 passed`；新增真实 Linux 能力门禁、outer/inner
  PID、proof schema/nonce/PID/errno、凭证顺序、Darwin hardening、第四条 FD、temp
  setup 失败与清理回归。
- 两轮 Autonomous 全量分别为 `1868 passed, 1 skipped` 和最终
  `1869 passed, 1 skipped, 1 warning`（最终 6m51s）；warning 仍为第三方
  `lark_channel` protobuf 的 `utcfromtimestamp` 弃用提示。
- 真实 Linux production worker 探针确认 bwrap inner PID verified、无密钥 SDK preflight
  完成后仍存活，锁文件可读且 canary 隐藏。`ruff check src/autonomous/`、涉及测试
  Ruff、配置校验、`uv sync --check`、`bash -n restart.sh`、diff-check 均通过。
- 当前没有 macOS 真机，因此只完成 Seatbelt contract 与 launcher 模拟回归。Apple 已把
  `sandbox-exec`/SBPL 视为 deprecated/非第三方稳定接口；目标 Intel/Apple Silicon 和
  macOS 版本仍必须验证 profile 编译、PT_DENY_ATTACH、DNS/TLS/WSS。长期产品边界应迁移
