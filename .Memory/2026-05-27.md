## Slock ACP 工具过滤与失败升级修复

- 问题：18:02-18:03 日志里 Slock reviewer 创建 Claude session 后直接报 `ACP session does not support Slock tool filtering`；随后 agent 失败升级又记录 `Invalid agent transition: idle -> idle`；Codex ACP 执行失败时还伴随 provider `Internal error` 和 event-loop/subprocess 清理噪声。
- 根因：
  - Slock `_run_acp_session()` 为所有 agent 创建 session 后要求 `set_tool_filter()`，但 `create_engine_session(agent_type="claude")` 固定走 legacy `SyncClaudeCLISession`，该 session 不支持 Slock 的 per-agent 工具过滤。
  - `SyncACPSession.set_tool_filter()` 只把 filter 存在 wrapper 上，没有下传到真实 `ACPSession`/`GhostAPClient`，所以即使是 ACP session，文件/terminal/permission callbacks 也没有实际执行 Slock 工具边界。
  - `_execute_agent()` 捕获 ACP 异常后先手动把 agent 置为 `IDLE`，再调用 `escalate()`；`EscalationManager` 也会做 `RUNNING -> IDLE` 收尾，导致第二次变成 `idle -> idle` 非法迁移。
  - ACP 自动授权入口依赖当前 `SandboxExecutor` 默认策略，明显危险命令在 ACP permission auto-approve 层缺少硬拒绝兜底。
- 修复：
  - `create_engine_session()` 增加 `require_tool_filter` 参数；Slock agent session 创建时传 `True`，Claude 在该场景改走 ACP backend，不再使用不支持过滤的 legacy CLI session；普通 Claude 编程入口保持原行为。
  - `ACPSession` 保存并下传 tool filter；`SyncACPSession.set_tool_filter()` 同步传播到内部 `ACPSession`；`GhostAPClient` 在 `request_permission`、`read_text_file`、`write_text_file`、`create_terminal` callback 层执行 filter，filter 异常按 fail-closed 处理。
  - ACP 自动授权前复用危险模式策略硬拒绝明显危险 execute 请求；被拒绝的虚拟 terminal 使用 exit code `1`，避免非负 exit status schema 再触发二次错误。
  - `_execute_agent()` 移除 escalation 前的重复 `set_agent_status(... IDLE)`，由 `EscalationManager` 统一完成失败收尾。
  - `_apply_tool_restrictions()` 失败时立即 `close_session_safely(session)`，避免 session 创建成功但安全策略降级后泄漏进程。
- 验证：
  - 新增红灯覆盖：ACP callback filter 拦截文件/terminal/permission；Slock 要求 tool filter 时 Claude 走 ACP；ACP 失败升级不再出现 `idle -> idle`。

## tui2acp 终端态启动修复

- 问题：`/tui2acp` 选择 `claude` 后，tui2acp ACP 会话创建成功但底层 TUI session 立即进入 terminal state，GhostAP 后续 prompt 报 `Session ... is in terminal state`。
- 根因：`logs.log` 中对应 session 先出现 tui2acp `session/new`，随后 tui2acp stderr 报 `component=pty errorType=spawn_failed errorMessage=posix_spawnp failed.`；直接用 `node-pty` 启动 `/bin/echo` 也复现同样错误，说明是当前 Node/macOS 环境下 node-pty spawn 失败，不是 GhostAP 官方 ACP provider 或卡片渲染问题。
