## Slock 最近错误日志修复

- 问题：17:42-17:45 的 `logs.log` 里集中出现三类 Slock 错误：新/未管理群里发送 `/role list` 时被动激活把 slash 命令当作首条任务入队；`/role list` 卡片发送失败，Feishu 返回 `230099 / 200621 parse card json err`，原因是 `div` 元素带了 Schema V2 不支持的 `elements` 字段；Agent 执行失败触发 escalation 时日志报 `on_escalation: engine not found`。
- 根因：
  - `MessageDispatcher` 的 `NEEDS_ACTIVATION` 分支没有排除显式 slash 命令，`should_auto_activate()` 也没有对未管理群 slash 文本短路，导致 `/role list` 进入 `activate_slock(requirement=text)` 并被 `First message enqueued`。
  - `build_mobile_card_row()` 为多行移动端布局返回 `{"tag": "div", "elements": [...]}`，该结构被角色列表卡复用，但 Feishu Card 2.0 不接受 `div.elements`。
  - `on_escalation` 只查 `get_active_engine()`，执行失败后 engine 可能已经回到 idle，但 Slock channel 仍是 activated，升级卡因此找不到 engine。
- 修复：
  - 在 `NEEDS_ACTIVATION` 被动激活分支显式排除 `/...` slash 命令；`should_auto_activate()` 对未管理群 slash 文本返回 `False`，防止控制面命令被当任务文本。
  - `build_mobile_card_row()` 改为返回合法 `column_set`，把 title/content/action 元素放入单列，不再生成非法 `div.elements`；同步更新卡片断言。
  - `on_escalation` 保持 running engine 优先，找不到时兜底使用 activated Slock engine，避免失败后升级卡漏发。

## Slock Review 缺陷修复

- 问题：最近 Slock 改动引入/暴露了 4 类缺口：dispatch loop 取全局 Agent 可能跨群派单；`wake_policy` 只精确匹配 `on_mention` 且默认配置无校验；agent-to-agent `@mention` 路由仍走旧 `@(\w+)` + `find_by_name()`，无法匹配 `agent-id` 等 token；队友 roster 直接拼接用户字段，换行和 Markdown 控制字符会进入 agent prompt 结构。
- 修复：
  - `_dispatch_single_task()` 改为 `list_agents(channel_id=channel_snapshot.channel_id)`，队列任务只在当前 Slock channel 内派发。
  - `slock_default_wake_policy` 增加 settings validator，支持 `ON-MENTION` / `on-mention` 归一化为 `on_mention`，非法值直接配置报错；engine 层同样归一化 agent/channel policy。
  - `_route_at_mentions()` 改为支持 `A-Za-z0-9_.:-` 和中文 token，并复用 `AgentRegistry.find_by_at_token()`，与 router/registry 的 at-token 规范一致。
  - `_render_team_roster()` 对 name/role/personality_traits 做单行、长度和 Markdown 控制字符清洗，避免用户字段破坏 prompt 结构。
  - `/task assign` parser 恢复 deprecated 契约：返回 `UNKNOWN` 并引导用户直接在 Slock 群发送任务文本；同步旧 E2E/quoted parsing 测试。

## tui2acp 终端态启动修复

- 问题：`/tui2acp` 选择 `claude` 后，tui2acp ACP 会话创建成功但底层 TUI session 立即进入 terminal state，GhostAP 后续 prompt 报 `Session ... is in terminal state`。
- 根因：`logs.log` 中对应 session 先出现 tui2acp `session/new`，随后 tui2acp stderr 报 `component=pty errorType=spawn_failed errorMessage=posix_spawnp failed.`；直接用 `node-pty` 启动 `/bin/echo` 也复现同样错误，说明是当前 Node/macOS 环境下 node-pty spawn 失败，不是 GhostAP 官方 ACP provider 或卡片渲染问题。
- 修复：在 `/Users/jiataorui/workspaces/githubgood/tui2acp/src/pty/driver.ts` 中为 macOS 增加 Python PTY fallback。仅当 node-pty 抛 spawn 异常时启用，通过 Python `pty.fork()` 创建真实 PTY 来运行原 TUI adapter 命令，Node 继续通过 stdin/stdout 管道读写；保留 `/tui2acp` 作为 TUI bridge 模式，不改成官方 ACP native adapter。
- GhostAP 侧只做 tui2acp 专属恢复：`ProgrammingModeHandler.handle_response()` 中仅当 `interaction_mode == TUI2ACP` 且错误文本为 terminal state 时清理对应 `ACPSessionManager` session，避免同一个已终止 tui2acp session 被反复复用；普通 Claude/Codex/Gemini ACP 逻辑不触发该清理。
