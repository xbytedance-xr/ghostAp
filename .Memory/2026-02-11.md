## 架构审查修复（8 项）
### 任务描述
从架构师视角对全部代码做评审，聚焦"飞书丝滑远程编程 + 多并发"目标，修复关键架构问题。

### 执行内容

**1. Engine 状态卡死修复（CRITICAL）**
- 问题：`DeepEngine/LoopEngine` 的 `plan_and_execute()`/`execute()` 在 ACP 异常时 `_run_state` 未重置
- 方案：`finally` 块中 `self._run_state = EngineRunState.IDLE`（已在前序修复中完成）
- 新增 `_close_session_safely()` 方法，安全关闭旧 session

**2. EngineManagers 线程安全（CRITICAL）**
- 问题：`DeepEngineManager/LoopEngineManager` 的 `_engines` dict 被多线程并发访问，无锁保护
- 方案：新增 `self._lock = threading.Lock()`，`get_or_create()`/`remove()`/`cleanup_all()` 包裹在 lock 中
- `src/deep_engine/engine.py` — `DeepEngineManager` 加 `_lock`
- `src/loop_engine/engine.py` — `LoopEngineManager` 加 `_lock`

**3. ACPSessionManager 并发保护（CRITICAL）**
- 问题：`start_session()` 中关闭旧 session + 注册新 session 之间存在竞态
- 方案：`start_session()` 在 lock 中检查/关闭旧 session，新 session 注册也在 lock 中
- 提取 `_end_session_unlocked()` 避免死锁
- `src/acp/manager.py` — 重构 lock 使用

**4. resume() 会话泄漏修复（HIGH）**
- 问题：`resume()` 直接创建新 session，未关闭旧 session，导致 orphan 进程
- 方案：`resume()` 前先调用 `_close_session_safely()`
- `src/deep_engine/engine.py` + `src/loop_engine/engine.py`

**5. Loop startup timeout 误配修复（HIGH）**
- 问题：`LoopEngine.execute()/resume()` 使用 `loop_execution_timeout`（7200s）作为 startup_timeout
- 方案：改为 `acp_startup_timeout`（20s）
- `src/loop_engine/engine.py`

**6. inject_context/inject_guidance 队列模式（HIGH）**
- 问题：`_pending_context: Optional[str]` 单值，多次注入只保留最后一条
- 方案：改为 `list[str]`，`inject_context()` append，`_drain_pending_context()` 批量 drain + join
- `src/deep_engine/engine.py` — `_pending_context: list[str]`
- `src/loop_engine/engine.py` — `_user_guidance: list[str]`

**7. send_prompt 超时后 cancel agent（MEDIUM）**
- 问题：`send_prompt()` 超时后只抛 TimeoutError，不 cancel agent 进程，agent 继续占用资源
- 方案：`except TimeoutError` 中先调用 `self.cancel()` 再 re-raise
- `src/acp/sync_adapter.py`

**8. on_event 错误可见性 + 用户错误反馈（MEDIUM）**
- 问题：`session.py` 的 on_event 异常 `pass` 吞掉；`ws_client.py` 顶层异常仅 log 无用户反馈
- 方案：on_event 异常改为 `logger.warning()`；ws_client 异常时 reply_message 错误提示
- `src/acp/session.py` — `on_event` 异常 pass→warning
- `src/feishu/ws_client.py` — 顶层 except 增加用户消息回复

### 技术要点
- `_end_session_unlocked()` 模式：避免 `end_session()` 在持有 `_lock` 的 caller 中重入死锁
- inject_context 队列：多条消息用 `\n\n---\n\n` 拼接后作为单个 follow-up prompt 发送
- 超时 cancel：确保 agent 进程在超时后释放资源，而不是继续执行直到自然结束

### 修改文件
- `src/acp/manager.py` — 并发保护、_end_session_unlocked
- `src/acp/sync_adapter.py` — 超时 cancel
- `src/acp/session.py` — on_event warning
- `src/deep_engine/engine.py` — _close_session_safely、_pending_context 队列、EngineManager lock
- `src/loop_engine/engine.py` — _close_session_safely、_user_guidance 队列、EngineManager lock、startup timeout
- `src/feishu/ws_client.py` — 用户错误反馈
- `tests/test_loop_engine.py` — 适配 inject_guidance 队列

## ACP 实现缺陷修复（5 项）
### 任务描述
对 ACP 重构后残留的 5 个实现问题进行修复：inject_context 空操作、resume 空操作、引擎绕过 ACPSessionManager、配置缺失、agent 进程崩溃导致挂起。

### 执行内容

**1. DeepEngine.inject_context() 空操作修复**
- 问题：仅打日志，不实际发送上下文到 agent session
- 方案：存储 pending_context（线程安全），在主 prompt 完成后通过 `_drain_pending_context()` 发送跟随 prompt
- `src/deep_engine/engine.py` — 新增 `_pending_context` + `_context_lock` + `_drain_pending_context()` 方法

**2. resume() 空操作修复**
- 问题：DeepEngine.resume() 和 LoopEngine.resume() 直接 return self._project
- 方案：实现真正的恢复逻辑 — 启动新 ACP session，发送延续 prompt，继续迭代
- `src/deep_engine/engine.py` — resume() 新建 session + 发送继续 prompt + 完整事件处理
- `src/loop_engine/engine.py` — resume() 新建 session + 从已有迭代数继续循环

**3. Deep/Loop 引擎绕过 ACPSessionManager 修复**
- 问题：引擎直接 `SyncACPSession()` 创建会话，没有重试/渐进超时
- 方案：新增 `start_session_with_retry()` 工具函数（提取自 ACPSessionManager 的重试逻辑）
- `src/acp/sync_adapter.py` — 新增 `start_session_with_retry(agent_type, cwd, startup_timeout)` 函数
- `src/acp/__init__.py` — 导出新函数
- `src/deep_engine/engine.py` — plan_and_execute() + resume() 使用 start_session_with_retry
- `src/loop_engine/engine.py` — execute() + resume() 使用 start_session_with_retry

**4. acp_permission_auto_approve 配置缺失修复**
- 问题：GhostAPClient 硬编码 auto_approve=True，无法通过配置控制
- `src/config.py` — 新增 `acp_permission_auto_approve: bool = True`
- `src/acp/session.py` — ACPSession.start() 从 settings 读取 auto_approve

**5. Agent 进程崩溃中途挂起修复**
- 问题：agent 进程在 prompt 执行中崩溃，send_prompt() 阻塞至超时（最长 7200s）
- 方案：新增 watchdog 线程，每 5 秒检查 agent 进程存活，进程死亡立即 cancel future
- `src/acp/sync_adapter.py` — send_prompt() 新增 `_watchdog` 守护线程 + `stop_watchdog` 事件

### 技术要点
- inject_context 的跨线程安全：使用 `threading.Lock` 保护 `_pending_context`
- resume 的限制：ACP session 无法继承上一次的对话历史（不同进程），只能发送新的延续 prompt
- start_session_with_retry 复用了 ACPSessionManager 的渐进超时策略 `timeout * (1 + 0.5 * (attempt-1))`
- watchdog 守护线程使用 `threading.Event.wait(timeout=5)` 实现可中断的定时检查
- `asyncio.CancelledError` 转为 `RuntimeError` 提供用户友好的错误信息

### 修改文件
- `src/acp/sync_adapter.py` — start_session_with_retry(), send_prompt() watchdog
- `src/acp/session.py` — auto_approve 从 config 读取
- `src/acp/__init__.py` — 导出 start_session_with_retry
- `src/config.py` — acp_permission_auto_approve
- `src/deep_engine/engine.py` — inject_context, resume, start_session_with_retry
- `src/loop_engine/engine.py` — resume, start_session_with_retry
