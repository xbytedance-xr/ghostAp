## Workflow 执行期错误卡根因暴露修复

### 背景 / 症状
- 用户反馈 `/wf` 执行阶段显示通用系统错误卡：`Workflow 失败` 场景下只看到
  “发生内部错误，请重试。如问题持续，请联系管理员。”，详情收起但主卡没有可操作根因。
- 复查 `logs.log`：最近一条明确 WF 根因是 2026-06-23 脚本生成校验失败
  `Unbalanced parentheses: () balance is -1`，随后旧卡片结构又被飞书拒绝；相关卡片 schema 问题已在当天前几轮修复。
- 本轮定位到仍残留的执行期错误通道：`WorkflowHandler._build_workflow_callbacks().on_error()` 手写窄分类，
  然后调用 `BaseHandler.reply_error()`，导致所有未知 RuntimeError/桥接错误/脚本校验类错误被二次包成通用系统错误卡，

### 改动

    `AGENT_LIMIT`/`RUNTIME_TIMEOUT`/`CANCELLED` 映射为 `invalid_state`，其他错误映射为 `internal_error`。
  - 改为调用 `_reply_workflow_error(message_id, category, detail=...)`，detail 先经过 `_strip_internal_details()`，
    保留安全根因提示，不再走通用 `reply_error()`。
- `tests/test_workflow_confirm.py`：
  - 新增执行期 validation failure 回归：必须调用 WF 专用错误卡并带 `invalid_argument` detail。
  - 新增未知 runtime error 回归：必须调用 WF 专用错误卡并带 `internal_error` 的脱敏 detail。

## Workflow 执行进度卡切换修复

### 背景 / 症状
- 复核用户指出的 3 个问题，确认同属 `/wf` 确认执行后的进度卡链路：
  1. `_build_workflow_callbacks()` 的 `on_progress`/`on_done` 将 renderer 返回的 `header/elements`
     重新塞进 `CardBuilder.build_info_card(title=..., elements=..., template=...)`。
  2. `CardBuilder.build_info_card()` 当前签名要求第一个位置参数 `project`，且没有 `elements`/`template` 参数；
     因此回调内会抛 `TypeError`，再被 `except Exception` 静默吞掉，导致进度/完成卡不更新。
  3. `handle_workflow_confirm_start()` 在 `start_execution()` 后直接提交后台任务，没有先把确认卡切到运行中；
     如果后台任务排队或首个进度事件延迟，用户会继续看到确认卡。

### 改动
- `src/feishu/handlers/workflow.py`：
  - 新增 `_build_workflow_card_from_renderer_data()`，把 Workflow renderer 的纯输出
    `{"header": ..., "elements": [...]}` 转成完整 Feishu CardKit 2.0：
    `{"schema": "2.0", "config": ..., "header": ..., "body": {"elements": [...]}}`。
  - 新增 `_replace_or_send_workflow_rendered_card()`，统一处理 `update_card` 失败时新发卡片，并返回后续应更新的
    `message_id`。
  - `on_progress` 和 `on_done` 不再调用 `CardBuilder.build_info_card()`，直接包装 renderer 输出后更新卡片。
  - `handle_workflow_confirm_start()` 在清理 pending 后立即将 `engine.project.status` 置为 `RUNNING`，补齐
    requirement/script_path/started_at/selected_tools，并用 `WorkflowProgressRenderer` 原地替换确认卡；如果 patch 失败，
    新发执行卡并把后续 callbacks 绑定到新 `message_id`。
  - 复审发现 `/wf_status` 仍保留同源错误调用；同步改为复用 renderer 输出包装函数，避免状态查询卡走
    `build_info_card(elements=...)`。
- `tests/test_workflow_confirm.py`：
  - 新增回归测试，验证确认执行会在提交后台任务前立即 `update_card` 为完整 CardKit 2.0 执行卡。
  - 新增回归测试，patch 掉 `CardBuilder.build_info_card()` 后直接调用 progress/done callbacks，锁定回调不再走错误
    builder，且输出卡片没有裸 root `elements`。
  - 按复审补强：后台任务提交 mock 会在提交时断言进度卡已先更新；新增 `update_card` 失败 fallback 新发卡后，
    下一次 progress 使用新 `message_id` 继续 patch 的覆盖；新增 `/wf_status` 使用完整 CardKit 2.0 卡的覆盖。

    防止重复点击排入多个生成任务。
  - 后台生成任务增加异常兜底，未预期异常会记录日志并回用户可见错误卡，而不是静默留在旧状态。

  - 新增回归测试，锁定 review 完成回调不再内联调用 `_generate_and_show_confirm_card()`，而是提交 scheduler；

## Slock 可用性问题分析与优化方案

### 背景 / 症状
- 用户反馈 Slock 模式也存在不能顺滑启动和运行的问题，需要先确认当前代码已提交推送，再深入分析并制定优化方案。
- 检查 `git status` 与 `git log` 时，当前分支 `slock` 已同步到 `origin/slock`，最近一次代码提交为
  `e46e9cb fix(workflow): avoid unsupported note cards`。

### 关键发现
- `logs.log` 显示启动时恢复了 9 个 Slock engine，其中包含 `test_chat_006`、`test_chat_007`、`t_claim_1`、
  `t_claim_2`、`t_claim_3`、`ch_templates`、`ch_workspace` 等测试/演示 marker，说明全局存储恢复链路会把非真实飞书群也拉起。
- 真实群 `oc_8db1260b409f93efa03a2a87681ea285` 的计划多次出现 `No agent for role planner/coder/reviewer/tester`，
  最终被标记为 `all steps skipped/timed out`；持久化 agent 里该群只有 `writer` 角色，缺少默认协作链。
- Slock 卡片模板仍输出飞书 Schema 2.0 不支持的 `progress`、`note` 和 legacy `action` tag；日志中已有
  `not support tag: progress` 的卡片拒绝记录。
- `src/config/settings.py` 与 `src/slock_engine/role_bootstrap.py` 仍允许 `ttadk` 且不接受 `traex` 作为默认角色工具，
  与当前工具选择过滤和“以后回退 Traex、精简 TTADK”方向不一致。
- 当前缺少一键诊断入口。用户只能看到计划失败或卡片不更新，无法知道是缺角色、旧 marker、卡片结构还是工具不可用。

### 输出
- 新增实施方案文档：`docs/superpowers/plans/2026-06-25-slock-usability-hardening.md`。
- 方案按 5 个可独立提交的任务拆分：
  1. 增加 Slock Schema 2.0 卡片契约测试，移除 `progress`/`note`/`action`。
  2. 默认角色改为 Traex-first，并从 Slock bootstrap 中移除 TTADK。
  3. 运行时恢复默认跳过非 `oc_` 测试/演示 marker。
  4. 缺少角色时将计划置为可操作的 blocked 状态，不再静默全部跳过后失败。
