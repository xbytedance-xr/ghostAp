# 2026-06-25

## Workflow 脚本生成异步化

### 背景 / 症状
- 用户再次反馈 `/wf` 仍不能正常运转，截图停留在 “Workflow — 生成脚本中...”。
- 复查 `logs.log`：6 月 24 日 16:03 的旧失败是确认卡 `tag: note` 被飞书 Schema V2 拒绝，已由上一轮修复；
  6 月 25 日重启后未出现新的 `/wf` 卡片 schema 错误。
- 同一段日志显示旧流程在 `workflow_review_finish` 后进入 Traex 脚本生成 session，并在约 72 秒后才继续更新确认卡；
  代码确认该耗时调用在飞书按钮回调处理链路内同步执行。
- 根因：脚本生成可能耗时数十秒到数分钟，但 `handle_workflow_review_finish()` 直接调用
  `_generate_and_show_confirm_card()`，长时间占住回调处理；用户只看到 loading 卡，且长耗时回调会增加长连接健康风险。

### 改动
- `src/feishu/handlers/workflow.py`：
  - 新增 `_schedule_generate_and_show_confirm_card()`，将耗时的脚本生成与确认卡替换提交到现有 `TaskScheduler`。
  - `handle_workflow_review_finish()` 保存评审 Agent 和工具 allowlist 后，将状态切到 `GENERATING_SCRIPT` 并提交后台任务，
    防止重复点击排入多个生成任务。
  - 后台生成任务增加异常兜底，未预期异常会记录日志并回用户可见错误卡，而不是静默留在旧状态。
- `tests/test_workflow_orchestrator_select.py`：
  - 新增回归测试，锁定 review 完成回调不再内联调用 `_generate_and_show_confirm_card()`，而是提交 scheduler；
    同时验证后台闭包仍带着正确 requirement、project、root_path 和 `["coco", "claude"]` 工具列表。

- 定向回归：
  - 新增测试修复后通过。
  - `uv run python -m pytest tests/test_workflow_orchestrator_select.py tests/test_workflow_confirm.py tests/test_workflow_selection_controller.py -q`

  - `uv run ruff check src/feishu/handlers/workflow.py tests/test_workflow_orchestrator_select.py` → All checks passed。
  - `uv run python -m src.main --validate` → 配置校验通过。
  - `./restart.sh rr` → 远程重启触发成功；`logs.log` 显示新进程启动并连接飞书长连接成功。

### 后续风险
- 重新生成脚本和旧的工具确认入口仍沿用同步路径；本次先修复三步 `/wf` 主路径，避免扩大行为面。
- 后台任务开始前不会进入真正的模型调用；如果队列被其他同项目任务占满，用户仍可能短暂等待，但回调线程不会被模型调用占住。

## Slock 可用性问题分析与优化方案

### 背景 / 症状
- 用户反馈 Slock 模式也存在不能顺滑启动和运行的问题，需要先确认当前代码已提交推送，再深入分析并制定优化方案。
- 检查 `git status` 与 `git log` 时，当前分支 `slock` 已同步到 `origin/slock`，最近一次代码提交为
  `e46e9cb fix(workflow): avoid unsupported note cards`。

### 关键发现
- `logs.log` 显示启动时恢复了 9 个 Slock engine，其中包含 `test_chat_006`、`test_chat_007`、`t_claim_1`、
  `t_claim_2`、`t_claim_3`、`ch_templates`、`ch_workspace` 等测试/演示 marker，说明全局存储恢复链路会把非真实飞书群也拉起。
- 真实群 `oc_8db1260b409f93efa03a2a87681ea285` 的计划多次出现 `No agent for role planner/coder/reviewer/tester`，
  最终被标记为 `all steps skipped/timed out`；持久化 agent 里该群只有 `writer` 角色，缺少默认协作链。
- Slock 卡片模板仍输出飞书 Schema 2.0 不支持的 `progress`、`note` 和 legacy `action` tag；日志中已有
  `not support tag: progress` 的卡片拒绝记录。
- `src/config/settings.py` 与 `src/slock_engine/role_bootstrap.py` 仍允许 `ttadk` 且不接受 `traex` 作为默认角色工具，
  与当前工具选择过滤和“以后回退 Traex、精简 TTADK”方向不一致。
- 当前缺少一键诊断入口。用户只能看到计划失败或卡片不更新，无法知道是缺角色、旧 marker、卡片结构还是工具不可用。

### 输出
- 新增实施方案文档：`docs/superpowers/plans/2026-06-25-slock-usability-hardening.md`。
- 方案按 5 个可独立提交的任务拆分：
  1. 增加 Slock Schema 2.0 卡片契约测试，移除 `progress`/`note`/`action`。
  2. 默认角色改为 Traex-first，并从 Slock bootstrap 中移除 TTADK。
  3. 运行时恢复默认跳过非 `oc_` 测试/演示 marker。
  4. 缺少角色时将计划置为可操作的 blocked 状态，不再静默全部跳过后失败。
  5. 增加 `/slock doctor` 只读诊断，报告缺角色、旧 marker、工具可用性和下一步操作。

### 验证
- 本次仅提交分析方案和 Memory 记录，未修改运行时代码。
- 已用日志、全局 Slock 持久化文件和相关源码路径交叉确认问题来源。

### 后续风险
- 后续实现会触及 Slock 配置默认值、角色 bootstrap、卡片模板和协作计划状态机，必须按任务拆小提交并补回归测试。
