## 智能/编程状态流转与 restart.sh 远程执行修复

### 背景

用户希望把普通编程状态和 Deep/Spec/WT 引擎话题明确区分：单聊或项目群在任一时刻只有 SMART 或某个普通编程工具状态；`/exit` 回到 SMART，`/codex`、`/coco` 等显式命令进入普通编程状态；Deep/Spec/WT 继续在话题内保持上下文的一次性 engine 模式。重点问题是项目群 SMART 模式下发送 `./restart.sh rr` 不能远程重启，实际会被项目群自由文本/编程话题链路干扰，而不是稳定走 shell 执行。

### 实现

- `ProgrammingModeHandler.enter_mode()` 移除普通编程入口的 one-shot pending 分支：即使 `thread_programming_enabled=True`，主群 `/coco`、`/codex` 选模型后也会直接启动 chat/project 级 ACP session，并记录普通 `enter_*_mode` 转换。
- `MessageDispatcher` 在已处于编程状态时不再把无 session 的顶层消息转成 `_dispatch_to_thread()`；消息直接交给当前工具 handler，由 handler 负责 session recovery 和上下文延续。
- `SystemHandler.handle_select_acp_model()` 不再在顶层模型选择后创建编程话题；pending prompt 直接转发给当前工具，保持当前群内编程状态。
- `_dispatch_empty_text()` 不再因为顶层无 thread 而提示“发送需求创建编程话题”，而是按当前工具正常转发。
- SMART/project chat shell 判定补齐本地可执行脚本和常用 shell runner：`./restart.sh rr`、`../scripts/...`、`~/...`、`sh/bash/zsh`、`uv/pnpm/python/node` 等都会被识别为 shell-like，避免被项目群自由文本编程入口抢走。
- 更新用户可见文案：普通编程提示改为“当前对话/当前工具持续使用”，切换工具以 `/exit` 后再 `/codex`、`/coco` 为主路径；Deep/Spec/WT topic-scoped 逻辑未改。

## CardSession terminal 去重与项目群保存工具复用

### 背景

用户反馈消息卡片经常出现重复更新，并怀疑锁机制是否过复杂；同时项目群里 `/codex` 等编程模式使用不顺畅，期望一次选择工具/模型后，后续自由文本直接按该模式处理，用户需要变更时再 `/exit` 后重新选择。排查确认群锁和仓库锁主要是业务互斥，不直接参与卡片 PATCH；更近的根因在 CardSession 的异步投递合并层：terminal delivery 已在途时，重复 terminal event 仍可进入 `_delivery_pending`，第一个 terminal 成功后 session 才关闭，但 finally 仍会继续提交 pending terminal，导致同一终态卡片重复 PATCH。

### 实现

- `CardSession._submit_delivery()` 在 session 已关闭时直接忽略，并在 terminal delivery 已在途时丢弃后续所有 pending delivery（包括重复 terminal），避免终态卡片重复刷新。
- `_submit_pending_delivery_if_any()` 在发现 session 已关闭时清空 pending 状态并复位 in-flight 标记，防止 terminal 成功后继续投递旧 pending。
- 项目群自由文本若命中已保存的 `project.acp_tool_name`（coco/claude/aiden/codex/gemini），改走 `SystemHandler.handle_enter_acp_saved_selection()`，直接复用 `project.acp_model_name` 或默认模型处理 pending prompt，不再展示模型选择卡；普通编程入口的最终状态契约已在上方“智能/编程状态流转”中改为当前群内顶层编程状态。
- 保留显式 `/codex`、`/model`、首次无保存工具时的模型选择流程；用户要换工具/模型仍通过显式命令触发。
- 顺手修正 `test_multiple_hooks_all_called` 对 fire-and-forget dispatched hook 的跨线程顺序假设，只验证调用次数与 terminal exactly-once 语义。

## Spec 启动前拆解与 stale binding 瀑布优化

### 背景

用户基于 `hermes-r` Spec 任务日志指出两个启动/早期执行瓶颈：自然语言需求在进入 Spec 循环前会先通过独立 ACP 子会话做验收标准 LLM 拆解；旧卡片 binding 过期后，`99992354` 会持续触发串行 PATCH 失败。当前本地 `logs.log` 未包含用户引用的 `hermes-r` / `95455662` 片段，但代码路径确认了两个风险：`SpecEngine.execute()` 会在创建主 ACP session 前调用 `parse_acceptance_criteria(..., self._decompose_criteria_with_llm)`，而 `CardDelivery` 对 stale binding 只删除单页，仍会在同一次多页 delivery 中继续尝试其它旧页。

### 实现

- Spec 自然语言需求不再启动前调用辅助 ACP 拆解；`parse_acceptance_criteria()` 只解析显式 `- ` / `* ` / checkbox 列表，普通文本先形成单条 provisional criterion。
- 首轮 Spec artifact 仍会通过既有 `merge_acceptance_criteria()` 用模型生成的 `acceptance_criteria` 替换 provisional criterion，保留正式验收标准质量，但不阻塞任务启动。
- `CardDelivery` 命中 `recreate:*` outcome（当前主要是 `99992354` stale message）后，删除整个 session binding、重置剩余 page sequence，并短路本次剩余旧页 PATCH。
- 保留 `PageMutator` 单页 stale 处理作为底层兜底；上层 delivery 负责把“一个 stale page 证明整个 session binding 已过期”的策略收口到 session 级。

## Spec 首轮乐观收敛防护

### 背景

Spec 收敛链路存在首轮误判风险：如果验收标准在一次评估中从 0% 直接跳到 100%，且 review 同时 PASS，`ContinuationPolicy` 会立即返回 success；同时 `CriteriaTracker` 只会把标准从未满足推进到已满足，不支持后续评估把标准回退为未满足。这会让 LLM 自评过度乐观时提前结束。

### 实现

- `CriteriaTracker.update()` 支持已满足标准在后续评估中回退为未满足，并清理对应 `satisfied_at_iteration`。
- `ContinuationPolicy.should_stop()` 对多验收标准的单轮 0% → 100% 跳变增加确认轮保护，不在同一轮立即 success。
- `discover_optimization_questions()` 新增 `min_cycles` 参数；当全满足发生在最小轮次之前时，不应用“全满足即跳过 discovery”的门控。
- `SpecEngine._discover_optimization_questions()` 传入 `settings.spec_min_cycles`，让 discovery 门控与 Spec 最小轮次配置保持一致。
- 更新 convergence 测试，锁定全满足 discovery 门控在 early-cycle bypass 之后才生效。
- 补齐全量测试暴露的相邻回归：ACP 默认模型按钮加入后，card builder 测试改为按 payload 定位具体模型按钮；Settings 空管理员测试隔离本机 `.env`；`ReviewCircuitState.recent_outcomes` 默认改为有界列表，直接 append 也会裁剪到最近 20 条。

## WT 带目标/话题目标交互与主控 agent 规划修复

## Topic-scoped engine sessions 完整落地

### 背景

用户确认 WT 不做全局任务拆分/调度，保持单任务模型；同项目多个 WT 通过飞书话题天然并发，每个话题各自承载一个目标。Deep、Spec、WT 以及后续 engine 都应复用同一套“话题继续对话即继续当前模式”的能力，但必须避免旧的 chat 级隐式续聊导致用户在错误话题里误触发任务。

### 实现

- `ThreadContextManager` 新增 engine context helper：按精确 `thread_root_id/root_id` 查询、判断 active engine、绑定 engine topic。
- `FeishuWSClient` 移除按 chat 找最近编程话题的 fallback：只有命中 `root_id` 的话题上下文才自动继承 Deep/Spec/WT/编程 mode；未知 root 或无 root 只解析项目，不再偷连到最近会话。同一话题绑定 engine 后，切到不同 engine 的命令会被拦截提示。
- WT handler 强制启用话题语义：即使项目配置不是 thread 模式，`/wt` 也会用当前消息创建/绑定 `thread_root_id`，后续同话题普通消息可作为目标继续推进。
- 新增 `WorktreeSessionStore`，WT runtime state 从项目级 `ProjectContext.worktree_state` 迁到 `project_id + chat_id + thread_root_id` scoped key；保留项目 state 镜像作为过渡兼容。
- WT selection flow 改为无独立开始按钮：`/wt <目标>` 选完工具/模型后自动执行；空 `/wt` 选完后进入 `AWAITING_GOAL`，等待同话题第一条普通消息作为目标。
- WT git unit/branch/worktree 命名携带 topic session slug，降低同项目多话题 WT 并发时的命名冲突。
- 新增轻量 `WorktreeReviewAdapter`：按目标和变更文件规划 Architect/Tester/Integration/Product 等角色，阻塞建议必须有证据，无证据 blocker 降级为 observation，为后续接入 Spec 式多角色审查保留结构。
- Worktree 卡片事件 payload/render/reducer 贯通 `thread_root_id`，确认卡按钮收敛为“重新选择/停止”，不再渲染 `WORKTREE_CONFIRM_START`。

### 验证

- `uv run python -m pytest tests/ -q` → `6418 passed in 187.09s`
- `uv run python -m src.main --validate` → 配置校验通过；仍有既有 retry 预算提示
- `git diff --check` → 通过

### 风险与后续

- 当前 WT review adapter 只记录计划与空聚合结果，尚未真正启动多角色 review worker；这是本轮有意保留的轻量接入点。
- WT merge 按现有项目级串行边界处理，不新增全局任务控制器。

## WT 自动合并清理与冲突自动处理原则落地

### 背景

用户明确项目基本原则：WT 产物应该尽量直接可用，不能把冲突处理留给人工；冲突要按最新 WT 逻辑自动优先选择 WT 产物，同时向用户披露影响，用户只决定是否发起额外修复。

### 实现

- `WorktreeGitService.merge_branch()` 使用 `git merge --no-ff -X theirs`，合并冲突时优先采用 WT 分支变更。
- 新增 `commit_worktree_changes()`，WT 执行完成后先把脏 worktree 产物提交到对应分支，再合并回 base，避免未提交改动被漏掉。
- `WorktreeHandler.handle_worktree_execute()` 在执行成功后自动 merge；merge 全部成功后自动 cleanup，不再要求用户手动点击合并/清理。
- `handle_worktree_merge()` 的手动兜底入口也会在 merge 成功后自动 cleanup。
- `cleanup_worktrees()` 改为重置 topic-scoped WT state，避免只清 `project.worktree_state` 但 session store 仍残留。
- cleanup 卡新增 `completed` phase，自动合并清理完成后不再显示合并/清理按钮；合并结果 detail 会展示“如有同文件冲突，已自动优先采用 Worktree 分支变更”。
- AGENTS.md 记录 durable rule：WT 不应要求人工修复 merge conflict；系统自动优先采用 WT 分支并披露影响。

### 验证

- `uv run python -m pytest tests/test_worktree_git_service.py tests/test_worktree_auto_execute.py tests/test_worktree_session_store.py tests/test_worktree_card.py tests/test_card_events.py tests/test_card_render_components.py tests/test_worktree_e2e.py tests/test_worktree_command_routing.py -q` → `253 passed`
- `uv run python -m pytest tests/ -q` → `6422 passed in 195.90s`

### 风险与后续

- `-X theirs` 只处理 Git 能自动解决的冲突类型；极端冲突仍会保留现场并展示失败结果，用户可用同一话题发起额外修复。

## /setadmin 首次管理员自助初始化

### 背景

日志提示 `admin_user_ids is empty`，导致 `/lock` 和 `/unlock` 不可用。用户希望第一次和机器人对话的人发送 `/setadmin` 后成为唯一管理员，之后只有这个管理员能修改，其他人发送同命令不生效，重启后也不能绕过。

### 实现

- 新增 `src/admin_bootstrap.py::AdminBootstrapService`，负责读取当前 `admin_user_ids`、在进程内全局锁下原子更新 `.env` 中的 `ADMIN_USER_IDS`，并同步当前进程内 settings。
- 当 `ADMIN_USER_IDS` 为空时，`/setadmin` 忽略参数，只把当前发送者 open_id 写成唯一管理员。
- 当已存在管理员时，只有现有管理员可调用 `/setadmin [open_id]` 替换唯一管理员；非管理员调用只回复拒绝，不改 `.env` 或进程状态。
- `/setadmin` 接入 `SystemHandler` slash 路由，带参数和不带参数都走统一 CommandMatch。
- `ChatLockManager` 把 `/setadmin` 作为锁定期间可到达命令，但真正权限仍由 `AdminBootstrapService` 判断；帮助卡管理员命令中补充 `/setadmin [open_id]`。
- `AGENTS.md` 记录该 bootstrap 规则：只有空配置时任何人可初始化，之后只能由当前管理员替换唯一管理员。

### 验证

- `uv run python -m pytest tests/test_admin_bootstrap.py tests/test_chat_lock.py tests/test_worktree_command_routing.py tests/test_lock_cards.py tests/test_ui_text_placeholders.py -q` → `323 passed`
- `uv run python -m pytest tests/test_refactoring_issue_matrix.py tests/test_acp_startup_utils.py tests/test_ws_event_router.py -q` → `39 passed`
- `uv run python -m src.main --validate` → 配置校验通过
- `git diff --check` → 通过
- `uv run python -m pytest tests/ -q` → `6432 passed`

### 风险与后续

- `/setadmin` 写入的是服务当前工作目录下的 `.env`；如果部署层用系统环境变量覆盖 `ADMIN_USER_IDS`，重启后系统环境变量仍会优先生效。
