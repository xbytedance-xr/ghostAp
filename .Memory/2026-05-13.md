## 执行日志 error/warning 收口

### 背景

执行日志里最新可操作错误集中在 Worktree 卡片更新：飞书拒绝卡片 JSON，错误路径为 `ROOT -> body -> elements -> [1](tag: column_set) -> columns -> [0](tag: column) -> elements -> [4](tag: collapsible_panel)`，原因是 `collapsible_panel` 上出现了 Schema V2 不支持的 `background_style` 字段。随后的 fallback patch warning 是同一张非法卡片触发后的二级症状。日志里还混有几类正常路径提示：Codex 本地模型缓存命中、空管理员 bootstrap 状态、Spec retry 预算提示、SIGTERM 优雅停机，它们不应继续以 warning 级别干扰执行日志。

### 实现

- 移除 `src/card/render/worktree.py` 中 Worktree units `collapsible_panel` 的非法 `background_style` 字段，保留合法的 border/padding/spacing。
- 在 `render_card()` 出口补 Schema V2 sanitizer，递归剥离任何 `collapsible_panel.background_style`，避免未来预构建 atom 或其他 renderer 再把同类非法字段送到飞书。
- 新增回归测试，递归检查 Worktree units 渲染出的 `collapsible_panel` 不再包含 `background_style`，并覆盖 `_assemble_card_json()` 对嵌套非法字段的兜底清理。
- 将正常路径提示从 warning 降为 info：Codex 使用本地模型缓存、ChatLock 空管理员 bootstrap 状态、Spec retry 软预算提示、SIGTERM 优雅停机。
- 更新配置软预算测试，锁定该提示仍存在但不再污染 warning 日志。

## WT 空目标等待目标续聊修复

### 背景

用户在飞书话题中发送空 `/wt`，选完工具模型后卡片进入“等待目标”；随后在同一话题直接发送任务目标时，机器人回复“处理消息时发生内部错误”。日志显示同话题上下文已解析为 `mode=worktree`，但 `FeishuWSClient._get_effective_mode()` 将该 engine-only mode 强转为 `InteractionMode`，触发 `ValueError: 'worktree' is not a valid InteractionMode`，导致还没进入 WT goal 执行分支就失败。

### 实现

- `_get_effective_mode()` 对 thread mode 做安全转换：Coco/Claude/Aiden/Codex/Gemini/TTADK 仍转换为 `InteractionMode`，`worktree` 等 engine-only mode 返回 `SMART/False`，让 dispatcher 继续走 WT awaiting-goal 判定。
- WT confirm 卡片在无 goal 时改为“等待目标”状态：标题、step hint、footer 都提示用户在当前话题直接发送任务目标，避免继续显示“等待确认/点击开始”。
- 增加回归测试覆盖 worktree topic 普通目标消息不会再因 InteractionMode 强转失败，并覆盖无 goal confirm 卡片的等待目标文案。

## Topic-scoped engine sessions 完整落地

### 背景

用户确认 WT 不做全局任务拆分/调度，保持单任务模型；同项目多个 WT 通过飞书话题天然并发，每个话题各自承载一个目标。Deep、Spec、WT 以及后续 engine 都应复用同一套“话题继续对话即继续当前模式”的能力，但必须避免旧的 chat 级隐式续聊导致用户在错误话题里误触发任务。

### 实现

- `ThreadContextManager` 新增 engine context helper：按精确 `thread_root_id/root_id` 查询、判断 active engine、绑定 engine topic。
- `FeishuWSClient` 移除按 chat 找最近编程话题的 fallback：只有命中 `root_id` 的话题上下文才自动继承 Deep/Spec/WT/编程 mode；未知 root 或无 root 只解析项目，不再偷连到最近会话。同一话题绑定 engine 后，切到不同 engine 的命令会被拦截提示。
- WT handler 强制启用话题语义：即使项目配置不是 thread 模式，`/wt` 也会用当前消息创建/绑定 `thread_root_id`，后续同话题普通消息可作为目标继续推进。
- 新增 `WorktreeSessionStore`，WT runtime state 从项目级 `ProjectContext.worktree_state` 迁到 `project_id + chat_id + thread_root_id` scoped key；保留项目 state 镜像作为过渡兼容。
- WT selection flow 改为无独立开始按钮：`/wt <目标>` 选完工具/模型后自动执行；空 `/wt` 选完后进入 `AWAITING_GOAL`，等待同话题第一条普通消息作为目标。
- WT git unit/branch/worktree 命名携带 topic session slug，降低同项目多话题 WT 并发时的命名冲突。
- 新增轻量 `WorktreeReviewAdapter`：按目标和变更文件规划 Architect/Tester/Integration/Product 等角色，阻塞建议必须有证据，无证据 blocker 降级为 observation，为后续接入 Spec 式多角色审查保留结构。
- Worktree 卡片事件 payload/render/reducer 贯通 `thread_root_id`，确认卡按钮收敛为“重新选择/停止”，不再渲染 `WORKTREE_CONFIRM_START`。

### 验证

- `uv run python -m pytest tests/ -q` → `6418 passed in 187.09s`
- `uv run python -m src.main --validate` → 配置校验通过；仍有既有 retry 预算提示
- `git diff --check` → 通过

### 风险与后续

- 当前 WT review adapter 只记录计划与空聚合结果，尚未真正启动多角色 review worker；这是本轮有意保留的轻量接入点。
- WT merge 按现有项目级串行边界处理，不新增全局任务控制器。

## WT 自动合并清理与冲突自动处理原则落地

### 背景

用户明确项目基本原则：WT 产物应该尽量直接可用，不能把冲突处理留给人工；冲突要按最新 WT 逻辑自动优先选择 WT 产物，同时向用户披露影响，用户只决定是否发起额外修复。

### 实现

- `WorktreeGitService.merge_branch()` 使用 `git merge --no-ff -X theirs`，合并冲突时优先采用 WT 分支变更。
- 新增 `commit_worktree_changes()`，WT 执行完成后先把脏 worktree 产物提交到对应分支，再合并回 base，避免未提交改动被漏掉。
- `WorktreeHandler.handle_worktree_execute()` 在执行成功后自动 merge；merge 全部成功后自动 cleanup，不再要求用户手动点击合并/清理。
- `handle_worktree_merge()` 的手动兜底入口也会在 merge 成功后自动 cleanup。
- `cleanup_worktrees()` 改为重置 topic-scoped WT state，避免只清 `project.worktree_state` 但 session store 仍残留。
- cleanup 卡新增 `completed` phase，自动合并清理完成后不再显示合并/清理按钮；合并结果 detail 会展示“如有同文件冲突，已自动优先采用 Worktree 分支变更”。
- AGENTS.md 记录 durable rule：WT 不应要求人工修复 merge conflict；系统自动优先采用 WT 分支并披露影响。

### 风险与后续

## /setadmin 首次管理员自助初始化

### 背景

日志提示 `admin_user_ids is empty`，导致 `/lock` 和 `/unlock` 不可用。用户希望第一次和机器人对话的人发送 `/setadmin` 后成为唯一管理员，之后只有这个管理员能修改，其他人发送同命令不生效，重启后也不能绕过。

### 实现

- 新增 `src/admin_bootstrap.py::AdminBootstrapService`，负责读取当前 `admin_user_ids`、在进程内全局锁下原子更新 `.env` 中的 `ADMIN_USER_IDS`，并同步当前进程内 settings。
- 当 `ADMIN_USER_IDS` 为空时，`/setadmin` 忽略参数，只把当前发送者 open_id 写成唯一管理员。
- 当已存在管理员时，只有现有管理员可调用 `/setadmin [open_id]` 替换唯一管理员；非管理员调用只回复拒绝，不改 `.env` 或进程状态。
- `/setadmin` 接入 `SystemHandler` slash 路由，带参数和不带参数都走统一 CommandMatch。
- `ChatLockManager` 把 `/setadmin` 作为锁定期间可到达命令，但真正权限仍由 `AdminBootstrapService` 判断；帮助卡管理员命令中补充 `/setadmin [open_id]`。
- `AGENTS.md` 记录该 bootstrap 规则：只有空配置时任何人可初始化，之后只能由当前管理员替换唯一管理员。
