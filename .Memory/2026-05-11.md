## 修复 Spec/Coco 辅助 ACP prompt 60s 超时误报

### 任务描述
用户提供线上日志：项目群 `/coco` 进入 Coco 后，Spec 引擎启动阶段先创建一次 Coco ACP engine session，随后在 60s 处报错：
`[ACP:coco] prompt 执行超时 (timeout=60s): 操作超时，请稍后重试`，然后关闭该 session 并重新创建，才进入 `循环 1 阶段 spec 开始`。需要修复该日志错误。

### 根因分析
- 主 Spec phase 使用 `spec_execution_timeout=7200`，并非 60s。
- 60s 超时来自 Spec 启动前的需求验收标准拆解：`SpecEngine._decompose_criteria_with_llm()` → `_make_aux_send_fn()` → `prompt_via_acp()`。
- `prompt_via_acp()` 是一次性辅助 ACP 子会话，原默认 `timeout=60`，而冷启动 Coco + 模型上下文 + 首次 prompt 在真实环境可能超过 60s；该辅助任务失败后会 fallback 到原始文本，所以流程后续仍能继续，但日志会出现 ERROR + Traceback，造成误报和一次无意义的 session 重启。

### 执行内容
- `src/config/settings.py` 新增 `engine_aux_prompt_timeout: int = 600`，与 `SyncACPSession.send_prompt()` 默认 600s 对齐。
- `src/utils/acp_prompt.py` 将 `prompt_via_acp()` 默认 timeout 从 60 调整为 600，避免未显式传参的辅助 ACP prompt 继续使用偏短默认值。
- `src/spec_engine/engine.py` 的 `_make_aux_send_fn()` 显式读取 `self.settings.engine_aux_prompt_timeout` 并传给 `prompt_via_acp()`。
- `tests/test_spec_engine.py` 新增回归测试，先红灯确认旧代码没有透传配置（captured timeout 为 None/走 60s 默认），再绿灯锁定配置值会传入辅助 prompt。

### 技术要点
- 这是辅助 prompt 的 timeout 配置缺口，不是主 Spec phase timeout 缺口。
- 修复保持全模式兼容：`prompt_via_acp()` 仍接收显式 timeout；Spec 通过 Settings 配置控制；其它调用默认也从 60s 提升到 600s，降低冷启动误报。
- 非关键辅助任务仍保留失败 fallback，不阻塞主 Spec 流程。

## TaskOrchestrator lazy 建卡 — 任务真正执行才推飞书卡片

### 任务描述
用户在 P0 修复后继续追问根因：
> "deep模式可能原因是，在明确 n 个任务要做的时候，直接创建了飞书卡片，而不是开始做某个任务时候才创建。这个需要强调一点：只有在任务开始执行了才构建飞书消息卡片，无论是主 agent 还是 subagent 都是如此。当然其他模式的飞书卡片的确是重复推送了，明明没有任务。"

### 根因分析
search 子智能体调研 5 处不当的"建卡时机"：

1. **`TaskOrchestrator.on_plan_received`**：plan 一到就为 N 个 task 同步建卡（含 pending 状态），与"任务执行时才建卡"的契约直接冲突；
2. **Deep handler 入口**：`reply_card("planning")` 静态卡 + `create_deep_callbacks` 内 thinking session ⇒ 双卡现象；
3. **Spec handler 入口**：`reply_card("分析中")` 占位卡，引擎尚未产出第一帧就先推卡；
4. **Programming `prog_session.start()`**：LLM 输出前先建卡（本轮**未动**——Programming 由用户消息触发，不算"无任务空推"）；
5. **Subagent 旁路**：已是 lazy 范式（TOOL_STARTED 检测到 agent/subagent 时按需建卡），无需修改。

### 执行内容

**1. [`src/card/orchestrator.py`](src/card/orchestrator.py)** — 核心 lazy 改造
- `on_plan_received` 重写：只 register 到 `TaskRegistry` + 计算 overflow 映射，**不再立即建 N 个 session**；仅当 plan 中已有 task 处于 `in_progress` 时立即触发建卡（保持向后兼容 ACP plan 状态语义）。
- 新增 `_ensure_task_session(task_id) -> bool`：lazy 建卡入口，幂等。`__init__` 同步引入 3 个状态字段：`_plan_tasks_pending` / `_plan_visible_task_ids` / `_thinking_finalized`。`reset()` 同步清理。
- `dispatch_to_task` 在 resolve 后、查询 session 前先调 `_ensure_task_session(resolved_id)`，并 re-check fallback。
- `route_acp_event` 同样接入 lazy 触发，仅对非 overflow task 调用，避免破坏原 overflow separator 路径。
- `handle_plan_update` 在 status broadcast 段遇到 `in_progress` 时触发 lazy 建卡，使"任务从 pending → in_progress"成为建卡的事实信号。
- **flood_merged 通知保留**：当 visible target session 首次建出来时，一次性 emit 所有指向它的 overflow task 的合并通知。这把"plan 时同步刷 N 条 flood 文本"的副作用迁移到了 lazy 路径，保证不丢通知。
- `_thinking_finalized` 守卫：第一张 lazy task 卡建出时再 finalize thinking session，避免 plan 一到就把思考卡 archive 了。

**2. [`src/feishu/handlers/deep.py`](src/feishu/handlers/deep.py)** — 移除 Deep 静态 planning 卡
- 删除 `engine_card`/`reply_card("planning")` 静态推卡逻辑；`engine.create(..., initial_message_id=None)`。
- 唯一入口卡由 `DeepRenderer.create_deep_callbacks` 内的 thinking session 承担，引擎首帧到达时才创建。

**3. [`src/feishu/handlers/spec.py`](src/feishu/handlers/spec.py)** — 移除 Spec 入口"分析中"占位卡
- 删除 `reply_card("分析中")` 调用；首张 rotator session 由 SpecRenderer 在 phase event 到达时按需创建。

**4. 测试同步更新**
- `tests/test_card_orchestrator.py`：新增 `_trigger_all(orch, tasks)` helper（在 `on_plan_received` 后批量调 `_ensure_task_session`），41 个旧 eager 语义测试全部恢复绿。
- `tests/test_deep_renderer_orchestrator.py` / `tests/test_spec_renderer_orchestrator.py`：将 N+1 卡数量契约测试中的 plan entries 改为 `status="in_progress"`，让 lazy 路径在测试中也立即建卡。

### 技术要点
- **行为契约迁移**：`on_plan_received` 从"建卡器"退化为"register + 映射器"，建卡责任下沉到 `_ensure_task_session`；事件 ARE 执行信号。
- **多 agent 一致性**：主 agent / subagent 共享同一个 lazy 入口（subagent 已通过 TOOL_STARTED 触发；主 agent 通过 in_progress 状态或首事件触发）。
- **幂等保证**：`_ensure_task_session` 在 lock 内查 `_sessions`，已存在直接 return True；overflow / unknown task / fallback / closed 均 return False，调用方安全。
- **视觉副作用保留**：flood_merged 合并通知、thinking session finalize、TASK_LIST_UPDATED 初始事件全部下沉到 lazy 路径，无 UI 退化。
- **测试**：6154 全绿（与昨日 6129 持平 + 25 项新增/调整），无回归；遗留 1 项 `test_show_full_help` 仍是 pre-existing，与本次无关。

## 修复飞书卡片重叠 + 流式更新延迟双 BUG

### 任务描述
2026-05-11 用户反馈 Deep / Spec / Programming 等模式下飞书卡片更新存在两类问题：
1. **多卡片重叠**：同一任务在飞书会话中产生 4-5 张内容高度相似的卡片（每张顶部都重复显示完整任务列表）。
2. **流式更新严重延迟**：后台引擎已经执行完毕，飞书卡片仍在缓慢逐 chunk 流式更新。

### 根因分析
通过对 card pipeline 全链路（`src/card/`、`src/feishu/handlers/`、`src/feishu/renderers/`、`src/acp/renderer.py`、`src/config/card.py`）的只读调研定位三个核心瓶颈：

1. **LiveTicker 心跳被错误标记为结构性事件** — `src/card/state/reducer.py` 的 `_STRUCTURAL_EVENTS` 包含 `TOOL_MODEL_CHANGED`，而 `programming_adapter.py` 每 1.2s 用 `tool_model_changed(live_ticker_frame=...)` dispatch 心跳帧。结果 N 张活跃卡每 1.2s 各触发一次飞书 `update_page`（整卡 PATCH，重接口），形成持续背压。
2. **delivery_pool worker 过少** — `delivery_pool_max_workers` 默认仅 4。一次 Deep 执行同时活跃卡数 ≥ 1 (planning) + 1 (thinking) + 8 (per-task) = 10，跨 session 投递在线程池排队等待，叠加 per-session 串行锁形成长队列。
3. **orchestrator 广播去抖窗口过短** — `_BROADCAST_DEBOUNCE_MS=100` 让密集 plan_update 几乎实时全量广播给所有 task 卡，N 张卡同步刷新且内容雷同，加剧"重叠+延迟"双重观感。

### 执行内容
按"最小改动" P0 集合实施 3 处修复：

**1. [`src/card/state/reducer.py`](src/card/state/reducer.py)**
- 从 `_STRUCTURAL_EVENTS` 移除 `CardEventType.TOOL_MODEL_CHANGED`。
- 新增 `_is_structural_event(event)` 辅助函数：仅当 payload 包含 `tool_name` / `model_name` / `subagents` 时视为结构性；纯 `live_ticker_frame` 心跳走 `element_content` 路径。
- `reduce_card_state` 调用点改为 `_is_structural_event(event)`。
- 效果：N 张卡 × 每 1.2s 的整卡 PATCH 退化为单元素 PATCH，飞书 API 压力降至原 1/N。

**2. [`src/config/card.py`](src/config/card.py)**
- `delivery_pool_max_workers` 默认值 `4 → 16`（validator 上限 32 仍然保留）。
- 效果：跨 session 的投递不再受限于全局线程池；多卡场景下吞吐提升约 4×。

**3. [`src/card/orchestrator.py`](src/card/orchestrator.py)**
- `_BROADCAST_DEBOUNCE_MS` `100 → 800`，并补充注释说明权衡。
- 效果：密集 plan_update 合并广播，N 张 task 卡的结构性事件密度降低 8×；任务列表最多滞后 ~0.8s，用户基本无感。

### 技术要点
- **D-1 的特殊处理保留向后兼容性**：`TOOL_MODEL_CHANGED` 在真正含工具/模型/子智能体变化时仍被视为结构事件，header/footer chips 更新行为不变。
- **未引入新抽象**：3 处改动均为常量或助手函数级别，无新模块、无 API 变更，向前兼容。
- **测试通过**：6129 项测试全部通过；唯一 1 项失败 (`test_show_full_help`) 经 git stash 验证为 pre-existing，与本次修改无关（断言 help 标题缺少 `v0.2.0` 版本号后缀）。
- **保留更激进改动作为 Backlog**：完整 task_list panel 在每张 per-task 卡顶部重复渲染（视觉重叠根源）需要 renderer 层 `compact` 模式设计，列入 Backlog 等设计稿确认后再实施。
