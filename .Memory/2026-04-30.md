## 卡片 Delivery 层 P1 缺陷修复
### 任务描述
审计另一个 agent 对新卡片系统的代码审查结论（6 个 P1），逐一验证真伪并修复真实问题。

### 执行内容
1. **P1-1 修复：4xx 错误被吞** — `feishu_client.py` 的 `update_card()` 和 `update_element()` 对非 300317 的失败只 log warning 不抛异常，导致 engine.py 误更新本地 binding。修复：所有非成功响应统一抛 `TransportError`。
2. **P1-5 修复：终态 close() 短路** — `session.py` 中 `dispatch()` 对终态事件设 `_closed=True`，但随后 `close()` 检测到 `_closed` 直接 return，`delivery.close()` 永不执行导致 binding/sequence 内存泄漏。修复：终态 dispatch 内直接调 `delivery.close()`；`close()` 增加双重检查保持线程安全。
3. **P1-2 修复：element_content ID 体系** — `create_card()` 通过 IM API 返回的 message_id 作为 card_id，但 CardKit element_content API 需要真正的 card_id。修复：Protocol 新增 `create_streaming_card`/`send_card_reference`；`_create_page` 对 streaming 卡片优先走 CardKit 创建路径，失败自动降级到 IM API。

### 审查结论
| 问题 | 真实性 | 处理 |
|------|--------|------|
| P1-1: 4xx 被吞 | ✅ 真实 | 已修复 |
| P1-2: element_content ID | ⚠️ 真实(自愈) | 已修复 |
| P1-3: PATCH 没传 sequence | ❌ 假问题 | im.v1.message.patch 不支持 sequence 参数 |
| P1-4: stale page 不完整 | ⚠️ 低优 | close() 已覆盖清理 |
| P1-5: close() 短路 | ✅ 真实 | 已修复 |
| P1-6: Deep/Loop/Spec 未切 | ❌ 设计如此 | 用 EngineCardSender 正确 |
| P1-7: conftest bridge | ✅ 已解决 | 上一轮已删除 |

### 技术要点
- `im.v1.message.patch` 是消息级 API 不支持 sequence，sequence 只在 CardKit element_content API 有效
- EngineCardSender 是为 Deep/Loop/Spec 设计的独立路径，有自己的 re-anchor 逻辑，不需要 CardSession 全链路
- CardKit 的 `create_streaming_card` 返回真正的 card_id，需配合 `send_card_reference` 投递到聊天
