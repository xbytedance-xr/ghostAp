## ACP 流式缓冲区溢出修复 — Deep 模式长时间执行崩溃

### 任务描述
Deep 模式执行久了会抛出异常：`Separator is found, but chunk is longer than limit`，导致 ACP 通信中断。

### 执行内容

**根因分析**：`asyncio.StreamReader` 默认缓冲区上限为 64KB。ACP 协议使用 JSON-RPC over stdio（换行分隔），当 agent 返回大量输出（长代码生成、文件内容等）时，单条 JSON-RPC 消息超过 64KB，`readline()` 在 `acp/connection.py:151` 抛出 `ValueError`。

调用链：
```
Agent 发送大 JSON-RPC 消息 (>64KB)
  → asyncio.StreamReader.readline() (默认 limit=64KB)
    → readuntil('\n') 发现换行符但 chunk > _high_water
      → ValueError: "Separator is found, but chunk is longer than limit"
        → Connection._receive_loop 崩溃 → ACP 通信断开
```

ACP SDK 已支持通过 `transport_kwargs={"limit": N}` 传递缓冲区大小给 `asyncio.create_subprocess_exec()`，但 GhostAP 的 `ACPSession.start()` 未传递此参数。

**修复方案**（2 个文件）：
1. `src/config.py` — 新增 `acp_stream_buffer_limit: int = 10 * 1024 * 1024`（10MB，可通过环境变量覆盖）
2. `src/acp/session.py` — `ACPSession.start()` 读取配置，通过 `transport_kwargs={"limit": buf_limit}` 传递给 `spawn_agent_process()`

### 技术要点
- asyncio.StreamReader 的 `limit` 参数控制 `_high_water` 阈值，`readline()`/`readuntil()` 在发现分隔符时如果累计 chunk > limit 会抛异常
- ACP SDK 的传递路径：`spawn_agent_process(transport_kwargs=)` → `spawn_stdio_transport(limit=)` → `asyncio.create_subprocess_exec(limit=)`
- 10MB 默认值足够覆盖大多数场景，且可通过 `ACP_STREAM_BUFFER_LIMIT` 环境变量调整
- 设为 0 时退回 asyncio 默认值（64KB），保持向后兼容
- 792 测试全通过

## Shell 命令执行无限递归修复

### 任务描述
用户反馈 Shell 命令执行后卡住无返回。

### 执行内容

**根因分析**：`SystemHandler.submit_shell_command` 的 `_run` 回调通过 `self.ctx.message_callback` 调用 `Application.handle_message`，后者又调用 `_submit_shell_command`，形成无限递归任务创建循环。Shell 命令（SandboxExecutor.execute）从未被实际调用。

调用链：
```
submit_shell_command._run()
  → message_callback (= Application.handle_message)
    → _submit_shell_command
      → 新任务._run() → message_callback → ... (无限循环)
```

**修复方案**：新增 `SystemHandler.execute_shell_and_reply()` 方法，直接调用 `SandboxExecutor.execute()` 并回复结果，打破递归循环。

**修改文件**（3 个）：
1. `src/feishu/handlers/system.py` — 新增 `execute_shell_and_reply()`；`submit_shell_command._run` 改为调用该方法
2. `src/feishu/ws_client.py` — `shell_fast_tracked` 路径和多任务步骤路径改为调用 `execute_shell_and_reply`
3. `src/main.py` — `Application.handle_message` 改为调用 `execute_shell_and_reply`

### 技术要点
- 无限递归的根因：`message_callback` 是 legacy 回调，在 Handler 架构拆分后变成了中间人，但 `submit_shell_command` 仍然通过它间接回调自己
- 修复策略：让所有 shell 执行路径直接调用 `SandboxExecutor.execute()` + `reply_message()`，而非通过 `message_callback` 间接调用
- 792 测试全通过

## Shell 命令结果卡片渲染优化

### 任务描述
Shell 命令执行后以 `msg_type="text"` 发送带 markdown 标记的纯文本，飞书 text 消息不支持 markdown 渲染，用户看到的是原始标记。需要改为 interactive 卡片（schema 2.0）与其他响应保持一致。

### 执行内容

**修改文件**（2 个）：
1. `src/card/builder.py` — 新增 `build_shell_result_card()` 静态方法，构建 shell 命令结果 interactive 卡片
2. `src/feishu/handlers/system.py` — `execute_shell_and_reply` 改为使用 `CardBuilder.build_shell_result_card()` 替代 `result.to_message()` 纯文本

**卡片结构**：
- Header: `✅ 命令执行成功` (turquoise) / `❌ 命令执行失败` (red)
- 工作目录行（复用 `_build_directory_element`）
- 分割线
- 命令引用：`> 🖥️ \`{cmd}\``
- stdout: `` ```BASH``` `` 代码块
- stderr: `⚠️ **错误输出**` + 代码块
- error_message: 安全拦截等场景显示
- 无输出时: `✅ 命令执行成功（无输出）`
- 返回码: notation 字号

### 技术要点
- 复用 `_wrap_card` + `_build_directory_element` 等既有基础设施
- 无需 footer 按钮（shell 结果是一次性的，不需要模式切换按钮）
- 792 测试全通过
