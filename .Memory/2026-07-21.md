## Deep 卡片 stale warning 瀑布与 phase 误告警修复

### 日志证据与根因

- 最新启动段在 13:49:06–13:49:24 连续出现 10 组 `Stale card binding ... code=99992354` 与 `delivery reconcile required`。Deep 最终内容从多页压缩后，最高 CardKit binding 失效；旧逻辑删除该页后用 `max(binding.pages)` 重新计算最高页，导致下一次投递依次 PATCH 每张冻结历史页，形成逐页 warning 瀑布。
- 13:29:05 的 `PHASE_DONE received without prior PHASE_STARTED: cycle=1 phase=build` 不是事件漏发。`MAX_TOTAL_BLOCKS=100` 的尾部裁剪在长任务中先移除了早期活动 phase block，但 `engine_ext.phase_info` 仍保留 build，终态关闭阶段因此产生不一致告警。
- 审查进一步证明初版自愈还存在三个边界：分页增长时替代边界会被后续页覆盖；替代创建失败后空 binding 会重用旧页幂等键；后续页创建失败时前一边界被冻结过早，随后单页终态可能被当作成功 skip。

### 实现

- `DeliveryBinding` 持久于 session 生命周期内维护单调 `message_high_watermark` 与最新逻辑来源页；`PageBinding` 分离可见 `page_index` 和 renderer `source_page_index`。历史页继续冻结，最新逻辑页即使经过 stale 替代、压缩和再次增长也能定位到正确的可见消息。
- 飞书确认缺失 `99992354` 时，同一次 delivery 在下一个可见页号追加替代页，成功恢复返回 applied 且只记 INFO，不再进入 `CardSession` failure warning。替代创建失败保留 high-watermark/source tombstone，下一次使用完全相同的替代页号和 idempotency key 重试；update timeout 首次仍返回 reconcile，避免把不确定晚写伪装成成功。
- 分页增长的 stale 边界与新最新页分别创建；冻结动作延后到后一可见页确认成功之后。若后页创建失败，已确认边界保持 live，可被后续压缩终态继续 PATCH。
- 创建阶段的飞书审核拒绝 fallback 透传 `source_page_index`，避免逻辑页与可见页偏移后破坏映射。
- reducer 的 100-block 安全裁剪优先保留 active phase anchor，再保留最近内容；`PHASE_DONE` 可正常完成 phase、清理 `phase_info`，容量上限不变。
