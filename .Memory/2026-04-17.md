# 2026-04-17 项目记录

## /exit 误报"不在模式中"二次修复（project_id 传递缺失）

### 任务描述
上次修复（`c806030`）解决了"进入模式未发消息时 /exit 误报"的场景，但在**项目级模式**下问题仍然复现。用户在编程模式中输入 `/exit` 仍收到 `⚠️ 当前不在 XX 模式中` 警告。

### 根因分析
`ModeManager` 采用双层存储：`_chat_modes`（chat 级）和 `_project_modes`（项目级），`get_mode(chat_id, project_id)` 按 project → chat → SMART 优先级查询。

问题出在 `_is_in_this_mode` 及相关模式判断方法：
- 所有 6 个子类（Coco/Claude/Aiden/Codex/Gemini/TTADK）的 `_is_in_this_mode(chat_id)` **只传 chat_id 不传 project_id**
- 当模式通过 `enter_mode` 存储在项目级 `_project_modes[project_id]` 时
- `_is_in_this_mode(chat_id)` 只查 chat 级 → 返回 **False**
- 导致 `was_in_this_mode = False`，所有分支不满足，走到误报路径

**关键矛盾**：`exit_current_mode`（system.py）正确传了 `project_id` 识别到当前模式并分发到对应 handler，但 handler 内部 `_is_in_this_mode` 不传 `project_id` 导致二次判断失败。

### 修复方案
为所有模式判断方法统一增加 `project_id` 参数传递：

1. **基类抽象方法签名**：`_is_in_this_mode(chat_id, project_id=None)` 和 `_is_in_opposite_mode(chat_id, project_id=None)`
2. **6 个子类实现**：全部透传 `project_id` 到 `ModeManager.is_xxx_mode(chat_id, project_id=project_id)`
3. **调用处修复**：
   - `exit_mode` 中 `was_in_this_mode = self._is_in_this_mode(chat_id, project_id=project_id)`
   - `enter_mode` 中 `self._is_in_this_mode(chat_id, project_id=project_id)` 和 `self._is_in_opposite_mode(chat_id, project_id=project_id)`
   - `previous_mode = self.mode_manager.get_mode(chat_id, project_id=project_id)`
4. **关联方法修复**：
   - `_is_any_other_programming_mode(chat_id, project_id=None)` 透传到 predicate 调用
   - `_exit_other_programming_modes` 从 project 提取 project_id 透传

### 技术要点
- `ModeManager` 的 `is_xxx_mode` 方法本身已支持 `project_id` 参数（签名为 `is_coco_mode(chat_id, project_id=None)`），问题仅在调用侧未传递
- 修复范围仅限 `programming.py`，`ModeManager` 无需改动
- 所有 6 种编程模式（Coco/Claude/Aiden/Codex/Gemini/TTADK）统一修复，保持全模式兼容
