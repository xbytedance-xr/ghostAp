# 2026-06-26

## 全模式 Traex 模型选择分页修复（11310 element exceeds limit）

### 背景 / 症状
- 用户反馈 traex 的模型选择在“所有模式”下都有问题。
- 复查 `logs.log`：
  - `124017 [ACP] selecting tool=traex` 后 `124018-124023` 普通 `/model`/工具选择卡 `update/reply` 连续失败
    `230099 - Failed to create card content, ext=ErrCode: 11310; ErrMsg: element exceeds the limit`。
- 本机 `fetch_acp_models("traex")` 返回约 90 个模型。`8ab814d` 的分页只覆盖了 Workflow
  （`selection_flow.py`），普通编程 / slock `/new-role` / Worktree / Spec 仍把全部模型一次性渲染成按钮，
  超出飞书单卡元素/节点上限，用户侧表现为“所有模式模型选不了”。

### 根因
- 共享卡片 `SystemBuilder.build_acp_model_select_card`（普通 `/model`、slock `/new-role` 共用）一次性
  `build_responsive_layout(全部模型按钮)`，无分页。
- Worktree/Spec 走独立渲染：`_render_worktree_model_option_grid`（双列网格）与
  `SpecReviewBuilder._build_model_select_card`，同样无上限，90 模型实测 45 行 / 315 节点，超 Feishu 预算。

### 改动
- `src/card/builders/system.py`：`build_acp_model_select_card` 新增 `model_page`，每页 20 个模型 +
  “第 X/Y 页”提示 + 上一页/下一页导航（导航按钮复用 `refresh_action_name` 携带 `model_page`）。
- `src/card/builder.py`：facade 透传 `model_page`。
- `src/feishu/handlers/system.py`：`_show_acp_model_selection_flow` 接收 `model_page`；
  `handle_refresh_acp_models` 从 payload 读取 `model_page` 并重渲对应页。
- `src/feishu/handlers/slock.py`：`/new-role` 模型卡把 `refresh_action_name` 改为
  `SLOCK_NEW_ROLE_SELECT_TOOL`（原为空导致无翻页），读取并透传 `model_page`，翻页时原卡 PATCH。
- Worktree/Spec 网格分页：
  - `src/card/render/worktree.py`：`_render_worktree_model_option_grid` 新增分页 + 翻页列；
    翻页按钮重新触发 `worktree_select_tool`（携带 `model_page`/`page_tool_name`/`page_provider`）重拉模型。
  - `src/card/events/worktree.py` + `payloads.py` + `state/reducers/worktree.py`：新增
    `model_page`/`page_action`/`page_tool_name`/`page_provider` 字段并在 reducer 透传到 block.data。
  - `src/feishu/handlers/worktree.py`：`handle_worktree_select_tool` 读取 `model_page` 并在模型 dispatch 传页参。
  - `src/card/builders/spec_review.py`（+ `builder.py` facade、`handlers/spec.py`）：
    `_build_model_select_card` 分页，导航复用 `SPEC_REVIEW_SELECT_TOOL`。
- 分页统一上限 20/页。
