### 执行内容
1. 并行审计三条主线：
   - `src/acp/`、`src/agent_session.py`、`src/ttadk/`：检查会话 key、启动/重试/回退、路径归一化和 orphan session 风险。
   - `src/deep_engine/`、`src/loop_engine/`、`src/spec_engine/`、`src/tasking/`、`src/project/`：检查调度 stop 语义、运行中 cleanup、持久化/恢复闭环和跨项目上下文污染。
   - `src/feishu/`、`src/card/`：检查 Aiden/Codex/TTADK 多模式路由、空文本/运行态消息分发、fallback 输出丢失、帮助卡与控制面白名单不一致等问题。
2. 重点识别出的高风险点：
   - `TaskScheduler.stop(shutdown_executor=True)` 存在 dispatcher 已标记 RUNNING、executor 已关闭的竞态，可能制造“永远 RUNNING”的僵尸任务。
   - `TaskScheduler.stop()` 不会将队列中未执行任务收敛到终态，`QUEUED` 任务可能永久挂起。
   - Aiden/Codex 运行态消息路由未在 `ws_client` 中显式接线，当前会回落到 Claude handler。
   - `ProgrammingModeHandler.enter_mode()` 的“已在该模式”判断只看 chat 级，不看 project 级，可能阻断项目级编程模式初始化。
   - TTADK 会话层仍存在并发双启动窗口、resume 语义只停留在本地 metadata、CLI model mismatch 检查字段不可靠等一致性风险。
3. 中风险点包括：
   - `update_project_id()` 只更新元数据索引，不迁移队列归属，可能破坏项目内串行语义。
   - Deep/Loop 引擎 manager 仍可能对运行中的 engine 直接 cleanup，引发 use-after-cleanup 风险。
   - `ProjectManager.close_project()` 未联动清理 `UnifiedContextStore`，重新复用同 `project_id` 时可能继承旧上下文。
   - `ProjectManager._save_projects()` 吞错导致上层收到假成功，重启后状态回滚。
4. 对关键问题做了主代码 spot-check，确认不是单纯理论推断；最终未修改业务代码，仅记录风险与证据。

### 技术要点
- 本轮审计将“确定问题”与“高概率设计风险”区分，避免把所有可疑点都等同对待。
- 对多模式兼容问题特别关注 chat 级与 project 级状态源是否一致，因为这是最近多次回归的共性根因。
- 对长运行服务类模块优先检查 stop/cleanup/持久化语义，而非仅检查 happy path。
