## WF 终态一致性收口

### 背景

用户贴出 `/wf` 进度卡截图：卡片头部为绿色 `✅ Workflow`，正文却仍显示“当前执行中”，Execution 阶段里 `execute-traex` 仍在运行，`execute-coco` 已失败/取消。这说明 runtime/bridge 已让 engine 进入 completed 渲染路径，但 state tree 里仍有未关闭 agent。

### 根因

- `WorkflowEngine.execute_workflow()` 在 `RuntimeBridge.run()` 普通返回后只解析上一轮新增的 `fallback:true/status:failed` JSON；如果 JS 最终返回顶层 `{error: ...}` 或普通 `"ok"`，即使状态树里仍有 `RUNNING/PENDING` agent，也会无条件设置 `WorkflowStatus.COMPLETED`。
- `WorkflowStateManager.on_workflow_done()` 只关闭最后一个 phase，不关闭 open agents；一旦 engine 误入 success path，后续 `on_workflow_failed()` 又会因为 COMPLETED sticky guard 拒绝纠正。
- `WorkflowProgressRenderer._render_summary_section()` 对终态仍固定输出“当前执行中”，且 0-agent phase 即使 `finished_at` 已设置也显示“等待中”，导致卡片进一步放大状态矛盾。

### 改动

- `src/workflow_engine/engine.py`
  - `_terminal_failure_from_result()` 支持解码 JSON 字符串包裹的 JSON，并把顶层 `{error: ...}` 视为终态失败。
  - 新增 `_terminal_failure_from_project()`：bridge 返回后先审计 state snapshot，只要存在 `RUNNING/PENDING` agent 或 metrics 显示 agent 未全部终止，就 fail-close 并走 `on_workflow_failed()` 关闭 open agents。
- `src/workflow_engine/renderer.py`
  - 终态摘要标题按 status 输出“执行已完成/执行已失败/执行已取消”，不再固定“当前执行中”。
  - 无 agent 的 phase 若已结束显示“已完成 0/0”，已启动未结束显示“进行中 0/0”，只有未启动才显示“等待中”。
  - 0-agent finished phase 的状态图标改为完成态。
- 测试
  - 新增顶层 `{error}` 和 JSON-string-wrapped fallback 解析测试。
  - 新增 bridge 正常返回但仍有 open agent 时必须 failed 且关闭所有 RUNNING/PENDING agent 的回归测试。
  - 新增终态摘要不显示“当前执行中”、finished empty phase 不显示“等待中”的渲染测试。
  - 更新旧 empty phase summary 断言：started empty phase 应显示“进行中 0/0”。

### 后续风险

- 本轮没有改变 JS `abort_request` 到 Python state 的即时状态更新语义；最终一致性由 engine 返回前的 fail-close 审计兜底。

## WF deadline 预算联动修复

### 背景

用户提供 5 个分析代理的 `/wf` 超时诊断报告，重点怀疑脚本生成仍使用 300s 默认超时、JS runtime 不知道 Python bridge 总 deadline、per-agent timeout 不随剩余预算裁剪，以及 bridge 消息队列满载时静默丢消息导致 Promise 悬挂。

### 结论

- **确认存在**：脚本生成阶段实际调用 `session.send_prompt(..., timeout=AGENT_CALL_TIMEOUT_S)`，仍是 300s。
- **确认存在**：`RuntimeBridge.run()` 有 1800s 总 deadline，但 `start()` init 只下发 `args/max_concurrent`，JS runtime 无 deadline 感知。
- **确认存在**：`agent_call` 入口直接把脚本传入的 timeout 交给 `AgentExecutor`，没有按剩余总预算裁剪。
- **部分存在**：`agent_call` 已有 backpressure reject 和 JS `-32000` 重试，所以“agent_call 队列满必然静默丢弃”不完整；但 reader `_msg_queue` 满时仍会丢弃任意 JSON-RPC 响应/通知，确实可能造成 JS pending 请求悬挂。
- **已有缓解**：降级路径串行 timeout 在 2026-07-03 已通过 generated-dynamic-workflow 本地 routing + 多工具 `race()` 缓解，本轮未重复修改。

### 改动

- `src/workflow_engine/constants.py`
  - 新增 `SCRIPT_GEN_TIMEOUT_S = 120`，脚本生成专用短超时。
  - 新增 `WORKFLOW_TIMEOUT_HEADROOM_S = 5`，host deadline 前保留收尾余量。
- `src/feishu/handlers/workflow_script.py` 和旧内联拷贝 `src/feishu/handlers/workflow.py`
  - 脚本生成 `send_prompt()` 改用 `SCRIPT_GEN_TIMEOUT_S`，不再占满 300s agent 执行默认超时。
- `src/workflow_engine/bridge.py`
  - `RuntimeBridge.start()` 初始化并向 JS init 下发 `started_unix_ms`、`deadline_unix_ms`、`total_timeout_s`。
  - `RuntimeBridge.run()` 复用同一个 monotonic deadline，避免 JS/Python 预算漂移。
  - `agent_call` 入口新增 host-side 剩余预算检查；不足 headroom 时直接 JSON-RPC error，足够时把 `params.timeout` 裁剪为 `min(requested, remaining - headroom)`。
  - 子 workflow 继承父 bridge deadline，避免嵌套 workflow 重新获得完整 30 分钟预算。
  - reader `_msg_queue` 满载时改为设置 `_error/_done` 并唤醒 run loop，不再静默丢消息。
- `src/workflow_engine/executor.py` 和 `src/workflow_engine/engine.py`
  - bridge deadline 透传到 `WorkflowEngine._handle_agent_call()` 和 `AgentExecutor.execute()`。
  - session creation timeout 与 prompt/schema retry timeout 均按剩余 workflow deadline 裁剪；deadline 已耗尽时不会再创建新 session。
- `src/workflow_engine/runtime/runtime.js`
  - 读取 host deadline，`agent()` 在发送请求前按 JS 侧剩余时间裁剪 timeout。
  - `sendRequest()` 为 pending JSON-RPC Promise 设置响应超时；agent 请求超时时主动发 `abort_request`。
  - cancel/response/abort 路径清理 pending timer。
- 测试
  - 新增脚本生成短 timeout、bridge init deadline、host-side timeout 裁剪、deadline 耗尽拒绝、真实 `_handle_agent_call` 传入裁剪后 timeout、reader queue full 不再丢消息的回归测试。

## WF 模型选择级联下拉与缓存

### 背景

用户反馈 `/wf` 选择模型非常慢，尤其 Traex 模型列表约 90+ 个变体时，卡片需要分页点击，交互噪音大且每次展开/选择都会等待模型发现。

### 根因

- `SelectionFlowController` 旧实现把模型列表扁平成按钮，Traex 这类 `openrouter-3o/low`、`openrouter-3o/max/high` 变体会渲染成大量按钮，只能靠 20/page 分页规避卡片节点上限。
- `WorkflowHandler._send_combined_selection_card()` 只要存在 `pending_tool_name` 就同步调用 `_get_workflow_models_for_tool()`；最终选择模型时又会再次调用该函数做白名单校验。
- `_get_workflow_models_for_tool()` 每次创建 `WorktreeToolDiscovery()` 并重新走 provider/model discovery，没有 Workflow 选择阶段专用短 TTL 缓存。
- `WorktreeToolDiscovery.get_models_for_tool(... provider="ttadk")` 默认强制 `force_refresh=True`，Workflow 选择阶段会绕过 TTADK 模型缓存。

### 改动

- `src/workflow_engine/selection_flow.py`
  - 工具选择改为 `select_static` 下拉。
  - 模型选择改为自适应级联：模型族 → Profile → Effort → “设为主编排 Agent/添加此组合”。
  - 尾部维度仅识别 `low/medium/high/xhigh/max`，不会误拆 `anthropic/claude-sonnet` 这类普通 provider/model 路径。
  - 无额外模型时仍保留“默认模型（推荐）”；无法拆维度的模型列表退化为单个模型族下拉。
- `src/feishu/handlers/workflow.py`
  - 下拉回调读取 Feishu 注入的 `_option`。
  - 新增 model group/profile/effort 回调，只更新选择状态并重绘卡片。
  - `_get_workflow_models_for_tool()` 增加 5 分钟短 TTL 缓存；选择模型时的白名单校验复用同一缓存。
- `src/worktree_engine/tool_discovery.py`
  - TTADK 模型选择默认不再 `force_refresh=True`，避免 Workflow 打开选择卡时强制慢探测。
- `src/card/actions/dispatch.py`、`src/feishu/action_registry.py`、`src/feishu/router.py`、`src/feishu/ws_card_action_handler.py`、`src/feishu/ws_client.py`、`src/card/events/payloads.py`

- `ux/workflow-model-cascade.html`
  - 新增 Workflow 模型级联选择预览。

## WF 选择性能与 fallback 终态修复

### 背景

用户继续反馈 `/wf` 新级联选择已经可用但仍慢，并贴出一次失败执行被卡片标成“✅ Workflow — 完成”的记录。日志对应链路为：脚本生成 120s 超时后进入 bounded fallback，fallback 中 `traex`/`coco` agent 调用在 180s host-response timeout 后返回 `{"fallback": true, "error": ...}`，但 Python engine 把该 JSON 当作正常结果处理。

### 根因

- `/wf` 初始工具列表调用 `workflow_engine.tool_registry.get_available_tools(require_available=True)`，内部对 ACP provider 做同步 `get_availability(... allow_sync_probe=True)`，会触发 `claude/coco update` 这类 auto-update。现场日志显示首卡前有 `claude auto-update`，约 5s 级阻塞。
- 选中 Traex 后 `_get_workflow_models_for_tool()` 先创建 `WorktreeToolDiscovery()` 并重新扫描所有工具以推断 provider，再做模型发现；Traex 冷模型 lookup 现场耗时 `5370.6ms`，Coco 为 `2482.3ms`。
- 上一轮为了 Workflow 性能把 `WorktreeToolDiscovery.get_models_for_tool()` 默认改成不强刷 TTADK，误伤了 Worktree 自己“避免陈旧模型”的测试契约。
- `WorkflowEngine.execute_workflow()` 只要 `RuntimeBridge.run()` 正常返回字符串就无条件标记 `COMPLETED`，没有识别 JS 侧结构化 fallback/error 结果。

### 改动

- `src/workflow_engine/tool_registry.py`
  - `require_available=True` 改为快速可选工具发现：PATH 上存在二进制即展示，兜底入口可用也展示；仅触发异步 availability warmup，不再同步 probe/auto-update。
- `src/feishu/handlers/workflow.py`
  - 初始主编排选择卡发送后，后台预热前 2 个推荐工具模型列表，减少用户点工具后的冷等待。
  - Workflow 模型 lookup 直接按 Workflow 已知 ACP 工具集合推断 provider，避免为单个工具选择重新扫描全量 Worktree 工具列表。
  - Workflow 调用 `WorktreeToolDiscovery.get_models_for_tool(... force_refresh=False)`，明确保持选择卡性能优化。
- `src/worktree_engine/tool_discovery.py`
  - 恢复 Worktree 默认 `force_refresh=True`，只让 Workflow 显式选择不强刷，避免性能优化跨策略串味。
- `src/workflow_engine/engine.py`
  - 新增终态结果解析：`{"fallback": true, "error": "..."}` 或 `{"status": "failed", "error": "..."}` 正常返回时标记 `FAILED`，触发 `on_error`，不再调用 `on_done`。
- 测试
  - 增加非阻塞工具发现、Workflow 模型 lookup 不重扫工具、不强刷 TTADK、bounded fallback 结果必须 failed 的回归测试。
