## WF 动态执行超时/状态修复

- 背景：用户继续反馈 `/wf` 复刻 Claude 动态 workflow 能力时总出问题，截图显示同一个 `task-analysis` 同时处于“执行中”和“失败”；随后补充真实错误卡为“状态不匹配”，细节却是 `Workflow execution exceeded total timeout of 1800s`。
- 根因：
  1. `WorkflowStateManager.on_agent_started()` 对相同 label 每次直接 append 新 `AgentProgress`，并用 label 覆盖 `_label_to_agent` 映射；一旦脚本重用 `task-analysis` 或重试/并发复用 label，旧 running 项留在 phase 列表里，新 failed 项进入失败桶，卡片就会同时显示同一代理“执行中/失败”。
  2. `AgentProgress` 没有 `started_at` / `finished_at` 字段，但 renderer 读取这些字段推断最近变更；终态失败时也没有把仍在 running/pending 的 agent 收口，导致失败后顶部 summary 仍显示“当前执行中”。
  3. `TimeoutError: ACP prompt 执行超时 (300s)` 被 `is_transient_error()` 当作可重试 timeout，`MAX_RETRIES=3` 导致同一 agent 最坏执行 4 次 300s，加 backoff 后接近截图中的 20m20s。
  4. `ErrorCategory.RUNTIME_TIMEOUT` 在 handler 回调里被映射成 `invalid_state`，所以 1800s 总超时对用户显示“状态不匹配”，掩盖真实根因。
  5. 脚本生成 prompt 虽然列了动态原语，但缺少可靠性硬约束，模型容易生成固定的 “Analysis -> Execution” 模板和单个慢速 `task-analysis` agent，而不是 Claude-style 的动态路由/fallback。
- 修复：
  - `AgentProgress` 增加 `started_at` / `finished_at`；`on_agent_started()` 返回 effective label，并对重复 label 生成 `label #2`、`label #3` 等唯一名称；`WorkflowEngine._handle_agent_call()` 后续 done/failed/callback 使用 effective label。
  - `on_agent_done()` / `on_agent_failed()` 写入 finished_at，并避免重复终态重复累计 completed/failed/cached 指标；`on_workflow_failed()` / `on_workflow_cancelled()` 将 remaining running/pending agent 收口为终态并关闭 phase。
  - `is_transient_error()` 对 ACP prompt 已耗尽的 per-call timeout 禁止重试，保留普通 network timeout 的原重试策略。
  - 新增 `runtime_timeout` Workflow 错误卡文案；handler `on_error()` 将 `RUNTIME_TIMEOUT` 映射到 `runtime_timeout`，不再报 invalid_state。
  - `build_script_gen_prompt()` 增加 Dynamic Workflow Reliability Rules：禁止先派大而慢 analysis agent、要求唯一 label、短超时、检查 `result.error` 并提供 fallback、容错合成并优先使用 classify/fanout/verify/loop/race。

## WF 状态一致性加固

- 背景：用户反馈 `/wf` 模式执行过程经常出现状态不一致。按日志和近期修复记录反向追踪后，问题集中在三步选择、异步脚本生成、确认执行、取消/停止之间的状态边界。
- 根因：
  1. `GENERATING_SCRIPT` 是 pending 会话态，但没有纳入 stale/block/cancel 状态集合；脚本生成中再次发送 `/wf` 会开启新会话，而旧后台生成任务完成后仍能把旧 requirement/pending 写回确认卡。
  2. `ProjectContext._wf_selection_snapshot` 跨 Workflow 复用，新 `/wf` 会恢复上一轮已选 Agent/模型，导致卡片 UI 与新的 `PendingConfirmation` 语义不一致。
  3. `WorkflowEngine.stop()` 置位 `_cancel_event` 后，同一 engine 被 manager 复用时新 run 没有清位，下一次执行会继承旧取消态。
  4. RuntimeBridge 取消路径先设 `WorkflowStatus.CANCELLED`，随后调用 `WorkflowStateManager.on_workflow_failed()` 又把终态覆盖成 `FAILED`。
- 修复：
  - 新增 `_workflow_pending_statuses()`，统一包含 `GENERATING_SCRIPT`、`AWAITING_AGENT_SELECT`、`AWAITING_TOOL_SELECT`、`AWAITING_CONFIRM`，供启动阻塞、stale reset 和取消入口复用。
  - `_schedule_generate_and_show_confirm_card()` 捕获提交时的 `engine_session_key`；后台任务执行前校验 session 仍匹配；`_generate_and_show_confirm_card()` 在模型生成返回后再次校验，若用户已取消或开启新会话则丢弃旧结果。
  - 新 `/wf` 初始化主编排选择时重建 `SelectionFlowController` 并覆盖 `_wf_selection_snapshot`，不再继承上一次选择。
  - `WorkflowEngine.execute_workflow()` 开始时清理 `_cancel_event`；取消终态改走 `WorkflowStateManager.on_workflow_cancelled()`，不再写成 failed。
  - `/stop_wf` 支持停止生成中的 pending 会话，使用 pending initiator 做权限校验并清理状态。
- 修改文件：
  - `src/feishu/handlers/workflow.py` — pending 状态集合、异步生成 session guard、fresh 选择快照、生成中停止/取消。
  - `src/workflow_engine/engine.py` — 新 run 复位 cancel event，取消路径使用 cancelled 状态写入。
  - `src/workflow_engine/state_manager.py` — 新增 `on_workflow_cancelled()`。

## Spec 完成度把控三阶段落地

- 背景：Spec 模式的"目标完成度"不可控——构建 Agent 自评 PASS/FAIL 后直接决定终止，缺乏客观验证；已有 `completion_control` 审查角色但只读文本无法真实验证；已有 `objective_anchor.py` / `graded_assessment.py` 死代码但从未接入。
- 方案设计：
  1. **Phase 1 客观验证锚点**：`criteria.py` 新增 `run_objective_verify()` 函数，在 `evaluate_criteria()` 中当 LLM 判定 all_satisfied=True 时自动执行 `project.verify_command`，非 0 退出直接覆盖为 all_satisfied=False。引擎 cycle loop 在 build→review 之间也运行 verify，结果传递给 review artifacts。
  2. **Phase 2 主动验证者**：`completion_control` 角色获得专属 prompt（`_build_completion_control_prompt`），明确指示"不择手段验证"——主动运行命令/检查文件/grep 代码，每条判定必须附带客观证据。`ReviewArtifacts` 新增 `acceptance_criteria`/`criteria_satisfied`/`verify_command`/`verify_passed`/`verify_output` 字段。
  3. **Phase 3 证据支撑早停/否决闸门**：`RoleReviewOutcome` 新增 `goal_verdict`/`goal_confidence`/`goal_evidence`；`AdaptiveReviewResult` 新增 `completion_gate_met`/`completion_gate_confidence`/`completion_gate_evidence`；引擎 `_finalize_successful_cycle` 中：(a) 当 completion_control=GOAL_MET(high) + verify 通过 + review 无 blocker → 允许绕过 streak 提前结束；(b) 当 completion_control 有 blocking 建议且 goal_verdict=GOAL_NOT_MET → 一票否决 success。
- 修改文件：
  - `src/config/settings.py` — 新增 4 个设置项
  - `src/spec_engine/criteria.py` — 新增 `run_objective_verify()`，`evaluate_criteria()` 增加 verify gate
  - `src/spec_engine/adaptive_review.py` — 新增 `_build_completion_control_prompt()`，`parse_role_review_output` 提取 goal_verdict，`_outcomes_to_review_result` 提取 completion gate
  - `src/spec_engine/review_aggregation.py` — `RoleReviewOutcome` 新增 3 个 goal 字段
  - `src/spec_engine/review_artifacts.py` — `ReviewArtifacts` 新增 5 个完成度字段，`collect_review_artifacts` 传递 verify 结果
  - `src/spec_engine/engine.py` — cycle loop 增加 pre-review verify 步骤；`_conduct_review` 传递 verify 结果；`_finalize_successful_cycle` 增加完成度闸门逻辑

- 设计决策：
  - 未接入已有的 `objective_anchor.py` / `graded_assessment.py` 重型机制（避免过度封装），而是用最精简的方式（15 行 `run_objective_verify` + 引擎内 3 个条件判断）达到同样效果。
  - 早停仅在有客观证据（verify_command 通过）时绕过 streak，纯 LLM 自评仍必须走满 streak，防止乐观假完成回退。
  - verify_command 默认开启但零副作用（仅当 `project.verify_command` 有值时才执行）。
- 风险：
  - completion_control 的专属 prompt 变更了输出格式（增加 `goal_verdict`/`goal_confidence`），旧模型如果不返回这些字段则 gate 不生效（安全降级）。
  - 非编程任务（writing/research/design）的 completion_control 也获得专属 prompt，但这些场景通常没有 verify_command，gate 不会误触发。
