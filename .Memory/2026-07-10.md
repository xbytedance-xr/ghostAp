## WF 报告展示压缩与启动失败熔断

### 背景
- `/wf` 终态卡片需要继续保持可读的紧凑摘要，同时 HTML 报告需要承担完整执行过程、阶段与 agent 明细展示。
- 当某个 ACP 工具启动失败时，Workflow 后续 agent 调用仍可能反复尝试同一不可用工具，每次都消耗启动超时预算。

### 改动
- `src/workflow_engine/reporting.py`：
  - HTML 报告新增顶部统计、状态徽标、阶段时间线和 agent 明细列表。
  - 报告章节改为更清晰的 section body 和 sticky 搜索工具栏，仍保持自包含 HTML 并对动态内容做 `html.escape()`。
- `src/workflow_engine/renderer.py`：
  - 完成卡片统计压缩为 4 列单行摘要。
  - 执行过程摘要前置，完整报告存在时卡片仍保留截断后的结果摘要，并把附件发送状态放在后面。
- `src/workflow_engine/executor.py` 与 `src/workflow_engine/errors.py`：
  - 检测 `ACPStartupError` 异常链，按工具在单次 Workflow run 内建立启动失败熔断。
  - 同一工具再次被调用时直接返回 circuit breaker 错误，避免重复等待启动超时。
  - 将 circuit breaker / ACP startup failure 归入永久错误关键词。
- `src/workflow_engine/script_gen.py`：
  - fallback 生成脚本的 verifier 固定使用主编排工具，减少评审工具误落到候选中不可用工具的概率。
- 更新 `tests/test_workflow_renderer.py` 对新卡片文案和报告摘要策略的断言。

## WF 结果简报卡去截断

### 背景与根因
- 终态卡同时承担结果正文和状态摘要，`renderer.py` 对 `project.result` 使用 4,000/2,000 字符上限，再由 `_enforce_card_size()` 做第二次按字符裁剪或尾部元素删除。
- 结果是同一卡片多处出现“内容已截断”，用户无法确认丢失的是结论、证据还是风险；7 月 9 日已有的完整 HTML/Markdown 报告只能保证归档，不能替代卡片上的关键行动信息。
- 两套 `on_done()` 异常降级仍回复 `result[:500]`，卡片投递失败时会再次把不完整原始正文发到话题。

### 改动
- 新增 `src/workflow_engine/result_brief.py`：
  - Pydantic 类型化 `WorkflowResultBrief`、verdict/severity 和完整 `BriefItem`。
  - 优先验证顶层 `card_summary`；旧结果只映射 `summary/conclusion/verification/findings/issues/risks/deliverables/artifacts/next_steps/recommendations` 等已知字段，不递归猜测未知对象。
  - 纯文本或无法可靠识别的结果使用“任务已完成，完整结果见报告”中性结论。
  - 按 UTF-8 预算选择完整条目；过长或超量条目整体省略并累计“另有 N 条”，绝不切半句。
- `src/workflow_engine/renderer.py`：
  - 完成卡只消费标准化简报，信息顺序改为结论→关键发现→验证→交付物→下一步，空栏目隐藏。
  - 统计改为耗时/阶段/验证/高风险或交付物；`needs_attention` 使用“需修正”终态。
  - 执行过程移到结果后的折叠面板，报告状态置底。
  - 结果简报在 Markdown 转义前执行 agent-output 防泄漏检查。
  - 完成卡超过 28KB 时重建最小卡，只保留完整结论、遗漏数量和报告/错误状态，不经过字符截断。
- `src/workflow_engine/script_gen.py`：
  - AI 脚本生成提示要求成功结果保留完整业务产物并返回 `card_summary`。
  - 静态 fail-closed 脚本和 `generate_simple_script()` 通过 `completionEnvelope()` 输出同一契约；执行错误仍保留原有顶层失败语义。
- `WorkflowScriptMixin` 增加共享安全终态降级，两套 callback 不再回复 `result[:500]`；只说明结果卡失败和完整报告状态。
- 测试基础设施最小修复：action registry 路径从当前 checkout 推导；并发压力测试仍执行 80 次写入，但用 8 个 writer 线程避免受限机器把线程创建本身当成被测行为。
- 设计与预览：`docs/superpowers/specs/2026-07-10-workflow-result-brief-design.md`、`ux/workflow-result-brief.html`。

## ACP 模型探测慢：single-flight + 负缓存

### 背景
- `logs.log` 尾部显示 `/wf` 选择模型时 `model_lookup ... cached=false` 从不命中缓存：traex 每次 5.3–8.7s，claude **每次固定 15s 超时**后 fallback（`TimeoutError: ACP claude 模型探测超时 (15.0s)`），是"模型选择确定很慢"的直接原因。
- 关键证据：相邻两行 `model_lookup tool=traex ... duration_ms=7688.3` 紧跟 `duration_ms=7702.9`，第二次探测在第一次完成前就启动——`_preheat_workflow_models` 后台预热线程与用户点击线程对同一 `(tool, cwd)` 并发探测，各自 spawn 一次 `traex acp serve` 全量探测（惊群）。

### 根因
1. `src/acp/helper.py` 的 `_acp_probe_cache`（TTL 300s）只在探测**完成后**写入，无法阻止并发首填 → 缺少 single-flight。
2. claude 不支持 `acp serve`，`conn.initialize` 挂起烧满 15s 超时；无负缓存，每次 lookup（点击 + 预热 + group/profile/effort 每步切换）都重复付 15s。
3. `workflow.py:2806` 硬编码 `cached=false`；实例缓存命中的早返回路径不打日志，导致指标长期误报、掩盖真相。

### 改动
- `src/acp/helper.py`：
  - 三层缓存统一改为 `(tool_name, cwd)` 键（原正缓存只按 `tool_name`，与 cwd 维度不一致）。
  - 新增负缓存 `_acp_neg_cache`（`_ACP_NEG_CACHE_TTL=45s`）+ `_is_negatively_cached`/`_set_negative_probe`：空/超时结果短期记住，窗口内直接走本地 fallback 不再重探，到期自动重探。
  - 新增 single-flight `_acp_probe_inflight`（`threading.Event`）+ `_fetch_non_coco_models_singleflight`：leader 线程跑一次真实探测并 `event.set()`（`finally` 保证释放），其余线程按 `probe_timeout+5s` 上限等待复用 leader 结果，不再各自 spawn 子进程。
  - 抽出 `_probe_models_blocking`（原探测逻辑）；coco 路径行为完全不变（继续走 `CocoModelManager`）。
  - 修复既有别名 bug：`_get_cached_probe` 现返回 `_copy_models`（`dataclasses.replace` 造新 `ACPModelOption`），`_mark_default` 只改每个调用方的副本；此前缓存命中路径直接改共享对象的 `is_default`，single-flight 下会成为并发写竞态。
- `src/feishu/handlers/workflow.py`：实例缓存命中路径补一条 `cached=true` 日志（miss 路径保留 `cached=false`），修正误导指标。
- `tests/test_acp_model_probe_timeout.py`：既有 3 个直接操作 `_acp_probe_cache` 的用例改用 `_probe_key(tool, cwd)` 元组键并清 `_acp_neg_cache`；新增 5 个回归：并发探测只跑一次、负缓存窗口内不重探、负缓存过期后重探、缓存命中返回独立副本（is_default 不污染缓存/其他调用方）、不同 cwd 独立探测。

### 验证
- 只读审计（Explore 子代理，与实现并行）：确认全部 `fetch_acp_models` 调用方（tool_discovery / system / slock / slock_roles，及经 `get_models_for_tool` 的 worktree/spec/workflow）均在**同步工作线程**执行、从不在运行中的事件循环内调用，无同线程重入自锁；无调用方原地改返回列表 → GO + 三条护栏（返回副本、finally 释放 Event、统一键）均已落实。
- `tests/test_acp_model_probe_timeout.py` → 17 passed（连跑 3 次无 flaky）。
- 相关面：`test_workflow_model_selection / test_worktree_tool_discovery / test_model_cascade / test_model_command / test_workflow_orchestrator_select` → 81 passed；`test_slock_role_creation / test_acp_client` → 71 passed。
- `uv run ruff check` 三个改动文件 → All checks passed；`uv run python -m src.main --validate` → 配置校验通过。

### 后续风险
- claude 恢复 ACP serve 能力后，负缓存窗口最长 45s 内仍走 fallback，之后自动重探恢复实时列表；可接受。
