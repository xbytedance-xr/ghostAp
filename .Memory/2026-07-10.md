# 2026-07-10

## Codex 模型/Effort 下拉恢复与协议贯通

### 背景与根因
- 7 月 6 日的普通模式级联 UI 依赖模型名自身携带 `/profile/effort` 后缀；7 月 10 日切到官方 `@agentclientprotocol/codex-acp@1.1.2` 后，adapter 改为在 `new_session.config_options` 中分别返回 `category=model` 的 `model` 和 `category=thought_level` 的 `reasoning_effort`。
- `src/acp/helper.py` 原实现只抽取 `category=model`，丢弃 `reasoning_effort`。因此 Codex 模型名不再带 `/high`、`/max`，`has_cascade_variants()` 判定无变体并退回按钮卡；启动链路也只设置 `config_id=model`，Effort 没有真正进入运行会话。
- 真实 adapter 显示不同模型的 Effort 集不同：例如 `gpt-5.5` 只有 `low/medium/high/xhigh`，`gpt-5.6-sol` 有 `low/medium/high/xhigh/max/ultra`，不能用当前模型的列表与所有模型做笛卡尔积。

### 设计与 Grill-Me 审查
- 设计文档：`docs/2026-07-10-codex-effort-selection-design.md`；实现计划：`docs/2026-07-10-codex-effort-selection-plan.md`；生产前预览同步更新 `ux/acp-model-cascade.html`。
- 用户授权自动采纳 grill 推荐。审查后落地：
  - 在一个临时 ACP session 中逐模型切换并读取响应后的 `reasoning_effort`，构造精确能力矩阵。
  - `ACPModelOption` 每个模型只保留一项，附加 `reasoning_efforts` 与 `adapted_reasoning_effort`；后者表示从临时 session 初始 Effort 切到该模型后 adapter 实际适配出的值，不冒充模型固有 default。
  - 只在共享级联渲染器内部展开 `model/effort`，Worktree/Slock 按钮模式仍是一模型一项。
  - `max` 对 Codex 明确是 Effort，不复用 Traex 单后缀 `/max` 的 Profile 语义；支持集合覆盖 `none/minimal/low/medium/high/xhigh/max/ultra`，未来未知值在 probe 边界告警并过滤。
  - 项目与 Workflow 继续持久化既有复合 `model/effort`，不新增 schema 字段；只在官方 Codex ACP 边界拆成 `model` 与 `reasoning_effort` 两次 config option。
  - non-Coco 共享模型缓存只存 provider 原始默认，项目 current model 仅标记返回副本；模型下架回退 adapter 默认，Effort 下架但基础模型仍存在时保留该模型并采用新的适配 Effort。
  - Workflow 模型缓存对嵌套 `reasoning_efforts` 做独立复制，避免调用方污染缓存。
  - 官方 Codex session 一致性优先比较 `_model_name`，不再假设模型一定出现在 process argv，避免 follow-up prompt 误重启。
  - 泛化后的 `set_config_option()` 日志只记录 `config_id`，不记录未来可能敏感的 value。
- 按 mloop 从产品、架构、工程、QA、UI/UX 视角持续复审并修正，最终连续两轮无实质建议后停止。

### 实现
- `src/acp/model_selection.py`：新增 Codex 复合选择值解析/组装和已知 Effort 集。
- `src/acp/helper.py`：官方 Codex adapter 精确能力探测、未知 Effort 过滤、共享默认隔离、失效模型/Effort 降级与缓存副本语义。

## Codex ACP 官方迁移收口：首选模型、SDK 回调与重启预热

### 补充根因
- 初版官方迁移已替换 adapter、升级 Python ACP SDK，并让模型列表只信任 adapter；复查发现仍遗留三处 0.8/Zed-era 假设。
- `CodexACPProvider` 仍向官方 stdio adapter 传 `-c model=...`。该参数只被旧 Zed adapter 解释；官方 adapter 会忽略它。因此卡片可显示用户选中的模型，实际新会话却可能继续使用 `~/.codex/config.toml` 默认模型。
- 虽然运行时已升到 ACP SDK 0.11，`GhostAPClient` 的 permission、file read/write、terminal callback 仍按旧参数顺序定义。纯文本 prompt 不触发这些回调，真实工具请求则可能把 session、路径、命令或选项错位。
- `restart.sh` 把官方 adapter 当作普通 CLI 执行 `--help`；官方入口只有 `--version` 会立即退出，`--help` 会进入 stdio server 并可能卡住重启预热。

### 收口改动
- `src/acp/providers/__init__.py`：官方 fallback 固定为 `npx --yes @agentclientprotocol/codex-acp@1.1.2`，不再追加旧 `-c model=...`。
- `src/acp/sync_adapter.py`：仅识别到官方 Codex fallback 时，`new_session` 后、ready 前调用已有的 `ACPSession.set_model()`（ACP 0.11 `set_config_option(config_id="model")`）。显式模型被拒绝时立即关闭会话并抛启动错误，复用既有 retry/失败卡链路；默认模型和 native/custom Codex 路径不发送多余 RPC。
- `src/acp/client.py`：对齐 ACP 0.11 Client callback 签名；文件读支持 `line`/`limit`，终端安全处理 `args`、项目内 `cwd`、受校验环境覆盖与 UTF-8 字节尾部截断。环境值不进入 filter payload、历史或日志；ACP 终端调用固定非交互，避免 shell 启动转义序列污染 byte-limit 输出。
- `src/sandbox/executor.py`：增加可选 `env_overrides`，在已有 pager 默认环境之后注入，保持所有原调用方行为不变。
- `restart.sh`：依赖预热改为 `npx ... --version`。
- 移除了实现过程中的 design/plan 文档，避免保留临时文档；最终事实仅保存在本记录。

### 验证

## Codex ACP 官方适配器迁移与选模 fail-close

### 背景与根因
- 现场 `/codex` 日志显示启动时预热的是 `@zed-industries/codex-acp@0.14.0`，随后 `/codex` 模型选择在 `10:08:34` 直接使用本机 `~/.codex/models_cache.json`（`using local codex model cache (5 models) before live probe`）。
- 用户选择 `gpt-5.6-sol` 后，Deep 以 `agent=codex model=gpt-5.6-sol` 创建 `npx` ACP session；旧 Zed adapter 内嵌 Codex 版本落后，不认识该模型元数据，prompt 两次重试后稳定 `Internal error`。
- 第二条隐患是 Python `agent-client-protocol` 锁在 0.8.0，运行时只会走旧 `session/set_model`；官方 Codex ACP 1.1.x 使用 session config option 暴露/切换模型，旧 RPC 会继续触发不兼容错误。

### 改动
- `src/acp/providers/__init__.py` 与 `restart.sh`：Codex ACP fallback 从 `@zed-industries/codex-acp@0.14.0` 切到官方 `@agentclientprotocol/codex-acp@1.1.2`，保留 `GHOSTAP_CODEX_ACP_NPM_PACKAGE` 覆盖口。
- `pyproject.toml` / `uv.lock`：`agent-client-protocol` 下限升级到 `>=0.11.0`，锁定 0.11.0。
- `src/acp/helper.py`：Codex 模型列表不再读取 `~/.codex/models_cache.json`；只信任 live ACP adapter probe/config_options。probe 为空、超时或命中负缓存时直接返回空列表，让既有错误卡 fail-close，而不是展示本机 Codex CLI 缓存里的潜在不兼容模型。删除旧 Codex local-cache helper，避免回归。
- `src/acp/session.py`：`set_model()` 优先调用 ACP 0.11 的 `set_config_option(config_id="model", value=<model>)`，兼容 camelCase 和旧实验名；若新协议存在但失败则返回 `False`，不再继续尝试旧 `session/set_model`。仅在连接完全没有 config-option 能力时保留 legacy `set_session_model` fallback。
- `tests/test_acp_stdio_integration.py`：ACP 0.11 的 `AgentSideConnection(listening=True)` 会自动启动读循环，测试 fake 改为 `listening=False` 后显式 `listen()`，避免双读循环导致 Internal error。

## WF 结果简报卡去截断

### 背景与根因
- 终态卡同时承担结果正文和状态摘要，`renderer.py` 对 `project.result` 使用 4,000/2,000 字符上限，再由 `_enforce_card_size()` 做第二次按字符裁剪或尾部元素删除。
- 结果是同一卡片多处出现“内容已截断”，用户无法确认丢失的是结论、证据还是风险；7 月 9 日已有的完整 HTML/Markdown 报告只能保证归档，不能替代卡片上的关键行动信息。
- 两套 `on_done()` 异常降级仍回复 `result[:500]`，卡片投递失败时会再次把不完整原始正文发到话题。

### 改动
- 新增 `src/workflow_engine/result_brief.py`：
  - Pydantic 类型化 `WorkflowResultBrief`、verdict/severity 和完整 `BriefItem`。
  - 优先验证顶层 `card_summary`；旧结果只映射 `summary/conclusion/verification/findings/issues/risks/deliverables/artifacts/next_steps/recommendations` 等已知字段，不递归猜测未知对象。
  - 纯文本或无法可靠识别的结果使用“任务已完成，完整结果见报告”中性结论。
  - 按 UTF-8 预算选择完整条目；过长或超量条目整体省略并累计“另有 N 条”，绝不切半句。
- `src/workflow_engine/renderer.py`：
  - 完成卡只消费标准化简报，信息顺序改为结论→关键发现→验证→交付物→下一步，空栏目隐藏。

## ACP 模型探测慢：single-flight + 负缓存

### 背景
- `logs.log` 尾部显示 `/wf` 选择模型时 `model_lookup ... cached=false` 从不命中缓存：traex 每次 5.3–8.7s，claude **每次固定 15s 超时**后 fallback（`TimeoutError: ACP claude 模型探测超时 (15.0s)`），是"模型选择确定很慢"的直接原因。
- 关键证据：相邻两行 `model_lookup tool=traex ... duration_ms=7688.3` 紧跟 `duration_ms=7702.9`，第二次探测在第一次完成前就启动——`_preheat_workflow_models` 后台预热线程与用户点击线程对同一 `(tool, cwd)` 并发探测，各自 spawn 一次 `traex acp serve` 全量探测（惊群）。

### 根因
1. `src/acp/helper.py` 的 `_acp_probe_cache`（TTL 300s）只在探测**完成后**写入，无法阻止并发首填 → 缺少 single-flight。
2. claude 不支持 `acp serve`，`conn.initialize` 挂起烧满 15s 超时；无负缓存，每次 lookup（点击 + 预热 + group/profile/effort 每步切换）都重复付 15s。
3. `workflow.py:2806` 硬编码 `cached=false`；实例缓存命中的早返回路径不打日志，导致指标长期误报、掩盖真相。

### 改动
- `src/acp/helper.py`：
  - 三层缓存统一改为 `(tool_name, cwd)` 键（原正缓存只按 `tool_name`，与 cwd 维度不一致）。
  - 新增负缓存 `_acp_neg_cache`（`_ACP_NEG_CACHE_TTL=45s`）+ `_is_negatively_cached`/`_set_negative_probe`：空/超时结果短期记住，窗口内直接走本地 fallback 不再重探，到期自动重探。
  - 新增 single-flight `_acp_probe_inflight`（`threading.Event`）+ `_fetch_non_coco_models_singleflight`：leader 线程跑一次真实探测并 `event.set()`（`finally` 保证释放），其余线程按 `probe_timeout+5s` 上限等待复用 leader 结果，不再各自 spawn 子进程。
  - 抽出 `_probe_models_blocking`（原探测逻辑）；coco 路径行为完全不变（继续走 `CocoModelManager`）。
  - 修复既有别名 bug：`_get_cached_probe` 现返回 `_copy_models`（`dataclasses.replace` 造新 `ACPModelOption`），`_mark_default` 只改每个调用方的副本；此前缓存命中路径直接改共享对象的 `is_default`，single-flight 下会成为并发写竞态。
- `src/feishu/handlers/workflow.py`：实例缓存命中路径补一条 `cached=true` 日志（miss 路径保留 `cached=false`），修正误导指标。
- `tests/test_acp_model_probe_timeout.py`：既有 3 个直接操作 `_acp_probe_cache` 的用例改用 `_probe_key(tool, cwd)` 元组键并清 `_acp_neg_cache`；新增 5 个回归：并发探测只跑一次、负缓存窗口内不重探、负缓存过期后重探、缓存命中返回独立副本（is_default 不污染缓存/其他调用方）、不同 cwd 独立探测。

### 验证
- 只读审计（Explore 子代理，与实现并行）：确认全部 `fetch_acp_models` 调用方（tool_discovery / system / slock / slock_roles，及经 `get_models_for_tool` 的 worktree/spec/workflow）均在**同步工作线程**执行、从不在运行中的事件循环内调用，无同线程重入自锁；无调用方原地改返回列表 → GO + 三条护栏（返回副本、finally 释放 Event、统一键）均已落实。
- `tests/test_acp_model_probe_timeout.py` → 17 passed（连跑 3 次无 flaky）。
- 相关面：`test_workflow_model_selection / test_worktree_tool_discovery / test_model_cascade / test_model_command / test_workflow_orchestrator_select` → 81 passed；`test_slock_role_creation / test_acp_client` → 71 passed。

### 后续风险
- claude 恢复 ACP serve 能力后，负缓存窗口最长 45s 内仍走 fallback，之后自动重探恢复实时列表；可接受。

## 普通 ACP 最终选模：异步激活与真实终态

### 背景与根因
- 最新 `logs.log` 显示用户在 `17:10:38` 点击最终 Codex 模型后，卡片回调同步进入 ACP 会话启动；第一次在 `17:10:58` 耗尽 20s，第二次在 `17:11:29` 耗尽 30s，最终约 51s 才返回。
- `SystemHandler.handle_select_acp_model()` 在调用 `_enter_mode_with_acp_model()` 返回前不会 PATCH 原模型卡，因此 ACP 冷启动/重试时用户感知为“选模型很慢”。
