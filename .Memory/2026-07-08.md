## Deep 多页终态卡输出不完整 + 卡片顺序错乱根因修复

### 背景

用户反馈 Deep 模式任务结束时：(1) 最终任务卡的输出内容不完整；(2) 卡片顺序看起来有问题。

### 根因（真实代码复现验证）

- Deep 是**单 session、多 page** 渲染（`DeepStreamProcessor._CYCLE=1`，不像 Spec 那样轮转 session），最终模型输出（`### 执行过程` + 总结报告）较长、累计 body 超过 `RenderBudget.byte_budget`(27KB) 时会分成多张 page：page0 = 第一条飞书消息，page1 = 续接消息……
- `src/card/delivery/engine.py` 的 `_deliver_unlocked()` 有个流式防抖优化 `freeze_history_pages`（引入于 `c8be46c`，2026-05-16）：当卡片 ≥2 页且 binding 已存在时，**无条件 skip 所有非最后一页**（`history_page_frozen`）。问题是这个冻结在**终态（completed/failed/cancelled）也生效**。
- 后果：完成时 page0 永远停留在"执行中"的中途快照——完成态的最终全文、绿色完成头、末尾追加的 `_summary` block（落在最后一页）都刷不到 page0。
  - **现象1"输出不完整"**：用户看到的第一张卡冻结在执行中途，缺完成时才 flush 的完整文本与总结。
  - **现象2"卡片顺序有问题"**：第一张卡显示"执行中"、第二张卡显示"已完成（绿）"，先发的卡看起来还没结束、后发的卡反而结束了，观感上就是顺序/状态错乱。
- 复现脚本 `/tmp/repro_deep_multipage.py`：终态投递 `[('skipped','history_page_frozen'),('applied','updated:card_2')]`，page0 从未到达 completed。

### 修改

给投递链路贯穿 `is_terminal` 语义：**终态投递不冻结历史页**，让每页都刷到最终态；流式期冻结优化保留不变。

- `src/card/delivery/engine.py`
  - `deliver()`/`_deliver_unlocked()` 新增 `is_terminal: bool = False` 形参。
  - `freeze_history_pages` 增加 `and not is_terminal` 条件：终态时豁免冻结，所有页（含 page0）都走 create/update/stream。
- `src/card/dispatch_coordinator.py`
  - `DispatchDeliveryCoordinator.deliver(rendered, *, is_terminal=False)` 透传到 `CardDelivery.deliver`。
- `src/card/session/core.py`
  - `_deliver_and_track()` 把已算好的 `is_terminal`（COMPLETED/FAILED/CANCELLED/ARCHIVED/BLOCKED）经 `_coordinator.deliver(rendered, is_terminal=is_terminal)` 下传。
- `src/card/session/_ttl_mixin.py`
  - force-close 投递（渲染 `cancelled` 终态）与 `force_deliver()`（终态重试路径，唯一调用方 `ttl.py:schedule_terminal_retry`）传 `is_terminal=True`。
- 测试
  - `tests/test_card_delivery_engine.py`：新增 `test_terminal_delivery_updates_all_pages_including_history`（终态两页都 applied 且 page0 被 PATCH）、`test_non_terminal_still_freezes_history_pages`（流式仍冻结，防回退）。
  - `tests/test_dispatch_coordinator.py`：`test_deliver_calls_delivery_engine` 断言补 `is_terminal=False`；新增 `test_deliver_propagates_is_terminal`。
  - `tests/test_card_session.py`(3)、`tests/test_terminal_fallback.py`(3)、`tests/test_static_session.py`(1) 的 mock `deliver` 签名补 `is_terminal=False`，避免新 kwarg 触发 TypeError。

### 验证

- `test_card_delivery_engine/test_dispatch_coordinator/test_card_session/test_terminal_fallback/test_static_session/test_ttl_handler`：221 passed。

### 后续风险

## WF完成报告卡补齐

### 背景

用户反馈 `/wf` 运行完成后卡片只显示终态进度：阶段、代理、进度条均显示完成，但没有任务执行结果、执行过程总结和最终报告。截图中最后一张可见卡仍是 `WorkflowProgressRenderer` 的终态进度卡，而不是明确的完成报告卡。

### 根因

- `render_completion_card()` 只把 `project.result[:500]` 作为“结果摘要”原样展示。Workflow runtime 会把 JS return 结果 JSON 编码成字符串，长 `final_report` / `verification` 内容容易被 500 字符截断，结构化报告也不可读。
- `WorkflowHandler._build_workflow_callbacks()` 和 `WorkflowScriptMixin._build_workflow_callbacks()` 都允许 `on_done()` 之后的迟到 `on_progress()` 继续 patch 同一张卡。若 heartbeat/coalescer 还有最后一帧终态进度卡，可能覆盖刚发送的完成卡。
- `workflow.py` 中存在一个类本体的 `_build_workflow_callbacks()`，会覆盖 mixin 里的同名实现；修复 callback 行为时必须两处同步，否则测试/运行时仍会走旧实现。

### 修改

- `src/workflow_engine/renderer.py`
  - 新增 completion result 解析与格式化：优先解析 JSON payload，识别 `final_report`、`report`、`summary`、`result`、`output`、`verification`、`reviews`、`risks`、`next_steps` 等字段。
  - 完成卡新增“执行报告”区块和“执行过程”区块；执行过程列出阶段/代理完成、失败、缓存数，并逐阶段展示完成状态。
  - 完成但 JS 未返回结果时，完成卡会明确提示“没有返回最终结果”，避免报告区块完全缺失。
  - completion card 使用 phase/agent 状态兜底 metrics，避免 metrics 未填时成功率/Token/代理数全为 0。
  - completion card 最后走 `_enforce_card_size()`，降低长报告触发飞书 payload 限制的风险。
- `src/feishu/handlers/workflow.py`
  - 在实际生效的 `_build_workflow_callbacks()` 中增加 `terminal_sent` guard；`on_done()` / `on_error()` 后忽略迟到 `on_progress()`。
  - 补齐运行态 progress card 的 stop button 注入，使类本体实现与 mixin 行为一致。
- `src/feishu/handlers/workflow_script.py`
  - 同步 `terminal_sent` guard，避免后续 refactor 切回 mixin 时行为回退。
- `tests/test_workflow_renderer.py`
  - 新增回归：长 `final_report` 关键结论位于 500 字后时仍应显示；结构化 `verification.summary` 和“执行过程”必须出现在完成卡；空 result 也必须显示报告占位。
- `tests/test_workflow_confirm.py`
  - 新增回归：`on_done()` / `on_error()` 后模拟迟到 progress callback，不得再次更新卡片或覆盖终态报告/错误卡。

## WF "closed stdout unexpectedly" 根因修复（stdout 截断 + EOF 竞态）

### 背景

用户反馈 `/wf` 报错：`Workflow 执行过程中发生内部错误。Workflow runtime error: Runtime process closed stdout unexpectedly`。挖 `logs.log`（138908、138571 两次失败，engine `25f0f8bf2e3b`/`a9907093e125`）发现：Node 启动正常（`Node.js runtime ready and initialized`，node v22.16.0 ≥ NODE_MIN_VERSION），两次都死在 Verification 阶段、紧跟 `Schema validation failed, retry 1/2 for tool=coco`，且日志中**完全没有** Node stderr/traceback/exit code —— 真实错误被通用文案掩盖。

### 根因（两层）

1. **runtime.js stdout 截断（主因）**：`runtime.js` 所有终态路径都是 `sendNotification('done'|'error', ...)` 紧跟同步 `process.exit()`。当 stdout 是管道（异步写）且有 backlog（真实 workflow 会 emit 大量 log/phase 帧）时，`process.exit()` 会**截断未 flush 的尾部**，包括终态 `done`/`error` 帧。Python bridge 只看到 stdout EOF 而无终态帧 → 落到 `bridge.py:660` 的通用串。用真实 Node 复现：写 20001 帧只有 6220 帧到达管道、`done` 帧丢失；drain-before-exit 后 20001 帧全达。**成功路径（done→exit 0）同样受影响**，会静默丢结果。
2. **bridge.py EOF 竞态 + stderr 埋没（次因/纵深）**：reader 线程 EOF 分支直接写 `self._error`，即使此刻队列里已有合法终态帧（flush 修复后 done-before-EOF 变得**可靠**，反而放大此竞态）→ run() 仍抛通用串掩盖真实结果。且 `_stderr_reader` 只把 Node stderr 打 DEBUG（`[runtime stderr]`），进程真正崩溃（SIGKILL/OOM）时临终 stderr 丢失，连"好路径"的 `_drain_stderr()` 也因 drain 线程已清空管道而返回空。

### 修改

- `src/workflow_engine/runtime/runtime.js`
  - 新增 `flushAndExit(code)`：pause stdin，对 stdout+stderr 各写空串等 drain 回调后再 `process.exit`，带 2s 兜底 timer（`unref`）防管道卡死。
  - 9 处终态 `process.exit()` 全部替换为 `flushAndExit()`（usage、读脚本失败、sandbox 加固失败、脚本 load 失败、meta 校验失败、workflow done/error、clean shutdown、unhandledRejection、uncaughtException）；因 `flushAndExit` 是异步 defer，每个 main() 内调用点补 `return` 保持原控制流。
- `src/workflow_engine/bridge.py`
  - 新增 `_eof_fallback_error` 字段：EOF 只写此**独立**字段而非 `_error`，`run()` 退出时先 `_drain_pending_messages()` 排空队列应用终态帧，优先级 `done/error 帧 > EOF fallback`。
  - 新增 `_stderr_tail`（`collections.deque(maxlen=STDERR_TAIL_MAX_LINES=50)` + leaf lock），`_stderr_reader` 逐行写入 tail；新增 `_stderr_tail_text()`/`_stderr_diagnostic()`/`_describe_unexpected_exit()`（含 exit/signal code + stderr tail，int 强制转换防非 int poll，isinstance 防非 str remainder）。
  - EOF 分支、health-check 分支、`start()` "ready" 超时分支统一用 `_stderr_diagnostic()`；修复 `start()` 里先 `_kill_process()`（置空 `_process`）导致 `_drain_stderr()` 拿不到 stderr 的顺序 bug。
  - `_drain_pending_messages()`：run() 退出路径排空队列并 dispatch，遇终态帧即停。
- `src/workflow_engine/constants.py`：新增 `STDERR_TAIL_MAX_LINES = 50`。
- `tests/test_workflow_stdout_flush.py`（新增）
  - Layer 1 真实 Node E2E：20000 log backlog 后 done/error 帧必须完整送达（node 缺失时 skip）。
  - Layer 2 bridge 单测：队列内 done/error 帧优先于 EOF fallback；无终态帧时用增强 fallback；stderr tail 进入 `_describe_unexpected_exit`；`_stderr_reader` 记录 tail。
- `tests/test_workflow_bridge_transport.py`：`test_reader_eof_sets_done` 更新为新契约（EOF 写 `_eof_fallback_error` 而非 `_error`）。

### 后续风险

- `flushAndExit` 的 2s 兜底：极端卡死管道下最多多等 2s 才退出，可接受（换取终态帧可靠送达）。
- health-check 分支 `poll()!=None` 后 `_drain_pending_messages()` 可能 dispatch 残留 `agent_call`，但 `_send()` 对已死进程走 BrokenPipe 静默，无副作用。
- runtime.js 与 bridge.py 各有独立的 EOF/exit 处理，未来若改传输协议需两侧同步。
