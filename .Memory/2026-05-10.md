## 修复编程卡片重复刷屏 + reasoning panel 非法 JSON
### 任务描述
用户反馈：同一编程任务在飞书里冒出大量重复卡片（截图里 0m00s/0m05s/0m06s/0m51s 各一张新卡），要求与 `ux/unified_card_v2_*.html` 的 v2 形态一致，并精简工具调用细节、保留中间思考总结。后续澄清：不要复杂算法，本身是流式信息，飞书卡片有节点上限，**快到上限再开新卡**即可。

### 执行内容
- **根因 A（重复卡片）**：`ProgrammingCardSession._handle_plan_update()` 每当 plan 的 in-progress 任务变化（`_extract_current_tasks` 签名变了）就调 `_rotate_primary_session()` → `SessionRotator.rotate()` 新建一张飞书卡片。Agent 在 TODO 列表里推进时 in-progress 项频繁变化 → 每变一次就刷一张新卡。
  - 修复：`_handle_plan_update()` 只 `self._rotator.dispatch(card_event)`（任务列表区原地更新）+ 同步给 subagent 会话；删除 `_rotate_primary_session`、`_primary_task_signature`、`_build_primary_task_label`、`_extract_current_tasks` 及未用的 `PlanInfo` typing import。新续卡只由渲染期分页（`paginate_layout`，`node_budget=180`）在接近飞书节点上限时自然产生——即用户要的"快到上限开新卡"，不另搞算法。
- **根因 B（卡片整体渲染失败 → 失败后被当作 permanent transport error 强制重建 → 又失败 → 刷屏）**：`render_reasoning_panel()` 窄边栏列里写了 `{"tag": "div", "elements": []}`，飞书 `div` 不支持 `elements` 属性，整卡 JSON 被拒（日志大量 `code=230099 / ErrCode: 200621 ... unknown property, property: elements, path: ... (tag: div)`）。2924c65 引入。
  - 修复：窄边栏元素改为 `{"tag": "div", "text": {"tag": "plain_text", "content": " "}}`——合法 `div` 形态；刻意不用 `markdown`，否则 `renderer._prepend_bridge_phrase()` 递归会把"续接："注入到这个 spacer 而不是右侧正文列（`test_bridge_phrase_prepended_to_reasoning_column_set` 会红）。
- 工具调用细节：现有 `flatten_to_atoms` 已是 v2 形态（完成工具→一行 `▣ **已探索 N 项…**` digest，active 工具→一行 `⏳ **Tool** · summary`，无大块 input/output），无需再改；subagent 独立卡（`_handle_agent_task_event`）保持不变。
- 测试：`test_task_switch_opens_new_card_instead_of_overwriting_previous_one` 断言的是旧（错误）行为，改写为 `test_plan_updates_stay_in_single_card`：start 后两次 PLAN_UPDATE 不应新增 `client.creates`，message_id 不变，plan block 内容反映最新 in-progress 任务。

### 技术要点
- "一个任务一张卡片"中的"任务"= 用户那次请求，不是 agent TODO 里的某一项；TODO 列表整体在一张流式卡里原地更新，超节点预算才分页成续卡。
- 续卡机制有两条：分页（同 session 内多 page，`deliver` 对新 page_index 调 `create_card`）和 `SessionRotator`（引擎迭代边界用，Spec/Loop）。本次去掉的是 programming 模式按 plan-item 触发的 rotator 误用。
- 遗留（Medium，已可不急）：`page_mutator.update_page()` 对 `code=230099` 一律当 permanent → `remove_page()` 强制重建；若内容本身非法，重建会产出同样非法 JSON 形成刷屏。根因 B 修了直接触发源，但该兜底策略仍是隐患，建议后续按错误子码区分（`99992354` 才重建，content-parse 错误应渲染 fallback 卡）。

## 修复项目群 /status 锁状态误显示“没有锁”
### 任务描述
用户反馈：项目与群应一一对应；当一个项目正在执行任务并持有仓库锁时，切换/进入另一个项目群查看状态却返回“没有锁”。预期应按项目群绑定关系解析项目，不同项目群查询各自项目锁。

### 执行内容
- 梳理锁链路：编程/引擎任务通过 `RepoLockManager` 按 `root_path` 持有仓库锁，`/status` 在 `DiagnosticsHandler._build_lock_status_lines()` 中依赖传入 `ProjectContext.root_path` 查询 repo lock。
- 定位根因：项目群绑定关系 `find_by_bound_chat_id()` 只在自由文本默认进入 Coco 的分支中使用；`/status` 属于 slash/system 命令，绕过自由文本分支，通用上下文解析只回退 `get_active_project(chat_id)`。当项目群没有 active project 时，`project=None`，诊断页自然显示“没有锁”。
- 修复 `FeishuWSClient._resolve_project_from_message()`：在消息引用 parent/root 未命中后，先通过 `bound_chat_id -> ProjectContext` 解析项目群一一绑定项目，最后再回退 active project。
- 补充回归测试：覆盖项目群绑定 fallback、parent/root 消息引用优先级仍高于 bound chat、无绑定时仍回退 active project。

### 技术要点
- 该修复不改变 repo lock 的 key 语义：锁仍按规范化 `root_path` 互斥；只是确保项目群中的系统命令能拿到正确项目上下文，从而查询正确的 `root_path`。
- 优先级保持为：thread/message reference 上下文 > bound project chat > active project，避免回复历史项目卡片时被群绑定项目错误覆盖。

## 修复飞书卡片内容重复渲染 Bug（card table number over limit）
### 任务描述
用户报告飞书编程卡片中出现大量重复信息（同一文本段落重复10+次），同时日志出现 `ErrCode: 11310; ErrMsg: card table number over limit`。

### 根因分析
- `block_index` 使用 **last-wins 语义**：`rebuild_block_index()` 对相同 `block_id` 只保留最后一个索引
- 多个 reasoning block 共享 `block_id="_active_reasoning"`（text block 有 per-turn ID，reasoning 没有）
- `_render_atom_reasoning()` 从 `block_index` 查找 block 并使用 `block.content` 渲染 → 所有 atom 都渲染最后一个 block 的内容
- 重复的 `collapsible_panel`（reasoning panels）被飞书计为 "table"，超过限制触发 11310 错误

### 执行内容
1. **`render_reasoning_panel`** 添加 `content_override` 参数，使用 `atom.content` 替代 `block.content`
2. **`_render_atom_reasoning`** 传递 `atom.content` 作为 override
3. **`ProgrammingCardSession`**（`programming_adapter.py`）添加 per-turn reasoning block ID 机制：`_reasoning_turn_seq` 计数器 + `_current_reasoning_block_id()` 方法
4. **`ACPStreamBridge`**（`stream_bridge.py`）为 Deep/Loop/Spec 路径添加 per-turn 序列计数器（`_text_turn_seq`/`_reasoning_turn_seq`），生成唯一 block_id 并 override CardEvent payload
5. **`render_plan_panel`** 防御性添加 `content_override` 参数
6. **`_render_atom_plan`** 传递 `atom.content` 防止潜在重复

### 技术要点
- `_render_text_element` 已使用 `atom.content`（安全），无需修改
- `block_index` last-wins 语义本身是合理的（用于 streaming element_id 定位），问题在于渲染函数不应依赖它获取内容
- 修复策略：双管齐下 — 生成唯一 ID 避免碰撞 + 渲染函数用 atom.content 兜底

## 修复 /coco 模型选择卡显示陈旧硬编码模型列表

### 任务描述
用户反馈 `/coco` 弹出的"💭 coco 模型选择"卡只显示 6 个明显过期的假模型（gpt-5.2 / gpt-4.1 / claude-3-opus / claude-3.5-sonnet / claude-3.7-sonnet / doubao-1.5-pro），不是 coco 真实模型列表（GPT-5.5/5.4、GLM-5、Kimi-K2.6、openrouter pools、Gemini preview 等 26 个）。

### 根因
- `/coco` → `dispatcher._handle_enter_coco` → `SystemHandler.handle_select_acp_tool("coco")` → `fetch_acp_models("coco")`（`src/acp/helper.py`）。

- 超时后两条路径都退化：`CocoModelManager._load_models_via_acp` 返回 `[]` → `_load_models()` 退到静态 `DEFAULT_MODELS` 并**缓存 5 分钟**；`fetch_acp_models` 自己的 `probe_acp_models("coco")` 也再超时一次 → 最终回退到 `DEFAULT_MODELS`。用户实际等了 ~12s 还看到假列表，连"刷新模型列表"按钮 5 分钟内也无效（因为 manager 缓存了假列表）。

### 执行内容
- `src/config/settings.py`：`acp_model_probe_timeout` 默认 6.0 → **15.0**（给冷启动握手公平窗口）；新增 `acp_model_preheat_on_startup: bool = True`。

- `src/main.py`：启动时（`acp_model_preheat_on_startup` 开关下）调用 `get_coco_model_manager().kickoff_preheat()`，与 TTADK preheat 并列；`/coco` 交互路径正常直接读热缓存，不再付冷启动探测代价。

### 技术要点
- `fetch_acp_models("coco")` 流程：先 `_coco_models_from_manager`（若 manager 仅有 DEFAULT_MODELS 则返回 `[]` 强制重探）→ 自行 `probe_acp_models("coco")`（用 `safe_wait_for(timeout=acp_model_probe_timeout)`）→ 仍失败才退 `DEFAULT_MODELS`。两层都吃 `acp_model_probe_timeout`，所以只需调一个参数即可同时放宽两处。
