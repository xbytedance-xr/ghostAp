## Deep Agent 完成卡片无内容修复

### 任务描述
Deep Agent 执行完成后，卡片"执行输出"区域几乎为空，进度条显示 0%，只有 `🔧 grep ✅` 一条工具调用行，无实质信息。

### 根因分析
1. **0% 进度条**：无 `PLAN_UPDATE` 事件时 `total_steps=0`，`make_progress_bar(0,0)="[░░░░░░░░░░] 0%"`，完成态仍显示 0%，误导用户。

3. **工具调用统计粗糙**：`format_summary()` 只输出 `"🔧 工具调用: 95 次"`，没有 kind 拆分，信息量低。

### 修复内容
- **`src/feishu/renderers/deep_renderer.py`** `on_project_done`：
  - 改用本地 closure `renderer.get_final_content()`（与流式更新一致），引擎 renderer 作兜底。
  - 当 `rendered_content` 为空时补充提示"（Agent 未产生文本输出，仅执行工具调用）"，避免区域空白。
  - 完成态且 `total_steps=0` 时不传 `progress_bar`，消除误导性的 0% 进度条。
- **`src/deep_engine/progress.py`** `format_summary()`：
  - 按 kind 统计工具调用次数，输出如 `🔧 工具调用: 95 次 (search: 90 · execute: 5)`。
- **`tests/test_deep_renderer.py`**：为 mock 补充 `engine.progress.total_steps = 0`，适配新的 `> 0` 判断。

## /exit 在编程模式中误报"不在模式中"修复

### 任务描述
用户确认已在编程模式中，但输入 `/exit` 仍收到 `⚠️ 当前不在 Claude 模式中` 警告。

### 根因分析
`exit_mode` 中的成功路径条件：
- `has_session`：ACP 会话存在并成功关闭
- `is_pending_slot`：仅在 `thread_programming_enabled=True` 时有效

当用户进入编程模式但**尚未发送任何消息**（lazy init，ACP 会话未创建），或会话已过期时：
- mode_manager 已标记为对应模式 ✓
- `end_session` 返回 None → `has_session=False`
- `thread_programming_enabled=False` → `is_pending_slot=False`
- 两者均 False → 误显示"不在模式中"警告

### 修复方案
在 `exit_to_smart` 重置模式之前捕获 `was_in_this_mode`，新增 `is_mode_only_exit` 分支（非 thread 模式、无 session、但确实在该模式中），三个条件任一满足均走正常退出路径，显示 `mode_exit_pending_msg`。

### 技术要点
- `_is_in_this_mode` 必须在 `exit_to_smart` 前调用，之后状态已被重置
- `is_mode_only_exit` 用 `mode_exit_pending_msg`（无会话可保存），避免显示"会话已保存"

## /spec_guide 目标重写修复（类 btw 命令语义）

### 任务描述
`/spec_guide` 原实现只是将引导消息追加到 `_user_guidance` 队列，下一轮 spec/build 阶段消费后即清除，不改变 `project.requirement`（原始目标）。用户要求：应像 Claude 的 `btw` 命令一样，将原始目标与本次引导通过 LLM 合并生成新目标，后续所有循环基于此新目标迭代。

### 执行内容
1. **`src/spec_engine/prompts.py`** — 新增 `build_goal_rewrite_prompt(original, guidance)` prompt 构建函数，用于指导 LLM 将原始需求与新引导合并为一个连贯完整的新需求文本
2. **`src/spec_engine/engine.py`** — 新增 `refine_goal_with_guidance(guidance)` 方法：调用 LLM 重写目标，更新 `project.requirement`，清空 `_user_guidance` 队列，持久化状态；无 LLM 配置时退化为直接追加；LLM 失败时返回 `(False, error_msg)` 供调用方降级
3. **`src/feishu/handlers/spec.py`** — `update_spec_guidance` 调用 `refine_goal_with_guidance`；成功时显示新目标（`format_goal_rewritten`），失败时降级调用 `inject_guidance` 显示旧提示
4. **`src/spec_engine/reporter.py`** — 新增 `format_goal_rewritten(guidance, new_req)` + `get_goal_rewritten_title()` 用于展示目标重写结果
5. **`tests/test_spec_engine.py`** — 更新 `test_update_spec_guidance_allows_when_clarifying` 适配新行为；新增 `test_update_spec_guidance_fallback_on_llm_failure` 测试降级路径

### 技术要点
- `refine_goal_with_guidance` 返回 `(bool, str)` tuple，handler 基于此决定走重写路径还是降级路径
- 目标一旦重写，`project.requirement` 持久更新，后续所有 `build_spec_prompt` 和 `build_review_prompt` 自动使用新目标
- 无需修改 cycle 内的 prompt 构建逻辑，因为它们都读 `project.requirement`
