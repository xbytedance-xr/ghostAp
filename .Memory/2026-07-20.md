## 员工 canonical partial 历史话题污染修复

### 日志证据与根因

- 12:53:44 与 12:53:51 的两个员工候选先记录 `employee canonical context unavailable; candidate deferred`，随后分别在 12:54:05 与 12:54:20 按既有 bounded retry 合约终态化；这不是 provider/ACP 卡死，而是上下文准备阶段持续拒绝。
- Journal 中当前普通群消息（序号 1423、1466）的 `thread_id` 都为空，但同群最近窗口仍包含序号 1361 的历史 `thread_id=om_...` 话题记录；`window()` 按 tenant+chat 收集最近 50 条，不按当前 thread 过滤。
- 前序修复把未来 producer 改为写真实 `omt_...` 并封住裸 `ValueError`，但 `assemble_partial()` 对窗口内任意无法映射的旧 `om_...` 都 fail-close，导致一条非当前历史话题记录持续污染此后同群普通消息的 canonical partial。

### 实现

- `GroupContextLedger.assemble_partial()` 只省略窗口内“非当前且语法合法”的旧 `om_...` 话题坐标；当前记录、任意未知 malformed 坐标以及权威绑定不足的当前 legacy root 仍按 `ROOT_THREAD_BINDING` fail-close。
- 省略发生时追加单次 `legacy_thread_record_omitted` ledger warning，并把全部 warning code 纳入 `snapshot_hash`；没有省略时保持原 hash 输入公式不变，避免无关快照身份漂移。
- 不重写历史 Journal，也不放宽 `ContextMessage` 的 `omt_...` 标识空间；代价只是丢弃无法权威映射的旧 group-recent 消息，当前消息仍必须完成身份与租户校验。

### TDD、审查与验证

- 首个红灯用线上同形态窗口稳定复现 `ContextUnavailableError(ROOT_THREAD_BINDING)`；第二个红灯证明省略路径尚缺审计 warning，随后分别以最小实现转绿。

## 员工话题派发 ValueError 死循环修复

### 日志证据与根因

- 12:20:46 起 `employee dispatch worker failed closed: ValueError` 约每 9 秒重复一次；Journal 停在两条员工 Ingress 的 `accepted -> authorized -> queued`，没有进入 `router_dispatching` 或 attempt 事件，证明故障发生在候选准备阶段而非 TraeX 执行/回复阶段。
- 两条队列项来自同一群话题消息。主 Bot 先写入的 canonical `group.event.recorded` 把 `message.root_id`（`om_...`）记录为 `thread_id`；员工 Channel 的真实 payload 则持有合法 `feishu_thread_id`（`omt_...`）。
- 根源位于 `FeishuWSClient._process_message_async()`：账本发布错误地使用 `get_current_thread_id() or message.root_id`，而这两个值都是 GhostAP/飞书根消息坐标，不是飞书 `thread_id`。
- 在线 Lark Context 获取失败转入 Journal canonical-partial fallback 后，`GroupContextLedger.assemble_partial()` 将旧 `om_...` 直接传给要求 `omt_...` 的 `ContextMessage.thread_id`，抛出未转换的 `ValueError`。worker 只记录异常类型并重试，queued 记录既不终态化也不前进，形成永久错误循环。

### 实现

- 主 Bot 账本入口改为只写官方消息对象的 `message.thread_id`；普通群消息保持空字符串，话题消息保存真实 `omt_...`，不再混用当前 GhostAP thread context 或 `root_id`。
- canonical-partial 读取增加受限历史兼容：仅当旧 `record.thread_id` 与当前权威请求的 `thread_root_message_id` 完全相等、且请求持有非空合法 `feishu_thread_id` 时，才替换为该 `omt_...`；空 thread 请求和其他非法账本坐标统一转成 `ContextUnavailableError(ROOT_THREAD_BINDING)`，进入既有 bounded retry/terminal 生命周期，不再泄漏裸 `ValueError`。
- WS 测试 fixture 显式初始化 `message.thread_id`，并保留真实 runtime 生命周期，只 mock 账本发布边界，避免覆盖 runtime 后产生 teardown 后台错误。

### TDD 与验证

- 红灯 1 证明话题消息旧实现实际发布 `thread_id=om_topic_root`，期望为 `omt_topic`。
- 红灯 2 用线上同形态旧账本记录稳定复现 `ValueError: thread_id must use omt_ identifier space`；实现后两项转绿且输出干净。
- 扩大回归共 420 项通过：WS/Context/账本 106，员工启动与网关 116，Ingress/Context/完整员工组合 198；其中新增负向用例覆盖空 `feishu_thread_id`、混合话题旧账本污染和 coordinator bounded retry。
- 修改文件 Ruff、`git diff --check` 与 `uv run python -m src.main --validate` 通过。
- 独立代码审查首轮发现空 thread 降级和其他话题旧记录裸异常两项 Important；补齐严格门禁与生命周期测试后复审为 Critical 0、Important 0、Ready。

### 部署与运行证据

- `./restart.sh rr` 只返回远程触发回执但没有新 PID 或 `[RESTART]` 标记，未误判为部署成功；随后使用本地 `./restart.sh restart` 完成安全重启并恢复旧队列，审查加固后的最终进程 PID 为 `40539`。

- 跨过原错误重试周期后，新进程持续存活，重启窗口没有新增 ERROR/CRITICAL，`employee dispatch worker failed closed: ValueError` 已停止。
