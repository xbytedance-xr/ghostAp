# Workflow 模式规格（Spec）

## 一、目标（Goals）
1. 修复当前 Workflow 模式无法正常启动运行的问题
2. 借鉴 Spec 模式的优秀设计，重构 Workflow 模式的核心流程
3. 实现支持工具与模型联合选择的编程模式
4. 建立动态角色分配机制，提升 Workflow 模式的灵活性与智能化水平

## 二、功能规格（Functional Spec）
### 1. 编程模式选择
- 进入 Workflow 模式后，主 Agent 选择界面需支持同时选择工具（如 Coco、Traex、Codex 等）和对应的模型版本
- 选择界面需采用与 Spec 模式一致的联合选择交互方式
- 支持通过展开工具面板查看并选择该工具下的所有可用模型
- 提供快捷选择默认模型的入口

### 2. 评判工具选择
- 评审工具选择界面需采用与主 Agent 选择相同的联合选择模式
- 支持同时选择多个评审工具与模型组合
- 提供自动评审选项，允许使用主 Agent 进行自评审
- 支持动态添加/移除评审工具与模型组合

### 3. 预算控制
- 完全移除所有与 token 预算控制相关的功能
- 删除预算配置、预算显示、预算验证等所有相关代码
- 确保 Agent 调用不受任何预算限制

### 4. 角色定义
- 移除固定角色选择功能
- 实现由主 Agent 动态判断并生成角色的机制
- 角色生成需借助评审工具的 AI 能力
- 支持根据任务需求动态调整角色分配

### 5. 流程控制
- 实现两阶段选择流程：主 Agent 选择 → 评审工具选择
- 完成选择后自动生成 JS 编排脚本
- 提供脚本确认界面，支持用户查看脚本详情并确认执行
- 执行过程中实时展示 workflow 进度

## 三、非功能需求（Non-Functional Requirements）
### 1. 用户体验
- 统一 Workflow 模式与其他模式（如 Worktree）的设计语言
- 提供清晰的步骤指示器，明确当前所处流程阶段
- 为选中项提供明显的视觉区分，提升选择状态感知
- 为工具展开/收起操作提供直观的状态指示器
- 使用醒目的错误提示方式，提升错误信息可见性

### 2. 性能
- 确保 workflow 启动时间控制在可接受范围内
- 支持并行执行多个 Agent 调用
- 优化脚本生成与执行效率

### 3. 兼容性
- 与现有工具和模型发现机制保持兼容
- 支持所有 Spec/Worktree 模式已支持的工具与模型
- 确保与 Feishu 卡片 API 的兼容性

### 4. 安全性
- 恢复 JavaScript 运行时沙箱中 Error.prototype 的冻结保护
- 确保所有 Agent 调用符合安全规范
- 防止恶意脚本执行

## 四、验收标准（Acceptance Criteria）
### 功能验收
- [x] AC1: 用户执行 `/wf <需求>` 后，能看到工具+模型联合选择卡片（与 Spec 模式 review agent 选择卡片视觉一致），可选择主编程 Agent 的工具和模型
- [x] AC2: 主 Agent 选择完成后，能看到评判工具选择卡片，同样支持工具+模型联合选择
- [x] AC3: 选择流程中不出现任何 budget/预算相关的 UI 元素或选项
- [x] AC4: 选择流程中不出现静态角色列表供用户勾选
- [x] AC5: 完成两步选择后，能成功生成 JS 编排脚本并进入确认执行阶段
- [x] AC6: 确认执行后，workflow 能实际启动 Node.js runtime 并完成至少一个 agent() 调用
- [x] AC7: agent() 调用执行时不再有 token budget exhausted 拒绝情况
- [x] AC8: 生成的脚本中角色分配由 LLM 自主决定，而非从用户选择列表注入

### 代码质量验收
- [x] AC9: constants.py 中不再存在 BUDGET_OPTIONS、DEFAULT_BUDGET_TOKENS、RESERVE_PER_AGENT_TOKENS 等预算常量
- [x] AC10: PendingConfirmation 模型中不再包含 selected_budget 字段
- [x] AC11: StateManager 中不再包含 try_reserve/settle 等预算相关方法
- [x] AC12: uv run python -m pytest tests/ -q 全部通过

### 架构验收
- AC13: router.py FORWARDING_MAP 中包含所有 workflow 相关转发条目
- AC14: 步骤指示器准确反映两阶段选择+确认的三步流程
- AC15: API 契约完整，所有按钮回调包含必要的 chat_id、project_id 等字段
- AC16: 模型发现机制正确集成到选择流程中

### UI/UX 验收
- AC17: 使用与 Worktree 模式一致的可视化 stepper 组件
- AC18: 为选中项提供明显的视觉区分
- AC19: 为工具展开按钮提供展开/收起状态图标指示器
- AC20: 使用醒目的 banner 组件显示错误信息

## 五、范围与约束（Scope & Constraints）
### 包含范围
- Workflow 模式的核心选择流程重构
- 工具与模型联合选择界面实现
- 动态角色分配机制实现
- 预算控制功能移除
- 相关 API 契约修复与完善
- UI/UX 统一与优化
- 安全漏洞修复

### 排除范围
- 其他模式（如 Spec、Worktree）的修改
- 工具与模型本身的功能修改
- 非 Workflow 模式相关的测试用例
- 文档更新（除必要的代码注释外）

### 约束条件
- 必须使用 subagent 实现复杂任务
- 优先使用 workflow 实现多步骤任务
- 必须与现有代码架构保持兼容
- 必须通过所有现有测试用例

## 六、风险与假设（Risks & Assumptions）
### 风险
- 流程重构可能引入新的兼容性问题
- 动态角色分配可能导致不可预测的结果
- UI/UX 统一可能需要额外的设计资源
- 移除预算控制可能导致 token 使用量增加

### 假设
- 假设用户熟悉 Spec 模式的交互方式
- 假设所有工具都支持模型版本选择
- 假设动态角色分配的 AI 能力足够可靠
- 假设用户能够接受无预算控制的模式

## 七、决策记录（Decisions）
1. 决策：采用两阶段选择流程（主 Agent → 评审工具）
   - 理由：简化用户操作，提升流程清晰度
2. 决策：完全移除预算控制功能
   - 理由：当前阶段不需要预算控制，简化流程优先级更高
3. 决策：实现动态角色分配
   - 理由：提升 Workflow 模式的智能化水平，减少用户干预
4. 决策：统一 UI/UX 设计语言
   - 理由：提升产品一致性，降低用户学习成本
