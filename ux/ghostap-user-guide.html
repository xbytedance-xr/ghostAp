普通编程模式与 Spec 模式实际操作流程

这份说明面向日常使用：普通模式适合直接对话编程、修 bug、跑命令和持续迭代； Spec 模式适合目标较明确、需要多轮规划、执行、审查和收敛的复杂任务。 WT 仍在迭代中，本页只保留当前使用边界。

  • Spec 过程文件写入 ~/.cache/ghostAp。
  • /spec_status 可扫描缓存。
  • 支持一键还原最新 Spec。

普通模式：直接对话编程的实际流程

适合明确修复、局部实现、跑测试和连续追问

1. 进入工具

在项目群里发送工具命令。Coco/Codex/Aiden 等会进入普通编程模式。

  • /coco：进入 Coco
  • /codex：进入 Codex
  • /aiden：进入 Aiden
  • /claude / /gemini / /ttadk

2. 选择模型

系统先在同一张卡上展示“正在查询模型”，拿到列表后 PATCH 成模型按钮。

  • 可选择具体模型。
  • 也可使用默认模型，由工具自身配置决定。
  • 选完后进入“编程模式已就绪”。

3. 连续对话

进入模式后，普通文本会持续发给当前工具，直到你显式退出。

  • /model 打开当前工具模型选择。
  • /model <模型名> 直接切换。
  • /exit 回到 SMART。

Spec 模式：目标驱动的多轮规划、执行与审查

适合复杂目标、长任务、需要审查收敛的改造
1
发送目标 在项目群或当前项目上下文中发送 /spec 修复登录流程并补测试。Spec 会绑定当前 Feishu 话题，后续同话题普通文本会继续按 Spec 策略处理。
2
选择审查工具 选择 Auto 表示沿用主 agent + 模型审查；也可以添加多个工具和模型组合，例如 Coco / Test-New-Thinking + Codex / gpt-5.5。
3
确认并启动 确认后同一张选择卡会 PATCH 成“正在启动”，随后进入原 Spec 流程，不会跳到 Worktree，也不会要求再次输入目标。
4
观察运行卡片 运行卡片展示当前轮次、总耗时、阶段输出、工具调用和 review 结果。多角色审查标题后会显示实际使用的工具和模型。
5
干预或恢复 长任务可用 /spec_status、/spec_pause、/spec_resume、/spec_guide <补充说明> 管理。进程重启后可通过缓存还原。

/coco / /codex / /aiden 进入普通编程模式 展示模型选择卡；选择后进入当前工具会话，后续普通消息持续发给该工具。

/exit 退出当前模式 普通编程模式回到 SMART；在 Spec/WT/Deep 话题内会退出对应话题策略。

/spec_history / /spec_metrics 查看历史和指标 读取 Spec 持久化记录，便于回看轮次结果和执行趋势。
/spec_save / /spec_export 保存或导出 Spec 结果 保存当前状态；导出会生成 Markdown 报告。

普通模式、Spec、WT 的边界

避免把三种工作方式混在一起

普通模式更像“远程结对”

它会保存当前工具和模型，适合一边看结果一边追加要求。你不需要每条消息都写命令。

  • 适合：小到中等修改、持续追问、代码解释、跑测试。
  • 退出：/exit。

Spec 更像“目标驱动执行器”

它围绕一个目标反复形成规格、计划、任务、构建和审查，直到满足收敛条件。

  • 适合：复杂改造、多文件变更、需要审查闭环。
  • 恢复：/spec_status 或 /spec_resume。

WT 仍在迭代中

WT 的核心是隔离 worktree 和多工具组合执行，目前正在优化交互与合并体验。

  • 适合：需要隔离分支尝试的任务。
  • 详细状态暂看 WT 流程图。
推荐经验：能一句话清楚描述“改哪、怎么验”的任务先用普通模式；目标复杂但验收明确的任务用 Spec；需要隔离 worktree 并行尝试时再使用 WT。
