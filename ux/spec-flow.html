GhostAP Spec 模式真实工作流程

代码口径：2026-05-14 当前实现。Spec 是 topic-scoped engine strategy：同一个 Feishu 话题持续进入 Spec；执行本身按项目/仓库串行加锁，通过 SpecEngine 的循环把需求转成规格、计划、任务、实现和审查，直到满足验收与审查收敛。

主路径总览

入口 → topic 绑定 → 任务调度 → SpecEngine 循环 → 审查收敛 → 持久化/恢复

话题持续策略

  • /spec <需求> 会绑定当前 Feishu topic 为 mode=spec。
  • 同 topic 普通文本继续走 Spec 策略，不回 SMART 意图识别。
  • 同 topic 里切 WT/Deep 会被拦截；/exit 只退出话题策略。

启动路径

  • 入口不先发“分析中”卡，第一帧由 Spec renderer 懒创建。
  • 普通需求只用 parse_acceptance_criteria 生成 provisional criteria。
  • 首轮 SPEC 产物再替换/合并正式验收标准。

循环主体

  • 每轮固定走 SPEC → PLAN → TASK → BUILD → REVIEW。
  • phase 通过 ACP session 流式执行，Build 阶段可拆任务卡。
  • 失败 phase 会保存 recoverable task，并尝试模型切换或下一轮恢复。

终止条件

  • 成功需要验收标准满足，并且阻塞审查连续通过。
  • 0% 到 100% 的首轮乐观跳变会要求下一轮确认。
  • 也可能因收敛、backlog 停滞、连续失败或 max cycles 停止。

逐步解释

按代码路径整理
1
入口与命令族 /spec <需求> 启动；/spec_status、/spec_history、/spec_metrics、/spec_config、/spec_export、/spec_save、/spec_pause、/spec_resume、/spec_recover 走同一个 handler。
2
topic 绑定 BaseEngineHandler._ensure_topic_engine_context() 把当前消息或话题绑定为 mode=spec。后续同话题普通文本继续进入 Spec 策略。
3
调度与仓库锁 SpecHandler.start_spec_engine() 创建/复用 SpecEngine，提交到 TaskScheduler，实际执行由 _safe_execute_engine() 放进 repo lock。
4
分析与会话 SpecEngine.execute() 校验输入、创建 SpecProject，用轻量 criteria seed 启动，再创建 ACP/CLI engine session。
5
五阶段循环 每轮依次跑 SPEC、PLAN、TASK、BUILD、REVIEW。每个 phase 产物可落盘，卡片通过 SpecStreamProcessor 流式更新。
6
审查与收敛 默认 adaptive_roles 规划审查角色，按依赖批次并发执行。阻塞建议必须带证据；无证据的高置信 blocker 会降级为 observation。
真实边界：Spec 不是一次性“跑完就退出”的上下文，同话题会持续保持 Spec 策略；但运行中的具体 engine 仍按 chat + root_path 管理，并通过 repo lock 保护同仓库修改。
