GhostAP WT 真实工作流程

代码口径：2026-05-14 当前实现。WT 是 topic-scoped engine strategy：同一个 Feishu 话题承载一个 WT 任务流；同项目不同话题可以并行拥有独立 WT session，但合并/清理仍走仓库级串行锁。

主路径总览

入口 → 话题绑定 → 选择 → 并行执行 → 自动合并清理 → 同话题续下一目标

话题是 WT 边界

  • WT session key 是 project_id + chat_id + thread_root_id。
  • 未知 root 不会偷连最近会话；同话题不同 engine 命令会被拦截。
  • /exit 退出 topic engine，使该话题回 SMART。

选择后才执行

  • /wt <目标> 先选工具/模型，选完自动执行。
  • 空 /wt 选完后进入 AWAITING_GOAL，同话题第一条普通消息就是目标。
  • 没有独立“开始执行”按钮作为主路径。

并行只在单任务内部

  • 一个 WT topic 是一个任务，不做全局任务调度器。
  • 已选工具/模型组合生成多个 unit，dispatcher 并行执行。
  • 主控 agent 负责目标梳理、拆分、验收，其它 unit 做实现/测试/审查。

集成默认自动化

  • 执行成功后自动提交 worktree 脏变更。
  • 自动合并回 base，冲突策略是 -X theirs，偏向 WT 分支产物。
  • 全部合并成功后自动删除 worktree 和本地 WT 分支。

生命周期状态

WorktreeJourneyStatus + unit status

逐步解释

按代码路径整理
1
入口 /wt、/worktree 或绑定 WT 话题里的普通消息进入路由。路由只认精确 root_id/thread_root_id，不会按 chat 猜最近会话。
2
绑定话题 WorktreeHandler 把当前消息或当前话题绑定为 ThreadContext(mode="worktree")，并设置当前 thread id。
3
选择工具/模型 工具来自 ACP 与 TTADK discovery；每个选择项最终成为一个 WorktreeSelectionItem，默认模型不会写死具体 model。
4
创建隔离工作区 WorktreeGitService 以 session slug 生成 ghostap/wt/.../{index}-unit 分支和 wt-...-{index} worktree。
5
并行执行 WorktreeDispatcher 选择主控 agent，给每个 unit 生成角色提示，并通过 ThreadPoolExecutor 并行启动 agent session。
6
自动集成 完成且有变更的 unit 会先提交，再合并回 base；成功后清理 worktree 目录和本地分支。失败或未完成单元不会被静默合入。
真实限制：并发来自“同项目不同 Feishu 话题”与“单 WT 内多个 unit”，不是一个全局任务调度器。合并/清理持有 repo lock，避免同一仓库多个 WT 同时改 base。

并发与锁边界

同项目多 topic 可并行，base 集成串行
